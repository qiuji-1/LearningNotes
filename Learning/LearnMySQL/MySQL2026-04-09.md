## 1. MySQL 中如何解决深度分页问题？

### 📌 回答重点

深度分页的核心问题在于 `LIMIT 99999990, 10` 这种写法会让 MySQL 扫描前 99999990 条记录再丢弃，白白浪费大量 IO。解决思路就是想办法减少扫描的数据量，常见的优化方案有三种：

#### 1) 子查询优化

把原本的查询拆成两步：先用子查询在二级索引上快速定位起始 id，再用这个 id 去主键索引取数据。

**原始写法，慢：**
```sql
SELECT * FROM mianshiya
WHERE name = 'yupi'
ORDER BY id
LIMIT 99999990, 10;
```

**优化写法：**
```sql
SELECT * FROM mianshiya
WHERE name = 'yupi'
  AND id >= (
    SELECT id FROM mianshiya
    WHERE name = 'yupi'
    ORDER BY id
    LIMIT 99999990, 1
  )
ORDER BY id
LIMIT 10;
```

`name` 字段有索引的情况下，子查询只扫描 `name` 的二级索引，二级索引只存了 `name` 和 `id`，数据量比主键索引小很多。拿到起始 id 后，再去主键索引取 10 条完整记录，速度很快。

用 JOIN 写法也是一样的原理：
```sql
SELECT * FROM mianshiya
INNER JOIN (
  SELECT id FROM mianshiya
  WHERE name = 'yupi'
  ORDER BY id
  LIMIT 99999990, 10
) AS t ON mianshiya.id = t.id;
```

#### 2) 游标分页

每次查询都返回当前页的最大 id，下次查询时带上这个 id 作为起点：

**第一页：**
```sql
SELECT * FROM mianshiya WHERE name = 'yupi' ORDER BY id LIMIT 10;
-- 假设最大 id 是 100
```

**第二页：**
```sql
SELECT * FROM mianshiya WHERE name = 'yupi' AND id > 100 ORDER BY id LIMIT 10;
```

这种方式利用 `id > max_id` 直接过滤，MySQL 可以从索引定位到起始位置，不用扫描前面的数据。缺点是只能连续翻页，没法跳到第 10000 页。

#### 3) 搜索引擎

把数据同步到 Elasticsearch，用 `search after` 做深度分页。不过 ES 也有深度分页问题，用 `search after` 配合 PIT 才能高效处理。如果对 ES 不熟，面试时慎用这个答案。

---

### 📚 扩展知识

##### 为什么深度分页会慢

深度分页之所以慢，是因为 MySQL 在执行 `LIMIT offset, size` 时，必须先扫描到 offset 条记录，然后丢弃这些记录，再取 size 条。当 offset 很大时，这个扫描和丢弃的过程会消耗大量的 CPU 和 IO 资源。

##### 子查询优化的原理

子查询优化的关键在于利用了二级索引（辅助索引）。二级索引只包含索引列和主键列，数据量远小于主键索引。子查询在二级索引上定位起始位置，避免了在主键索引上扫描大量数据。

##### 游标分页的局限

- 只能连续翻页，无法跳页
- 排序字段必须是唯一的，否则可能出现重复或遗漏
- 需要客户端记录上一页的游标值
- 不支持排序方式的动态变化

##### 业务层面的解决思路

在实际业务中，可以考虑：
- 限制分页的最大页数，比如最多翻到第 1000 页
- 使用"加载更多"代替传统分页
- 前端缓存已加载的数据，避免重复请求
- 对超深分页的场景改用游标分页或搜索引擎

### 💬 面试官追问

#### 提问1：游标分页如果按创建时间排序，时间相同的记录怎么处理？

**回答：** 时间相同的记录没法用单字段游标区分，需要用联合游标。比如 `WHERE (create_time, id) > (上一页最后一条的create_time, id) ORDER BY create_time, id LIMIT 10`。MySQL 8.0 支持 Row Constructor 比较，写起来简洁。如果是低版本，就得拆成 `WHERE create_time > x OR (create_time = x AND id > y)` 这种写法。

#### 提问2：ES 的 search after 和 MySQL 的游标分页有什么区别？

**回答：** 原理差不多，都是记录上一页的排序值作为下一页的起点。区别在于 ES 的 `search_after` 必须配合 PIT（Point In Time）使用才能保证数据一致性。PIT 会创建一个数据快照，分页期间即使有新数据写入也不会影响翻页结果。MySQL 没有这个机制，分页期间数据变化可能导致重复或遗漏。ES 的 `search_after` 不支持跳页，只能往后翻，这点和游标分页一样。

## 2. MySQL 主从同步机制及实现原理？

### 📌 回答重点

MySQL 主从同步的核心就是 binlog 复制：主库把写操作记到二进制日志里，从库拉过来重放一遍，数据就同步了。整个流程涉及三个线程配合：

#### 1) 主库的 dump 线程

监听 binlog 变更，有新内容就推事件给从库。

#### 2) 从库的 I/O 线程

拉取主库数据，把收到的 binlog 写进本地的 relay log。

#### 3) 从库的 SQL 线程

读 relay log，逐条执行 SQL 语句。

---

### 📚 扩展知识

##### 三种复制模式

MySQL 支持三种复制模式：

1. **异步复制（Asynchronous）**：主库执行完事务后立即返回，不等待从库确认。这是默认模式，性能最好但数据一致性最弱。

2. **半同步复制（Semi-synchronous）**：主库执行完事务后，至少等待一个从库确认收到 binlog 才返回。平衡了性能和数据一致性。

3. **全同步复制（Fully synchronous）**：主库执行完事务后，等待所有从库确认收到 binlog 才返回。数据一致性最强但性能最差。

##### 半同步复制

半同步复制是 MySQL 5.5 引入的特性，需要在主库和从库都安装插件：

```sql
-- 主库安装插件
INSTALL PLUGIN rpl_semi_sync_master SONAME 'semisync_master.so';
SET GLOBAL rpl_semi_sync_master_enabled = 1;

-- 从库安装插件
INSTALL PLUGIN rpl_semi_sync_slave SONAME 'semisync_slave.so';
SET GLOBAL rpl_semi_sync_slave_enabled = 1;
```

半同步复制的优势：
- 提高数据安全性，避免主库宕机导致数据丢失
- 当没有从库响应时，自动降级为异步复制，保证可用性
- 可以配置超时时间，超时后自动切换到异步模式

##### 并行复制演进

MySQL 的并行复制经历了几个阶段：

1. **MySQL 5.6 - 基于库的并行复制**：`slave_parallel_workers` 参数控制线程数，按 database 维度并行。缺点是不同库之间无法并行。

2. **MySQL 5.7 - 基于组提交的并行复制**：`binlog_group_commit_sync_delay` 参数优化组提交，支持 `MASTER_AUTO_POSITION` 自动定位 binlog 位置。引入 `MTS（Multi-Threaded Slave）`，基于组提交信息并行执行事务。

3. **MySQL 8.0 - 基于 WriteSet 的并行复制**：`binlog_transaction_dependency_tracking` 参数控制依赖关系跟踪模式。WriteSet 模式通过记录事务修改的主键集合，精确判断事务间的依赖关系，实现更细粒度的并行。

```sql
-- MySQL 8.0 开启并行复制
SET GLOBAL slave_parallel_workers = 4;
SET GLOBAL slave_parallel_type = LOGICAL_CLOCK;
SET GLOBAL binlog_transaction_dependency_tracking = WRITESET;
```

##### 主从延迟问题

主从延迟是指从库数据落后于主库的时间，常见原因：

1. **大事务**：主库执行大事务（如删除百万数据），从库回放时间长
2. **网络延迟**：主从之间网络带宽不足或不稳定
3. **从库性能差**：从库硬件配置低于主库
4. **并行度不够**：从库 SQL 线程单线程回放，处理不过来
5. **锁竞争**：从库回放时与其他查询产生锁冲突

**监控主从延迟：**
```sql
-- 查看从库状态
SHOW SLAVE STATUS\G

-- 关键字段
Seconds_Behind_Master: 0  -- 延迟秒数，0 表示无延迟
Slave_IO_Running: Yes      -- IO 线程状态
Slave_SQL_Running: Yes    -- SQL 线程状态
```

### 💬 面试官追问

#### 提问1：binlog 有几种格式，它们有什么区别？

**回答：** 三种格式：statement、row 和 mixed。

- **statement（基于语句）**：记录 SQL 语句本身，日志小但可能因为函数执行结果不同导致主从不一致，比如 `NOW()` 在主从执行时间不一样。

- **row（基于行）**：记录每行数据的变更前后值，日志大但绝对一致。

- **mixed（混合模式）**：让 MySQL 自己判断，一般用 statement，遇到不安全的语句自动切 row。

生产环境推荐用 row，虽然日志大点但数据一致性有保障。

```sql
-- 查看 binlog 格式
SHOW VARIABLES LIKE 'binlog_format';
-- 临时修改
SET SESSION binlog_format = 'ROW';
-- 永久修改（my.cnf）
binlog_format = ROW
```

#### 提问2：主从延迟太大怎么办？有没有什么应急手段？

**回答：** 先定位原因，是从库配置差、大事务还是并行度不够。应急手段有几个：

1. **强制读主库**：对一致性要求高的业务直接走主库
2. **跳过错误**：如果是某条 SQL 执行失败导致卡住，可以用 `SET GLOBAL sql_slave_skip_counter = 1` 跳过，然后 `START SLAVE` 继续
3. **并行复制调优**：把 `slave_parallel_workers` 调大，8.0 用 WriteSet 模式
4. **过滤复制**：从库只复制需要的表或库，减少回放压力

```sql
-- 跳过错误
STOP SLAVE;
SET GLOBAL sql_slave_skip_counter = 1;
START SLAVE;

-- 过滤复制（my.cnf）
replicate_do_db = db1,db2  -- 只复制指定库
replicate_ignore_db = db3  -- 忽略指定库
```

长期来看还是要升级从库配置、拆分大事务、做好读写分离的负载均衡。

#### 提问3：如果主库挂了，怎么把从库提升为主库？

**回答：** 先确认从库已经把 relay log 全部回放完，执行 `STOP SLAVE` 停掉复制，再 `RESET SLAVE ALL` 清掉主库信息。然后修改应用配置指向新主库，其他从库也要 `CHANGE MASTER TO` 指向新主库。

```sql
-- 在从库执行
STOP SLAVE;
RESET SLAVE ALL;

-- 查看当前 binlog 位置
SHOW MASTER STATUS;

-- 在其他从库重新指向新主库
CHANGE MASTER TO
  MASTER_HOST='新主库IP',
  MASTER_USER='repl',
  MASTER_PASSWORD='password',
  MASTER_LOG_FILE='mysql-bin.000001',
  MASTER_LOG_POS=154;

START SLAVE;
```

如果用了 MHA（Master High Availability）或 Orchestrator 这类工具，切换是自动的，它们会选延迟最小的从库提升，还会处理好各从库的 binlog 位点对齐。

（现在很多是云厂商提供的云数据库，这块简单了解下即可）

## 3. 如何处理 MySQL 主从同步延迟？

### 📌 回答重点

先明确一点：主从延迟是必然存在的，不可能完全消除，只能尽量缩短延迟时间或者在业务层面做规避。业务层面常见的处理方案：

#### 1) 关键业务强制走主库

比如用户注册完立马登录，这种写后读的场景直接走主库，不走从库。虽然牺牲了读写分离的意义，但这类操作频次不高，对主库压力有限。

```java
// 示例：注册后强制走主库
public User register(RegisterDTO dto) {
    // 写主库
    userService.save(dto);
    // 读主库
    return userService.getById(dto.getId(), DataSourceType.MASTER);
}
```

#### 2) 延迟感知

写操作后记录时间戳，短时间内的读请求强制走主库，过了延迟窗口再走从库。可以用 `ThreadLocal` 或 Redis 记录写操作时间。

```java
// 示例：使用 ThreadLocal 记录写时间
private ThreadLocal<Long> writeTime = new ThreadLocal<>();

public void update(User user) {
    userDao.update(user);
    writeTime.set(System.currentTimeMillis());
}

public User get(Long id) {
    Long time = writeTime.get();
    if (time != null && System.currentTimeMillis() - time < 1000) {
        // 1秒内的读请求走主库
        return userDao.findById(id, DataSourceType.MASTER);
    }
    return userDao.findById(id, DataSourceType.SLAVE);
}
```

#### 3) 二次查询兜底

从库查不到就再查一次主库，属于兜底策略。问题是如果有人恶意查询不存在的数据，每次都打到主库，变相攻击。

```java
// 示例：二次查询
public User get(Long id) {
    try {
        // 先查从库
        return userDao.findById(id);
    } catch (Exception e) {
        // 从库失败，查主库
        log.warn("从库查询失败，降级到主库", e);
        return userDao.findById(id, DataSourceType.MASTER);
    }
}
```

#### 4) 缓存前置

写入主库的同时写入缓存，读请求先查缓存。不过又引入了缓存一致性问题，属于拿一个问题换另一个问题。

```java
// 示例：缓存前置
public void update(User user) {
    // 写主库
    userDao.update(user);
    // 写缓存
    redis.set("user:" + user.getId(), user);
}

public User get(Long id) {
    // 先查缓存
    User user = redis.get("user:" + id);
    if (user != null) {
        return user;
    }
    // 缓存没有，查从库
    return userDao.findById(id);
}
```

除此之外，也可以提一提配置问题，例如主库的配置高，从库的配置太低了，可以提升从库的配置等。如果面试官对 MySQL 比较熟，可能会追问一些偏 DBA 侧的问题，例如并行复制等。

### 📚 扩展知识

##### 延迟的根本原因

主从延迟的根本原因在于：

1. **单线程回放**：从库的 SQL 线程是单线程回放 binlog，而主库可以多线程并发写，导致回放速度跟不上写入速度

2. **大事务**：大事务在主库执行时间长，从库回放也需要同样时间，期间其他事务必须等待

3. **网络延迟**：binlog 从主库传输到从库需要时间，网络带宽不足或不稳定会加剧延迟

4. **从库负载**：从库除了回放 binlog，还要承接读请求，CPU、IO 资源紧张会影响回放速度

5. **锁冲突**：从库回放时可能与业务查询产生锁冲突，导致回放阻塞

##### 并行复制优化

并行复制是解决主从延迟的核心手段：

```sql
-- MySQL 5.6：基于库的并行复制
SET GLOBAL slave_parallel_workers = 4;

-- MySQL 5.7：基于组提交的并行复制
SET GLOBAL slave_parallel_workers = 4;
SET GLOBAL slave_parallel_type = LOGICAL_CLOCK;
SET GLOBAL binlog_group_commit_sync_delay = 100;  -- 优化组提交

-- MySQL 8.0：基于 WriteSet 的并行复制
SET GLOBAL slave_parallel_workers = 4;
SET GLOBAL slave_parallel_type = LOGICAL_CLOCK;
SET GLOBAL binlog_transaction_dependency_tracking = WRITESET;
SET GLOBAL transaction_write_set_extraction = XXHASH64;
```

**并行复制对比：**

| 版本 | 并行策略 | 优点 | 缺点 |
|------|---------|------|------|
| MySQL 5.6 | 按库并行 | 实现简单 | 单库无法并行 |
| MySQL 5.7 | 按组提交并行 | 提升明显 | 依赖组提交 |
| MySQL 8.0 | 按 WriteSet 并行 | 精确判断依赖 | CPU 开销稍大 |

##### 监控延迟

**基础监控：**

##### 半同步复制

半同步复制可以减少延迟，但不是彻底消除：


### 💬 面试官追问

#### 提问1：`Seconds_Behind_Master` 为 0 就说明没延迟吗？

**回答：** 不一定。这个值为 0 只能说明从库已经把 relay log 里的事件都执行完了，但不代表主从完全同步。

比如网络断了，从库收不到新 binlog，relay log 执行完了这个值就是 0，但实际主库已经有新数据了。还有一种情况是主库长时间没写入，这个值也是 0。

想要准确监控得用 `pt-heartbeat`，主动写入时间戳来计算真实延迟。

#### 提问2：大事务为什么会导致主从延迟？

**回答：** binlog 是事务提交后才写入的，一个大事务在主库执行 10 分钟，提交后 binlog 一次性发给从库，从库也要执行 10 分钟才能完成。这 10 分钟里从库就落后了。

而且大事务期间主库可能还有其他小事务提交了，这些小事务得排在大事务后面重放，延迟会进一步放大。

所以生产环境要避免大事务，批量更新拆成多次小批量：

```sql
-- 不推荐：一次性删除百万数据
DELETE FROM user WHERE create_time < '2020-01-01';

-- 推荐：分批删除
DELETE FROM user WHERE create_time < '2020-01-01' LIMIT 1000;
-- 循环执行，每次删除 1000 条
```

#### 提问3：级联复制是什么意思？

**回答：** 就是从库再挂从库。主库只同步给一级从库，一级从库再同步给二级从库。这样主库的 dump 线程压力小了，不用给每个从库都发 binlog。

适合从库数量特别多的场景，比如要做多机房容灾，每个机房一个一级从库，机房内再挂多个二级从库。

**配置示例：**
```sql
-- 一级从库配置
CHANGE MASTER TO
  MASTER_HOST='主库IP',
  MASTER_USER='repl',
  MASTER_PASSWORD='password';

-- 二级从库配置（指向一级从库）
CHANGE MASTER TO
  MASTER_HOST='一级从库IP',
  MASTER_USER='repl',
  MASTER_PASSWORD='password';
```

缺点是延迟会叠加，二级从库的延迟是一级从库延迟加上自身重放时间。

**级联复制拓扑：**
```
主库
  └─ 一级从库（延迟 1s）
      ├─ 二级从库A（延迟 1.5s）
      ├─ 二级从库B（延迟 1.6s）
      └─ 二级从库C（延迟 1.4s）
```
