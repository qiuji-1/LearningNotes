## 1. MySQL 中有哪些锁类型？

### 📌 回答重点

MySQL InnoDB 的锁可以从两个维度来分：粒度和模式。

**粒度上分：表锁和行锁**
- 表锁：锁整张表，粒度粗，并发度低
- 行锁：只锁具体的行，粒度细，并发度高

**模式上分：共享锁(S锁)和排他锁(X锁)**
- S锁（共享锁）：允许多个事务同时读，读读不冲突
- X锁（排他锁）：独占锁，读写、写写都冲突

**行锁细分为三种：**
- 记录锁：锁住具体那一行记录
- 间隙锁：锁住两条记录之间的空隙，防止插入
- 临键锁：记录锁 + 间隙锁的组合，防止幻读

### 📚 扩展知识

#### 共享锁和排他锁

共享锁(S锁)和排他锁(X锁)是两种最基本的锁模式，它们的兼容关系决定了事务之间的冲突情况。

**锁的兼容性矩阵：**

| 锁类型 | S锁（共享锁） | X锁（排他锁） |
|--------|-------------|-------------|
| S锁（共享锁） | ✅ 兼容 | ❌ 冲突 |
| X锁（排他锁） | ❌ 冲突 | ❌ 冲突 |

**共享锁(S锁)的特点：**

```sql
-- 获取共享锁
START TRANSACTION;
SELECT * FROM account WHERE id = 1 LOCK IN SHARE MODE;
-- 或
SELECT * FROM account WHERE id = 1 FOR SHARE;

-- 多个事务可以同时获取S锁
-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR SHARE;
-- 成功获取S锁

-- 事务B
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR SHARE;
-- 也能成功获取S锁，因为S锁之间兼容

COMMIT;
```

**排他锁(X锁)的特点：**

```sql
-- 获取排他锁
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- 或
UPDATE account SET balance = 1000 WHERE id = 1;

-- X锁与其他事务的所有锁都冲突

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- 成功获取X锁

-- 事务B
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR SHARE;  -- 被阻塞
SELECT * FROM account WHERE id = 1 FOR UPDATE;  -- 被阻塞
UPDATE account SET balance = 2000 WHERE id = 1;   -- 被阻塞

-- 只有事务A提交后，事务B才能获取锁
COMMIT;
```

**自动加锁情况：**

| 操作 | 自动加锁类型 | 说明 |
|------|------------|------|
| 普通SELECT | 无锁 | 快照读，使用MVCC |
| SELECT ... LOCK IN SHARE MODE | S锁 | 共享锁 |
| SELECT ... FOR UPDATE | X锁 | 排他锁 |
| INSERT | X锁 | 插入意向锁 + 排他锁 |
| UPDATE | X锁 | 排他锁 |
| DELETE | X锁 | 排他锁 |

#### 元数据锁(MDL)

元数据锁(Metadata Lock, MDL)用于保护表结构不被并发修改。

**MDL的作用：**

```sql
-- 事务A:查询数据
START TRANSACTION;
SELECT * FROM orders WHERE id = 1;
-- 持有MDL读锁

-- 事务B:修改表结构
ALTER TABLE orders ADD COLUMN status VARCHAR(20);
-- 被阻塞!需要MDL写锁
-- 必须等待事务A释放MDL读锁
```

**MDL的类型：**

| MDL类型 | 持有场景 | 兼容性 |
|---------|---------|--------|
| MDL读锁 | SELECT、INSERT、UPDATE、DELETE | 与MDL读锁兼容，与MDL写锁冲突 |
| MDL写锁 | DDL语句(ALTER TABLE、DROP TABLE等) | 与所有MDL冲突 |

**线上事故案例：**

```sql
-- 场景:线上加字段导致查询全部超时

-- 步骤1:长事务持有MDL读锁
START TRANSACTION;
SELECT * FROM large_table WHERE id = 1;
-- ... 长时间未提交 ...

-- 步骤2:执行DDL
ALTER TABLE large_table ADD COLUMN new_col INT;
-- 等待MDL写锁，被长事务阻塞

-- 步骤3:后续查询也被阻塞
SELECT * FROM large_table WHERE id = 2;
-- 被阻塞!需要MDL读锁
-- 但MDL写锁优先级更高，所以排在DDL后面

-- 结果:整个表的所有查询都超时
```

**解决方法：**

1. **杀死长事务：**

```sql
-- 查找持有MDL锁的会话
SELECT * FROM information_schema.processlist
WHERE COMMAND != 'Sleep' AND TIME > 60;

-- 杀掉长事务
KILL <process_id>;
```

2. **使用在线DDL工具：**

```bash
# 使用 pt-online-schema-change
pt-online-schema-change \
  --alter "ADD COLUMN new_col INT" \
  D=database,t=large_table \
  --execute
```

3. **NOWAIT和WAIT选项：**

```sql
-- MySQL 8.0+ 支持
ALTER TABLE orders ADD COLUMN status VARCHAR(20) NOWAIT;
-- 如果有锁冲突，立即报错而不是等待

ALTER TABLE orders ADD COLUMN status VARCHAR(20) WAIT 10;
-- 等待10秒，超时后报错
```

#### 意向锁

意向锁是表级锁，用于快速判断表锁和行锁的兼容性。

**为什么需要意向锁？**

```sql
-- 假设没有意向锁

-- 事务A:给某行加了X锁
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- 给id=1的记录加X锁

-- 事务B:想加表锁
LOCK TABLES account READ;
-- 问题:需要检查表中的每一行是否有行锁
-- 如果表有1000万行，需要检查1000万次!
-- 性能极差!

-- 有意向锁后
-- 事务A加行锁时，自动加表级意向锁
-- 事务B只需要检查意向锁，不需要检查每一行
-- 性能大幅提升
```

**意向锁的类型：**

| 锁类型 | 含义 | 加锁时机 |
|--------|------|---------|
| 意向共享锁(IS) | 事务打算在表的某些行上加S锁 | SELECT ... LOCK IN SHARE MODE |
| 意向排他锁(IX) | 事务打算在表的某些行上加X锁 | SELECT ... FOR UPDATE / INSERT / UPDATE / DELETE |

**意向锁的兼容性矩阵：**

| 锁类型 | IS | IX | S | X |
|--------|----|----|---|---|
| 意向共享锁(IS) | ✅ | ✅ | ✅ | ❌ |
| 意向排他锁(IX) | ✅ | ✅ | ❌ | ❌ |
| 共享锁(S) | ✅ | ❌ | ✅ | ❌ |
| 排他锁(X) | ❌ | ❌ | ❌ | ❌ |

**实际示例：**

```sql
-- 事务A:给某行加X锁
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- 自动加表级IX锁 + 行级X锁

-- 事务B:尝试加表读锁
LOCK TABLES account READ;
-- 检查表级锁:有IX锁
-- IX与S冲突
-- 立即失败，不需要检查每一行

-- 事务C:加行锁
START TRANSACTION;
SELECT * FROM account WHERE id = 2 FOR UPDATE;
-- 检查表级锁:有IX锁
-- IX与IX兼容
-- 只检查行锁，成功加锁
```

**总结：**

- 意向锁是表级锁，自动加锁，手动无法控制
- 目的：提高表锁和行锁冲突检查的性能
- IS表示事务在表的某些行上加了S锁
- IX表示事务在表的某些行上加了X锁

#### 记录锁

记录锁(Record Lock)锁住索引记录本身，不锁间隙。

**记录锁的特点：**

```sql
-- 等值查询,记录存在
CREATE TABLE account (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    balance INT
);

INSERT INTO account VALUES (1, '张三', 1000), (5, '李四', 2000), (10, '王五', 3000);

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- 加锁:Record Lock(id=5)
-- 只锁定id=5这一行

-- 事务B:尝试操作其他行
SELECT * FROM account WHERE id = 1 FOR UPDATE;  -- 成功
SELECT * FROM account WHERE id = 10 FOR UPDATE;  -- 成功
UPDATE account SET balance = 100 WHERE id = 1;    -- 成功

-- 事务B:尝试操作被锁的行
SELECT * FROM account WHERE id = 5 FOR UPDATE;  -- 被阻塞
UPDATE account SET balance = 1500 WHERE id = 5;  -- 被阻塞
```

**记录锁与索引的关系：**

```sql
-- 没有索引,全表扫描
CREATE TABLE no_index (
    id INT,
    name VARCHAR(50)
);

INSERT INTO no_index VALUES (1, '张三'), (5, '李四'), (10, '王五');

-- 事务A
START TRANSACTION;
SELECT * FROM no_index WHERE id = 5 FOR UPDATE;
-- 没有索引,全表扫描
-- 对所有扫描到的记录加锁
-- 效果等同于表锁!

-- 事务B
UPDATE no_index SET name = '赵六' WHERE id = 1;  -- 被阻塞
UPDATE no_index SET name = '钱七' WHERE id = 10; -- 被阻塞
-- 所有行都被锁住
```

**有索引 vs 没有索引：**

| 情况 | 加锁范围 | 性能 | 并发度 |
|------|---------|------|--------|
| 有索引，命中 | 单行记录 | 好 | 高 |
| 有索引，未命中 | 间隙锁(可重复读级别) | 一般 | 中 |
| 没有索引 | 全表扫描所有记录 | 差 | 低 |

#### 间隙锁和临键锁

间隙锁(Gap Lock)和临键锁(Next-Key Lock)主要用于防止幻读。

**间隙锁(Gap Lock)：**

```sql
-- 间隙锁锁住索引记录之间的空隙

-- 表中数据: id = 1, 5, 10
-- 间隙:
--   (-∞, 1)
--   (1, 5)
--   (5, 10)
--   (10, +∞)

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- Next-Key Lock锁定:
--   - Record Lock: id=5
--   - Gap Lock: (1, 5) 和 (5, 10)

-- 事务B:尝试插入
INSERT INTO account VALUES (2, '赵六', 100);   -- 被阻塞,在(1,5)间隙中
INSERT INTO account VALUES (6, '钱七', 200);   -- 被阻塞,在(5,10)间隙中
INSERT INTO account VALUES (11, '孙八', 300);  -- 成功,在(10,+∞)间隙外

-- 事务B:尝试更新其他行
UPDATE account SET balance = 2000 WHERE id = 1;  -- 成功
UPDATE account SET balance = 3000 WHERE id = 10; -- 成功
```

**临键锁(Next-Key Lock)：**

Next-Key Lock = Record Lock + Gap Lock

```sql
-- 临键锁锁定范围: (上一条记录, 当前记录]

-- 表中数据: id = 1, 5, 10

-- 执行: SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- Next-Key Lock锁定: (1, 5]

-- 包含:
--   - Gap Lock: (1, 5)
--   - Record Lock: id=5

-- 防止:
--   - 插入id=2,3,4 (在(1,5)间隙中)
--   - 更新id=5 (记录锁)
```

**Next-Key Lock的加锁规则：**

**1) 等值查询，记录存在：**

```sql
-- id=5存在
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- 加锁: Next-Key Lock(5) = (1, 5]

-- 包含:
--   - Record Lock: id=5
--   - Gap Lock: (1, 5)
```

**2) 等值查询，记录不存在：**

```sql
-- id=3不存在(在1和5之间)
SELECT * FROM account WHERE id = 3 FOR UPDATE;
-- 加锁: Gap Lock(1, 5)

-- 只锁定间隙,不包含记录
-- 防止插入id=2,3,4
```

**3) 范围查询：**

```sql
-- 范围查询
SELECT * FROM account WHERE id >= 5 AND id <= 10 FOR UPDATE;
-- 加锁:
--   - Next-Key Lock(5): (1, 5]
--   - Next-Key Lock(10): (5, 10]

-- 防止:
--   - 插入id=2,3,4 (在(1,5)中)
--   - 插入id=6,7,8,9 (在(5,10)中)
```

**Gap Lock vs Record Lock vs Next-Key Lock：**

| 锁类型 | 锁定范围 | 作用 | 是否阻塞插入 |
|--------|---------|------|------------|
| Record Lock | 索引记录 | 锁定单个记录 | 否 |
| Gap Lock | 索引之间的间隙 | 防止其他事务插入 | 是 |
| Next-Key Lock | 记录+前面的间隙 | 防止幻读 | 是 |

**间隙锁的兼容性：**

```sql
-- 间隙锁之间是兼容的!

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- 持有Gap Lock: (1, 5) 和 (5, 10)

-- 事务B
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR SHARE;
-- 成功!Gap Lock之间兼容
-- 也持有Gap Lock: (1, 5) 和 (5, 10)

-- 事务A和B都持有相同的Gap Lock
-- 但它们都尝试插入
INSERT INTO account VALUES (3, '赵六', 100);
-- 事务A尝试插入 → 被B的Gap Lock阻塞
-- 事务B尝试插入 → 被A的Gap Lock阻塞
-- 死锁!
```

**不同隔离级别的差异：**

```sql
-- 可重复读级别
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- 使用Next-Key Lock,包含Gap Lock
-- 防止幻读

-- 读已提交级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- 只使用Record Lock,不使用Gap Lock
-- 不会防止幻读
```

#### 插入意向锁

插入意向锁(Insert Intention Lock)是一种特殊的间隙锁，用于多个事务并发插入到同一间隙的不同位置。

**为什么需要插入意向锁？**

```sql
-- 表中数据: id = 1, 10

-- 事务A
START TRANSACTION;
INSERT INTO account VALUES (5, '张三', 100);
-- 获取插入意向锁
-- 锁定: (1, 10)间隙中的"插入位置5"

-- 事务B
START TRANSACTION;
INSERT INTO account VALUES (6, '李四', 200);
-- 获取插入意向锁
-- 锁定: (1, 10)间隙中的"插入位置6"

-- 两个事务的插入意向锁互相兼容
-- 都成功插入!

-- 如果有事务持有了这个间隙的Gap Lock
-- 事务C
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- 持有Gap Lock: (1, 10)

-- 事务A和B都会被阻塞
-- 等待Gap Lock释放
```

**插入意向锁的特点：**

| 特性 | 说明 |
|------|------|
| 锁类型 | 特殊的Gap Lock |
| 锁定对象 | 间隙中的"插入位置" |
| 兼容性 | 插入意向锁之间互相兼容 |
| 作用 | 允许并发插入到同一间隙的不同位置 |

**实际案例：**

```sql
-- 表中数据: id = 1, 10

-- 场景1:多个事务并发插入不同位置
-- 事务A
START TRANSACTION;
INSERT INTO account VALUES (3, '张三', 100);
-- 插入意向锁: 位置3

-- 事务B
START TRANSACTION;
INSERT INTO account VALUES (5, '李四', 200);
-- 插入意向锁: 位置5

-- 事务C
START TRANSACTION;
INSERT INTO account VALUES (7, '王五', 300);
-- 插入意向锁: 位置7

-- 所有事务都成功,互不阻塞!

-- 场景2:插入相同位置(冲突)
-- 事务A
START TRANSACTION;
INSERT INTO account VALUES (5, '张三', 100);
-- 插入意向锁: 位置5

-- 事务B
START TRANSACTION;
INSERT INTO account VALUES (5, '李四', 200);
-- 等待唯一索引锁
-- 因为id=5已经有插入意向锁
-- 被阻塞

-- 场景3:有Gap Lock存在
-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- 持有Gap Lock: (1, 10)

-- 事务B
START TRANSACTION;
INSERT INTO account VALUES (5, '李四', 200);
-- 被阻塞!插入意向锁与Gap Lock冲突
```

**插入意向锁 vs 普通Gap Lock：**

| 锁类型 | 作用 | 兼容性 |
|--------|------|--------|
| Gap Lock | 防止插入 | 与插入意向锁冲突 |
| 插入意向锁 | 标记插入位置 | 插入意向锁之间兼容 |

#### Auto-Inc Lock

自增锁(Auto-Inc Lock)用于保护自增ID的唯一性。

**Auto-Inc Lock的两种模式：**

**1) 传统模式(traditional):**

```sql
-- 所有INSERT语句都使用表级AUTO-INC锁
-- 锁住整个表的自增ID生成过程
-- 保证自增ID严格递增，但性能差

-- 事务A
START TRANSACTION;
INSERT INTO account (name, balance) VALUES ('张三', 100);
-- 持有AUTO-INC表锁
-- 生成id=1

-- 事务B
START TRANSACTION;
INSERT INTO account (name, balance) VALUES ('李四', 200);
-- 被阻塞,等待事务A释放AUTO-INC锁

COMMIT;  -- 事务A提交,释放AUTO-INC锁
-- 事务B获取AUTO-INC锁,生成id=2
-- 严格递增: 1, 2, 3, 4...
```

**2) 交错模式(interleaved):**

```sql
-- 批量INSERT使用AUTO-INC锁
-- 简单INSERT不使用AUTO-INC锁
-- 性能更好，但自增ID可能不连续

-- 事务A
START TRANSACTION;
INSERT INTO account (name, balance) VALUES ('张三', 100);
-- 不持AUTO-INC锁
-- 快速生成id=1

-- 事务B
START TRANSACTION;
INSERT INTO account (name, balance) VALUES ('李四', 200);
-- 不持AUTO-INC锁
-- 快速生成id=2

-- id序列可能: 1, 2, 5, 6, 3, 4...
-- 不连续,但性能好
```

**配置Auto-Inc Lock模式：**

```sql
-- 查看当前模式
SHOW VARIABLES LIKE 'innodb_autoinc_lock_mode';

-- 0 = 传统模式(traditional)
-- 1 = 连续模式(consecutive, 默认值)
-- 2 = 交错模式(interleaved)

-- 设置为交错模式(性能最好)
SET GLOBAL innodb_autoinc_lock_mode = 2;
```

**三种模式对比：**

| 模式 | 值 | 特点 | 性能 | ID连续性 |
|------|---|------|------|---------|
| 传统模式 | 0 | 所有INSERT都用表锁 | 差 | 严格连续 |
| 连续模式 | 1 | 批量用表锁,简单不用锁 | 好 | 基本连续 |
| 交错模式 | 2 | 简单INSERT不用锁 | 最好 | 不连续 |

**实际案例：**

```sql
-- 连续模式(innodb_autoinc_lock_mode = 1)

-- 表结构
CREATE TABLE account (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(50),
    balance INT
);

-- 事务A:批量INSERT
START TRANSACTION;
INSERT INTO account (name, balance) VALUES
    ('张三', 100),
    ('李四', 200),
    ('王五', 300);
-- 使用AUTO-INC表锁
-- 生成id: 1, 2, 3

-- 事务B:简单INSERT
START TRANSACTION;
INSERT INTO account (name, balance) VALUES ('赵六', 400);
-- 被阻塞,等待事务A释放AUTO-INC锁

COMMIT;  -- 事务A提交,释放AUTO-INC锁
-- 事务B获取AUTO-INC锁,生成id=4
-- id序列: 1, 2, 3, 4, 5, 6... (连续)

-- 交错模式(innodb_autoinc_lock_mode = 2)

-- 事务A和事务B可以并发INSERT
-- id序列: 1, 3, 5, 2, 4, 6...
-- 不连续,但性能好
```

**总结：**

- 传统模式：保证ID严格连续，但性能差，适用于需要严格连续ID的场景
- 连续模式：平衡性能和连续性，默认模式
- 交错模式：性能最好，ID不连续，适用于大多数场景

#### 谓词锁(扩展学习)

谓词锁(Predicate Lock)是MySQL 8.0+引入的新特性，用于支持空间索引。

**什么是谓词锁？**

```sql
-- 谓词锁锁定的是查询条件本身,而不是数据

-- 传统锁:锁定具体数据
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- Next-Key Lock: (1, 5]

-- 谓词锁:锁定查询条件
-- 用于空间索引的复杂查询条件
SELECT * FROM locations
WHERE ST_Contains(geo, POINT(120, 30))
FOR UPDATE;
-- 谓词锁:锁定"ST_Contains(geo, POINT(120, 30))"条件
```

**谓词锁的作用：**

```sql
-- 空间索引表
CREATE TABLE locations (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    geo GEOMETRY,
    SPATIAL INDEX idx_geo (geo)
);

-- 插入空间数据
INSERT INTO locations VALUES
    (1, '杭州', ST_GeomFromText('POINT(120.2, 30.3)')),
    (2, '上海', ST_GeomFromText('POINT(121.5, 31.2)')),
    (3, '南京', ST_GeomFromText('POINT(118.8, 32.1)'));

-- 事务A:查询某个区域内的点
START TRANSACTION;
SELECT * FROM locations
WHERE ST_Contains(
    ST_GeomFromText('POLYGON((120 30, 121 30, 121 31, 120 31, 120 30))'),
    geo
)
FOR UPDATE;
-- 谓词锁:锁定这个多边形区域内的查询条件

-- 事务B:尝试插入新点
START TRANSACTION;
INSERT INTO locations VALUES
    (4, '苏州', ST_GeomFromText('POINT(120.5, 30.5)'));
-- 检查谓词锁
-- 如果这个点在事务A锁定的区域内
-- 被阻塞
-- 否则,成功

-- 事务B:插入区域外的点
INSERT INTO locations VALUES
    (5, '北京', ST_GeomFromText('POINT(116.4, 39.9)'));
-- 成功,不在锁定区域内
```

**谓词锁 vs 传统锁：**

| 特性 | 谓词锁 | 传统锁 |
|------|--------|--------|
| 锁定对象 | 查询条件 | 具体数据 |
| 适用场景 | 空间索引 | 普通索引 |
| 精确度 | 更精确 | 较粗 |
| 复杂度 | 复杂 | 简单 |

**总结：**

谓词锁是MySQL 8.0+的高级特性,主要用于空间索引的复杂查询条件锁定。对于普通业务,主要使用Record Lock、Gap Lock和Next-Key Lock就足够了。

### 💬 面试官追问

#### 提问1:什么情况下会触发表锁而不是行锁?

**回答:**

三种情况。一是查询没走索引，InnoDB只能全表扫描，锁就加到所有扫描到的记录上，效果等于表锁。二是显式用LOCK TABLES 语句加表锁。三是DDL语句比如ALTER TABLE,会加MDL写锁。

**详细说明:**

**情况1:查询没走索引**

```sql
-- 没有索引的表
CREATE TABLE no_index (
    id INT,
    name VARCHAR(50),
    balance INT
);

INSERT INTO no_index VALUES
    (1, '张三', 1000),
    (5, '李四', 2000),
    (10, '王五', 3000);

-- 事务A:查询没走索引
START TRANSACTION;
SELECT * FROM no_index WHERE id = 5 FOR UPDATE;
-- 没有索引,全表扫描
-- 对所有扫描到的记录加锁
-- 等同于表锁!

-- 事务B:所有操作都被阻塞
SELECT * FROM no_index WHERE id = 1 FOR UPDATE;  -- 被阻塞
UPDATE no_index SET balance = 100 WHERE id = 10;  -- 被阻塞
INSERT INTO no_index VALUES (15, '赵六', 400);    -- 被阻塞

-- 解决方法:添加索引
CREATE INDEX idx_id ON no_index(id);

-- 添加索引后
SELECT * FROM no_index WHERE id = 5 FOR UPDATE;
-- 使用索引,只锁定id=5这一行
-- 其他行不受影响
```

**情况2:显式使用LOCK TABLES**

```sql
-- 显式加表锁
LOCK TABLES account READ;
-- 加表级读锁

-- 其他事务
SELECT * FROM account;                   -- 成功,可以读
INSERT INTO account VALUES (...);          -- 被阻塞
UPDATE account SET ...;                   -- 被阻塞
DELETE FROM account WHERE ...;            -- 被阻塞

-- 解锁
UNLOCK TABLES;
```

**情况3:DDL语句加MDL锁**

```sql
-- DDL语句自动加MDL写锁
ALTER TABLE account ADD COLUMN status VARCHAR(20);
-- 加MDL写锁

-- 其他事务的所有操作都要等MDL读锁
-- 但MDL写锁优先级更高
-- 所以所有查询都被阻塞

-- 详见[元数据锁](#元数据锁mdl)部分
```

**性能对比:**

| 情况 | 锁类型 | 性能 | 并发度 |
|------|--------|------|--------|
| 有索引,命中 | 行锁 | 好 | 高 |
| 有索引,未命中 | 间隙锁 | 一般 | 中 |
| 没有索引 | 表锁 | 差 | 低 |
| LOCK TABLES | 表锁 | 差 | 低 |
| DDL | MDL锁 | 极差 | 极低 |

**最佳实践:**

```sql
-- 1. 确保查询走索引
-- 检查执行计划
EXPLAIN SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- key列应该显示使用的索引

-- 2. 避免全表扫描加锁
-- 错误示例
SELECT * FROM large_table WHERE name LIKE '%张%' FOR UPDATE;
-- 全表扫描,锁住所有行

-- 正确示例
SELECT * FROM large_table WHERE name = '张三' FOR UPDATE;
-- 使用索引,只锁一行

-- 3. 添加合适索引
CREATE INDEX idx_name ON account(name);
CREATE INDEX idx_balance ON account(balance);

-- 4. DDL操作在低峰期执行
-- 避免阻塞线上查询
```

#### 提问2:两个事务都用间隙锁锁住同一个间隙,会死锁吗?

**回答:**

单纯持有间隙锁不会，因为间隙锁之间不冲突。但如果两个事务各自持有间隙锁，然后都想往对方锁住的间隙里插数据，就会死锁。比如事务A持有(3,5)的间隙锁想插(5,10)，事务B持有(5,10)的间隙锁想插(3,5),互相等着对方释放，死锁触发。

**详细说明:**

**情况1:单纯持有间隙锁(不会死锁)**

```sql
-- 表中数据: id = 3, 5, 10

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- Next-Key Lock: (3, 5] + (5, 10)
-- 持有Gap Lock: (3, 5) 和 (5, 10)

-- 事务B
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR SHARE;
-- Next-Key Lock: (3, 5] + (5, 10)
-- 持有Gap Lock: (3, 5) 和 (5, 10)

-- Gap Lock之间是兼容的!
-- 两个事务都持有相同的Gap Lock
-- 不会死锁
```

**情况2:持有间隙锁后尝试插入(可能死锁)**

```sql
-- 表中数据: id = 3, 5, 10

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- 持有Gap Lock: (3, 5) 和 (5, 10)

-- 事务B
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- 持有Gap Lock: (3, 5) 和 (5, 10)

-- 事务A:尝试插入
INSERT INTO account VALUES (7, '张三', 100);
-- 尝试在(5, 10)间隙插入
-- 但(5, 10)被事务B的Gap Lock锁住
-- 被阻塞

-- 事务B:尝试插入
INSERT INTO account VALUES (4, '李四', 200);
-- 尝试在(3, 5)间隙插入
-- 但(3, 5)被事务A的Gap Lock锁住
-- 被阻塞

-- 死锁!
-- A等B释放(5, 10)
-- B等A释放(3, 5)
-- 互相等待

-- MySQL检测到死锁,自动回滚其中一个事务
-- 查看死锁日志
SHOW ENGINE INNODB STATUS;
```

**时间线示意图:**

```

----  ----------------------------------------  ----------------------------------------

      持有Gap Lock: (3, 5), (5, 10)

T4                                                SELECT ... FOR UPDATE id=5

                                                  (成功,Gap Lock兼容)
T5    INSERT VALUES (7, ...)  想插在(5,10)
      被阻塞,等B的(5,10)
T6                                                INSERT VALUES (4, ...)  想插在(3,5)
                                                  被阻塞,等A的(3,5)
T7    ❌ 死锁!互相等待
T8    MySQL回滚事务B
T9    事务A成功插入
```

**避免死锁的方法:**

```sql
-- 方法1:减少锁持有时间
-- ❌ 错误:长时间持有锁
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- ... 执行5分钟的业务逻辑 ...
INSERT INTO account VALUES (7, '张三', 100);
COMMIT;

-- ✅ 正确:快速提交
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
INSERT INTO account VALUES (7, '张三', 100);
COMMIT;

-- 方法2:按固定顺序访问
-- ❌ 错误:不同顺序
-- 事务A: 先锁(3,5),再插(5,10)
-- 事务B: 先锁(5,10),再插(3,5)
-- 容易死锁

-- ✅ 正确:固定顺序
-- 所有事务都按id从小到大顺序处理
-- 减少死锁概率

-- 方法3:使用读已提交隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
-- 读已提交级别不使用Gap Lock
-- 只使用Record Lock
-- 减少死锁概率

-- 方法4:设置锁超时
SET innodb_lock_wait_timeout = 10;
-- 超过10秒自动回滚
-- 避免长时间等待

-- 方法5:重试机制
-- 应用层捕获死锁错误
-- 自动重试事务
```

**总结:**

- Gap Lock之间是兼容的,单纯持有不会死锁
- 但如果尝试插入被Gap Lock锁住的间隙,就会死锁
- 解决方法:减少锁持有时间、按固定顺序、使用RC级别、设置超时、重试机制
- 详见[间隙锁和临键锁](#间隙锁和临键锁)部分的详细说明

#### 提问3:为什么说间隙锁是可重复读级别特有的?

**回答:**

可重复读需要防止幻读，间隙锁就是干这个的。读已提交级别不需要防幻读，所以间隙锁关闭只保留记录锁，锁范围小死锁概率也低。这也是很多互联网公司选择读已提交级别的原因之一。

**详细说明:**

**可重复读级别的间隙锁:**

```sql
-- 可重复读级别
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 表中数据: id = 1, 5, 10

-- 事务A:当前读
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- Next-Key Lock: (1, 5] + (5, 10)
-- 包含Gap Lock

-- 事务B:尝试插入
START TRANSACTION;
INSERT INTO account VALUES (3, '张三', 100);   -- 被阻塞,在(1,5)间隙
INSERT INTO account VALUES (7, '李四', 200);   -- 被阻塞,在(5,10)间隙
-- Gap Lock防止插入,避免幻读

-- 事务A:再次查询
SELECT * FROM account WHERE id BETWEEN 1 AND 10;
-- 结果一致,没有新记录
-- 没有幻读

COMMIT;
```

**读已提交级别不使用间隙锁:**

```sql
-- 读已提交级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 表中数据: id = 1, 5, 10

-- 事务A:当前读
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- Record Lock: id=5
-- 不使用Gap Lock!

-- 事务B:成功插入
START TRANSACTION;
INSERT INTO account VALUES (3, '张三', 100);   -- 成功!
INSERT INTO account VALUES (7, '李四', 200);   -- 成功!
COMMIT;

-- 事务A:再次查询
SELECT * FROM account WHERE id BETWEEN 1 AND 10;
-- 结果: 1, 3, 5, 7, 10
-- 多了两条记录(3和7)
-- 幻读!

-- 但事务A的当前读不会受影响
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- 还是只能看到id=5
-- Record Lock不影响其他事务
```

**两种级别的锁对比:**

| 特性 | 可重复读(RR) | 读已提交(RC) |
|------|-------------|-------------|
| Next-Key Lock | 使用 | 不使用 |
| Gap Lock | 使用 | 不使用 |
| Record Lock | 使用 | 使用 |
| 防止幻读 | ✅ 是 | ❌ 否 |
| 死锁概率 | 较高 | 较低 |
| 并发性能 | 较好 | 更好 |

**为什么读已提交不使用Gap Lock?**

```sql
-- 读已提交的设计目标:
-- 1. 能读到已提交的数据
-- 2. 不需要防止幻读(可以接受)
-- 3. 性能更好

-- 如果读已提交也用Gap Lock:
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- Gap Lock: (1, 5) 和 (5, 10)
-- 其他事务插入被阻塞
-- 并发性能下降
-- 但读已提交级别本身就可能出现幻读
-- 用Gap Lock也没意义

-- 所以读已提交只用Record Lock:
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- Record Lock: id=5
-- 其他事务可以插入
-- 性能更好
```

**实际案例对比:**

```sql
-- 场景:库存管理

-- 可重复读级别
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 事务A
START TRANSACTION;
SELECT * FROM inventory WHERE product_id = 100 FOR UPDATE;
-- Next-Key Lock锁定product_id=100的范围
-- 防止其他事务插入product_id=100的新记录
-- 保证库存记录不变

-- 事务B
INSERT INTO inventory (id, product_id, quantity) VALUES (10, 100, 5);
-- 被阻塞,Gap Lock防止插入

-- 优点:防止幻读
-- 缺点:死锁概率高

-- 读已提交级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 事务A
START TRANSACTION;
SELECT * FROM inventory WHERE product_id = 100 FOR UPDATE;
-- Record Lock锁定已有记录
-- 不防止插入

-- 事务B
INSERT INTO inventory (id, product_id, quantity) VALUES (10, 100, 5);
-- 成功插入

-- 优点:死锁概率低,性能好
-- 缺点:可能出现幻读
```

**互联网公司为什么选择读已提交?**

```sql
-- 原因1:避免间隙锁导致的死锁
-- 可重复读级别
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- Gap Lock: (1, 5) 和 (5, 10)
-- 容易死锁

-- 读已提交级别
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- Record Lock: id=5
-- 死锁概率低

-- 原因2:性能更好
-- 可重复读级别
-- Next-Key Lock锁定范围大
-- 并发性能受限

-- 读已提交级别
-- Record Lock锁定范围小
-- 并发性能更好

-- 原因3:业务可以接受幻读
-- 大部分业务场景
-- 查询统计可以容忍少量不准确
-- 不需要防止幻读

-- 只有少数场景需要防止幻读
-- 如:库存扣减、订单处理
-- 可以使用SELECT FOR UPDATE + 应用层逻辑
```

**如何选择隔离级别?**

| 业务场景 | 推荐隔离级别 | 原因 |
|---------|-------------|------|
| 银行转账 | REPEATABLE READ | 需要防止幻读,数据一致性最重要 |
| 电商订单 | REPEATABLE READ | 订单查询需要数据一致 |
| 电商秒杀 | READ COMMITTED | 性能重要,可以容忍少量不准确 |
| 日志分析 | READ COMMITTED/READ UNCOMMITTED | 性能重要,数据准确性要求不高 |
| 监控系统 | READ COMMITTED | 需要看到最新数据 |
| 统计报表 | REPEATABLE READ | 需要数据一致 |

**配置建议:**

```sql
-- 传统企业应用
-- 使用默认的可重复读
SET GLOBAL TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 互联网高并发应用
-- 使用读已提交
SET GLOBAL TRANSACTION ISOLATION LEVEL READ COMMITTED;
SET GLOBAL binlog_format = ROW;

-- 原因:
-- 1. 避免间隙锁死锁
-- 2. 提高并发性能
-- 3. 业务可以容忍少量幻读
```

**总结:**

- Gap Lock是可重复读级别特有的,用于防止幻读
- 读已提交级别不使用Gap Lock,只使用Record Lock
- 可重复读:防幻读,但死锁概率高
- 读已提交:不防幻读,但死锁概率低,性能好
- 互联网公司倾向于选择读已提交,避免死锁,提高性能
- 关键业务场景仍使用可重复读,保证数据一致性

#### 提问4:线上加字段导致查询全部超时,原因是什么?

**回答:**

ALTER TABLE 需要拿MDL写锁，如果有长事务持有 MDL读锁没释放，ALTER TABLE就一直等。更惨的是后面的查询也要拿MDL读锁，但写锁请求优先级更高，所以后面的查询全都堵在ALTER TABLE后面。解决办法是先干掉长事务，或者用 pt-online-schema-change 这类工具做无锁 DDL。

**详细说明:**

**完整事故过程:**

```sql
-- 步骤1:长事务开始
-- 事务A
START TRANSACTION;
SELECT * FROM large_table WHERE id = 1;
-- ... 长时间未提交,比如5分钟 ...
-- 持有MDL读锁

-- 步骤2:执行DDL
-- 事务B
ALTER TABLE large_table ADD COLUMN new_col INT;
-- 需要MDL写锁
-- 但事务A持有MDL读锁
-- MDL写锁等待MDL读锁释放
-- 状态:Waiting for table metadata lock

-- 步骤3:后续查询也被阻塞
-- 事务C
SELECT * FROM large_table WHERE id = 2;
-- 需要MDL读锁
-- 但MDL写锁优先级更高
-- 队列: [事务B(MDL写锁)] → [事务C(MDL读锁)]
-- 事务C等待事务B,事务B等待事务A
-- 状态:Waiting for table metadata lock

-- 步骤4:所有查询都超时
-- 事务D, E, F...
SELECT * FROM large_table WHERE ...;
-- 全部被阻塞
-- 整个表的所有查询都超时!

-- 结果:线上故障!
```

**时间线示意图:**

```
时间  事务A(长事务)                     事务B(DDL)                     事务C(查询)
----  ------------------------------  -----------------------------  ---------------------------
T1    START TRANSACTION
T2    SELECT * FROM large_table ...
T3    持有MDL读锁
T4                                    ALTER TABLE ADD COLUMN ...
T5                                    等待MDL写锁(被A阻塞)
T6                                                                  SELECT * FROM large_table
T7                                                                  等待MDL读锁(被B阻塞)
T8    ... (5分钟后) ...
T9    COMMIT
      释放MDL读锁
T10                                   获取MDL写锁
                                      执行DDL
T11                                   COMMIT
                                      释放MDL写锁
T12                                                                  获取MDL读锁
                                                                     成功查询
```

**为什么MDL写锁优先级高?**

```sql
-- MySQL的设计:
-- DDL操作通常比较关键
-- 需要尽快完成
-- 所以MDL写锁优先级更高

-- 队列顺序:
-- [DDL(MDL写锁)] → [查询(MDL读锁)] → [查询(MDL读锁)] → ...

-- 即使DDL后面有很多查询
-- DDL也会先执行

-- 为什么这样设计?
-- 避免DDL长时间等待
-- 如果MDL读锁优先级高
-- DDL可能永远等不到
```

**MDL锁的类型和兼容性:**

| MDL类型 | 说明 | 持有场景 | 兼容性 |
|---------|------|---------|--------|
| MDL读锁(共享) | 允许读,不允许改表结构 | SELECT、INSERT、UPDATE、DELETE | 与MDL读锁兼容 |
| MDL写锁(排他) | 独占,不允许任何操作 | DDL(ALTER TABLE、DROP TABLE等) | 与所有MDL冲突 |

**查看MDL锁状态:**

```sql

WHERE COMMAND != 'Sleep' AND TIME > 60;

-- 查看锁等待

```

**解决方法:**

**方法1:杀死长事务**

## 2. MySQL 事务的二阶段提交是什么？

### 📌 回答重点

二阶段提交是MySQL用来保证redo log和binlog数据一致性的机制。因为这两个日志分属不同层，一个是InnoDB 引擎层的，一个是Server层的，如果写入过程中宕机，就可能出现两边数据对不上的问题。二阶段提交就是用来解决这个问题。

整个过程分两步走：

**1) Prepare阶段：**
事务提交时，InnoDB先把修改写到redo log，但状态标记为prepare，表示"我准备好了，但还没真正提交"

**2) Commit 阶段：**
redo log写完后，Server 层把操作写到binlog。binlog落盘成功，再通知InnoDB把 redo log状态改成commit，整个事务才算提交完成

---

### 📚 扩展知识

#### binlog和 redo log各管什么

MySQL有两个非常重要的日志：redo log和binlog，它们各自负责不同的职责。

**redo log（重做日志）：**

```sql
-- redo log是InnoDB引擎特有的日志
-- 作用：崩溃恢复
-- 特点：循环写入、物理日志、InnoDB私有

-- 工作原理：
-- 1. 事务修改数据时，先写到内存中的数据页
-- 2. 同时把修改记录到redo log buffer
-- 3. 定期刷到磁盘的redo log file

-- 示例：更新操作
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1;
-- InnoDB会记录：
-- - 哪个数据页被修改
-- - 修改前后的值
-- - 事务ID等

COMMIT;

-- 如果在写入数据页到磁盘之前宕机
-- 重启后通过redo log恢复数据
-- 保证数据不丢失
```

**redo log的特点：**

| 特性 | 说明 |
|------|------|
| 归属 | InnoDB引擎层 |
| 作用 | 崩溃恢复，保证持久性 |
| 记录方式 | 物理日志（记录数据页的物理修改） |
| 写入方式 | 循环写入（固定大小，写满覆盖） |
| 位置 | InnoDB内部 |
| 大小 | 配置innodb_log_file_size |

**binlog（二进制日志）：**

```sql
-- binlog是MySQL Server层的日志
-- 作用：主从复制、数据归档、恢复
-- 特点：追加写入、逻辑日志、所有存储引擎通用

-- 工作原理：
-- 1. 事务提交时，Server层把操作的SQL语句或行变更记录到binlog
-- 2. 写到binlog cache
-- 3. 定期刷到磁盘的binlog file

-- 示例：相同的更新操作
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1;
-- binlog会记录：
-- - SQL语句（statement格式）或
-- - 行变更（row格式）或
-- - 混合格式（mixed格式）
-- - 事务信息

COMMIT;

-- 从库通过读取binlog
-- 重放相同的操作
-- 实现主从同步
```

**binlog的特点：**

| 特性 | 说明 |
|------|------|
| 归属 | MySQL Server层 |
| 作用 | 主从复制、数据恢复、审计 |
| 记录方式 | 逻辑日志（记录SQL或行变更） |
| 写入方式 | 追加写入（不覆盖，可以保留历史） |
| 位置 | MySQL Server |
| 大小 | 配置max_binlog_size，自动分割 |

**两者的对比：**

| 对比维度 | redo log | binlog |
|---------|----------|--------|
| 归属 | InnoDB引擎层 | MySQL Server层 |
| 作用 | 崩溃恢复 | 主从复制、数据恢复 |
| 记录内容 | 物理日志（数据页修改） | 逻辑日志（SQL或行变更） |
| 写入方式 | 循环写入 | 追加写入 |
| 适用范围 | 仅InnoDB | 所有存储引擎 |
| 持久化时机 | 事务提交时（prepare/commit） | 事务提交时 |
| 恢复能力 | 崩溃后恢复 | 历史数据恢复、主从同步 |

**为什么需要两个日志？**

```sql
-- 场景1:崩溃恢复
-- 宕机后，需要恢复未写入磁盘的数据
-- redo log记录了数据页的物理修改
-- 可以快速恢复到崩溃前状态

-- 场景2:主从复制
-- 需要把主库的操作传给从库
-- binlog记录了SQL或行变更
-- 从库重放相同操作，保持数据一致

-- 场景3:数据归档
-- 需要保留历史操作记录
-- binlog追加写入，不覆盖
-- 可以长期保存用于审计和恢复
```

**实际案例：**

```sql
-- 场景:银行转账
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1;
UPDATE account SET balance = balance + 100 WHERE id = 2;
COMMIT;

-- 这个事务会同时写入两个日志：

-- redo log记录：
-- - 页号X，偏移量Y，修改前值1000，修改后值900
-- - 页号A，偏移量B，修改前值2000，修改后值2100
-- - 事务ID、时间戳等

-- binlog记录（row格式）：
-- - UPDATE account SET balance=900 WHERE id=1
-- - UPDATE account SET balance=2100 WHERE id=2
-- - 事务开始时间、提交时间等

-- 如果在这两个日志之间宕机：
-- - 只有redo log：主库可以恢复，但从库不会同步
-- - 只有binlog：主库丢失数据，但可以从库恢复
-- - 两个都有：完美恢复，主从一致
```

#### 为什么需要二阶段提交

如果没有二阶段提交，直接先写redo log再写binlog（或反过来），会出现什么问题？

**方案1：先写redo log，再写binlog**

```sql
-- 事务A:扣款100元
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1;

-- 步骤1:写入redo log
-- redo log状态：prepare
-- 内容：id=1的账户余额从1000改成900

-- 步骤2:写入binlog
-- ❌ 在写binlog之前宕机！
-- binlog还没写

-- 重启后：
-- - redo log还在（prepare状态）
-- - binlog没有这个事务
-- - 从库不会同步这个操作
-- - 问题：主库扣了100元，从库没扣
-- - 结果：主从不一致！
```

**方案2：先写binlog，再写redo log**

```sql
-- 事务A:扣款100元
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1;

-- 步骤1:写入binlog
-- binlog写成功
-- 从库同步了这个操作

-- 步骤2:写入redo log
-- ❌ 在写redo log之前宕机！
-- redo log还是旧状态

-- 重启后：
-- - binlog有这个事务
-- - 从库已经同步（余额900）
-- - redo log没有这个事务
-- - 主库恢复后还是旧值（余额1000）
-- - 问题：从库扣了100元，主库没扣
-- - 结果：主从不一致！
```

**二阶段提交如何解决？**

```sql
-- 事务A:扣款100元
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1;

-- 阶段1:Prepare阶段
-- InnoDB写入redo log，状态标记为prepare
-- redo log: id=1, 1000→900, XID=1001, status=PREPARE

-- 阶段2:Commit阶段
-- 步骤2.1:写入binlog
-- Server层写入binlog
-- binlog: UPDATE account SET balance=900 WHERE id=1, XID=1001
-- binlog落盘成功

-- 步骤2.2:提交redo log
-- 通知InnoDB把redo log状态改成commit
-- redo log: id=1, 1000→900, XID=1001, status=COMMIT

-- 整个事务提交完成

-- 如果在某个阶段宕机：
```

**情况1：redo log写完prepare后宕机**

```sql
-- 宕机时状态：
-- - redo log: PREPARE状态
-- - binlog: 没有这个事务

-- 重启后恢复：
-- 1. InnoDB检查redo log的XID
-- 2. 拿XID去binlog里查找
-- 3. 发现binlog里没有这个XID
-- 4. 判断：binlog未写成功，事务回滚
-- 5. 结果：主从都未扣款，一致
```

**情况2：redo log写完prepare，binlog也写完，但redo log还没改成commit**

```sql
-- 宕机时状态：
-- - redo log: PREPARE状态
-- - binlog: 有这个事务（XID=1001）

-- 重启后恢复：
-- 1. InnoDB检查redo log的XID
-- 2. 拿XID=1001去binlog里查找
-- 3. 发现binlog里有这个XID
-- 4. 判断：binlog写成功了，应该提交
-- 5. 把redo log状态改成COMMIT
-- 6. 结果：主从都扣款成功，一致
```

**情况3：redo log已经改成commit**

```sql
-- 宕机时状态：
-- - redo log: COMMIT状态
-- - binlog: 有这个事务

-- 重启后恢复：
-- - redo log已经是COMMIT状态
-- - 无需恢复，直接继续
-- - 结果：正常，主从一致
```

**二阶段提交的核心逻辑：**

```sql
-- 伪代码
function two_phase_commit(txn):
    # 阶段1: Prepare阶段
    redo_log.prepare(txn)
    
    # 阶段2: Commit阶段
    try:
        binlog.write(txn)
        binlog.fsync()  # 落盘
        redo_log.commit(txn)  # 改成commit状态
    except:
        # 如果binlog写失败，回滚redo log
        redo_log.rollback(txn)
        return False
    
    return True

# 崩溃恢复时
function crash_recovery():
    for txn in redo_log.transactions:
        if txn.status == PREPARE:
            # 检查binlog里是否有这个事务
            if binlog.contains(txn.xid):
                # binlog成功了，提交
                redo_log.commit(txn)
            else:
                # binlog失败了，回滚
                redo_log.rollback(txn)
```

**总结：**

二阶段提交通过XID（事务ID）来关联redo log和binlog，确保两个日志的一致性：

- **如果redo log是PREPARE，binlog里有XID → 提交**（因为binlog写成功了）
- **如果redo log是PREPARE，binlog里没有XID → 回滚**（因为binlog没写成功）
- **如果redo log是COMMIT → 无需处理**（事务已提交）

这样无论在哪个阶段宕机，都能保证主从数据一致。

#### 如何对比redo log和binlog是一致的

MySQL通过XID（事务ID）来对比redo log和binlog是否一致。

**XID的作用：**

```sql
-- XID（Transaction ID）是MySQL中唯一标识一个事务的ID
-- 它贯穿整个事务生命周期，关联redo log和binlog

-- 示例事务
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1;
COMMIT;

-- 在这个事务中，XID会在以下地方出现：
-- 1. redo log: 记录XID和事务状态
-- 2. binlog: 记录XID
-- 3. 事务表: 记录XID（如果开启了事务日志）

-- 通过对比XID，可以判断redo log和binlog是否一致
```

**崩溃恢复时的对比流程：**

```sql
-- 步骤1:扫描redo log
-- InnoDB重启后，扫描所有状态为PREPARE的redo log记录

-- 伪代码
for redo_log in innodb.redo_logs:
    if redo_log.status == PREPARE:
        xid = redo_log.xid
        # 去binlog里查找这个XID
        found = binlog.find_xid(xid)
        
        if found:
            # binlog里有这个XID，说明binlog写成功了
            # 需要提交这个事务
            redo_log.commit()
        else:
            # binlog里没有这个XID，说明binlog没写成功
            # 需要回滚这个事务
            redo_log.rollback()

-- 步骤2:根据对比结果恢复
-- - 提交的事务：binlog和redo log都写成功了
-- - 回滚的事务：只有redo log，binlog没写
```

**实际案例：**

```sql
-- 场景:3个并发事务
-- 事务A: id=1001, UPDATE account SET balance=balance-100 WHERE id=1
-- 事务B: id=1002, UPDATE account SET balance=balance+50 WHERE id=2
-- 事务C: id=1003, UPDATE account SET balance=balance+50 WHERE id=3

-- 执行过程：
-- T1: 事务A写入redo log PREPARE
-- T2: 事务A写入binlog
-- T3: 事务A提交redo log COMMIT
-- T4: 事务B写入redo log PREPARE
-- T5: 事务B写入binlog
-- T6: ❌ 宕机！（事务B的redo log还没改成commit）
-- T7: 事务C写入redo log PREPARE
--     （binlog还没写）

-- 重启后恢复：
-- 检查redo log状态为PREPARE的事务

-- 事务B (XID=1002):
-- redo log: PREPARE状态
-- binlog: 找到XID=1002
-- 判断: binlog写成功了，提交
-- 操作: redo_log.commit(XID=1002)

-- 事务C (XID=1003):
-- redo log: PREPARE状态
-- binlog: 找不到XID=1003
-- 判断: binlog没写成功，回滚
-- 操作: redo_log.rollback(XID=1003)

-- 结果:
-- 事务A: 提交（已经COMMIT）
-- 事务B: 提交（binlog有，恢复后提交）
-- 事务C: 回滚（binlog没有，恢复后回滚）
```

**binlog的XID:**

```sql
-- binlog的XID格式
# Xid = 1001

-- 包含信息:
-- - 事务ID（1001）
-- - 用于在崩溃恢复时与redo log对比
-- - 格式: Xid = <transaction_id>

-- GTID模式下的XID:
GTID = server_uuid:transaction_id
# 例如: 3A11C477-CA6A-11D1-BF9B-0000C029B634:1001
```

**一致性检查的时序：**

```
时序  redo log状态            binlog状态            操作
----  ---------------------  -------------------  ------------------
T1    PREPARE (XID=1001)     无                    写入redo log
T2    PREPARE (XID=1001)     有 (XID=1001)        写入binlog成功
T3    COMMIT (XID=1001)      有 (XID=1001)        提交redo log
T4    COMMIT (XID=1001)      有 (XID=1001)        事务完成

如果T2之后宕机：
- redo log: PREPARE
- binlog: 有 (XID=1001)
- 恢复时发现XID=1001在binlog中 → 提交

如果T1之后宕机：
- redo log: PREPARE
- binlog: 无
- 恢复时发现XID=1001不在binlog中 → 回滚
```

**总结：**

通过XID关联redo log和binlog，MySQL可以在崩溃恢复时：

1. **扫描redo log**：找到所有PREPARE状态的事务
2. **对比binlog**：用XID在binlog中查找
3. **判断状态**：
   - binlog有XID → binlog写成功，提交
   - binlog无XID → binlog没写，回滚
4. **恢复数据**：提交或回滚对应事务

这样就能保证redo log和binlog的一致性，确保主从数据同步。

#### 组提交优化

二阶段提交虽然保证了数据一致性，但每次事务提交都要写两次磁盘（redo log和binlog），性能开销较大。MySQL引入了组提交（Group Commit）来优化这个问题。

**组提交的原理：**

```sql
-- 传统提交方式（串行）
-- 事务A提交 → 写redo log → 写binlog → 完成（耗时10ms）
-- 事务B提交 → 写redo log → 写binlog → 完成（耗时10ms）
-- 事务C提交 → 写redo log → 写binlog → 完成（耗时10ms）
-- 总耗时: 30ms

-- 组提交方式（并行）
-- 事务A、B、C几乎同时提交
-- 一起写redo log → 一起写binlog → 一起完成
-- 总耗时: 10ms（只写一次磁盘）

-- 组提交把多个事务的日志合并，一次性写入磁盘
-- 减少了磁盘IO次数，提升性能
```

**组提交的三个阶段：**

```sql
-- MySQL 5.6+的组提交分为三个阶段

-- 阶段1: Flush阶段
-- 将redo log buffer刷到磁盘
-- 多个事务的redo log可以一起刷盘

-- 阶段2: Sync阶段
-- 将binlog cache刷到磁盘
-- 多个事务的binlog可以一起刷盘

-- 阶段3: Commit阶段
-- 将redo log状态改成commit
-- 多个事务可以一起提交

-- 每个阶段都有一个队列
-- 后来的事务加入队列，等待前面的写完
-- 一起提交，减少IO
```

**组提交的时序图：**

```
时间轴: T1 → T2 → T3 → T4 → T5 → T6 → T7 → T8 → T9

事务A提交:
T1: 加入Flush队列
T2: Flush阶段：刷redo log到磁盘
T3: 加入Sync队列
T4: Sync阶段：刷binlog到磁盘
T5: 加入Commit队列
T6: Commit阶段：修改redo log状态
T7: 提交完成

事务B提交（在T2时刻）:
T2: 加入Flush队列（和A一起）
T3: Flush阶段：一起刷redo log
T4: 加入Sync队列（和A一起）
T5: Sync阶段：一起刷binlog
T6: 加入Commit队列（和A一起）
T7: Commit阶段：一起修改状态
T8: 提交完成

事务C提交（在T3时刻）:
T3: 加入Flush队列
T4: Flush阶段：刷redo log（和B一起）
T5: 加入Sync队列
T6: Sync阶段：刷binlog（和B一起）
T7: 加入Commit队列
T8: Commit阶段：修改状态
T9: 提交完成

结果:
- 事务A、B的redo log和binlog是合并写的
- 事务C和A、B不是一组，分开写
- IO次数减少了
```

**组提交的队列机制：**

```sql
-- 每个阶段都有一个队列
-- MySQL会等待一段时间，把同一批事务合并写入

-- Flush队列
-- 队列: [事务A, 事务B, 事务C]
-- 操作: 一起刷redo log到磁盘
-- 耗时: 1次IO

-- Sync队列
-- 队列: [事务A, 事务B, 事务C]
-- 操作: 一起刷binlog到磁盘
-- 耗时: 1次IO

-- Commit队列
-- 队列: [事务A, 事务B, 事务C]
-- 操作: 一起修改redo log状态
-- 耗时: 内存操作，不涉及IO
```

**组提交的配置参数：**

```sql
-- 查看组提交相关配置
SHOW VARIABLES LIKE 'binlog_group_commit%';

-- 参数说明:

-- binlog_group_commit_sync_delay
-- 等待微秒数，延迟提交以合并更多事务
-- 默认: 0 (不延迟)
-- 生产: 建议设置100-1000微秒

-- binlog_group_commit_sync_no_delay_count
-- 累积多少个事务后立即提交，不等待
-- 默认: 0 (不限制)
-- 生产: 建议设置5-10个事务

-- 示例配置:
SET GLOBAL binlog_group_commit_sync_delay = 500;
-- 等待500微秒，合并更多事务

SET GLOBAL binlog_group_commit_sync_no_delay_count = 10;
-- 累积10个事务后立即提交
```

**组提交的性能对比：**

```sql
-- 测试场景: 1000个事务并发提交

-- 不开启组提交
-- 每个事务独立提交
-- IO次数: 2000次 (1000次redo log + 1000次binlog)
-- 耗时: 约2000ms
-- 吞吐量: 500 TPS

-- 开启组提交（延迟500微秒）
-- 每10个事务合并提交
-- IO次数: 200次 (100组 × 2次IO)
-- 耗时: 约200ms
-- 吞吐量: 5000 TPS

-- 性能提升: 10倍!
```

**组提交的缺点和权衡：**

```sql
-- 优点:
-- 1. 减少磁盘IO次数
-- 2. 提升吞吐量
-- 3. 提高并发性能

-- 缺点:
-- 1. 增加延迟（等待合并）
-- 2. 耗费内存（缓存事务）

-- 权衡:
-- 延迟时间设置:
-- - 0微秒: 立即提交，无延迟，低吞吐
-- - 100-500微秒: 适中延迟，高吞吐（推荐）
-- - >1000微秒: 高延迟，最高吞吐

-- 实际案例:
-- 秒杀场景:
-- - 设置binlog_group_commit_sync_delay=0
-- - 要求低延迟，牺牲部分吞吐

-- 日志写入场景:
-- - 设置binlog_group_commit_sync_delay=500
-- - 允许稍高延迟，追求高吞吐
```

**如何观察组提交效果：**

```sql

-- 组提交时，多个事务合并到同一个binlog事件中

# Xid = 1001
# Xid = 1002
# Xid = 1003
# ...
# Xid = 1010

# 这10个XID可能在同一个binlog文件中

-- 方法2:监控IO
-- 观察磁盘IO次数

-- 每秒1000次IO

-- 组提交后:
-- 每秒100次IO (合并了10倍)

-- 方法3:监控提交延迟
SHOW STATUS LIKE 'Innodb_row_lock_time';
SHOW STATUS LIKE 'Innodb_row_lock_waits';
```

**组提交的最佳实践：**

```sql
-- 1. 根据业务场景配置
-- 低延迟场景（秒杀、实时交易）:
SET GLOBAL binlog_group_commit_sync_delay = 0;
SET GLOBAL binlog_group_commit_sync_no_delay_count = 0;

-- 高吞吐场景（日志写入、批量处理）:
SET GLOBAL binlog_group_commit_sync_delay = 500;
SET GLOBAL binlog_group_commit_sync_no_delay_count = 10;

-- 2. 监控性能指标
-- 监控延迟: 平均提交时间
-- 监控吞吐: 每秒事务数
-- 监控IO: 磁盘IO次数

-- 3. 定期调优
-- 根据监控数据调整参数
-- 找到延迟和吞吐的平衡点
```

**总结：**

组提交通过合并多个事务的日志写入，显著减少了磁盘IO次数，提升了事务提交的吞吐量。主要优化点：

1. **减少IO次数**：多个事务的redo log和binlog合并写入
2. **提升吞吐量**：并发事务一起提交，提高并发性能
3. **可配置延迟**：通过`binlog_group_commit_sync_delay`和`binlog_group_commit_sync_no_delay_count`调优

但需要注意权衡延迟和吞吐，根据业务场景选择合适的配置。

### 💬 面试官追问

#### 提问1:为什么不直接把redo log和binlog合成一个日志?

**回答:**

因为它们本来就是给不同场景用的。redo log是InnoDB引擎私有的，记录物理修改，循环写，主要保证崩溃恢复。binlog是Server 层的，所有存储引擎都能用，记录逻辑操作，追加写，主要做主从复制和归档。MySQL架构上Server层和存储引擎层是分开的，强行合并会破坏分层设计，而且binlog的格式要兼容各种存储引擎，没法跟InnoDB 的物理日志混在一起。

**详细说明:**

**MySQL的分层架构:**

```sql
-- MySQL采用分层架构设计:
-- Server层: 连接管理、权限校验、查询解析、binlog等
-- 存储引擎层: 数据存储、索引、事务、redo log等

-- Server层是公共层，支持多种存储引擎:
-- - InnoDB
-- - MyISAM
-- - Memory
-- - CSV
-- - 等等

-- 存储引擎层是私有层，每个引擎有自己的实现:
-- - InnoDB: B+树索引、MVCC、redo log、undo log
-- - MyISAM: 压缩、全文索引、表级锁
-- - Memory: 基于内存，速度快
```

**为什么要分层?**

```sql
-- 优势1: 存储引擎可插拔
-- 用户可以根据业务选择不同的存储引擎

CREATE TABLE user (
    id INT PRIMARY KEY,
    name VARCHAR(50)
) ENGINE=InnoDB;  -- 事务安全

CREATE TABLE log (
    id INT PRIMARY KEY,
    message TEXT
) ENGINE=MyISAM;  -- 查询快，不要求事务

CREATE TABLE cache (
    id INT PRIMARY KEY,
    data BLOB
) ENGINE=Memory;  -- 临时数据，基于内存

-- 优势2: 存储引擎可以独立发展
-- InnoDB: 不断优化性能、添加新特性
-- MyISAM: 保持简单、兼容性好
-- 新引擎: 可以随时引入

-- 优势3: 降低耦合
-- Server层不关心底层存储
-- 存储引擎层不关心上层逻辑
-- 各自独立开发和维护
```

**如果合并redo log和binlog会怎样？**

```sql
-- 假设: 把redo log和binlog合并成一个统一的日志

-- 问题1: 如何兼容多种存储引擎?

-- InnoDB的redo log记录的是物理修改:
-- 页号X, 偏移量Y, 修改前值A, 修改后值B

-- MyISAM不使用redo log, 如何处理?

-- Memory引擎在内存中, 如何记录物理修改?

-- 不同引擎的物理格式完全不同, 统一日志无法表示

-- 问题2: 如何实现主从复制?

-- InnoDB的redo log:
-- 记录物理修改, 只能恢复到相同版本的InnoDB

-- 主库: InnoDB 5.7
-- 从库: InnoDB 8.0

-- 如果用物理日志:
-- 页结构、数据格式可能不同
-- 从库无法应用主库的redo log

-- binlog的逻辑日志:
-- UPDATE account SET balance=900 WHERE id=1
-- 所有版本都能理解和应用

-- 问题3: 如何进行数据恢复?

-- redo log:
-- 循环写入, 只保留最近的修改
-- 无法恢复到历史某个时间点

-- binlog:
-- 追加写入, 保留所有修改
-- 可以恢复到任意时间点

-- 问题4: 破坏分层设计

-- Server层:
-- 需要知道InnoDB的物理日志格式
-- 耦合度增加, 无法支持其他存储引擎

-- 存储引擎层:
-- 需要依赖Server层的binlog逻辑
-- 独立性降低
```

**binlog的设计目标:**

```sql
-- 目标1: 所有存储引擎通用
-- InnoDB: 能用
-- MyISAM: 能用
-- Memory: 能用
-- 新引擎: 都能用

-- 目标2: 主从复制
-- 记录逻辑操作, 跨版本兼容
-- 主从可以运行不同版本的MySQL

-- 目标3: 数据恢复
-- 追加写入, 保留所有修改
-- 可以恢复到任意时间点
-- 可以做增量备份

-- 目标4: 审计和追踪
-- 记录所有SQL或行变更
-- 可以审计谁在什么时候做了什么操作
```

**redo log的设计目标:**

```sql
-- 目标1: 崩溃恢复
-- 记录物理修改, 快速恢复
-- 循环写入, 节省磁盘空间

-- 目标2: 保证持久性
-- WAL (Write-Ahead Logging)
-- 先写日志, 再写数据
-- 崩溃后通过日志恢复

-- 目标3: 提升性能
-- 内存写, 磁盘刷盘异步
-- 减少磁盘IO

-- 目标4: InnoDB私有
-- 不需要考虑其他存储引擎
-- 可以使用InnoDB特有的优化
```

**实际案例对比:**

```sql
-- 场景: 执行一个UPDATE操作
UPDATE account SET balance = balance - 100 WHERE id = 1;

-- redo log记录:
-- 页号: 100
-- 偏移量: 1024
-- 修改前值: 0x000003E8 (1000)
-- 修改后值: 0x00000384 (900)
-- LSN (Log Sequence Number): 1000000

-- 这是InnoDB的物理格式
-- 只有InnoDB能理解
-- 其他存储引擎无法应用

-- binlog记录(row格式):
-- Timestamp: 2026-04-08 21:30:15
-- Database: test
-- Table: account
-- Operation: UPDATE
-- Before: id=1, name='张三', balance=1000
-- After:  id=1, name='张三', balance=900

-- 这是逻辑格式
-- 所有存储引擎都能理解
-- 跨版本兼容
```

**总结:**

不合并redo log和binlog的原因：

1. **分层设计**：MySQL采用分层架构，Server层和存储引擎层分离
2. **用途不同**：redo log用于崩溃恢复，binlog用于主从复制和恢复
3. **格式不同**：redo log是物理日志，binlog是逻辑日志
4. **兼容性**：binlog需要兼容所有存储引擎，redo log是InnoDB私有
5. **功能独立**：各自优化，互不影响

如果强行合并，会破坏MySQL的分层架构，降低灵活性和可扩展性。

#### 提问2:如果binlog写成功了但通知InnoDB commit失败了会怎样?

**回答:**

这时候redo log还是prepare状态。重启恢复时，MySQL会拿redo log里的XID去binlog查，能查到说明binlog完整，就把事务提交掉。所以不会丢数据，也不会不一致。

**详细说明:**

**异常场景分析:**

```sql
-- 正常的二阶段提交流程
-- 阶段1: Prepare阶段
InnoDB.write_redo_log(status=PREPARE, XID=1001)

-- 阶段2: Commit阶段
Server.write_binlog(XID=1001)  # 成功
InnoDB.commit_redo_log(XID=1001)  # 修改为COMMIT

-- 异常场景: binlog写成功，但通知InnoDB commit失败
-- 阶段1: Prepare阶段
InnoDB.write_redo_log(status=PREPARE, XID=1001)

-- 阶段2: Commit阶段
Server.write_binlog(XID=1001)  # ✅ 成功
InnoDB.commit_redo_log(XID=1001)  # ❌ 失败!
# 可能原因:
# - 网络断开
# - InnoDB崩溃
# - 磁盘错误
```

**失败时状态:**

```sql
-- 宕机时的状态:
redo log:
- XID=1001
- status=PREPARE  # 注意：还是PREPARE，没有改成COMMIT
- 内容: UPDATE account SET balance=900 WHERE id=1

binlog:
- XID=1001
- 内容: UPDATE account SET balance=900 WHERE id=1
- 状态: 已落盘，持久化

-- 问题: redo log状态是PREPARE，但binlog已经写成功了
```

**崩溃恢复时的处理:**

```sql
-- MySQL重启后，会执行崩溃恢复流程

-- 步骤1: 扫描redo log
-- 找到所有状态为PREPARE的事务

-- 步骤2: 对每个PREPARE事务，检查binlog
for txn in redo_log.prepare_transactions:
    xid = txn.xid
    found = binlog.find_xid(xid)
    
    if found:
        # binlog里有这个XID，说明binlog写成功了
        # 需要提交这个事务
        txn.commit()
    else:
        # binlog里没有这个XID，说明binlog没写成功
        # 需要回滚这个事务
        txn.rollback()

-- 步骤3: 根据XID在binlog中的存在与否，决定提交或回滚

-- 针对异常场景:
redo log: XID=1001, status=PREPARE
binlog: 有XID=1001

-- 判断: binlog里找到了XID=1001
-- 操作: 提交事务
InnoDB.commit_redo_log(XID=1001)

-- 结果:
redo log状态: COMMIT
数据: balance=900
```

**为什么不会丢数据？**

```sql
-- binlog已经写成功了，数据已经持久化到磁盘
-- 从库已经同步了这个事务（如果开启了主从复制）

-- 恢复后提交redo log，主库的数据也是一致的

-- 数据流程:
1. 用户执行: UPDATE account SET balance=balance-100 WHERE id=1
2. InnoDB修改内存: balance=900
3. 写redo log: PREPARE状态
4. 写binlog: 成功落盘
5. 从库同步: balance=900
6. 通知InnoDB commit: 失败!
7. 宕机重启

8. 恢复:
   - 扫描redo log: XID=1001, PREPARE
   - 检查binlog: 找到XID=1001
   - 提交redo log: 状态改为COMMIT
   - 恢复数据: balance=900

9. 结果:
   - 主库: balance=900
   - 从库: balance=900
   - 数据一致，没有丢失
```

**为什么不会不一致？**

```sql
-- binlog和redo log都记录了相同的修改
-- 通过XID关联起来

-- binlog: XID=1001, UPDATE account SET balance=900 WHERE id=1
-- redo log: XID=1001, 物理修改, PREPARE

-- 如果binlog有，说明事务应该提交
-- 如果binlog没有，说明事务应该回滚

-- 这样就保证了两边的一致性
```

**其他可能的不一致场景:**

```sql
-- 场景1: binlog写失败，通知InnoDB commit也失败
redo log: PREPARE
binlog: 没有XID

恢复:
- 检查binlog: 没找到XID
- 操作: 回滚
结果: 数据不变，一致

-- 场景2: binlog写失败，但通知InnoDB commit成功
redo log: COMMIT
binlog: 没有XID

恢复:
- redo log已经是COMMIT
- 不会恢复这个事务
结果: 
- 主库: balance=900
- 从库: balance=1000 (没有同步)
- 问题: 主从不一致!

-- 这种情况会发生吗?
# 不会!
# 因为binlog写失败时，Server层不会通知InnoDB commit
# InnoDB不会收到commit指令
# redo log状态不会变成COMMIT

-- 流程:
Server.write_binlog(XID=1001)
  if success:
    InnoDB.commit_redo_log(XID=1001)
  else:
    # 写失败，不通知InnoDB
    # 或者通知InnoDB rollback
    InnoDB.rollback_redo_log(XID=1001)
```

**如何验证恢复结果？**

```sql
-- 验证方法1: 查看redo log状态
SHOW ENGINE INNODB STATUS;

-- 查看事务信息
Transactions
------------
Trx id counter 1002
-- ...
-- 查看LSN (Log Sequence Number)
-- 确认redo log已提交

-- 验证方法2: 查看binlog内容
SHOW BINARY LOGS;
SHOW MASTER STATUS;

mysqlbinlog mysql-bin.000001 | grep "Xid = 1001"
# Xid = 1001

-- 验证方法3: 查看数据一致性
-- 主库
SELECT * FROM account WHERE id = 1;
-- 从库
SELECT * FROM account WHERE id = 1;

-- 对比主从数据，应该一致
```

**总结:**

如果binlog写成功但通知InnoDB commit失败：

1. **redo log状态**：PREPARE（未改成COMMIT）
2. **binlog状态**：已成功落盘，有XID
3. **崩溃恢复**：用XID在binlog中查找，找到则提交
4. **结果**：数据一致，不丢失

MySQL通过XID关联redo log和binlog，在任何异常情况下都能保证数据一致性。这是二阶段提交的核心机制。

#### 提问3:组提交攒得太久会不会有问题?

**回答:**

会。攒得久了事务响应时间就长，用户感知到的就是"提交卡了一下"。而且万一这段时间宕机，这一批事务可能全丢。

所以生产环境不会把delay设太大，几百微秒到几毫秒就差不多了，主要看业务能接受多大延迟。

**详细说明:**

**组提交的延迟问题:**

```sql
-- 组提交需要等待一段时间，以合并更多事务
-- binlog_group_commit_sync_delay = 1000  # 等待1000微秒

-- 场景: 用户转账
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1;
UPDATE account SET balance = balance + 100 WHERE id = 2;
COMMIT;

-- 如果是组提交:
-- 1. 用户执行COMMIT
-- 2. 事务加入队列，等待其他事务
-- 3. 等待1000微秒（1毫秒）
-- 4. 合并提交

-- 用户感知:
-- 提交命令执行后，需要等待1毫秒才返回成功
-- 如果延迟是10毫秒，用户会感觉"卡了一下"
```

**延迟与吞吐的权衡:**

```sql
-- 场景1: 延迟设置很小（0微秒）
SET GLOBAL binlog_group_commit_sync_delay = 0;

-- 优点:
-- - 响应时间短，用户体验好
-- - 事务快速提交

-- 缺点:
-- - 合并的事务少
-- - IO次数多
-- - 吞吐量低

-- 适用场景:
-- - 秒杀系统
-- - 实时交易
-- - 对延迟敏感的业务

-- 性能数据:
-- 延迟: <1ms
-- 吞吐量: 1000 TPS

-- 场景2: 延迟设置适中（500微秒）
SET GLOBAL binlog_group_commit_sync_delay = 500;

-- 优点:
-- - 响应时间适中
-- - 合并的事务较多
-- - IO次数减少
-- - 吞吐量提升

-- 缺点:
-- - 有一定延迟（0.5ms）

-- 适用场景:
-- - 大部分业务场景
-- - 日志写入
-- - 批量处理

-- 性能数据:
-- 延迟: 0.5ms
-- 吞吐量: 5000 TPS

-- 场景3: 延迟设置很大（10毫秒）
SET GLOBAL binlog_group_commit_sync_delay = 10000;

-- 优点:
-- - 合并的事务很多
-- - IO次数大幅减少
-- - 吞吐量最高

-- 缺点:
-- - 响应时间长（10ms）
-- - 用户感知明显

-- 适用场景:
-- - 批量导入
-- - 数据归档
-- - 离线处理

-- 性能数据:
-- 延迟: 10ms
-- 吞吐量: 10000 TPS
```

**数据丢失风险:**

```sql
-- 场景: 组提交期间宕机

-- T0: 事务A提交，加入队列
-- T1: 事务B提交，加入队列
-- T2: 事务C提交，加入队列
-- T3: 等待1毫秒（binlog_group_commit_sync_delay=1000）
-- T4: 准备合并提交
-- T5: ❌ 宕机!

-- 问题:
-- - 事务A、B、C都提交了，但还没落盘
-- - redo log和binlog都还在内存中
-- - 宕机后，这些数据都丢失了

-- 用户体验:
-- 事务A、B、C都返回了"提交成功"
-- 但实际上数据丢失了
-- 严重的业务问题！

-- 如何避免?
-- 方法1: 减少延迟
SET GLOBAL binlog_group_commit_sync_delay = 100;
-- 减少等待时间，减少丢失风险

-- 方法2: 设置事务数量限制
SET GLOBAL binlog_group_commit_sync_no_delay_count = 5;
-- 累积5个事务后立即提交，不等待

-- 方法3: 使用更可靠的磁盘
-- 使用SSD而不是HDD
-- 减少宕机概率

-- 方法4: 做好备份
-- 定期备份
-- 宕机后可以恢复
```

**不同业务场景的建议配置:**

```sql
-- 1. 金融交易系统
-- 要求: 低延迟，数据安全
SET GLOBAL binlog_group_commit_sync_delay = 0;
SET GLOBAL binlog_group_commit_sync_no_delay_count = 0;

-- 原因:
-- - 用户转账需要快速响应
-- - 数据丢失风险高
-- - 不能接受任何延迟

-- 2. 电商订单系统
-- 要求: 中等延迟，高吞吐

SET GLOBAL binlog_group_commit_sync_no_delay_count = 5;

-- 原因:
-- - 订单提交可以接受0.1ms延迟
-- - 高并发需要高吞吐
-- - 数据丢失风险可接受

-- 3. 日志系统
-- 要求: 高延迟，最高吞吐
SET GLOBAL binlog_group_commit_sync_delay = 1000;
SET GLOBAL binlog_group_commit_sync_no_delay_count = 10;

-- 原因:
-- - 日志写入不实时
-- - 可以接受较大延迟
-- - 需要高吞吐处理大量日志

-- 4. 批量导入
-- 要求: 最高吞吐，延迟不限
SET GLOBAL binlog_group_commit_sync_delay = 5000;
SET GLOBAL binlog_group_commit_sync_no_delay_count = 20;

-- 原因:
-- - 批量导入不需要实时响应
-- - 最大化吞吐量
-- - 延迟不是问题
```

**监控组提交效果:**

```sql
-- 方法1: 监控提交延迟
-- 使用SHOW STATUS查看

SHOW STATUS LIKE 'Innodb_row_lock_time';

-- 计算TPS:

-- 方法3: 监控IO
-- 查看磁盘IO次数

-- Linux命令:
iostat -x 1
# 查看磁盘IO

-- 方法4: 监控组提交队列长度
-- 查看有多少事务在等待组提交

SHOW STATUS LIKE 'binlog_group_commit%';
```

**调优建议:**

```sql
-- 步骤1: 确定业务需求
-- - 能接受多大延迟?
-- - 需要多大的吞吐?
-- - 数据丢失风险有多大?

-- 步骤2: 初始配置
-- 根据业务类型选择合适的配置
-- 金融: delay=0
-- 电商: delay=100
-- 日志: delay=1000

-- 步骤3: 压力测试
-- 模拟实际负载
-- 测试延迟和吞吐

-- 步骤4: 监控调整
-- 根据监控数据调整
-- 找到平衡点

-- 步骤5: 定期 review
-- 业务变化，配置也要调整
```

**实际案例:**

```sql
-- 案例: 电商订单系统

-- 初始配置:
SET GLOBAL binlog_group_commit_sync_delay = 0;

-- 问题:
-- 吞吐量不足，高峰期响应慢

-- 调整:
SET GLOBAL binlog_group_commit_sync_delay = 100;
SET GLOBAL binlog_group_commit_sync_no_delay_count = 5;

-- 延迟: 0.1ms (用户无感知)
-- 吞吐量: 提升5倍
-- 问题解决

-- 继续优化:
SET GLOBAL binlog_group_commit_sync_delay = 200;

-- 结果:
-- 延迟: 0.2ms (仍可接受)
-- 吞吐量: 再提升2倍
-- 性能最优
```

**总结:**

组提交的延迟设置需要权衡：

1. **延迟太小**：合并的事务少，IO次数多，吞吐量低
2. **延迟太大**：用户感知明显，数据丢失风险高
3. **推荐配置**：几百微秒到几毫秒，根据业务调整

生产环境建议：

- **金融交易**：delay=0（零延迟）
- **电商订单**：delay=100（0.1ms）
- **日志系统**：delay=1000（1ms）
- **批量处理**：delay=5000（5ms）

关键是找到延迟和吞吐的平衡点，满足业务需求。

#### 提问4:主从复制时从库也有二阶段提交吗?

**回答:**

从库执行relay log里的事务时也会写redo log和binlog，所以也走二阶段提交。不过从库的binlog是可选的，如果不需要级联复制可以关掉，关掉后就只写redo log，不需要二阶段了。

**详细说明:**

**主从复制的基本流程:**

```sql
-- 主库
-- 1. 事务提交
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1;
COMMIT;

-- 2. 写binlog
binlog: UPDATE account SET balance=900 WHERE id=1

-- 3. 从库读取binlog
-- 从库IO线程读取主库的binlog

-- 4. 从库写入relay log
relay log: 和binlog一样的内容

-- 从库
-- 1. SQL线程读取relay log
-- 2. 执行relay log里的事务
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1;
COMMIT;

-- 3. 写redo log和binlog
redo log: 记录物理修改
binlog: 记录逻辑操作（如果开启）
```

**从库的二阶段提交:**

```sql
-- 从库执行relay log时，也会走二阶段提交

-- 场景: 从库执行relay log中的事务

-- 步骤1: 读取relay log
SQL线程: 读取relay log
内容: UPDATE account SET balance=900 WHERE id=1

-- 步骤2: 执行事务
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1;

-- 步骤3: 二阶段提交
-- 阶段1: Prepare阶段
InnoDB.write_redo_log(status=PREPARE, XID=2001)

-- 阶段2: Commit阶段
-- 如果binlog开启:
Server.write_binlog(XID=2001)  # 写从库自己的binlog
InnoDB.commit_redo_log(XID=2001)

-- 如果binlog关闭:
# 不写binlog
InnoDB.commit_redo_log(XID=2001)  # 直接提交

-- 步骤4: 标记relay log已处理
relay log: 标记已执行
```

**为什么从库也要写redo log？**

```sql
-- 原因1: 保证数据持久性
-- 从库也需要崩溃恢复能力
-- 如果从库宕机，重启后通过redo log恢复

-- 原因2: 保持主从一致
-- 主库有redo log，从库也应该有
-- 主从的恢复机制一致

-- 原因3: 性能优化
-- redo log是WAL机制
-- 先写日志，再写数据
-- 提升写入性能

-- 示例: 从库崩溃恢复
-- 从库执行relay log时:
- 写redo log
- 应用事务
- 标记relay log

-- 如果在标记relay log之前宕机:
- redo log: PREPARE/COMMIT状态
- relay log: 未标记

-- 重启后:
- 扫描redo log
- 恢复未标记的relay log事务
- 保证数据一致性
```

**为什么从库要写binlog?**

```sql
-- 原因1: 级联复制
-- 主库 → 从库A → 从库B
-- 从库A需要binlog给从库B使用

-- 原因2: 备份恢复
-- 从库也可以做备份
- binlog恢复到任意时间点
- 数据归档

-- 原因3: 数据审计
-- 记录所有数据变更
-- 审计追踪

-- 示例: 级联复制
-- 架构:
-- 主库 (Master)
--   ↓ binlog
-- 从库A (Slave 1)
--   ↓ binlog (从库A的binlog)
-- 从库B (Slave 2)

-- 如果从库A不写binlog:
- 从库B无法从从库A同步
- 只能从主库同步
- 增加主库压力
```

**配置从库的binlog:**

```sql
-- 方法1: 开启从库binlog（默认）
-- my.cnf配置
[mysqld]
log_bin = mysql-bin
binlog_format = ROW
-- 从库会写binlog

-- 优点:
-- - 支持级联复制
-- - 可以做备份恢复
-- - 数据审计

-- 缺点:
-- - 占用磁盘空间
-- - 写binlog有性能开销
-- - 需要管理binlog文件

-- 方法2: 关闭从库binlog
-- my.cnf配置
[mysqld]
# log_bin = mysql-bin  # 注释掉或删除
-- 从库不写binlog

-- 优点:
-- - 节省磁盘空间
-- - 提升性能（不写binlog）
-- - 简化管理

-- 缺点:
-- - 不支持级联复制
-- - 无法从从库做备份恢复
-- - 无法审计

-- 方法3: 临时关闭
-- 动态配置
SET SESSION sql_log_bin = 0;
-- 当前会话不写binlog

-- 注意:
-- 只影响当前会话
-- 其他会话仍然写binlog
```

**不写binlog时的提交流程:**

```sql
-- 从库关闭binlog后

-- 步骤1: 读取relay log
SQL线程: 读取relay log

-- 步骤2: 执行事务
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1;

-- 步骤3: 提交（不需要二阶段）
# 不写binlog，直接提交redo log
InnoDB.write_redo_log(status=COMMIT)

-- 步骤4: 标记relay log已处理
relay log: 标记已执行

-- 优势:
-- - 流程简化
-- - 性能提升（不写binlog）
-- - 延迟降低

-- 注意:
-- - 没有binlog，无法崩溃恢复到特定时间点
-- - 只能依赖redo log的崩溃恢复
```

**主从复制的对比:**

| 特性 | 从库写binlog | 从库不写binlog |
|------|------------|--------------|
| 二阶段提交 | 是 | 否 |
| 级联复制 | 支持 | 不支持 |
| 备份恢复 | 支持 | 有限支持 |
| 数据审计 | 支持 | 不支持 |
| 性能 | 较低 | 较高 |
| 磁盘占用 | 较大 | 较小 |
| 管理复杂度 | 较高 | 较低 |

**实际案例:**

```sql
-- 案例1: 标准主从复制
-- 主库
-- - 写redo log
-- - 写binlog
-- - 二阶段提交

-- 从库A
-- - 读relay log
-- - 执行事务
-- - 写redo log
-- - 写binlog
-- - 二阶段提交

-- 从库B
-- - 读relay log（从从库A读取）
-- - 执行事务
-- - 写redo log
-- - 写binlog
-- - 二阶段提交

-- 优点: 支持级联复制
-- 缺点: 性能较低

-- 案例2: 单层主从复制
-- 主库
-- - 写redo log
-- - 写binlog
-- - 二阶段提交

-- 从库
-- - 读relay log
-- - 执行事务
-- - 写redo log
-- - 不写binlog
-- - 只写redo log，不需要二阶段

-- 优点: 性能高，简单
-- 缺点: 不支持级联复制
```

**如何选择配置:**

```sql
-- 场景1: 需要级联复制
-- 架构: 主 → 从A → 从B → 从C
-- 配置: 所有从库都开启binlog

-- 原因:
-- - 从A需要给从B提供binlog
-- - 从B需要给从C提供binlog
-- - 必须开启binlog

-- 场景2: 只有一主一从
-- 架构: 主 → 从
-- 配置: 从库关闭binlog

-- 原因:
-- - 不需要级联复制
-- - 提升性能
-- - 简化管理

-- 场景3: 从库做备份
-- 架构: 主 → 从（备份节点）
-- 配置: 从库开启binlog

-- 原因:
-- - 从库需要做备份恢复
-- - binlog是必须的
-- - 可以恢复到任意时间点

-- 场景4: 从库只读，不备份
-- 架构: 主 → 从（只读查询）
-- 配置: 从库关闭binlog

-- 原因:
-- - 从库只做查询，不备份
-- - 提升性能
```

**监控从库的binlog:**

```sql
-- 查看从库binlog状态
SHOW VARIABLES LIKE 'log_bin';
-- 结果: ON (开启) 或 OFF (关闭)

-- 查看从库binlog文件
SHOW BINARY LOGS;

-- 查看从库relay log
SHOW SLAVE STATUS\G
-- 查看Relay_Log_File, Relay_Log_Pos等

-- 查看从库执行状态
SHOW PROCESSLIST;
-- 查看SQL线程状态
```

**总结:**

从库的二阶段提交：

1. **如果开启binlog**：
   - 执行事务 → 写redo log → 写binlog → 二阶段提交
   - 支持级联复制、备份恢复、数据审计
   - 性能略低，磁盘占用大

2. **如果关闭binlog**：
   - 执行事务 → 写redo log → 直接提交
   - 不需要二阶段提交
   - 性能高，简单，但不支持级联复制

建议：

- **需要级联复制**：从库开启binlog
- **一主一从**：从库关闭binlog
- **从库做备份**：从库开启binlog
- **从库只读**：从库关闭binlog

## 3. MySQL 中如果发生死锁应该如何解决？

### 📌 回答重点

死锁发生后解决方式分两种:自动处理和手动干预。

MySQL InnoDB 自带死锁检测机制，由`innodb_deadlock_detect`参数控制，默认开启。一旦检测到死锁，存储引擎会自动选择一个代价最小的事务回滚掉，释放它持有的锁让另一个事务继续执行。另外还有个兜底机制是锁等待超时`innodb_lock_wait_timeout`，默认50秒，等锁超过这个时间就自动放弃并回滚。手动干预主要用在自动机制不够快或者需要立刻恢复的场景。先用`SHOW ENGINE INNODB STATUS`或者查`INFORMATION_SCHEMA`里的锁相关表找到阻塞的线程ID，然后`KILL <thread_id>`杀掉它。

```sql
-- 查看表级锁等待

-- 示例查询: 查找阻塞的事务
SELECT
    r.trx_id AS waiting_trx_id,
    r.trx_mysql_thread_id AS waiting_thread,
    r.trx_query AS waiting_query,
    b.trx_id AS blocking_trx_id,
    b.trx_mysql_thread_id AS blocking_thread,
    b.trx_query AS blocking_query
FROM information_schema.innodb_lock_waits w
INNER JOIN information_schema.innodb_trx b ON b.trx_id = w.blocking_trx_id
INNER JOIN information_schema.innodb_trx r ON r.trx_id = w.requesting_trx_id;
```

#### 常见避免死锁的手段

**1. 保持事务简短，减少锁持有时间**

```sql
-- ❌ 错误: 事务时间长，锁持有太久
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- ... 执行5分钟的业务逻辑 ...
UPDATE account SET balance = 900 WHERE id = 1;
COMMIT;
-- 问题: 锁持有5分钟，容易死锁

-- ✅ 正确: 事务简短，快速提交
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
UPDATE account SET balance = 900 WHERE id = 1;
COMMIT;
-- 锁持有时间短，死锁概率低
```

**2. 按固定顺序访问表和记录**

```sql
-- ❌ 错误: 访问顺序不一致
-- 事务A: 先锁id=1，再锁id=2
START TRANSACTION;
UPDATE account SET balance = 900 WHERE id = 1;
UPDATE account SET balance = 2100 WHERE id = 2;

-- 事务B: 先锁id=2，再锁id=1
START TRANSACTION;
UPDATE account SET balance = 2100 WHERE id = 2;
UPDATE account SET balance = 900 WHERE id = 1;
-- 容易死锁: A等id=2，B等id=1

-- ✅ 正确: 统一访问顺序
-- 事务A: 按id从小到大
START TRANSACTION;
UPDATE account SET balance = 900 WHERE id = 1;
UPDATE account SET balance = 2100 WHERE id = 2;

-- 事务B: 也按id从小到大
START TRANSACTION;
UPDATE account SET balance = 2100 WHERE id = 2;
UPDATE account SET balance = 900 WHERE id = 1;
-- 避免死锁
```

**3. 为表添加合适的索引**

```sql
-- ❌ 错误: 没有索引，全表扫描
CREATE TABLE no_index (
    id INT,
    name VARCHAR(50),
    balance INT
);

START TRANSACTION;
SELECT * FROM no_index WHERE id = 5 FOR UPDATE;
-- 全表扫描，所有行都被锁住
-- 相当于表锁，并发度低

-- ✅ 正确: 添加索引
CREATE TABLE with_index (
    id INT,
    name VARCHAR(50),
    balance INT,
    INDEX idx_id (id)
);

START TRANSACTION;
SELECT * FROM with_index WHERE id = 5 FOR UPDATE;
-- 使用索引，只锁定id=5一行
-- 并发度高，死锁概率低
```

**4. 使用较低的隔离级别**

```sql
-- 可重复读级别 (默认，死锁概率高)
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- 使用Next-Key Lock，包含Gap Lock
-- 锁定范围大，容易死锁

-- 读已提交级别 (死锁概率低)
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- 只使用Record Lock，不使用Gap Lock
-- 锁定范围小，死锁概率低
```

**5. 避免在事务中执行用户交互**

```sql
-- ❌ 错误: 事务中等待用户输入
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- ... 等待用户确认 ...
-- 如果用户长时间不确认
-- 锁一直持有，容易死锁
UPDATE account SET balance = 900 WHERE id = 1;
COMMIT;

-- ✅ 正确: 业务逻辑和事务分离
-- 步骤1: 查询数据，展示给用户
SELECT * FROM account WHERE id = 1;
-- 用户查看和确认

-- 步骤2: 用户确认后，快速提交事务
START TRANSACTION;
UPDATE account SET balance = 900 WHERE id = 1;
COMMIT;
-- 锁持有时间短
```

**6. 使用乐观锁替代悲观锁**

```sql
-- ❌ 悲观锁: SELECT ... FOR UPDATE
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- 持有锁，其他事务等待
UPDATE account SET balance = balance - 100 WHERE id = 1;
COMMIT;

-- ✅ 乐观锁: 使用version字段
CREATE TABLE account (
    id INT PRIMARY KEY,
    balance INT,
    version INT DEFAULT 0  -- 版本号
);

START TRANSACTION;
SELECT * FROM account WHERE id = 1;
-- 获取balance=1000, version=0

-- 业务逻辑: 计算新余额
-- ...

UPDATE account
SET balance = 900, version = version + 1
WHERE id = 1 AND version = 0;
-- 只更新version没变的记录

-- 检查影响行数
-- 如果affected_rows=0，说明版本冲突
-- 可以重试或提示用户
COMMIT;
```

**7. 批量操作分批处理**

```sql
-- ❌ 错误: 一次性更新大量数据
START TRANSACTION;
UPDATE account SET balance = balance + 100 WHERE balance > 0;
-- 锁定所有符合条件的行
-- 锁持有时间长，容易死锁
COMMIT;

-- ✅ 正确: 分批处理
-- 每次处理1000条
SET @batch_size = 1000;
SET @processed = 0;

REPEAT
    START TRANSACTION;
    UPDATE account
    SET balance = balance + 100
    WHERE id > @processed
    ORDER BY id
    LIMIT @batch_size;
    COMMIT;
    SET @processed = @processed + ROW_COUNT();
UNTIL ROW_COUNT() < @batch_size END REPEAT;
-- 每次只锁少量数据
```

**8. 使用SELECT ... LOCK IN SHARE MODE代替FOR UPDATE**

```sql
-- 如果只需要读取数据，不需要修改
-- 使用共享锁而不是排他锁

-- ❌ 排他锁: 阻塞其他事务
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- 其他事务无法读取或修改
COMMIT;

-- ✅ 共享锁: 允许其他事务读取
START TRANSACTION;
SELECT * FROM account WHERE id = 1 LOCK IN SHARE MODE;
-- 其他事务可以读取，但不能修改
-- 减少阻塞，降低死锁概率
COMMIT;
```

**9. 设置合理的锁等待超时**

```sql
-- 查看当前锁等待超时时间
SHOW VARIABLES LIKE 'innodb_lock_wait_timeout';
-- 默认: 50秒

-- 调短超时时间
SET GLOBAL innodb_lock_wait_timeout = 10;
-- 超过10秒自动回滚，避免长时间等待

-- 注意:
-- - 超时太短可能导致业务失败
-- - 超时太长可能导致系统响应慢
-- - 建议: 10-30秒
```

**10. 应用层实现重试机制**

```sql
-- 应用层代码示例(伪代码)
function update_with_retry(sql, max_retries=3):
    retries = 0
    while retries < max_retries:
        try:
            START TRANSACTION
            execute(sql)
            COMMIT
            return True
        except DeadlockError:
            retries += 1
            if retries < max_retries:
                sleep(retries * 0.1)  -- 指数退避
            else:
                ROLLBACK
                return False
        except other_error:
            ROLLBACK
            return False

-- 使用
success = update_with_retry("UPDATE account SET balance = 900 WHERE id = 1")
if success:
    print("更新成功")
else:
    print("更新失败，请稍后重试")
```

### 💬 面试官追问

#### 提问1: innodb_deadlock_detect关掉会怎么样?什么场景下会考虑关掉?

**回答:**

关掉之后MySQL不再主动检测死锁，只能靠锁等待超时来解除。死锁检测本身有开销，高并发场景下大量事务同时检测会消耗CPU。比如秒杀系统几万个请求同时抢一行数据，死锁检测的开销可能比业务逻辑还大。这时候可以考虑关掉检测，把`innodb_lock_wait_timeout`调短到1-2秒，靠快速超时来处理，应用层做重试。

**详细说明:**

**死锁检测的开销:**

```sql
-- 死锁检测需要:
-- 1. 维护锁等待图
--    - 记录每个事务持有的锁
--    - 记录每个事务等待的锁
--    - 构建有向图

-- 2. 检测循环等待
--    - 深度优先搜索
--    - 时间复杂度: O(n^2)
--    - n是事务数量

-- 3. 选择回滚事务
--    - 比较事务权重
--    - 选择代价最小的回滚

-- 示例: 高并发场景
-- 10000个事务同时更新同一行
-- 每个事务都要检测死锁
-- 开销: O(10000^2) = 100,000,000次比较
```

**关闭死锁检测的影响:**

```sql
-- 配置
SET GLOBAL innodb_deadlock_detect = OFF;

-- 影响:
-- 1. 不再主动检测死锁
-- 2. 依赖锁等待超时
-- 3. 死锁会一直等待，直到超时

-- 优点:
-- - 减少CPU开销
-- - 提升高并发场景性能

-- 缺点:
-- - 死锁可能长时间等待
-- - 需要依赖超时机制
```

**适用的场景:**

```sql
-- 场景1: 秒杀系统

-- 特点:
-- - 几万请求同时抢购
-- - 每个请求都要更新库存
-- - 高并发，高竞争

-- 解决方案:
SET GLOBAL innodb_deadlock_detect = OFF;
SET GLOBAL innodb_lock_wait_timeout = 1;  -- 1秒超时

-- 应用层重试:
function seckill_with_retry(user_id, product_id, max_retries=5):
    retries = 0
    while retries < max_retries:
        try:
            START TRANSACTION
            UPDATE inventory 
            SET stock = stock - 1 
            WHERE product_id = ? AND stock > 0
            IF affected_rows == 0:
                ROLLBACK
                return False  -- 抢购失败
            INSERT INTO orders (user_id, product_id) VALUES (?, ?)
            COMMIT
            return True  -- 抢购成功
        except DeadlockError or TimeoutError:
            retries += 1
            sleep(retries * 0.01)  -- 指数退避
    return False

-- 效果:
-- - 减少死锁检测CPU开销
-- - 1秒超时快速失败
-- - 应用层重试，用户无感知
```

#### 提问2: 为什么MySQL选择回滚代价小的事务而不是代价大的?

**回答:**

回滚代价小的事务意味着undo log少、已修改的数据少，回滚本身更快，对系统冲击更小。回滚大事务可能要花几秒甚至更久，这段时间锁还是拿着，其他事务还是被阻塞。选小的先回滚，让系统尽快恢复正常。

**详细说明:**

**事务回滚的代价:**

```sql
-- 事务回滚需要:
-- 1. 读取undo log
--    - 撤销已执行的修改
--    - 恢复到事务开始前的状态

-- 2. 释放持有的锁
--    - 释放所有行锁
--    - 释放所有间隙锁
--    - 释放所有意向锁

-- 3. 更新事务状态
--    - 标记事务为回滚
--    - 更新事务表

-- 回滚时间与以下因素相关:
-- - undo log的大小
-- - 修改的数据量
-- - 持有的锁的数量
-- - 事务执行时间
```

**事务权重计算:**

```sql
-- MySQL通过TRX_WEIGHT来衡量事务的代价
-- TRX_WEIGHT = undo log行数 + 修改的行数

-- 查看事务权重
SHOW ENGINE INNODB STATUS;

-- 示例输出:
-- *** (1) TRANSACTION:
-- TRANSACTION 1234, ACTIVE 5 sec starting index read
-- TRX WEIGHT: 153
-- ^^^^^^^^^^
-- 事务权重: 153

-- *** (2) TRANSACTION:
-- TRANSACTION 1235, ACTIVE 3 sec starting index read
-- TRX WEIGHT: 274
-- ^^^^^^^^^^
-- 事务权重: 274

-- 选择权重小的事务回滚:
-- 事务1: TRX_WEIGHT = 153
-- 事务2: TRX_WEIGHT = 274
-- 回滚事务1 (代价小)
```

**回滚代价对比:**

```sql
-- 场景1: 小事务 vs 大事务

-- 小事务
START TRANSACTION;
UPDATE account SET balance = 900 WHERE id = 1;  -- 只修改1行
-- undo log: 1条
-- 回滚时间: ~1ms
-- 持有锁: 1个行锁

-- 大事务
START TRANSACTION;
UPDATE account SET balance = balance - 10 WHERE balance > 0;  -- 修改10000行
-- undo log: 10000条
-- 回滚时间: ~1000ms (1秒)
-- 持有锁: 10000个行锁

-- 死锁时，选择回滚小事务
-- 代价: 1ms
-- 释放1个行锁
-- 大事务继续执行
-- 系统快速恢复正常

-- 如果回滚大事务:
-- 代价: 1000ms
-- 释放10000个行锁
-- 但这1000ms内，其他事务还被阻塞
-- 系统恢复慢
```

```sql
-- 场景2: 复杂事务 vs 简单事务

-- 简单事务
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
UPDATE account SET balance = 900 WHERE id = 1;
-- undo log: 1条
-- 修改: 1行
-- 持有锁: 1个行锁

-- 复杂事务
START TRANSACTION;

UPDATE orders SET status = 'completed' WHERE user_id = 1;

-- undo log: 3条
-- 修改: 3行
-- 持有锁: 3个行锁

-- 死锁时，选择回滚简单事务
-- 回滚: 1条undo log
-- 释放: 1个行锁
-- 系统快速恢复
```

**回滚对系统的影响:**

```sql
-- 回滚小事务的影响:
-- 回滚时间: 1ms
-- 持有锁: 1个
-- 阻塞其他事务: 1ms
-- 系统影响: 极小

-- 回滚大事务的影响:
-- 回滚时间: 1000ms
-- 持有锁: 10000个
-- 阻塞其他事务: 1000ms
-- 系统影响: 较大

-- 如果有100个事务等待:
-- 小事务回滚: 100个事务只等1ms
-- 大事务回滚: 100个事务要等1000ms
```

**死锁回滚的决策逻辑:**

```sql
-- 伪代码
function choose_rollback_transaction(txn1, txn2):
    weight1 = calculate_weight(txn1)  # 事务1的权重
    weight2 = calculate_weight(txn2)  # 事务2的权重
    
    if weight1 < weight2:
        return txn1  # 回滚权重小的事务
    else:
        return txn2

-- 计算权重的因素:
function calculate_weight(txn):
    weight = 0
    weight += txn.undo_log_count     # undo log数量
    weight += txn.modified_rows      # 修改的行数
    weight += txn.lock_count         # 持有的锁数量
    weight += txn.execution_time / 1000  # 执行时间(毫秒)
    return weight
```

```sql
-- 案例: 小事务 vs 大事务

-- 小事务: 查询余额
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- undo log: 0
-- 修改: 0
-- TRX_WEIGHT: 0

-- 大事务: 批量更新
START TRANSACTION;
UPDATE account SET balance = balance + 1 WHERE balance > 0;
-- undo log: 10000
-- 修改: 10000
-- TRX_WEIGHT: 10000

-- 死锁时:
-- 事务A: TRX_WEIGHT = 0
-- 事务B: TRX_WEIGHT = 10000
-- 回滚事务A (小事务)
-- 快速释放锁，系统恢复
```

**影响回滚决策的其他因素:**

```sql
-- 除了权重，MySQL还考虑以下因素:

-- 1. 事务开始时间
--    - 早开始的事务可能更"重要"
--    - 但权重仍然是主要因素

-- 2. 事务是否为系统事务
--    - 系统事务(如复制线程)的优先级更高

-- 3. 隔离级别
--    - 不同隔离级别可能有不同的选择策略

-- 但权重始终是主要因素
```

**优化建议:**

```sql
-- 1. 保持事务简短
--    减少修改的数据量
--    减少undo log
--    减少锁持有时间

-- 2. 避免大事务
--    大事务更容易被回滚
--    大事务回滚代价大

-- 3. 批量操作分批
--    减少单次修改的数据量
--    降低被回滚的概率

-- 4. 监控死锁
--    定期查看死锁日志
--    分析被回滚的事务特征
```

**总结:**

MySQL选择回滚代价小的事务，目的是：

1. **快速恢复**：小事务回滚快，系统快速恢复正常
2. **减少阻塞**：释放锁快，其他事务等待时间短
3. **最小影响**：对系统整体影响最小

这是MySQL优化的一个基本原则：以最小代价恢复系统。

#### 提问3: 间隙锁和临键锁跟死锁有什么关系?

**回答:**

间隙锁锁的是索引记录之间的空隙，临键锁是行锁加间隙锁的组合。它们主要出现在可重复读隔离级别下，用来防止幻读。但锁的范围一大，两个事务锁住相邻间隙然后互相等待插入的场景就容易发生死锁。典型的比如两个事务都先`SELECT ... FOR UPDATE`锁住一个范围，然后各自往对方锁住的范围里`INSERT`，直接死锁。

**详细说明:**

**间隙锁导致死锁的原理:**

```sql
-- 间隙锁之间是兼容的
-- 但如果尝试插入被间隙锁锁住的记录
-- 就会发生冲突

-- 示例:
-- 表中数据: id = 1, 5, 10
-- 间隙: (-∞,1), (1,5), (5,10), (10,+∞)

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- Next-Key Lock: (1,5] + (5,10)
-- 持有Gap Lock: (1,5) 和 (5,10)

-- 事务B
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- Next-Key Lock: (1,5] + (5,10)
-- 持有Gap Lock: (1,5) 和 (5,10)
-- Gap Lock之间兼容，成功

-- 事务A: 尝试插入
INSERT INTO account VALUES (3, '张三', 100);
-- 想插入到(1,5)间隙中
-- 但(1,5)被事务B的Gap Lock锁住
-- 被阻塞

-- 事务B: 尝试插入
INSERT INTO account VALUES (7, '李四', 200);
-- 想插入到(5,10)间隙中
-- 但(5,10)被事务A的Gap Lock锁住
-- 被阻塞

-- 死锁!
-- A等B释放(5,10)
-- B等A释放(1,5)
-- 互相等待
```

**临键锁导致死锁的原理:**

```sql
-- 临键锁 = Record Lock + Gap Lock

-- 示例:
-- 表中数据: id = 1, 5, 10

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- Next-Key Lock: (1,5]
-- 包含:
--   - Gap Lock: (1,5)
--   - Record Lock: id=5

-- 事务B
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- Next-Key Lock: (1,5]
-- 包含:
--   - Gap Lock: (1,5)
--   - Record Lock: id=5

-- 事务A: 尝试插入
INSERT INTO account VALUES (3, '张三', 100);
-- 想插入到(1,5)间隙中
-- 被事务B的Gap Lock阻塞

-- 事务B: 尝试插入
INSERT INTO account VALUES (3, '李四', 200);
-- 想插入到(1,5)间隙中
-- 被事务A的Gap Lock阻塞

-- 死锁!
```

**典型死锁场景:**

```sql
-- 场景1: 两个事务锁住相邻间隙

-- 表中数据: id = 1, 5, 10

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- 锁住: (1,5) 和 (5,10)

-- 事务B
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- 锁住: (1,5) 和 (5,10)

-- 事务A: 插入到(5,10)
INSERT INTO account VALUES (7, '张三', 100);
-- 被事务B阻塞

-- 事务B: 插入到(1,5)
INSERT INTO account VALUES (3, '李四', 200);
-- 被事务A阻塞

-- 死锁
```

```sql
-- 场景2: 范围查询导致间隙锁

-- 表中数据: id = 1, 5, 10, 15, 20

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id BETWEEN 5 AND 15 FOR UPDATE;
-- 锁住: (1,5], (5,10], (10,15], (15,20]

-- 事务B
START TRANSACTION;
SELECT * FROM account WHERE id BETWEEN 5 AND 15 FOR UPDATE;
-- 锁住: (1,5], (5,10], (10,15], (15,20]

-- 事务A: 插入
INSERT INTO account VALUES (12, '张三', 100);
-- 插入到(10,15)间隙
-- 被事务B阻塞

-- 事务B: 插入
INSERT INTO account VALUES (8, '李四', 200);
-- 插入到(5,10)间隙
-- 被事务A阻塞

-- 死锁
```

```sql
-- 场景3: 等值查询不存在的记录

-- 表中数据: id = 1, 10

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- 记录不存在
-- Gap Lock: (1,10)

-- 事务B
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- 记录不存在
-- Gap Lock: (1,10)

-- 事务A: 插入
INSERT INTO account VALUES (5, '张三', 100);
-- 插入到(1,10)间隙
-- 被事务B阻塞

-- 事务B: 插入
INSERT INTO account VALUES (5, '李四', 200);
-- 插入到(1,10)间隙
-- 被事务A阻塞

-- 死锁
```

**为什么读已提交级别不容易死锁:**

```sql
-- 读已提交级别不使用Gap Lock

SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- Record Lock: id=5
-- 不包含Gap Lock

-- 事务B
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- Record Lock: id=5

-- 事务A: 插入
INSERT INTO account VALUES (3, '张三', 100);
-- 成功! 不被阻塞

-- 事务B: 插入
INSERT INTO account VALUES (7, '李四', 200);
-- 成功! 不被阻塞

-- 不会死锁
```

**死锁日志分析:**

```sql
-- 等待: 间隙锁 + 插入意图锁

```

**避免间隙锁死锁的方法:**

```sql
-- 方法1: 使用读已提交隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
-- 不使用Gap Lock，减少死锁概率

-- 方法2: 减少锁持有时间
START TRANSACTION;
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- 立即执行插入，不等待
INSERT INTO account VALUES (3, '张三', 100);
COMMIT;

-- 方法3: 按固定顺序插入
-- 所有事务都按从小到大顺序插入
-- 避免互相等待

-- 方法4: 避免范围查询
-- 使用精确查询
SELECT * FROM account WHERE id = 5 FOR UPDATE;
-- 而不是
SELECT * FROM account WHERE id BETWEEN 5 AND 10 FOR UPDATE;

-- 方法5: 添加唯一索引
CREATE UNIQUE INDEX idx_id ON account(id);
-- 插入时只检查唯一性，不需要间隙锁
```

**总结:**

间隙锁和临键锁是可重复读级别防止幻读的重要机制，但也是死锁的主要原因：

1. **Gap Lock兼容但插入冲突**：Gap Lock之间兼容，但插入时冲突
2. **锁定范围大**：Next-Key Lock锁定范围大，容易互相等待
3. **常见场景**：两个事务锁住相邻间隙，然后互相插入
4. **读已提交级别**：不使用Gap Lock，死锁概率低
5. **避免方法**：使用RC级别、减少锁持有时间、按固定顺序操作

#### 提问4: 有没有遇到过比较诡异的死锁场景?

**回答:**

最常见的诡异场景是单条SQL自己跟自己死锁。比如一条`UPDATE`语句要修改多行，修改过程中涉及多个二级索引,锁的获取顺序不一致就可能自己死锁。还有批量插入时如果有唯一索引，插入顺序不同的两个事务各自持有一部分行的锁然后等对方释放。解决办法就是保证操作顺序一致，或者拆成更小的批次。

**详细说明:**

**诡异场景1: 单条SQL自己死锁**

```sql
-- 场景: 一条UPDATE语句自己跟自己死锁

-- 表结构
CREATE TABLE account (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    balance INT,
    INDEX idx_name (name),  -- 二级索引
    INDEX idx_balance (balance)  -- 二级索引
);

INSERT INTO account VALUES
    (1, '张三', 1000),
    (2, '李四', 2000),
    (3, '王五', 3000);

-- 诡异死锁
START TRANSACTION;
UPDATE account SET balance = balance + 100 WHERE name IN ('张三', '李四');
-- ❌ 死锁!

-- 为什么会自己死锁?

-- 原因: 多个二级索引锁的获取顺序不一致

-- UPDATE的执行过程:
-- 步骤1: 扫描idx_name索引
--   找到name='张三' → id=1
--   找到name='李四' → id=2
--   顺序: id=1, id=2

-- 步骤2: 加锁
--   对idx_name索引: 加锁name='张三', name='李四'
--   对idx_balance索引: 加锁id=1, id=2
--   对主键索引: 加锁id=1, id=2

-- 步骤3: 如果idx_name和idx_balance的扫描顺序不同
--   idx_name: id=1, id=2
--   idx_balance: id=2, id=1 (可能因为平衡值排序)
--   锁获取顺序不一致
--   可能导致死锁!

-- 死锁日志:
------------------------
LATEST DETECTED DEADLOCK
------------------------
*** (1) TRANSACTION:
TRANSACTION 1234, ACTIVE 5 sec updating
mysql tables in use 1, locked 1
*** (1) WAITING FOR THIS LOCK TO BE GRANTED:
RECORD LOCKS space id 1 page no 3 n bits 72 index idx_balance
trx id 1234 lock_mode X locks rec but not gap waiting
-- ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
-- 等待idx_balance索引的锁

*** (2) TRANSACTION:
TRANSACTION 1234, ACTIVE 5 sec updating
mysql tables in use 1, locked 1
2 lock struct(s), heap size 376, 2 row lock(s)
*** (2) HOLDS THE LOCK(S):
RECORD LOCKS space id 1 page no 3 n bits 72 index idx_balance
trx id 1234 lock_mode X locks rec but not gap
-- ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
-- 持有idx_balance索引的锁
*** (2) WAITING FOR THIS LOCK TO BE GRANTED:
RECORD LOCKS space id 1 page no 3 n bits 72 index idx_name
trx id 1234 lock_mode X locks rec but not gap waiting
-- 等待idx_name索引的锁

*** WE ROLL BACK TRANSACTION (1)
-- 同一个事务的不同锁等待
```

**诡异场景2: 批量插入唯一索引死锁**

```sql
-- 场景: 批量插入，有唯一索引

-- 表结构
CREATE TABLE account (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50) UNIQUE,  -- 唯一索引
    balance INT
);

-- 事务A
START TRANSACTION;
INSERT INTO account (name, balance) VALUES ('张三', 100), ('李四', 200);
-- 插入张三、李四
-- 持有张三、李四的唯一索引锁

-- 事务B
START TRANSACTION;
INSERT INTO account (name, balance) VALUES ('李四', 300), ('张三', 400);
-- 插入李四、张四
-- 持有李四的唯一索引锁
-- 等待张三的唯一索引锁

-- 事务A
INSERT INTO account (name, balance) VALUES ('王五', 500);
-- 等待李四的唯一索引锁

-- 死锁!
-- A等B释放李四
-- B等A释放张三
```

**诡异场景3: 外键约束死锁**

```sql
-- 场景: 外键约束导致死锁

-- 表结构
CREATE TABLE orders (
    id INT PRIMARY KEY,
    user_id INT,
    amount INT,
    INDEX idx_user_id (user_id),
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(50)
);

-- 事务A
START TRANSACTION;
DELETE FROM orders WHERE user_id = 1;
-- 删除订单，需要获取users表的外键锁
-- 持有users.id=1的锁

-- 事务B
START TRANSACTION;
UPDATE users SET name = '新名字' WHERE id = 1;
-- 更新用户，需要获取users.id=1的锁
-- 被事务A阻塞

-- 事务A
UPDATE users SET name = '新名字' WHERE id = 2;
-- 被事务B阻塞? (如果事务B也锁了users.id=2)

-- 复杂的锁依赖关系可能导致死锁
```

**诡异场景4: 自增锁死锁**

```sql
-- 场景: 自增锁导致死锁

-- 表结构
CREATE TABLE account (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50),
    balance INT
);

-- 事务A
START TRANSACTION;
INSERT INTO account (name, balance) VALUES ('张三', 100);
-- 获取AUTO-INC锁
-- 插入id=1

-- 插入id=2

-- 如果AUTO-INC锁模式是traditional
-- 事务A一直持有AUTO-INC锁

-- 但系统性能严重下降
```

**诡异场景5: 隐式转换导致死锁**

```sql
-- 场景: 隐式类型转换导致锁范围变大

-- 表结构
CREATE TABLE account (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    balance INT
);

INSERT INTO account VALUES (1, '123', 1000), (2, '张三', 2000);

-- 诡异死锁
START TRANSACTION;
UPDATE account SET balance = 900 WHERE name = 123;
-- ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
-- 注意: name是VARCHAR类型
-- 但查询时用的是INT类型123
-- MySQL会进行隐式类型转换
-- name = 123 → 所有字符串转换为数字比较
-- 可能导致全表扫描
-- 锁定所有行!

-- 事务B
START TRANSACTION;
UPDATE account SET balance = 900 WHERE name = '123';
-- 正确的查询
-- 锁定name='123'这一行

-- 事务A持有所有行的锁
-- 事务B持有name='123'的锁
-- 可能导致死锁
```

**诡异场景6: 多表操作死锁**

```sql
-- 场景: 多表操作顺序不一致

-- 表结构
CREATE TABLE account (
    id INT PRIMARY KEY,
    balance INT
);

CREATE TABLE orders (
    id INT PRIMARY KEY,
    account_id INT,
    amount INT,
    FOREIGN KEY (account_id) REFERENCES account(id)
);

-- 事务A
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1;
UPDATE orders SET amount = amount + 100 WHERE account_id = 1;
-- 顺序: account → orders

-- 事务B
START TRANSACTION;
UPDATE orders SET amount = amount - 100 WHERE account_id = 1;
UPDATE account SET balance = balance + 100 WHERE id = 1;
-- 顺序: orders → account

-- 死锁!
-- A等B释放orders
-- B等A释放account
```

**诡异场景7: 子查询导致锁范围异常**

```sql
-- 场景: 子查询导致锁范围变大

-- 表结构
CREATE TABLE account (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    balance INT
);

INSERT INTO account VALUES
    (1, '张三', 1000),
    (2, '李四', 2000),
    (3, '王五', 3000);

-- 诡异死锁
START TRANSACTION;
UPDATE account SET balance = 900
WHERE id IN (SELECT id FROM account WHERE name = '张三');
-- ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
-- 子查询可能导致临时表
-- 临时表可能影响锁的范围
-- 可能锁定更多行

-- 如果优化器选择不同的执行计划
-- 可能导致锁范围不一致
-- 可能引起死锁
```

**诡异场景8: 触发器导致死锁**

```sql
-- 场景: 触发器中隐式锁表

-- 表结构
CREATE TABLE account (
    id INT PRIMARY KEY,
    balance INT
);

CREATE TABLE audit_log (
    id INT PRIMARY KEY AUTO_INCREMENT,
    account_id INT,
    old_balance INT,
    new_balance INT,
    change_time TIMESTAMP
);

-- 创建触发器
DELIMITER $$
CREATE TRIGGER before_update_account
BEFORE UPDATE ON account
FOR EACH ROW
BEGIN
    INSERT INTO audit_log (account_id, old_balance, new_balance)
    VALUES (OLD.id, OLD.balance, NEW.balance);
END$$
DELIMITER ;

-- 诡异死锁
START TRANSACTION;
UPDATE account SET balance = 900 WHERE id = 1;
-- 触发器自动执行
-- 插入audit_log
-- 可能锁定audit_log表

-- 事务B
START TRANSACTION;
INSERT INTO audit_log (account_id, old_balance, new_balance)
VALUES (2, 2000, 1900);
-- 锁定audit_log表

-- 事务A继续
UPDATE account SET balance = 1900 WHERE id = 2;
-- 触发器插入audit_log
-- 被事务B阻塞

-- 事务B继续
UPDATE account SET balance = 2100 WHERE id = 1;
-- 被事务A阻塞

-- 死锁!
```

**解决方案:**

```sql
-- 解决方案1: 保证操作顺序一致
-- 所有事务都按相同的顺序访问表和记录

-- 统一顺序
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1;
UPDATE orders SET amount = amount + 100 WHERE account_id = 1;
COMMIT;

-- 所有事务都用这个顺序
-- 避免死锁

-- 解决方案2: 拆分大事务
-- 把大事务拆成多个小事务

-- 错误: 大事务
START TRANSACTION;
UPDATE account SET balance = balance + 100 WHERE id IN (1, 2, 3, ... 10000);
-- 一次性更新10000行
-- 容易死锁
COMMIT;

-- 正确: 拆分小事务
FOR i IN 1..10000 STEP 100:
    START TRANSACTION;
    UPDATE account SET balance = balance + 100 WHERE id >= i AND id < i+100;
    COMMIT;

-- 解决方案3: 避免隐式转换
-- 使用正确的数据类型

-- 错误
UPDATE account SET balance = 900 WHERE name = 123;

-- 正确
UPDATE account SET balance = 900 WHERE name = '123';

-- 解决方案4: 添加合适的索引
-- 确保查询走索引

-- 查看执行计划
EXPLAIN UPDATE account SET balance = 900 WHERE name = '张三';
-- 确保使用了索引

-- 解决方案5: 避免复杂的SQL
-- 简化SQL，减少锁的复杂性

-- 错误: 复杂的子查询
UPDATE account SET balance = 900
WHERE id IN (SELECT id FROM account WHERE name = '张三');

-- 正确: 简化
START TRANSACTION;
SELECT id FROM account WHERE name = '张三';
UPDATE account SET balance = 900 WHERE id = ?;
COMMIT;

-- 解决方案6: 使用读已提交隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
-- 减少Gap Lock，降低死锁概率
```

**总结:**

诡异死锁场景的共同特点：

1. **单条SQL自己死锁**：多个索引锁顺序不一致
2. **批量插入**：唯一索引导致锁依赖复杂
3. **外键约束**：隐式加锁
4. **自增锁**：AUTO-INC锁模式问题
5. **隐式转换**：锁范围变大
6. **多表操作**：访问顺序不一致
7. **子查询**：执行计划不确定
8. **触发器**：隐式加锁

解决方法：
- 保证操作顺序一致
- 拆分大事务
- 避免隐式转换
- 添加合适的索引
- 简化SQL
- 使用较低隔离级别

## 📝 总结

**MySQL InnoDB的锁类型:**

**1. 按粒度分:**
- 表锁:锁整张表,并发度低
- 行锁:只锁具体行,并发度高

**2. 按模式分:**
- 共享锁(S锁):允许多个事务同时读
- 排他锁(X锁):独占锁,读写都冲突

**3. 行锁细分:**
- 记录锁:锁住具体记录
- 间隙锁:锁住记录之间的空隙
- 临键锁:记录锁+间隙锁,防止幻读

**4. 特殊锁:**
- 意向锁:表级锁,提高冲突检查性能
- 元数据锁(MDL):保护表结构
- 插入意向锁:允许并发插入同一间隙
- Auto-Inc Lock:保护自增ID
- 谓词锁:支持空间索引(MySQL 8.0+)

**核心知识点:**

- **锁的兼容性**:S锁与S锁兼容,X锁与所有锁冲突
- **意向锁**:提高表锁和行锁冲突检查的性能
- **间隙锁**:可重复读级别特有,防止幻读,读已提交不使用
- **Next-Key Lock**:Record Lock + Gap Lock,防止幻读
- **MDL锁**:DDL操作需要MDL写锁,可能阻塞所有查询
- **Auto-Inc Lock**:有三种模式,默认连续模式
- **死锁**:间隙锁互相等待插入会导致死锁,减少锁持有时间
- **表锁触发**:没走索引、LOCK TABLES、DDL语句

**最佳实践:**

1. 确保查询走索引,避免全表扫描加锁
2. 减少锁持有时间,快速提交事务
3. 按固定顺序访问表,避免死锁
4. 大表DDL使用在线工具(pt-online-schema-change)
5. 在低峰期执行DDL,避免阻塞业务
6. 监控长事务和MDL锁
7. 根据业务场景选择合适的隔离级别
