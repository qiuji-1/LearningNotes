## 📝 测试代码详情

### 1. 聚簇索引和二级索引测试

**文件**: `Question01_ClusteredAndSecondaryIndex.sql`

**测试目的**: 演示通过二级索引查询时需要回表的过程

**主要内容**:
- 创建带有二级索引的测试表
- 通过二级索引查询并观察回表过程
- 对比主键查询和二级索引查询的执行计划

**关键知识点**:
- 聚簇索引：主键索引，叶子节点存储完整数据行
- 二级索引：普通索引，叶子节点存储索引值和主键值
- 回表：通过二级索引找到主键值，再通过主键索引查找完整数据

**执行方式**:
```sql
-- 创建表并插入数据
source Question01_ClusteredAndSecondaryIndex.sql

-- 查看执行计划，验证是否回表
EXPLAIN SELECT * FROM user WHERE name = '张三';
```

### 2. 自增主键测试

**文件**: `Question02_PrimaryKey_AutoIncrement.sql`

**测试目的**: 测试自增主键的存储效率和性能

**主要内容**:
- 创建自增主键表
- 插入测试数据
- 分析表状态和存储情况

**关键知识点**:
- 自增主键顺序插入，减少页分裂
- 聚簇索引有序，范围查询效率高
- 适合大多数场景，性能最优

**推荐场景**:
- 单表数据量小于1000万
- 没有分布式ID生成需求
- 需要较好的范围查询性能

**文件**: `Question03_PrimaryKey_UUID.sql`

### 5. 联合索引最左前缀匹配

**文件**: `Question05_LeftmostPrefix_Match.sql`

**测试目的**: 测试联合索引的最左前缀匹配规则

**主要内容**:
- 创建联合索引 (a, b, c)
- 测试7种不同查询场景
- 验证索引使用情况

**关键知识点**:
- 必须从最左列开始匹配
- 不能跳过中间列
- 部分匹配时只使用前缀

**匹配规则**:
```
✅ a = 1                          (使用a)
✅ a = 1 AND b = 2               (使用a, b)
✅ a = 1 AND b = 2 AND c = 3     (使用a, b, c)
❌ b = 2                          (全表扫描)
❌ c = 3                          (全表扫描)
❌ b = 2 AND c = 3               (全表扫描)
✅ a = 1 AND c = 3               (只使用a，索引下推)
```

### 6. 范围查询特殊规则

**文件**: `Question06_RangeQuery.sql`

**测试目的**: 测试联合索引在范围查询时的行为

**主要内容**:
- 创建联合索引 (user_id, order_date, amount)
- 测试7种范围查询场景
- 验证范围查询对索引使用的影响

**关键知识点**:
- 范围查询列之后的索引列无法使用
- `>=` 操作后续列仍可使用
- `>` 操作后续列无法使用
- 等值查询后的列仍可使用

**规则总结**:
```
✅ user_id = 100 AND order_date = '2026-04-03' AND amount = 300.00  (使用全部)
✅ user_id = 100 AND order_date > '2026-04-01'                         (使用user_id, order_date)
✅ user_id >= 100 AND order_date = '2026-04-03'                         (使用user_id, order_date)
❌ user_id > 100 AND order_date = '2026-04-03'                          (只使用user_id)
```

### 7. Skip Scan优化

**文件**: `Question07_SkipScan.sql`

**测试目的**: 测试MySQL 8.0的Skip Scan优化特性

**主要内容**:
- 创建联合索引 (user_id, order_date)
- 使用索引第二列查询
- 观察Skip Scan优化效果

**关键知识点**:
- Skip Scan：跳过索引第一列，直接使用第二列
- 需要MySQL 8.0.13及以上版本
- 第一列基数要小，第二列基数要大
- 通过EXPLAIN的Extra字段判断

**使用场景**:
- 只能使用联合索引的非首列
- 第一列的值比较少（区分度低）
- 第二列的值比较多（区分度高）

### 8. 查询条件顺序影响

**文件**: `Question08_ConditionOrder.sql`

**测试目的**: 测试MySQL优化器自动调整条件顺序的功能

**主要内容**:
- 创建联合索引 (a, b, c)
- 测试不同条件顺序的查询
- 验证优化器重写效果

**关键知识点**:
- MySQL优化器会自动调整条件顺序
- 不必手动按照索引顺序写WHERE条件
- 关键是包含所有需要的列
- 通过SHOW WARNINGS查看优化器重写

**结论**:
```sql
-- 这两个查询效果完全一样
SELECT * FROM test_condition WHERE a = 2 AND b = 1;
SELECT * FROM test_condition WHERE b = 1 AND a = 2;
```

### 9. 联合索引列顺序设计

**文件**: `Question09_IndexColumnOrder.sql`

**测试目的**: 测试高频查询、区分度、范围查询对索引列顺序的影响

**主要内容**:
- 创建订单表
- 计算各列区分度
- 创建不同顺序的索引
- 测试索引使用效果

**关键知识点**:
- 高频查询列放前面
- 区分度高的列放前面
- 范围查询列放最后
- 排序列放在查询列之后

**设计原则**:
1. **高频原则**: 查询频率最高的列放前面
2. **区分度原则**: 区分度高的列放前面
3. **范围查询原则**: 范围查询列放在等值查询列之后
4. **排序原则**: ORDER BY的列放在WHERE列之后

### 10. B+树容量计算

**文件**: `Question10_PageSize_Capacity.sql`

**测试目的**: 计算不同页大小和主键类型下B+树的容量

**主要内容**:
- 查看当前页大小
- 计算三层B+树理论容量
- 对比不同页大小、主键类型、行大小的影响

**关键知识点**:
- 页大小默认16KB (16384字节)
- 非叶子节点key数 = 页大小 / (主键长度 + 指针长度)
- 三层容量 = 非叶子节点key数² × 每页记录数
- 主键长度对容量影响巨大

**容量对比** (页大小16KB):
| 主键类型 | 非叶子节点key数 | 每页记录数 | 三层容量 |
|---------|---------------|-----------|---------|
| INT     | 1029          | 16        | 1.69亿  |
| BIGINT  | 1170          | 16        | 2.19亿  |
| CHAR(36)| 390           | 16        | 0.24亿  |

### 11. 二级索引容量计算

**文件**: `Question11_SecondaryIndex_Capacity.sql`

**测试目的**: 测试二级索引相比主键索引能支撑的数据量

**主要内容**:
- 创建订单表
- 创建二级索引
- 计算二级索引容量
- 测试覆盖索引和回表

**关键知识点**:
- 主键索引叶子存储完整数据行（约100字节）
- 二级索引叶子存储主键值+索引值（约16字节）
- 二级索引能支撑的数据量更多
- 回表需要额外的I/O开销

**容量对比** (页大小16KB, 三层):
| 索引类型 | 叶子节点内容 | 每页记录数 | 容量(万条) |
|---------|------------|-----------|-----------|
| 主键索引 | 完整数据行 | 16        | 219        |
| 二级索引 | 主键+索引值| 1024      | 13996      |

**回表优化**:
- 覆盖索引：SELECT只包含索引列，避免回表
- LIMIT限制：限制返回数量，减少回表次数
- 合理选择查询字段

## 🚀 使用方法

### 1. 环境准备

```bash
# 确保MySQL已安装并启动
mysql --version

# 创建测试数据库
mysql -u root -p
CREATE DATABASE test_mysql;
USE test_mysql;
```

### 2. 执行测试代码

**方式一：直接执行SQL文件**
```bash
mysql -u root -p test_mysql < Question01_ClusteredAndSecondaryIndex.sql
```

**方式二：MySQL命令行执行**
```sql
USE test_mysql;
source /path/to/Question01_ClusteredAndSecondaryIndex.sql;
```

**方式三：工具执行**
- 使用Navicat、DataGrip等工具打开SQL文件
- 选择测试数据库后执行

### 3. 查看执行计划

```sql
-- 查看查询的执行计划
EXPLAIN SELECT * FROM user WHERE name = '张三';

-- 查看详细执行计划（MySQL 8.0+）
EXPLAIN FORMAT=JSON SELECT * FROM user WHERE name = '张三';

-- 查看优化器重写
EXPLAIN SELECT * FROM user WHERE b = 1 AND a = 2;
SHOW WARNINGS;
```

### 4. 分析执行计划

重点关注以下字段：
- **type**: 访问类型（const、ref、range、index、ALL）
- **key**: 使用的索引
- **rows**: 扫描的行数
- **Extra**: 额外信息（Using index、Using where、Using filesort）

## 💡 最佳实践

### 1. 索引使用原则
- 经常作为WHERE条件的列加索引
- 经常ORDER BY、GROUP BY的列加索引
- 区分度低的列不适合单独加索引
- 避免过多索引，影响插入性能

### 2. 主键选择
- 首选自增主键
- 避免使用UUID
- 复合主键只用于中间表
- 无合适主键时考虑雪花算法

### 3. 联合索引设计
- 高频查询列放前面
- 区分度高的列放前面
- 范围查询列放最后
- 不要超过5列

### 4. 性能优化
- 使用EXPLAIN分析慢查询
- 关注rows和Extra字段
- 优先优化全表扫描查询
- 合理使用覆盖索引
