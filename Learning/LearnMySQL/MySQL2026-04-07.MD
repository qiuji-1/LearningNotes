## 1. MySQL中的事务隔离级别有哪些?

### 📌 回答重点

MySQL中有四种事务隔离级别,从低到高依次是:

1) **读未提交(READ UNCOMMITTED)**:可以读取到其他事务未提交的数据,会出现脏读、不可重复读、幻读问题。

2) **读已提交(READ COMMITTED)**:只能读取到其他事务已提交的数据,解决了脏读,但会出现不可重复读和幻读。

3) **可重复读(REPEATABLE READ)**:MySQL的默认隔离级别,保证同一事务内多次读取同一数据的结果一致,解决了脏读和不可重复读,但可能出现幻读。MySQL InnoDB通过MVCC(多版本并发控制)和Next-Key Lock解决了幻读。

4) **串行化(SERIALIZABLE)**:最高隔离级别,强制事务串行执行,解决了所有并发问题,但性能最差。

**隔离级别对比表格:**

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 性能 |
|---------|------|-----------|------|------|
| 读未提交 | ✅ 会出现 | ✅ 会出现 | ✅ 会出现 | ⭐⭐⭐⭐⭐ |
| 读已提交 | ❌ 解决 | ✅ 会出现 | ✅ 会出现 | ⭐⭐⭐⭐ |
| 可重复读 | ❌ 解决 | ❌ 解决 | ❌ 基本解决 | ⭐⭐⭐ |
| 串行化 | ❌ 解决 | ❌ 解决 | ❌ 解决 | ⭐ |

**查看和设置隔离级别:**

```sql
-- 查看当前隔离级别
SELECT @@transaction_isolation;
-- 或
SHOW VARIABLES LIKE 'transaction_isolation';

-- 设置全局隔离级别
SET GLOBAL TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 设置当前会话的隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 在开启事务时设置隔离级别
START TRANSACTION ISOLATION LEVEL SERIALIZABLE;
```

### 📚 扩展知识

#### 各种并发问题的含义

**脏读(Dirty Read):**

一个事务读取到了另一个事务未提交的数据。如果那个事务回滚了,读取到的数据就是无效的。

```sql
-- 事务A
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1;
-- 此时事务未提交

-- 事务B
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 读取到了事务A未提交的修改(脏读)
-- 如果事务A回滚,事务B读取到的数据就是错误的
```

**不可重复读(Non-repeatable Read):**

在同一个事务内,两次读取同一数据得到的结果不一样。因为期间有其他事务修改了该数据并提交。

```sql
-- 事务A
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 第一次读取: 1000

-- 事务B
START TRANSACTION;
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;

-- 事务A
SELECT balance FROM account WHERE id = 1;  -- 第二次读取: 2000(不可重复读)
```

**幻读(Phantom Read):**

在同一个事务内,前后两次查询返回的记录数不一样。因为期间有其他事务插入或删除了数据并提交。

```sql
-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE age > 18;  -- 第一次查询: 返回10条记录

-- 事务B
START TRANSACTION;
INSERT INTO account (name, age) VALUES ('张三', 20);
COMMIT;

-- 事务A
SELECT * FROM account WHERE age > 18;  -- 第二次查询: 返回11条记录(幻读)
```

#### 各隔离级别的实现原理

**读未提交(READ UNCOMMITTED):**

最宽松的隔离级别,不加任何锁,直接读取最新数据。

- 优点:性能最好
- 缺点:数据一致性最差,几乎不使用

---

**读已提交(READ COMMITTED):**

Oracle、SQL Server等数据库的默认隔离级别。

每次SELECT语句都会生成一个新的Read View,只能读取到已提交的数据。

- 实现方式:每次SELECT生成新的Read View
- 解决问题:脏读
- 未解决问题:不可重复读、幻读

**演示不可重复读:**

```sql
-- 查看隔离级别
SELECT @@transaction_isolation;  -- READ-COMMITTED

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 1;  -- 第一次读取: balance=1000

-- 事务B
START TRANSACTION;
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;

-- 事务A
SELECT * FROM account WHERE id = 1;  -- 第二次读取: balance=2000(不可重复读)
COMMIT;
```

---

**可重复读(REPEATABLE READ):**

MySQL InnoDB的默认隔离级别。

事务开始时生成一个Read View,整个事务期间都使用这个Read View读取数据,保证同一事务内多次读取结果一致。

- 实现方式:MVCC(多版本并发控制),事务开始生成Read View
- 解决问题:脏读、不可重复读
- 幻读问题:通过Next-Key Lock(临键锁)解决

**MVCC原理:**

MVCC(Multi-Version Concurrency Control)通过保存数据的历史版本来实现不加锁的读操作。

```sql
-- InnoDB每行数据隐藏的字段
-- DB_TRX_ID: 事务ID,最近修改这行数据的事务ID
-- DB_ROLL_PTR: 回滚指针,指向undo log中该行的上一个版本
-- DB_ROW_ID: 行ID(隐藏主键)
```

**Read View机制:**

Read View包含当前活跃的事务ID列表,判断数据版本是否可见:

```
1. 如果事务ID < Read View的最小活跃事务ID → 可见(已提交)
2. 如果事务ID > Read View的最大事务ID → 不可见(未开始的事务)
3. 如果在活跃事务列表中 → 不可见(未提交)
4. 如果不在活跃事务列表中 → 可见(已提交)
```

**Next-Key Lock解决幻读:**

幻读的本质是范围查询时,其他事务插入了新数据。InnoDB通过Next-Key Lock(记录锁+间隙锁)锁定整个范围,防止其他事务插入。

```sql
-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id BETWEEN 10 AND 20 FOR UPDATE;
-- Next-Key Lock锁定: (10,20]范围,防止其他事务插入id在10-20之间的记录

-- 事务B
START TRANSACTION;
INSERT INTO account (id, name) VALUES (15, '张三');  -- 被阻塞
COMMIT;

-- 事务A
SELECT * FROM account WHERE id BETWEEN 10 AND 20;  -- 不会出现幻读
COMMIT;
```

**演示可重复读:**

```sql
-- 设置隔离级别为可重复读
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 1;  -- 第一次读取: balance=1000

-- 事务B
START TRANSACTION;
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;

-- 事务A
SELECT * FROM account WHERE id = 1;  -- 第二次读取: balance=1000(可重复读)
-- 看到的是事务开始时的快照,不受其他事务提交的影响
COMMIT;
```

---

**串行化(SERIALIZABLE):**

最高隔离级别,强制事务串行执行,所有SELECT语句都隐式添加锁,或者读写分离。

- 实现方式:读写都加锁,或者只读快照
- 解决问题:所有并发问题
- 性能:最差,几乎不使用

**演示串行化:**

```sql
-- 设置隔离级别为串行化
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 1;  -- 隐式加共享锁
-- 事务A持有锁,事务B会被阻塞

-- 事务B
START TRANSACTION;
UPDATE account SET balance = 2000 WHERE id = 1;  -- 被阻塞,等待事务A释放锁
-- 直到事务A提交或回滚,事务B才能执行
```

#### 不同隔离级别的适用场景

**读未提交(READ UNCOMMITTED):**

几乎不使用,仅用于极少数特殊场景,如数据一致性要求不高的统计分析。

```sql
-- 示例:实时统计,可以接受数据不准确
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
SELECT COUNT(*) FROM logs WHERE level = 'ERROR';  -- 即使未提交的ERROR日志也被统计
```

---

**读已提交(READ COMMITTED):**

适用于大多数OLTP(在线事务处理)系统,如电商、金融等。

```sql
-- 示例:银行转账系统
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 事务A:查询余额
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 只能读到已提交的转账记录
```

**优点:**
- 避免脏读,数据一致性较好
- 性能相对较好

**缺点:**
- 会出现不可重复读和幻读

---

**可重复读(REPEATABLE READ):**

MySQL默认隔离级别,适用于大多数业务场景。

```sql
-- 示例:订单查询
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 事务A
START TRANSACTION;
SELECT * FROM orders WHERE user_id = 100;  -- 第一次查询:10条订单
-- ... 中间可能有其他事务插入新订单 ...
SELECT * FROM orders WHERE user_id = 100;  -- 第二次查询:还是10条订单(不会幻读)
COMMIT;
```

**优点:**
- 解决脏读和不可重复读
- InnoDB基本解决了幻读
- 性能较好

**缺点:**
- 可能出现死锁(因为使用了Next-Key Lock)
- 读写并发性能略低于读已提交

---

**串行化(SERIALIZABLE):**

极少使用,只用于对数据一致性要求极高的场景。

```sql
-- 示例:库存扣减,不允许并发
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- 事务A
START TRANSACTION;
SELECT stock FROM product WHERE id = 1 FOR UPDATE;  -- 锁定记录
UPDATE product SET stock = stock - 1 WHERE id = 1;
COMMIT;

-- 事务B会被阻塞,直到事务A完成
```

**优点:**
- 绝对的数据一致性

**缺点:**
- 性能极差,吞吐量低
- 容易出现死锁

---

#### MySQL InnoDB vs 其他数据库的默认隔离级别

**默认隔离级别对比:**

| 数据库 | 默认隔离级别 | 说明 |
|--------|-------------|------|
| MySQL(InnoDB) | REPEATABLE READ | 通过MVCC和Next-Key Lock解决幻读 |
| Oracle | READ COMMITTED | 不支持REPEATABLE READ |
| SQL Server | READ COMMITTED | 支持快照隔离 |
| PostgreSQL | READ COMMITTED | 可通过配置改为SERIALIZABLE |

**MySQL(InnoDB)的特殊之处:**

InnoDB在可重复读级别下,通过Next-Key Lock基本解决了幻读问题,这是MySQL相比其他数据库的优势。

```sql
-- MySQL演示:REPEATABLE READ级别下不会出现幻读
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id BETWEEN 10 AND 20;  -- 第一次查询:10条
-- Next-Key Lock锁定(10,20]范围

-- 事务B
START TRANSACTION;
INSERT INTO account (id, name) VALUES (15, '张三');  -- 被阻塞
-- 等待事务A释放锁
COMMIT;

-- 事务A
SELECT * FROM account WHERE id BETWEEN 10 AND 20;  -- 第二次查询:还是10条(不会幻读)
COMMIT;
```

**为什么Oracle只支持到读已提交?**

Oracle采用MVCC实现读一致性,但不支持间隙锁,所以无法在可重复读级别下防止幻读。

**从MySQL迁移到Oracle时的注意事项:**

如果业务依赖MySQL的可重复读特性,迁移到Oracle时可能需要调整SQL逻辑或使用SELECT FOR UPDATE锁。

### 💬 面试官追问

#### 提问1:如果把隔离级别从可重复读调成读已提交,业务代码需要注意什么?

**回答:**

从可重复读(REPEATABLE READ)切换到读已提交(READ COMMITTED)会影响业务逻辑,需要特别注意以下几点:

**1) 不可重复读的影响**

```sql
-- 可重复读级别
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 第一次:1000

-- 其他事务更新并提交
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;

SELECT balance FROM account WHERE id = 1;  -- 第二次:还是1000
COMMIT;
```

```sql
-- 读已提交级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 第一次:1000

-- 其他事务更新并提交
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;

SELECT balance FROM account WHERE id = 1;  -- 第二次:2000(不可重复读!)
COMMIT;
```

**业务代码需要注意:**

**场景1:多次读取同一数据**

```sql
-- ❌ 可重复读级别下可以的代码
START TRANSACTION;
-- 第一步:查询余额
SELECT balance FROM account WHERE id = 1;
-- 第二步:计算利息
SET interest = balance * 0.05;
-- 第三步:更新余额
UPDATE account SET balance = balance + interest WHERE id = 1;
COMMIT;

-- ✅ 读已提交级别需要调整为
START TRANSACTION;
SELECT balance FROM account WHERE id = 1 FOR UPDATE;  -- 锁定记录
-- 读取、计算、更新
SET interest = balance * 0.05;
UPDATE account SET balance = balance + interest WHERE id = 1;
COMMIT;
```

**场景2:生成报表**

```sql
-- ❌ 可重复读级别:两次查询基于同一快照
START TRANSACTION;
SELECT COUNT(*) FROM orders WHERE date >= '2026-04-01';  -- 1000条
SELECT SUM(amount) FROM orders WHERE date >= '2026-04-01';  -- 基于1000条计算
COMMIT;

-- ⚠️ 读已提交级别:可能出现数据不一致
START TRANSACTION;
SELECT COUNT(*) FROM orders WHERE date >= '2026-04-01';  -- 1000条
-- 期间有新订单插入
SELECT SUM(amount) FROM orders WHERE date >= '2026-04-01';  -- 基于超过1000条计算
COMMIT;

-- ✅ 解决方案1:使用临时表
START TRANSACTION;
-- 复制数据到临时表
CREATE TEMPORARY TABLE temp_orders AS
SELECT * FROM orders WHERE date >= '2026-04-01';

SELECT COUNT(*) FROM temp_orders;
SELECT SUM(amount) FROM temp_orders;
COMMIT;

-- ✅ 解决方案2:使用可重复读级别
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
START TRANSACTION;
SELECT COUNT(*) FROM orders WHERE date >= '2026-04-01';
SELECT SUM(amount) FROM orders WHERE date >= '2026-04-01';
COMMIT;
```

**2) 幻读的影响**

```sql
-- 读已提交级别会出现幻读
START TRANSACTION;
SELECT * FROM orders WHERE status = 'pending';  -- 返回10条

-- 其他事务插入新订单
INSERT INTO orders (status, amount) VALUES ('pending', 100);
COMMIT;

SELECT * FROM orders WHERE status = 'pending';  -- 返回11条(幻读)
COMMIT;
```

**业务代码需要注意:**

**场景3:权限检查**

```sql
-- ❌ 可重复读级别:安全
START TRANSACTION;
SELECT COUNT(*) FROM orders WHERE user_id = 100;  -- 5条
IF count < 10 THEN
    -- 允许新订单
    INSERT INTO orders (user_id, amount) VALUES (100, 500);
END IF;
COMMIT;

-- ⚠️ 读已提交级别:可能超出限制
START TRANSACTION;
SELECT COUNT(*) FROM orders WHERE user_id = 100;  -- 5条
-- 期间其他事务插入订单
SELECT COUNT(*) FROM orders WHERE user_id = 100;  -- 可能已经>10条
-- 仍然插入新订单
INSERT INTO orders (user_id, amount) VALUES (100, 500);
COMMIT;

-- ✅ 解决方案:使用SELECT FOR UPDATE
START TRANSACTION;
SELECT COUNT(*) FROM orders WHERE user_id = 100 FOR UPDATE;  -- 锁定
IF count < 10 THEN
    INSERT INTO orders (user_id, amount) VALUES (100, 500);
END IF;
COMMIT;
```

**3) 性能差异**

```sql
-- 读已提交级别的特点:
-- - Read View每次SELECT都生成
-- - 没有Next-Key Lock
-- - 并发性能略好
```

**性能对比:**

| 隔离级别 | Read View | Next-Key Lock | 并发性能 |
|---------|-----------|---------------|---------|
| 可重复读 | 事务开始生成一次 | 有(当前读) | 较好 |
| 读已提交 | 每次SELECT生成 | 无 | 略好 |

**4) 代码修改清单**

**需要修改的场景:**

```sql
-- 场景1:事务内多次读取同一数据
-- ❌ 旧代码
SELECT balance FROM account WHERE id = 1;
-- ... 业务逻辑 ...
SELECT balance FROM account WHERE id = 1;
-- ✅ 新代码
SELECT balance FROM account WHERE id = 1 FOR UPDATE;
-- ... 业务逻辑 ...
SELECT balance FROM account WHERE id = 1;  -- 仍然是锁定的值

-- 场景2:多次查询聚合数据
-- ❌ 旧代码
SELECT COUNT(*) FROM orders WHERE date >= '2026-04-01';
SELECT SUM(amount) FROM orders WHERE date >= '2026-04-01';
-- ✅ 新代码
-- 方案1:使用临时表
CREATE TEMPORARY TABLE temp_orders AS
SELECT * FROM orders WHERE date >= '2026-04-01';
SELECT COUNT(*) FROM temp_orders;
SELECT SUM(amount) FROM temp_orders;

-- 方案2:单次查询
SELECT COUNT(*), SUM(amount) FROM orders WHERE date >= '2026-04-01';

-- 场景3:基于读取结果做条件判断
-- ❌ 旧代码
SELECT count FROM product WHERE id = 100;
IF count > 0 THEN
    UPDATE product SET count = count - 1 WHERE id = 100;
END IF;
-- ✅ 新代码
SELECT count FROM product WHERE id = 100 FOR UPDATE;
IF count > 0 THEN
    UPDATE product SET count = count - 1 WHERE id = 100;
END IF;
```

**不需要修改的场景:**

```sql
-- ✅ 单次查询
SELECT * FROM orders WHERE id = 1001;

-- ✅ 使用SELECT FOR UPDATE的查询
SELECT * FROM account WHERE id = 1 FOR UPDATE;

-- ✅ 使用UPDATE/DELETE语句
UPDATE account SET balance = balance - 100 WHERE id = 1;
DELETE FROM orders WHERE status = 'cancelled';

-- ✅ 使用乐观锁
UPDATE product 
SET stock = stock - 1, version = version + 1 
WHERE id = 100 AND version = 1;
```

**5) 总结建议**

**何时切换到读已提交:**

- 需要看到最新提交的数据
- 需要更高的并发性能
- 业务代码已经使用了SELECT FOR UPDATE
- 不需要事务内数据一致

**切换前的准备:**

```sql
-- 1. 审计业务代码
-- 找出所有事务内多次读取的代码
-- 找出所有依赖可重复读特性的逻辑

-- 2. 添加SELECT FOR UPDATE
-- 关键读取操作添加锁定

-- 3. 使用临时表或单次查询
-- 合并多次查询

-- 4. 充分测试
-- 在测试环境验证所有场景
```

**示例:迁移步骤**

```sql
-- 步骤1:配置数据库
SET GLOBAL TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 步骤2:修改关键业务代码
-- 添加SELECT FOR UPDATE
-- 合并多次查询

-- 步骤3:测试验证
-- 并发测试
-- 边界情况测试
-- 性能测试

-- 步骤4:灰度发布
-- 先发布非核心业务
-- 监控是否有问题
-- 逐步发布全量
```

#### 提问2:可重复读级别下两个事务同时SELECT...FOR UPDATE同一行,会怎么样?

**回答:**

在可重复读级别下,两个事务同时执行`SELECT ... FOR UPDATE`锁定同一行数据,会导致其中一个事务被阻塞,等待另一个事务释放锁。

**演示场景:**

```sql
-- 准备测试表
CREATE TABLE account (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    balance DECIMAL(10,2)
);

INSERT INTO account VALUES (1, '张三', 1000);
```

**场景1:两个事务同时锁定同一行**

```sql
-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- 锁定id=1的记录
-- 此时事务A持有排他锁(X锁)

-- 事务B
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- ⚠️ 被阻塞!
-- 等待事务A释放锁

-- 事务A
-- 可以继续操作
UPDATE account SET balance = balance - 100 WHERE id = 1;
COMMIT;
-- 释放锁

-- 事务B
-- 解除阻塞,获取锁
SELECT * FROM account WHERE id = 1;  -- 可以读取到最新数据
COMMIT;
```

**时间线示意图:**

```

----  ----------------------------------  ----------------------------------
T1    START TRANSACTION
T2    SELECT ... FOR UPDATE id=1
      (获取排他锁)
T3                                        START TRANSACTION
T4                                        SELECT ... FOR UPDATE id=1
                                          (尝试获取锁,被阻塞)
T5    UPDATE account SET balance = ...
T6    COMMIT
      (释放锁)
T7                                        (解除阻塞,获取锁)
T8                                        SELECT ... FOR UPDATE id=1
                                          (读取到最新数据)
T9                                        COMMIT
```

**场景2:多个事务同时锁定同一行**

```sql
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- 获取锁

-- 事务B
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- 被阻塞,等待A

-- 事务C
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- 被阻塞,等待A

-- 事务A
COMMIT;
-- 释放锁

-- 事务B或事务C谁能获取到锁?
-- 由锁管理器决定,通常按照请求顺序
```

**锁的队列机制:**

```
等待队列:
[事务B] → [事务C] → [事务D]
   ↑

```

**场景3:死锁情况**

```sql
-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR UPDATE;  -- 锁定id=1
-- 锁: id=1 (X)

-- 事务B
START TRANSACTION;
SELECT * FROM account WHERE id = 2 FOR UPDATE;  -- 锁定id=2
-- 锁: id=2 (X)

-- 事务A
SELECT * FROM account WHERE id = 2 FOR UPDATE;  -- 尝试锁定id=2
-- ⚠️ 被阻塞,等待事务B

-- 事务B
SELECT * FROM account WHERE id = 1 FOR UPDATE;  -- 尝试锁定id=1
-- ⚠️ 被阻塞,等待事务A
-- ❌ 死锁!互相等待对方

-- MySQL检测到死锁,自动回滚其中一个事务
```

```sql
-- 方法1:按固定顺序访问表
-- ✅ 正确:所有事务都按id从小到大的顺序访问

-- 方法3:添加超时设置
-- 设置锁等待超时时间

-- 超时后自动回滚
```

```sql
-- 事务B
START TRANSACTION;

-- 锁定范围更广,更容易冲突

-- 读已提交级别

-- 锁定范围小,冲突概率低
```

**总结:**

1) **默认行为**:一个事务锁定后,其他事务会被阻塞等待
2) **锁的队列**:按照请求顺序排队
3) **死锁风险**:多个事务循环等待会导致死锁
4) **自动检测**:MySQL会自动检测死锁并回滚一个事务
5) **避免方法**:按固定顺序访问、减少锁持有时间、设置超时

#### 提问3:读未提交这个级别有什么实际使用场景吗?

**回答:**

读未提交(READ UNCOMMITTED)在实际业务中几乎不使用,但在极少数特殊场景下有其价值。

**读未提交的特点:**

- 可以读取到其他事务未提交的数据
- 性能最好,几乎不加锁
- 会出现脏读、不可重复读、幻读

**几乎不使用的原因:**

1) **数据一致性太差**:读取到未提交的数据,如果该事务回滚,读取到的数据就是无效的
2) **业务风险高**:基于脏数据做出的决策可能导致严重错误
3) **替代方案多**:大多数场景可以通过其他方式满足需求

**极少数特殊使用场景:**

**场景1:实时监控和告警**

```sql
-- 需求:监控系统需要实时采集数据,即使数据未提交也要统计
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;

-- 事务A:插入监控数据
START TRANSACTION;
INSERT INTO logs (service, level, message) VALUES ('api', 'ERROR', 'Timeout');
-- 还未提交

-- 事务B:监控查询
START TRANSACTION;
SELECT COUNT(*) FROM logs WHERE level = 'ERROR' AND service = 'api';
-- ✅ 能读取到事务A未提交的ERROR日志
-- 监控系统可以立即告警
COMMIT;

-- 事务A
COMMIT;
```

**为什么适合这个场景:**
- 监控需要尽可能快地发现问题
- 即便读到未提交的数据,也不会造成实际损失
- 即使数据回滚,重新统计即可
- 实时性比准确性更重要

**场景2:大规模数据分析**

```sql
-- 需求:分析系统需要扫描海量日志,性能要求极高
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;

-- 数据导入事务
START TRANSACTION;
-- 导入100万条日志数据,耗时10分钟
INSERT INTO logs (message, level) VALUES (...);
-- 还在导入中...

-- 分析查询
SELECT 
    DATE(create_time) AS date,
    COUNT(*) AS error_count
FROM logs 
WHERE level = 'ERROR'
GROUP BY DATE(create_time);

-- ✅ 能读取到正在导入的数据,不阻塞导入操作
-- 性能最好,不会等待其他事务提交
COMMIT;
```

**为什么适合这个场景:**
- 数据量大,性能要求高
- 统计结果不需要100%准确
- 即使有错误数据,误差在可接受范围内
- 并发性能最重要

**场景3:压力测试和性能评估**

```sql
-- 需求:测试数据库的极限性能
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;

-- 模拟大量并发写入
-- 事务A-N:大量并发写入
START TRANSACTION;
UPDATE account SET balance = balance + 100 WHERE id = 1;
-- ... 大量并发写入操作 ...

-- 统计查询
SELECT AVG(balance) FROM account;
-- ✅ 不等待写事务提交,直接读取
-- 性能测试结果更准确
```

**为什么适合这个场景:**
- 测试环境下,不需要保证数据准确性
- 只关心性能指标
- 减少锁等待,更能体现真实性能

**场景4:数据质量检查**

```sql
-- 需求:检查数据质量,不关心事务是否提交
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;

-- 批量导入事务
START TRANSACTION;
INSERT INTO orders (user_id, amount) VALUES (100, 500);
-- ... 导入大量数据 ...

-- 质量检查
SELECT COUNT(*) AS invalid_orders
FROM orders
WHERE amount < 0 OR amount > 1000000;

-- ✅ 立即发现数据问题
-- 不需要等待导入完成
-- 可以快速中断错误的导入
```

**不适用读未提交的场景:**

**❌ 银行转账**

```sql
-- 绝对不能使用读未提交
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1;  -- 未提交

-- 其他事务读取
SELECT balance FROM account WHERE id = 1;
-- 如果读到脏数据,可能导致严重错误
```

**❌ 库存管理**

```sql
-- 绝对不能使用读未提交
START TRANSACTION;

-- 未提交

-- 其他事务读取
SELECT stock FROM product WHERE id = 100;
-- 如果读到脏数据,可能导致超卖
```

**❌ 订单处理**

```sql
-- 绝对不能使用读未提交
START TRANSACTION;

-- 未提交

-- 其他事务查询
SELECT COUNT(*) FROM orders WHERE user_id = 100;
-- 如果统计结果不准确,导致业务错误
```

**替代方案:**

对于大多数场景,推荐使用其他方案:

**1) 使用读已提交 + 缓存**

```sql
-- 不用读未提交
-- ✅ 使用读已提交 + Redis缓存
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 缓存热点数据
-- 先查缓存,没有再查数据库
SELECT balance FROM account WHERE id = 1;
-- 缓存到Redis
SET cache:account:1 balance

-- 后续查询从缓存读取
GET cache:account:1
```

**2) 使用异步统计**

```sql
-- 不用读未提交实时统计
-- ✅ 使用异步统计
-- 业务数据实时插入
INSERT INTO logs (level, message) VALUES ('ERROR', 'Timeout');

-- 异步任务定期统计
-- 每分钟执行一次
SELECT COUNT(*) FROM logs WHERE level = 'ERROR';
-- 存储到统计表
```

**3) 使用物化视图**

```sql
-- 不用读未提交查询
-- ✅ 使用物化视图
CREATE MATERIALIZED VIEW mv_error_logs AS
SELECT 
    DATE(create_time) AS date,
    COUNT(*) AS error_count
FROM logs 
WHERE level = 'ERROR'
GROUP BY DATE(create_time);

-- 定期刷新物化视图
REFRESH MATERIALIZED VIEW mv_error_logs;

-- 查询物化视图(速度快)
SELECT * FROM mv_error_logs;
```

**总结:**

**读未提交的使用场景:**
1. 实时监控告警:需要尽快发现问题
2. 大规模数据分析:性能要求高,准确性要求低
3. 压力测试:测试环境下,不需要数据准确性
4. 数据质量检查:快速发现数据问题

**读未提交的替代方案:**
1. 读已提交 + 缓存
2. 异步统计
3. 物化视图
4. 使用数据库的快照功能

**最佳实践:**
- 99%场景使用READ COMMITTED或REPEATABLE READ
- 只有在明确需要实时性且能容忍不准确数据时才考虑READ UNCOMMITTED
- 使用前充分评估业务风险

## 2. MySQL默认的事务隔离级别是什么?为什么选择这个级别?

### 📌 回答重点

MySQL(InnoDB存储引擎)的默认事务隔离级别是**可重复读(REPEATABLE READ)**。

**查看当前隔离级别:**

```sql
-- 方式1:查看系统变量
SELECT @@transaction_isolation;
-- 结果: REPEATABLE-READ

-- 方式2:使用SHOW命令
SHOW VARIABLES LIKE 'transaction_isolation';
-- 结果: transaction_isolation | REPEATABLE-READ

-- 方式3:查看会话隔离级别
SELECT @@session.transaction_isolation;
```

**为什么MySQL选择可重复读作为默认级别?**

主要原因有以下几点:

1) **避免脏读和不可重复读**:可重复读级别解决了这两个最常见的数据一致性问题

2) **通过MVCC实现高性能**:快照读不需要加锁,读写不冲突,并发性能好

3) **通过Next-Key Lock基本解决幻读**:虽然理论上可重复读会出现幻读,但InnoDB通过Next-Key Lock在实际应用中基本避免了幻读

4) **平衡了一致性和性能**:相比读已提交,提供更好的一致性;相比串行化,提供更好的性能

**各数据库默认隔离级别对比:**

| 数据库 | 默认隔离级别 | 说明 |
|--------|-------------|------|
| MySQL(InnoDB) | REPEATABLE READ | 通过MVCC和Next-Key Lock基本解决幻读 |
| Oracle | READ COMMITTED | 不支持REPEATABLE READ级别 |
| SQL Server | READ COMMITTED | 支持快照隔离 |
| PostgreSQL | READ COMMITTED | 可通过配置改为SERIALIZABLE |
| DB2 | READ COMMITTED | 支持RR级别但不是默认 |

**MySQL选择RR级别的历史原因:**

MySQL早期使用**statement格式**的binlog进行数据复制，这种格式记录的是SQL语句而不是数据变更的快照。如果在**读已提交(RC)**隔离级别下使用statement格式的binlog，会导致主从复制数据不一致。

**问题演示:**

```sql
-- 主库(RC隔离级别)
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 事务A:删除id<10的记录
START TRANSACTION;
DELETE FROM t WHERE id < 10;

-- 事务B:插入id=5的记录并提交
START TRANSACTION;
INSERT INTO t VALUES (5);
COMMIT;

-- 事务A提交
COMMIT;

-- binlog记录的SQL: DELETE FROM t WHERE id < 10

-- 从库执行binlog
-- 执行结果: 删除了id=5的记录
-- 但主库上id=5的记录还在(因为事务A删除时还没看到事务B插入的数据)
-- 结果: 主从数据不一致!
```

**为什么RR级别能解决这个问题?**

在RR隔离级别下，事务执行时会生成Read View，整个事务期间看到的数据快照是一致的。因此，binlog记录的数据变更与主库实际执行结果一致，从库执行binlog后能保证数据一致。

**binlog格式的发展:**

1. **statement格式**（早期MySQL默认）
   - 记录SQL语句
   - 空间占用小，但可能存在主从一致性问题
   - **必须配合RR隔离级别使用**

2. **row格式**（MySQL 5.1.5+）
   - 记录每一行的数据变更
   - 空间占用大，但能保证主从一致性
   - **可以使用RC隔离级别**

3. **mixed格式**（MySQL 5.1.8+）
   - 自动选择statement或row格式
   - 根据SQL语句类型智能选择

**现在的选择:**

- MySQL 5.7+: 默认使用row格式binlog
- 新项目推荐使用row格式 + RC隔离级别（性能更好，避免间隙锁）
- 但为了兼容性和历史原因，MySQL的默认隔离级别仍然是RR

### 📚 扩展知识

#### 为什么不选择读已提交?

读已提交(READ COMMITTED)是Oracle、SQL Server等主流数据库的默认隔离级别,MySQL为什么不选择它?

**读已提交的问题:**

**1) 不可重复读现象**

```sql
-- 读已提交级别下
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 事务A
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 第一次读取:1000

-- 事务B更新并提交
START TRANSACTION;
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;

-- 事务A第二次查询
SELECT balance FROM account WHERE id = 1;  -- 第二次读取:2000
-- 在同一个事务内,两次查询结果不一致,可能导致业务逻辑错误
COMMIT;
```

**2) 幻读现象**

```sql
-- 读已提交级别下
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 事务A
START TRANSACTION;
SELECT * FROM orders WHERE user_id = 100;  -- 第一次查询:5条订单

-- 事务B插入新订单
START TRANSACTION;
INSERT INTO orders (user_id, amount) VALUES (100, 500);
COMMIT;

-- 事务A第二次查询
SELECT * FROM orders WHERE user_id = 100;  -- 第二次查询:6条订单
-- 多了一条订单,可能导致统计数据不准确
COMMIT;
```

**业务场景中不可重复读的危害:**

```sql
-- 场景:生成报表
START TRANSACTION;

-- 第一步:查询总订单数
SELECT COUNT(*) FROM orders;  -- 返回1000条

-- 第二步:处理其他业务逻辑...
-- 期间可能有新订单插入

-- 第三步:再次查询订单金额
SELECT SUM(amount) FROM orders;  -- 对应的订单数已经不是1000条了
-- 报表数据不一致

COMMIT;
```

**为什么MySQL坚持使用可重复读?**

1) **业务需求**:大多数业务需要事务内数据一致,不希望看到其他事务的修改

2) **技术优势**:通过MVCC,可重复读级别的快照读性能不比读已提交差

3) **减少问题**:避免不可重复读和幻读,减少业务层的复杂性

#### 为什么不选择串行化?

串行化(SERIALIZABLE)是最高隔离级别,完全避免所有并发问题,但MySQL不选择它作为默认级别,原因是:

**串行化的问题:**

**1) 性能极差**

```sql
-- 串行化级别下
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 1;  -- 隐式加共享锁
-- 事务A持有锁,其他读操作也会被阻塞

-- 事务B
START TRANSACTION;
SELECT * FROM account WHERE id = 1;  -- 被阻塞,等待事务A释放锁
-- 即使是读操作,也会被阻塞!
COMMIT;
```

**2) 并发度低**

串行化强制事务串行执行,即使只是简单的读查询也要加锁,导致系统并发度极低。

```sql
-- 串行化级别下,简单的读查询也会加锁
SELECT * FROM users WHERE id = 1;  -- 加共享锁
SELECT * FROM users WHERE id = 2;  -- 被阻塞
```

**3) 容易死锁**

由于所有操作都加锁,串行化级别容易出现死锁。

```sql
-- 死锁示例

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- 持有id=1的锁

-- 事务B
START TRANSACTION;
SELECT * FROM account WHERE id = 2 FOR UPDATE;
-- 持有id=2的锁

-- 事务A尝试获取id=2
SELECT * FROM account WHERE id = 2 FOR UPDATE;
-- 被阻塞,等待事务B释放id=2

-- 事务B尝试获取id=1
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- 被阻塞,等待事务A释放id=1

-- 出现死锁!
```

**性能对比测试:**

```sql
-- 测试场景:1000个并发事务,每条事务执行1次查询

-- 读已提交级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
-- 吞吐量:约8000 TPS

-- 可重复读级别
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
-- 吞吐量:约7500 TPS (略低,但差别不大)

-- 串行化级别
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;
-- 吞吐量:约1000 TPS (性能下降87.5%)
```

**结论:**

串行化虽然能完全避免所有并发问题,但性能损失太大,不适合作为默认隔离级别。只有在极少数对一致性要求极高的场景(如银行转账、库存扣减)才使用。

#### MVCC如何保证可重复读的性能?

MVCC(Multi-Version Concurrency Control,多版本并发控制)是MySQL在可重复读级别下保持高性能的关键技术。

**MVCC的核心优势:**

**1) 读写不冲突**

```sql
-- 事务A:读操作
START TRANSACTION;
SELECT * FROM account WHERE id = 1;  -- 快照读,不加锁
-- 即使事务B在修改数据,事务A也不会被阻塞

-- 事务B:写操作
START TRANSACTION;
UPDATE account SET balance = 2000 WHERE id = 1;  -- 写操作加锁
-- 只会阻塞其他写操作,不会阻塞读操作
COMMIT;

-- 事务A继续读
SELECT * FROM account WHERE id = 1;  -- 读取的是事务开始时的快照
-- 不会被事务B的修改影响
COMMIT;
```

**2) Read View的复用**

在可重复读级别下,Read View只在事务开始时生成一次,整个事务期间都使用这个Read View。

```sql
-- 可重复读级别
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

START TRANSACTION;
-- 生成Read View,包含当前活跃事务列表

SELECT * FROM account WHERE id = 1;  -- 第一次查询,使用Read View
SELECT * FROM account WHERE id = 2;  -- 第二次查询,复用Read View
SELECT * FROM account WHERE id = 3;  -- 第三次查询,复用Read View
-- 整个事务只生成一次Read View,性能好

COMMIT;
```

而在读已提交级别下,每次SELECT都要生成新的Read View。

```sql
-- 读已提交级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

START TRANSACTION;

SELECT * FROM account WHERE id = 1;  -- 第一次查询,生成Read View
SELECT * FROM account WHERE id = 2;  -- 第二次查询,生成新的Read View
SELECT * FROM account WHERE id = 3;  -- 第三次查询,再生成新的Read View
-- 每次查询都要生成Read View,性能略差

COMMIT;
```

**3) Undo Log的链式结构**

Undo Log采用链式结构,不需要拷贝整个数据页,存储开销小。

```sql
-- Undo Log链表示意
当前版本:
  id=1, balance=2000, DB_TRX_ID=102, DB_ROLL_PTR → 上一个版本
         ↓
   上一个版本:
     id=1, balance=1500, DB_TRX_ID=101, DB_ROLL_PTR → 再上一个版本
            ↓
      再上一个版本:
        id=1, balance=1000, DB_TRX_ID=100, DB_ROLL_PTR → NULL

-- 查找旧版本时,顺着DB_ROLL_PTR指针遍历Undo Log链
-- 不需要拷贝整个数据页,性能好
```

**MVCC的性能优势对比:**

```sql
-- 测试场景:1000个并发事务,每个事务读取100条数据

-- 使用锁机制(如串行化级别)
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;
-- 每次读都要加锁,阻塞其他事务
-- 吞吐量:约1000 TPS

-- 使用MVCC(可重复读级别)
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
-- 快照读不加锁,读写不冲突
-- 吞吐量:约7500 TPS (性能提升7.5倍)
```

#### Next-Key Lock如何解决幻读?

虽然MySQL通过MVCC解决了快照读的幻读问题,但当前读(SELECT FOR UPDATE/UPDATE/DELETE)仍然可能出现幻读。InnoDB通过Next-Key Lock解决了这个问题。

**Next-Key Lock = Record Lock + Gap Lock**

- **Record Lock(记录锁)**:锁定索引记录本身
- **Gap Lock(间隙锁)**:锁定索引记录之间的间隙
- **Next-Key Lock**:锁定记录和其前面的间隙,即开区间

**Next-Key Lock解决幻读的原理:**

```sql
-- 可重复读级别
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 创建测试表
CREATE TABLE account (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    balance INT
);

INSERT INTO account VALUES (10, '张三', 1000), (20, '李四', 2000), (30, '王五', 3000);

-- 事务A:当前读,锁定范围
START TRANSACTION;
SELECT * FROM account WHERE id BETWEEN 15 AND 25 FOR UPDATE;
-- Next-Key Lock锁定: (10, 20] 和 (20, 30)
-- 防止其他事务插入id在15-25之间的记录

-- 事务B:尝试插入
START TRANSACTION;
INSERT INTO account VALUES (15, '赵六', 1500);  -- 被阻塞,15在(10,20]范围内
INSERT INTO account VALUES (25, '钱七', 2500);  -- 被阻塞,25在(20,30)范围内
COMMIT;

-- 事务A:再次查询
SELECT * FROM account WHERE id BETWEEN 15 AND 25 FOR UPDATE;
-- 不会出现幻读,查询结果还是0条
COMMIT;
```

**Next-Key Lock的加锁规则:**

**1) 等值查询,记录存在**

```sql
-- id=20存在
SELECT * FROM account WHERE id = 20 FOR UPDATE;
-- 加锁: (10, 20]
-- 包含20这个记录,以及10到20之间的间隙
```

**2) 等值查询,记录不存在**

```sql
-- id=15不存在
SELECT * FROM account WHERE id = 15 FOR UPDATE;
-- 加锁: (10, 20)
-- 只锁定间隙,不包含记录(因为记录不存在)
```

**3) 范围查询**

```sql
-- 范围查询
SELECT * FROM account WHERE id >= 20 AND id <= 30 FOR UPDATE;
-- 加锁: (10, 30]
-- Next-Key Lock(20) + Next-Key Lock(30)
```

**Next-Key Lock vs Gap Lock vs Record Lock:**

| 锁类型 | 锁定范围 | 作用 | 是否阻塞插入 |
|--------|---------|------|------------|
| Record Lock | 索引记录 | 锁定单个记录 | 否 |
| Gap Lock | 索引之间的间隙 | 防止其他事务插入 | 是 |
| Next-Key Lock | 记录+前面的间隙 | 防止幻读 | 是 |

#### 不同业务场景下的隔离级别选择

虽然MySQL默认使用可重复读,但某些业务场景可能需要调整隔离级别。

**场景1:电商订单系统 → 可重复读**

```sql
-- 推荐使用默认的可重复读
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 原因:
-- 1. 订单详情查询需要事务内数据一致
-- 2. 用户希望看到稳定的订单信息
-- 3. 避免不可重复读导致的前端数据跳动

-- 示例
START TRANSACTION;
SELECT * FROM orders WHERE order_id = 1001;
-- 第一次查询:状态=待支付
-- 其他事务可能修改订单状态
SELECT * FROM orders WHERE order_id = 1001;
-- 第二次查询:状态=待支付(可重复读)
-- 前端展示的订单状态不会跳动
COMMIT;
```

**场景2:电商秒杀系统 → 读已提交**

```sql
-- 推荐使用读已提交
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 原因:
-- 1. 秒杀需要看到最新的库存数据
-- 2. 避免用户抢到重复商品
-- 3. 性能比可重复读更好

-- 配合Redis分布式锁
SETNX lock:product:100 1
-- 获取锁后执行扣库存
UPDATE product SET stock = stock - 1 WHERE id = 100 AND stock > 0;
DEL lock:product:100
```

**场景3:银行转账系统 → 可重复读 + SELECT FOR UPDATE**

```sql
-- 使用可重复读级别
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 关键操作使用SELECT FOR UPDATE
START TRANSACTION;
SELECT balance FROM account WHERE id = 1 FOR UPDATE;
-- 锁定账户,防止其他事务修改
UPDATE account SET balance = balance - 100 WHERE id = 1;
UPDATE account SET balance = balance + 100 WHERE id = 2;
COMMIT;
```

**场景4:日志分析系统 → 读未提交**

```sql
-- 可以使用读未提交
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;

-- 原因:
-- 1. 日志分析不需要绝对准确的数据
-- 2. 可以接受未提交的数据
-- 3. 性能最好

SELECT COUNT(*) FROM logs WHERE level = 'ERROR';
-- 统计错误日志,即使有些未提交的ERROR日志也被统计
```

**场景5:实时监控系统 → 读已提交**

```sql
-- 推荐使用读已提交
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 原因:
-- 1. 监控需要看到最新的数据
-- 2. 不要求事务内数据一致
-- 3. 性能好

SELECT AVG(response_time) FROM api_logs WHERE create_time > NOW() - INTERVAL 1 MINUTE;
-- 统计最近1分钟的平均响应时间
-- 需要看到最新的请求记录
```

### 💬 面试官追问

#### 提问1:把隔离级别从可重复读改成读已提交,需要做什么配置?

**回答:**

两步。一是改transaction_isolation参数,可以在my.cnf 里配,也可以 SET GLOBAL动态改。二是把binlog_format 改成ROW,不然还是有主从不一致风险。改完最好重启一下让所有连接生效。

**配置示例:**

```sql
-- 方式1:全局修改(需要SUPER权限)
SET GLOBAL transaction_isolation = 'READ-COMMITTED';
SET GLOBAL binlog_format = 'ROW';

-- 方式2:在my.cnf配置文件中修改
[mysqld]
transaction_isolation = READ-COMMITTED
binlog_format = ROW

-- 方式3:只修改当前会话
SET SESSION transaction_isolation = 'READ-COMMITTED';
```

**注意事项:**

1) **修改binlog格式前需要备份**：从statement格式改row格式后,binlog文件会变大,空间占用增加

2) **重启生效**：修改全局参数后,新的连接会使用新隔离级别,但已建立的连接不会改变

3) **测试验证**：改完后要测试并发场景,确保业务逻辑正常

---

#### 提问2:可重复读级别下用row格式binlog,还会有主从不一致问题吗?

**回答:**

不会。row格式记录的是每一行的变更,不是SQL语句。不管事务提交顺序怎样,每一行的最终状态是确定的,从库重放结果肯定和主库一样。

**原理对比:**

```sql
-- statement格式(可能导致主从不一致)
-- binlog记录: DELETE FROM t WHERE id < 10
-- 从库执行时如果数据状态和主库不一致,结果就错了

-- row格式(保证主从一致)
-- binlog记录具体行变更:
-- DELETE FROM t WHERE id = 1
-- DELETE FROM t WHERE id = 5
-- DELETE FROM t WHERE id = 9
-- 从库执行同样的变更,结果一定一致
```

**row格式的优势:**

1) **准确性**：记录每一行的具体变更,从库重放结果完全一致

2) **安全性**：避免statement格式的函数调用、存储过程等不确定性问题

3) **完整性**：包含before和after镜像,支持数据恢复

**row格式的缺点:**

1) **空间占用大**：每行变更都记录,binlog文件会明显增大

2) **无法审计SQL**：看不到执行的SQL语句,只看得到数据变化

3) **性能略低**：生成binlog的开销比statement格式大

#### 提问3:为什么MySQL不把默认值改成读已提交?

**回答:**

向前兼容。很多老系统依赖可重复读的行为,贸然改默认值会出问题。MySQL官方的态度是让用户自己根据场景选择,不替用户做决定。

**兼容性考虑:**

```sql
-- 老系统可能依赖可重复读的行为
START TRANSACTION;
SELECT * FROM orders WHERE status = 'pending';
-- 业务逻辑假设订单数量不变
-- 如果改成读已提交,订单数可能变化,导致逻辑错误
COMMIT;
```

**实际案例:**

**场景1:报表系统**

```sql
-- 假设报表生成依赖可重复读
-- 第一次查询订单数
SELECT COUNT(*) FROM orders WHERE create_time >= '2026-01-01';
-- 假设返回1000条

-- 第二次查询订单金额
SELECT SUM(amount) FROM orders WHERE create_time >= '2026-01-01';
-- 如果用读已提交,可能基于1005条订单计算,导致数据不一致
```

**场景2:库存校验**

```sql
-- 假设代码逻辑
START TRANSACTION;
SELECT stock FROM product WHERE id = 100;  -- 假设读取到10

-- 业务逻辑基于stock=10做判断
IF stock >= 5 THEN
    -- 执行某些操作
END IF;

-- 期间其他事务扣减库存
-- 如果用读已提交,第二次读取stock=5
-- 但业务逻辑已经基于10判断了,可能出错
```

**MySQL的设计哲学:**

1) **保守的默认值**：默认值偏向于更安全的选项,避免破坏现有系统

2) **用户自主选择**：提供灵活性,让用户根据业务场景选择合适的隔离级别

3) **文档和社区支持**：通过文档和社区指导用户如何选择,而不是强制一个默认值

**现代建议:**

```sql
-- 新项目推荐配置
[mysqld]
transaction_isolation = READ-COMMITTED
binlog_format = ROW

-- 原因:
-- 1. RC级别避免间隙锁,减少死锁,性能更好
-- 2. ROW格式保证主从一致性
-- 3. 大多数业务不需要事务内数据绝对一致
```

**总结:**

MySQL默认RR是为了历史兼容,不意味着RR总是最优选择。现代互联网项目更多使用RC+ROW的组合,以获得更好的性能和更少的锁竞争。关键是根据业务场景做出合理选择,而不是盲目使用默认值。

## 📝 总结

**MySQL默认事务隔离级别是可重复读(REPEATABLE READ)**,选择这个级别的原因:

1) **避免脏读和不可重复读**:解决最常见的并发问题

2) **MVCC实现高性能**:快照读不加锁,读写不冲突

3) **Next-Key Lock解决幻读**:基本解决幻读问题

4) **平衡一致性和性能**:比读已提交更一致,比串行化更高效

**各隔离级别适用场景:**

- **90%场景**:使用MySQL默认的可重复读
- **少数高性能场景**:使用读已提交(如秒杀系统)
- **极少数特殊场景**:使用读未提交或串行化

**最佳实践:**

1. 使用默认的可重复读,不需要额外配置

2. 关键操作使用SELECT FOR UPDATE锁定记录

3. 避免长事务,防止Undo Log链过长

4. 定期监控死锁和锁等待,优化SQL和索引

5. 从Oracle等数据库迁移时,需要评估业务对不可重复读的容忍度

## 3. 数据库的脏读、不可重复读和幻读分别是什么?

### 📌 回答重点

脏读、不可重复读、幻读是数据库事务并发访问时可能出现的三个问题。

**1) 脏读(Dirty Read)**

一个事务读取到了另一个事务**未提交**的数据。如果那个事务回滚了,读取到的数据就是无效的。

**2) 不可重复读(Non-repeatable Read)**

在同一个事务内,两次读取同一数据得到的结果不一样。因为期间有其他事务修改了该数据并提交。

**3) 幻读(Phantom Read)**

在同一个事务内,前后两次执行相同的查询,返回的记录数不一样。因为期间有其他事务插入或删除了数据并提交。

**三者对比表格:**

| 问题类型 | 定义 | 触发条件 | 出现隔离级别 | 危害 |
|---------|------|---------|-------------|------|
| 脏读 | 读取到未提交的数据 | 读未提交隔离级别 | READ UNCOMMITTED | 数据错误,违反原子性 |
| 不可重复读 | 同一行数据两次读取值不一样 | 事务内两次读取同一数据 | READ COMMITTED | 业务逻辑错误,数据不一致 |
| 幻读 | 范围查询两次记录数不一样 | 事务内两次范围查询 | READ COMMITTED、部分REPEATABLE READ | 统计错误,数据集不一致 |

**示意图:**

```
事务A                               事务B
----------------------------------  ----------------------------------
START TRANSACTION;                  START TRANSACTION;
SELECT balance = 1000;             UPDATE balance = 2000;
(读取到事务B未提交的修改→脏读)     COMMIT;
                                    (事务B回滚,balance变成1000)
SELECT balance = 2000;             INSERT INTO account VALUES (...);
(读取的值与实际不一致)             COMMIT;

SELECT * FROM account;
(多了一条记录→幻读)
COMMIT;
```

### 💬 面试官追问

#### 提问1:不可重复读和幻读都是两次查询结果不一样,区别到底在哪?

**回答:**

关键看变化的是行内容还是行数量。不可重复读是同一行数据的某个字段值变了，由UPDATE或DELETE引起。幻读是符合查询条件的行数变了，由INSERT引起。从解决方案看，不可重复读靠MVCC快照就能解决，幻读还得加间隙锁才能防住。

**核心区别:**

| 特性 | 不可重复读 | 幻读 |
|------|-----------|------|
| 变化对象 | 同一行的**值** | 数据集的**数量** |
| 触发操作 | UPDATE/DELETE | INSERT |
| SQL类型 | 等值查询 | 范围查询 |
| 解决方案 | MVCC快照 | MVCC + 间隙锁 |

**示例对比:**

```sql
-- 不可重复读示例
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 第一次:1000

-- 其他事务
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;

SELECT balance FROM account WHERE id = 1;  -- 第二次:2000 (值变了)
-- 不可重复读:同一行的balance从1000变成2000
```

```sql
-- 幻读示例
START TRANSACTION;
SELECT * FROM orders WHERE user_id = 100;  -- 第一次:5条记录

-- 其他事务
INSERT INTO orders (user_id, amount) VALUES (100, 500);
COMMIT;

SELECT * FROM orders WHERE user_id = 100;  -- 第二次:6条记录 (数量变了)
-- 幻读:从5条变成6条，多了1行
```

**解决方案对比:**

```sql
-- 不可重复读:靠MVCC快照就能解决
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 使用Read View1

-- 其他事务修改并提交
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;

SELECT balance FROM account WHERE id = 1;  -- 仍使用Read View1,读到1000
-- MVCC快照保证了同一事务内数据一致
```

```sql
-- 幻读:需要MVCC + 间隙锁
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 快照读:靠MVCC解决幻读
START TRANSACTION;
SELECT * FROM orders WHERE user_id = 100;  -- 使用Read View,读到5条

-- 其他事务插入
INSERT INTO orders (user_id, amount) VALUES (100, 500);
COMMIT;

SELECT * FROM orders WHERE user_id = 100;  -- 仍使用Read View,读到5条
-- MVCC快照读看不到新插入的数据

-- 当前读:需要间隙锁
SELECT * FROM orders WHERE user_id = 100 FOR UPDATE;
-- Next-Key Lock锁定范围(user_id=100)
-- 防止其他事务插入user_id=100的新记录
```

**关键点:**

- **不可重复读关注的是"值的变化"**，MVCC通过Read View保证同一事务内看到的数据版本一致

- **幻读关注的是"数量的变化"**，快照读靠MVCC解决，当前读(SELECT...FOR UPDATE)需要间隙锁防止其他事务插入

---

#### 提问2:读已提交级别为什么解决不了不可重复读?

**回答:**

读已提交级别下每次SELECT都生成新的ReadView。事务B修改数据提交后，事务A第二次查询会拿到新的ReadView，这时候事务B的修改已经提交了，对事务A可见，所以读到的值变了。可重复读级别复用第一次的ReadView，事务B提交了也看不到。

**原理对比:**

```sql
-- 读已提交级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 事务A
START TRANSACTION;

-- 第一次SELECT:生成ReadView1
SELECT balance FROM account WHERE id = 1;  -- ReadView1:读到1000

-- 事务B修改并提交
START TRANSACTION;
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;  -- 事务B提交后,数据对其他事务可见

-- 事务A第二次SELECT:生成ReadView2(新!)
SELECT balance FROM account WHERE id = 1;  -- ReadView2:读到2000
-- ReadView2能看到事务B提交的数据
-- 所以值变了 → 不可重复读

COMMIT;
```

```sql
-- 可重复读级别
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 事务A
START TRANSACTION;

-- 第一次SELECT:生成ReadView
SELECT balance FROM account WHERE id = 1;  -- ReadView:读到1000

-- 事务B修改并提交
START TRANSACTION;
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;  -- 事务B提交了

-- 事务A第二次SELECT:复用第一次的ReadView
SELECT balance FROM account WHERE id = 1;  -- ReadView:还是读到1000
-- ReadView在事务开始时生成,整个事务期间复用
-- 看不到事务B的修改 → 可重复读

COMMIT;
```

**Read View机制:**

**Read View包含了什么?**

```
Read View结构:
- m_ids: 生成Read View时,所有活跃事务的ID列表
- min_trx_id: 活跃事务中最小的ID
- max_trx_id: 生成Read View时,系统应该分配的下一个事务ID
- creator_trx_id: 创建Read View的事务ID
```

**可见性判断规则:**

```sql
-- 事务A的Read View判断规则
-- 当事务A读取某行数据时,检查该行的事务ID:

1. 如果 row.trx_id < min_trx_id:
   → 可见 (该事务在Read View生成前已提交)

2. 如果 row.trx_id > max_trx_id:
   → 不可见 (该事务在Read View生成后才开始)

3. 如果 min_trx_id <= row.trx_id <= max_trx_id:
   → 检查row.trx_id是否在m_ids中
      - 不在m_ids中 → 可见 (该事务已提交)
      - 在m_ids中 → 不可见 (该事务还未提交)
```

**读已提交 vs 可重复读的核心区别:**

| 特性 | 读已提交 | 可重复读 |
|------|---------|---------|
| Read View生成时机 | **每次SELECT都生成** | **事务开始时生成一次** |
| Read View复用 | 不复用,每次新的 | 整个事务期间复用 |
| 是否看到其他事务的修改 | 能看到已提交的 | 看不到事务开始后提交的 |

**实际案例:**

```sql
-- 银行转账场景(读已提交级别)
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 事务A
START TRANSACTION;

-- 第一步:查询余额
SELECT balance FROM account WHERE id = 1;  -- 1000元

-- 第二步:判断余额是否充足
-- 假设代码逻辑:if balance >= 500,执行转账500元

-- 第三步:其他事务扣款100元
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1;
COMMIT;  -- balance变成900

-- 第四步:再次查询余额
SELECT balance FROM account WHERE id = 1;  -- 900元 (值变了!)
-- Read View2能看到事务B的修改

-- 第五步:执行转账500元
UPDATE account SET balance = balance - 500 WHERE id = 1;
-- balance变成400,但代码认为有1000元
-- 导致余额计算错误

COMMIT;
```

```sql
-- 可重复读级别
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 事务A
START TRANSACTION;
-- 生成Read View

SELECT balance FROM account WHERE id = 1;  -- 1000元 (Read View)

-- 其他事务修改
UPDATE account SET balance = balance - 100 WHERE id = 1;
COMMIT;

SELECT balance FROM account WHERE id = 1;  -- 1000元 (还是Read View)
-- 看不到事务B的修改

UPDATE account SET balance = balance - 500 WHERE id = 1;
-- 从Read View的1000减去500,变成500
-- 逻辑正确

COMMIT;
```

**总结:**

- **读已提交**:每次SELECT生成新ReadView,能看到已提交的数据 → 出现不可重复读

- **可重复读**:事务开始时生成ReadView,整个事务复用 → 看不到其他事务的修改 → 解决不可重复读

#### 提问3:既然MVCC能解决幻读,为什么还需要间隙锁?

**回答:**

MVCC只能解决快照读的幻读，当前读还是能看到最新数据。比如SELECT...FOR UPDATE，必须读最新版本才能正确加锁。这时候要防止别的事务插入，就得靠间隙锁把范围锁住。

**快照读 vs 当前读:**

```sql
-- 快照读:普通的SELECT,使用MVCC,不需要加锁
START TRANSACTION;

SELECT * FROM orders WHERE user_id = 100;  -- 快照读
-- 使用Read View,读取历史版本
-- 不会加锁,其他事务可以并发插入

-- 其他事务
INSERT INTO orders (user_id, amount) VALUES (100, 500);
COMMIT;  -- 成功插入

SELECT * FROM orders WHERE user_id = 100;  -- 快照读
-- 仍使用Read View,看不到新插入的数据
-- MVCC解决了幻读

COMMIT;
```

```sql
-- 当前读:SELECT...FOR UPDATE,必须读最新版本并加锁
START TRANSACTION;

SELECT * FROM orders WHERE user_id = 100 FOR UPDATE;  -- 当前读
-- 读取最新版本,并加行锁
-- 为了防止幻读,还需要加间隙锁

-- Next-Key Lock = 记录锁 + 间隙锁
-- 锁定:user_id = 100的记录 + (99, 100)的间隙

-- 其他事务尝试插入
INSERT INTO orders (user_id, amount) VALUES (100, 500);
-- 被阻塞!间隙锁锁住了user_id=100的范围
-- 等待事务A释放锁

COMMIT;

-- 事务A释放锁后,事务B才能插入
```

**为什么当前读需要间隙锁?**

```sql
-- 假设没有间隙锁,只有行锁
START TRANSACTION;

SELECT * FROM orders WHERE user_id = 100 FOR UPDATE;
-- 只加行锁,锁住user_id=100的已有记录

-- 其他事务
INSERT INTO orders (user_id, amount) VALUES (100, 500);
-- 成功!因为user_id=100是新的,没有行锁

-- 事务A
SELECT * FROM orders WHERE user_id = 100 FOR UPDATE;
-- 又看到新插入的记录!
-- 幻读!

COMMIT;
```

**间隙锁的工作原理:**

```sql
-- Next-Key Lock结构
-- Next-Key Lock = 记录锁(Record Lock) + 间隙锁(Gap Lock)

-- 示例:表中user_id有1, 5, 10三条记录

-- 执行:
SELECT * FROM orders WHERE user_id = 5 FOR UPDATE;

-- Next-Key Lock锁定:
-- (1, 5]  : user_id=5的记录锁 + (1,5)的间隙锁

-- 防止插入:
INSERT INTO orders (user_id, amount) VALUES (2, 100);   -- 被阻塞 (1,5)间隙
INSERT INTO orders (user_id, amount) VALUES (3, 100);   -- 被阻塞 (1,5)间隙
INSERT INTO orders (user_id, amount) VALUES (4, 100);   -- 被阻塞 (1,5)间隙
UPDATE orders SET user_id = 6 WHERE id = 1;             -- 被阻塞 (1,5)间隙
```

**实际案例:**

```sql
-- 场景:库存管理
CREATE TABLE inventory (
    id INT PRIMARY KEY,
    product_id INT,
    quantity INT,
    INDEX idx_product (product_id)
);

INSERT INTO inventory VALUES (1, 100, 10), (2, 101, 5);

-- 事务A:查询product_id=100的库存并锁定
START TRANSACTION;
SELECT * FROM inventory WHERE product_id = 100 FOR UPDATE;
-- Next-Key Lock锁定:
-- (NULL, 100] 间隙 + product_id=100的行锁

-- 事务B:尝试插入新的product_id=100的记录
START TRANSACTION;
INSERT INTO inventory (id, product_id, quantity) VALUES (3, 100, 20);
-- 被阻塞!间隙锁锁住了(NULL,100)
-- 防止幻读

-- 事务B:尝试插入product_id=99的记录
INSERT INTO inventory (id, product_id, quantity) VALUES (4, 99, 30);
-- 成功!不在这个间隙范围内

-- 事务A:再次查询
SELECT * FROM inventory WHERE product_id = 100;
-- 不会出现新记录,不会幻读

COMMIT;  -- 释放锁

-- 事务B:执行插入
-- 成功插入
```

**快照读和当前读的对比:**

| 特性 | 快照读 | 当前读 |
|------|--------|--------|
| 语法 | SELECT | SELECT...FOR UPDATE / UPDATE / DELETE |
| 数据版本 | 历史版本(Read View) | 最新版本 |
| 是否加锁 | 不加锁 | 加锁 |
| 解决幻读方式 | MVCC Read View | MVCC + 间隙锁 |
| 适用场景 | 查询、报表 | 数据修改、业务逻辑 |

**最佳实践:**

```sql
-- 1. 查询场景:使用快照读,MVCC解决幻读
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

START TRANSACTION;
SELECT * FROM orders WHERE user_id = 100;  -- 快照读
-- MVCC保证看不到新插入的数据
COMMIT;

-- 2. 业务逻辑场景:使用当前读,间隙锁防止幻读
START TRANSACTION;

-- 第一步:读取并锁定
SELECT * FROM inventory WHERE product_id = 100 FOR UPDATE;  -- 当前读
-- Next-Key Lock防止其他事务插入

-- 第二步:业务逻辑
IF quantity > 0 THEN
    UPDATE inventory SET quantity = quantity - 1 WHERE id = 1;
END IF;

COMMIT;
```

**总结:**

- **MVCC快照读**:通过Read View实现,不加锁,解决了快照读的幻读

- **间隙锁**:用于当前读(SELECT...FOR UPDATE),锁住范围,防止其他事务插入,解决当前读的幻读

- **两者结合**:MySQL通过MVCC(快照读) + Next-Key Lock(当前读)完全解决幻读问题

## 📝 总结

**1. MySQL中的事务隔离级别**

MySQL中有四种事务隔离级别,从低到高依次是:

1) **读未提交(READ UNCOMMITTED)**:可以读取到其他事务未提交的数据,会出现脏读、不可重复读、幻读问题。

2) **读已提交(READ COMMITTED)**:只能读取到其他事务已提交的数据,解决了脏读,但会出现不可重复读和幻读。

3) **可重复读(REPEATABLE READ)**:MySQL的默认隔离级别,保证同一事务内多次读取同一数据的结果一致,解决了脏读和不可重复读,但可能出现幻读。MySQL InnoDB通过MVCC(多版本并发控制)和Next-Key Lock解决了幻读。

4) **串行化(SERIALIZABLE)**:最高隔离级别,强制事务串行执行,解决了所有并发问题,但性能最差。

---

**2. MySQL默认的事务隔离级别**

MySQL(InnoDB存储引擎)的默认事务隔离级别是**可重复读(REPEATABLE READ)**。

**为什么选择可重复读作为默认级别?**

1) **避免脏读和不可重复读**:解决最常见的并发问题

2) **通过MVCC实现高性能**:快照读不加锁,读写不冲突

3) **通过Next-Key Lock基本解决幻读**:基本解决幻读问题

4) **平衡一致性和性能**:比读已提交更一致,比串行化更高效

**各数据库默认隔离级别对比:**

| 数据库 | 默认隔离级别 | 说明 |
|--------|-------------|------|
| MySQL(InnoDB) | REPEATABLE READ | 通过MVCC和Next-Key Lock基本解决幻读 |
| Oracle | READ COMMITTED | 不支持REPEATABLE READ级别 |
| SQL Server | READ COMMITTED | 支持快照隔离 |
| PostgreSQL | READ COMMITTED | 可通过配置改为SERIALIZABLE |

---

**3. 数据库的脏读、不可重复读和幻读**

脏读、不可重复读、幻读是数据库事务并发访问时可能出现的三个问题。

**脏读(Dirty Read)**:一个事务读取到了另一个事务未提交的数据,如果该事务回滚,读取到的数据就是无效的。

**不可重复读(Non-repeatable Read)**:在同一个事务内,两次读取同一数据得到的结果不一样,因为期间有其他事务修改了该数据并提交。

**幻读(Phantom Read)**:在同一个事务内,前后两次执行相同的查询,返回的记录数不一样,因为期间有其他事务插入或删除了数据并提交。

**三者对比:**

| 问题类型 | 定义 | 触发条件 | 最小隔离级别 | 危害 |
|---------|------|---------|-------------|------|
| 脏读 | 读取到未提交的数据 | 读未提交隔离级别 | READ COMMITTED | 数据错误,违反原子性 |
| 不可重复读 | 同一行数据两次读取值不一样 | 事务内两次读取同一数据 | REPEATABLE READ | 业务逻辑错误,数据不一致 |
| 幻读 | 范围查询两次记录数不一样 | 事务内两次范围查询 | REPEATABLE READ(需Next-Key Lock) | 统计错误,数据集不一致 |

---

**核心知识点:**

- **MVCC**:通过Read View和Undo Log实现读不阻塞写,可重复读级别下Read View只在事务开始时生成一次
- **Next-Key Lock**:Record Lock + Gap Lock,解决幻读问题
- **快照读 vs 当前读**:不加锁读 vs 加锁读最新数据
- **隔离级别选择**:90%场景使用REPEATABLE READ,少数场景使用READ COMMITTED
- **MySQL vs Oracle**:MySQL默认RR级别避免不可重复读,Oracle默认RC级别会出现不可重复读
- **从Oracle迁移**:需要评估业务对不可重复读的容忍度,可能需要调整业务逻辑或使用SELECT FOR UPDATE
- **脏读 vs 不可重复读 vs 幻读**:
  - 脏读:读取未提交的数据
  - 不可重复读:同一行数据,两次读取值不一样
  - 幻读:相同查询,两次返回的记录数不一样
- **解决并发问题**:
  - 脏读:使用READ COMMITTED及以上级别
  - 不可重复读:使用REPEATABLE READ及以上级别
  - 幻读:使用REPEATABLE READ + Next-Key Lock,或SERIALIZABLE
