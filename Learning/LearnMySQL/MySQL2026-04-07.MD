## 1. MySQL中的事务隔离级别有哪些?

### 📌 回答重点

MySQL中有四种事务隔离级别,从低到高依次是:

1) **读未提交(READ UNCOMMITTED)**:可以读取到其他事务未提交的数据,会出现脏读、不可重复读、幻读问题。

2) **读已提交(READ COMMITTED)**:只能读取到其他事务已提交的数据,解决了脏读,但会出现不可重复读和幻读。

3) **可重复读(REPEATABLE READ)**:MySQL的默认隔离级别,保证同一事务内多次读取同一数据的结果一致,解决了脏读和不可重复读,但可能出现幻读。MySQL InnoDB通过MVCC(多版本并发控制)和Next-Key Lock解决了幻读。

4) **串行化(SERIALIZABLE)**:最高隔离级别,强制事务串行执行,解决了所有并发问题,但性能最差。

**隔离级别对比表格:**

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 性能 |
|---------|------|-----------|------|------|
| 读未提交 | ✅ 会出现 | ✅ 会出现 | ✅ 会出现 | ⭐⭐⭐⭐⭐ |
| 读已提交 | ❌ 解决 | ✅ 会出现 | ✅ 会出现 | ⭐⭐⭐⭐ |
| 可重复读 | ❌ 解决 | ❌ 解决 | ❌ 基本解决 | ⭐⭐⭐ |
| 串行化 | ❌ 解决 | ❌ 解决 | ❌ 解决 | ⭐ |

**查看和设置隔离级别:**

```sql
-- 查看当前隔离级别
SELECT @@transaction_isolation;
-- 或
SHOW VARIABLES LIKE 'transaction_isolation';

-- 设置全局隔离级别
SET GLOBAL TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 设置当前会话的隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 在开启事务时设置隔离级别
START TRANSACTION ISOLATION LEVEL SERIALIZABLE;
```

### 📚 扩展知识

#### 各种并发问题的含义

**脏读(Dirty Read)**:

一个事务读取到了另一个事务未提交的数据。如果那个事务回滚了,读取到的数据就是无效的。

```sql
-- 事务A
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1;
-- 此时事务未提交

-- 事务B
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 读取到了事务A未提交的修改(脏读)
-- 如果事务A回滚,事务B读取到的数据就是错误的
```

**不可重复读(Non-repeatable Read)**:

在同一个事务内,两次读取同一数据得到的结果不一样。因为期间有其他事务修改了该数据并提交。

```sql
-- 事务A
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 第一次读取: 1000

-- 事务B
START TRANSACTION;
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;

-- 事务A
SELECT balance FROM account WHERE id = 1;  -- 第二次读取: 2000(不可重复读)
```

**幻读(Phantom Read)**:

在同一个事务内,前后两次查询返回的记录数不一样。因为期间有其他事务插入或删除了数据并提交。

```sql
-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE age > 18;  -- 第一次查询: 返回10条记录

-- 事务B
START TRANSACTION;
INSERT INTO account (name, age) VALUES ('张三', 20);
COMMIT;

-- 事务A
SELECT * FROM account WHERE age > 18;  -- 第二次查询: 返回11条记录(幻读)
```

#### 各隔离级别的实现原理

**读未提交(READ UNCOMMITTED)**:

最宽松的隔离级别,不加任何锁,直接读取最新数据。

- 优点:性能最好
- 缺点:数据一致性最差,几乎不使用

---

**读已提交(READ COMMITTED)**:

Oracle、SQL Server等数据库的默认隔离级别。

每次SELECT语句都会生成一个新的Read View,只能读取到已提交的数据。

- 实现方式:每次SELECT生成新的Read View
- 解决问题:脏读
- 未解决问题:不可重复读、幻读

**演示不可重复读:**

```sql
-- 查看隔离级别
SELECT @@transaction_isolation;  -- READ-COMMITTED

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 1;  -- 第一次读取: balance=1000

-- 事务B
START TRANSACTION;
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;

-- 事务A
SELECT * FROM account WHERE id = 1;  -- 第二次读取: balance=2000(不可重复读)
COMMIT;
```

---

**可重复读(REPEATABLE READ)**:

MySQL InnoDB的默认隔离级别。

事务开始时生成一个Read View,整个事务期间都使用这个Read View读取数据,保证同一事务内多次读取结果一致。

- 实现方式:MVCC(多版本并发控制),事务开始生成Read View
- 解决问题:脏读、不可重复读
- 幻读问题:通过Next-Key Lock(临键锁)解决

**MVCC原理:**

MVCC(Multi-Version Concurrency Control)通过保存数据的历史版本来实现不加锁的读操作。

```sql
-- InnoDB每行数据隐藏的字段
-- DB_TRX_ID: 事务ID,最近修改这行数据的事务ID
-- DB_ROLL_PTR: 回滚指针,指向undo log中该行的上一个版本
-- DB_ROW_ID: 行ID(隐藏主键)
```

**Read View机制:**

Read View包含当前活跃的事务ID列表,判断数据版本是否可见:

```
1. 如果事务ID < Read View的最小活跃事务ID → 可见(已提交)
2. 如果事务ID > Read View的最大事务ID → 不可见(未开始的事务)
3. 如果在活跃事务列表中 → 不可见(未提交)
4. 如果不在活跃事务列表中 → 可见(已提交)
```

**Next-Key Lock解决幻读:**

幻读的本质是范围查询时,其他事务插入了新数据。InnoDB通过Next-Key Lock(记录锁+间隙锁)锁定整个范围,防止其他事务插入。

```sql
-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id BETWEEN 10 AND 20 FOR UPDATE;
-- Next-Key Lock锁定: (10,20]范围,防止其他事务插入id在10-20之间的记录

-- 事务B
START TRANSACTION;
INSERT INTO account (id, name) VALUES (15, '张三');  -- 被阻塞
COMMIT;

-- 事务A
SELECT * FROM account WHERE id BETWEEN 10 AND 20;  -- 不会出现幻读
COMMIT;
```

**演示可重复读:**

```sql
-- 设置隔离级别为可重复读
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 1;  -- 第一次读取: balance=1000

-- 事务B
START TRANSACTION;
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;

-- 事务A
SELECT * FROM account WHERE id = 1;  -- 第二次读取: balance=1000(可重复读)
-- 看到的是事务开始时的快照,不受其他事务提交的影响
COMMIT;
```

---

**串行化(SERIALIZABLE)**:

最高隔离级别,强制事务串行执行,所有SELECT语句都隐式添加锁,或者读写分离。

- 实现方式:读写都加锁,或者只读快照
- 解决问题:所有并发问题
- 性能:最差,几乎不使用

**演示串行化:**

```sql
-- 设置隔离级别为串行化
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 1;  -- 隐式加共享锁
-- 事务A持有锁,事务B会被阻塞

-- 事务B
START TRANSACTION;
UPDATE account SET balance = 2000 WHERE id = 1;  -- 被阻塞,等待事务A释放锁
-- 直到事务A提交或回滚,事务B才能执行
```

---

#### 不同隔离级别的适用场景

**读未提交(READ UNCOMMITTED)**:

几乎不使用,仅用于极少数特殊场景,如数据一致性要求不高的统计分析。

```sql
-- 示例:实时统计,可以接受数据不准确
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
SELECT COUNT(*) FROM logs WHERE level = 'ERROR';  -- 即使未提交的ERROR日志也被统计
```

---

**读已提交(READ COMMITTED)**:

适用于大多数OLTP(在线事务处理)系统,如电商、金融等。

```sql
-- 示例:银行转账系统
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 事务A:查询余额
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 只能读到已提交的转账记录
```

**优点:**
- 避免脏读,数据一致性较好
- 性能相对较好

**缺点:**
- 会出现不可重复读和幻读

---

**可重复读(REPEATABLE READ)**:

MySQL默认隔离级别,适用于大多数业务场景。

```sql
-- 示例:订单查询
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 事务A
START TRANSACTION;
SELECT * FROM orders WHERE user_id = 100;  -- 第一次查询:10条订单
-- ... 中间可能有其他事务插入新订单 ...
SELECT * FROM orders WHERE user_id = 100;  -- 第二次查询:还是10条订单(不会幻读)
COMMIT;
```

**优点:**
- 解决脏读和不可重复读
- InnoDB基本解决了幻读
- 性能较好

**缺点:**
- 可能出现死锁(因为使用了Next-Key Lock)
- 读写并发性能略低于读已提交

---

**串行化(SERIALIZABLE)**:

极少使用,只用于对数据一致性要求极高的场景。

```sql
-- 示例:库存扣减,不允许并发
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- 事务A
START TRANSACTION;
SELECT stock FROM product WHERE id = 1 FOR UPDATE;  -- 锁定记录
UPDATE product SET stock = stock - 1 WHERE id = 1;
COMMIT;

-- 事务B会被阻塞,直到事务A完成
```

**优点:**
- 绝对的数据一致性

**缺点:**
- 性能极差,吞吐量低
- 容易出现死锁

---

#### MySQL InnoDB vs 其他数据库的默认隔离级别

**默认隔离级别对比:**

| 数据库 | 默认隔离级别 | 说明 |
|--------|-------------|------|
| MySQL(InnoDB) | REPEATABLE READ | 通过MVCC和Next-Key Lock解决幻读 |
| Oracle | READ COMMITTED | 不支持REPEATABLE READ |
| SQL Server | READ COMMITTED | 支持快照隔离 |
| PostgreSQL | READ COMMITTED | 可通过配置改为SERIALIZABLE |

**MySQL(InnoDB)的特殊之处:**

InnoDB在可重复读级别下,通过Next-Key Lock基本解决了幻读问题,这是MySQL相比其他数据库的优势。

```sql
-- MySQL演示:REPEATABLE READ级别下不会出现幻读
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id BETWEEN 10 AND 20;  -- 第一次查询:10条
-- Next-Key Lock锁定(10,20]范围

-- 事务B
START TRANSACTION;
INSERT INTO account (id, name) VALUES (15, '张三');  -- 被阻塞
-- 等待事务A释放锁
COMMIT;

-- 事务A
SELECT * FROM account WHERE id BETWEEN 10 AND 20;  -- 第二次查询:还是10条(不会幻读)
COMMIT;
```

**为什么Oracle只支持到读已提交?**

Oracle采用MVCC实现读一致性,但不支持间隙锁,所以无法在可重复读级别下防止幻读。

**从MySQL迁移到Oracle时的注意事项:**

如果业务依赖MySQL的可重复读特性,迁移到Oracle时可能需要调整SQL逻辑或使用SELECT FOR UPDATE锁。

### 💬 面试官追问

#### 提问1:什么是MVCC?它如何实现可重复读?

**回答:**

MVCC(Multi-Version Concurrency Control,多版本并发控制)是InnoDB实现高并发的核心技术,通过保存数据的历史版本,实现读写不冲突。

**MVCC的核心思想:**

每个事务执行时,都能看到一个一致的数据快照,不受其他并发事务的影响。

**MVCC的实现机制:**

**1) 隐式字段**

InnoDB每行数据都有三个隐藏字段:

```sql
-- DB_TRX_ID: 事务ID,最后修改这行数据的事务ID(6字节)
-- DB_ROLL_PTR: 回滚指针,指向undo log中该行的上一个版本(7字节)
-- DB_ROW_ID: 行ID,如果没有主键时自动生成(6字节)
```

**2) Undo Log(回滚日志)**

当事务修改数据时,InnoDB会将旧版本的数据写入Undo Log。

```sql
-- 示例:更新操作的Undo Log链
-- 原始数据: id=1, balance=1000, DB_TRX_ID=100

-- 事务101更新: balance=1500
-- Undo Log记录: id=1, balance=1000, DB_TRX_ID=100
-- 当前数据: id=1, balance=1500, DB_TRX_ID=101, DB_ROLL_PTR指向Undo Log

-- 事务102更新: balance=2000
-- Undo Log记录: id=1, balance=1500, DB_TRX_ID=101
-- 当前数据: id=1, balance=2000, DB_TRX_ID=102, DB_ROLL_PTR指向Undo Log
```

**Undo Log链表示意:**

```
当前版本(事务102):
  id=1, balance=2000, DB_TRX_ID=102, DB_ROLL_PTR → 上一个版本
         ↓
   上一个版本(事务101):
     id=1, balance=1500, DB_TRX_ID=101, DB_ROLL_PTR → 再上一个版本
            ↓
      再上一个版本(事务100):
        id=1, balance=1000, DB_TRX_ID=100, DB_ROLL_PTR → NULL
```

**3) Read View(读视图)**

Read View是事务启动时生成的一个快照,包含当前活跃事务的ID列表。

**Read View的结构:**

```sql
-- Read View包含以下信息
-- m_ids: 当前活跃事务ID列表
-- min_trx_id: 活跃事务中最小的ID
-- max_trx_id: 下一个将要分配的事务ID(当前最大事务ID+1)
-- creator_trx_id: 创建该Read View的事务ID
```

**可见性判断规则:**

当事务访问一行数据时,根据该行的DB_TRX_ID和Read View判断是否可见:

```
1. 如果 DB_TRX_ID < min_trx_id
   → 该版本在Read View创建前已提交 → 可见

2. 如果 DB_TRX_ID >= max_trx_id
   → 该版本在Read View创建后才启动 → 不可见

3. 如果 DB_TRX_ID 在 m_ids 中
   → 该版本对应的活跃事务还未提交 → 不可见
   → 通过DB_ROLL_PTR找上一个版本

4. 如果 DB_TRX_ID 不在 m_ids 中
   → 该版本已提交 → 可见
```

**演示MVCC实现可重复读:**

```sql
-- 设置隔离级别为可重复读
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 初始数据
-- id=1, balance=1000, DB_TRX_ID=100

-- 事务A启动,生成Read View
START TRANSACTION;
-- Read View: m_ids=[], min_trx_id=101, max_trx_id=101, creator_trx_id=101

-- 事务A第一次查询
SELECT balance FROM account WHERE id = 1;  -- 返回1000

-- 事务B更新并提交
START TRANSACTION;  -- trx_id=102
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;
-- 数据变为: id=1, balance=2000, DB_TRX_ID=102

-- 事务A第二次查询
SELECT balance FROM account WHERE id = 1;
-- DB_TRX_ID=102 >= max_trx_id=101 → 不可见
-- 通过DB_ROLL_PTR找上一个版本
-- 上一个版本: balance=1000, DB_TRX_ID=100 < min_trx_id=101 → 可见
-- 返回1000(可重复读)
COMMIT;
```

**读已提交 vs 可重复读的Read View生成时机:**

| 隔离级别 | Read View生成时机 | 特点 |
|---------|------------------|------|
| 读已提交 | 每次SELECT生成 | 能读取最新提交的数据 |
| 可重复读 | 事务开始时生成 | 整个事务使用同一Read View |

**演示:**

```sql
-- 读已提交
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 事务A
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 第一次查询:1000

-- 事务B更新并提交
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;

-- 事务A第二次查询(生成新的Read View)
SELECT balance FROM account WHERE id = 1;  -- 第二次查询:2000(不可重复读)
COMMIT;
```

**MVCC的优点:**

1) **读写不冲突**:读操作不加锁,写操作加锁
2) **高性能**:相比锁机制,并发性能更高
3) **实现简单**:通过Undo Log和Read View实现,逻辑清晰

**MVCC的缺点:**

1) **占用空间**:Undo Log占用额外存储空间
2) **需要清理**:需要定期清理不用的Undo Log
3) **回滚慢**:长事务会导致Undo Log链很长,回滚开销大

#### 提问2:Next-Key Lock是什么?它如何解决幻读?

**回答:**

Next-Key Lock是InnoDB在可重复读隔离级别下用于解决幻读的锁机制,它由**记录锁(Record Lock)**和**间隙锁(Gap Lock)**组成,锁定一个记录及其前面的间隙。

**Next-Key Lock = Record Lock + Gap Lock**

**锁的类型:**

1) **记录锁(Record Lock)**:锁定索引记录本身
2) **间隙锁(Gap Lock)**:锁定索引记录之间的间隙,防止其他事务插入
3) **Next-Key Lock**:锁定记录和其前面的间隙,即开区间

**Next-Key Lock示意图:**

```
索引: 5, 10, 15, 20

Next-Key Lock(10) 锁定范围: (5, 10]
-- 包含10这个记录,以及5到10之间的间隙

Next-Key Lock(15) 锁定范围: (10, 15]
-- 包含15这个记录,以及10到15之间的间隙
```

**Next-Key Lock如何解决幻读:**

幻读的原因是:事务A读取了一个范围的数据,事务B在这个范围内插入了新数据,事务A再次读取时发现了"幻影"记录。

Next-Key Lock通过锁定整个范围,防止其他事务插入新记录,从而解决幻读。

**演示:**

```sql
-- 设置隔离级别为可重复读
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 创建测试表
CREATE TABLE account (
    id INT PRIMARY KEY,
    balance INT
);

INSERT INTO account VALUES (5, 500), (10, 1000), (15, 1500), (20, 2000);

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id BETWEEN 10 AND 20 FOR UPDATE;
-- Next-Key Lock锁定范围: (5, 20]
-- 包含: (5,10], (10,15], (15,20]
-- 防止其他事务插入id在(5,20]范围内的记录

-- 事务B
START TRANSACTION;
INSERT INTO account VALUES (8, 800);  -- 被阻塞,8在(5,10]范围内
INSERT INTO account VALUES (12, 1200);  -- 被阻塞,12在(10,15]范围内
INSERT INTO account VALUES (18, 1800);  -- 被阻塞,18在(15,20]范围内
INSERT INTO account VALUES (25, 2500);  -- 成功,25不在(5,20]范围内
COMMIT;

-- 事务A
SELECT * FROM account WHERE id BETWEEN 10 AND 20;
-- 不会出现幻读,查询结果还是4条记录
COMMIT;
```

**Next-Key Lock的加锁规则:**

**规则1:主键等值查询**

```sql
-- 等值查询,记录存在,使用Next-Key Lock
SELECT * FROM account WHERE id = 10 FOR UPDATE;
-- 加锁: (5, 10]

-- 等值查询,记录不存在,使用间隙锁
SELECT * FROM account WHERE id = 12 FOR UPDATE;  -- id=12不存在
-- 加锁: (10, 15)  -- 间隙锁,不包含记录
```

**规则2:主键范围查询**

```sql
-- 范围查询
SELECT * FROM account WHERE id >= 10 AND id <= 20 FOR UPDATE;
-- 加锁: (5, 20]
-- Next-Key Lock(10) + Next-Key Lock(15) + Next-Key Lock(20)

-- 范围查询,半开区间
SELECT * FROM account WHERE id > 10 AND id < 20 FOR UPDATE;
-- 加锁: (10, 20)
-- 间隙锁,不包含10和20
```

**规则3:非唯一索引查询**

```sql
-- 创建非唯一索引
CREATE INDEX idx_balance ON account(balance);

-- 非唯一索引等值查询
SELECT * FROM account WHERE balance = 1000 FOR UPDATE;
-- 加锁: 
-- - 非唯一索引上: (500, 1000], (1000, 1500)
-- - 主键索引上: 对应的主键记录的Record Lock
```

**为什么非唯一索引要加两个Next-Key Lock?**

因为非唯一索引可能有相同值的记录,为了防止幻读,需要锁定该值的前后间隙。

**Next-Key Lock的优缺点:**

**优点:**
- 完全解决幻读问题
- 保证数据一致性

**缺点:**
- 锁的范围大,影响并发性能
- 容易导致死锁

**如何减少Next-Key Lock的影响:**

```sql
-- 方案1:使用唯一索引
CREATE UNIQUE INDEX idx_id ON account(id);
-- 唯一索引等值查询只加Record Lock,不加Gap Lock

-- 方案2:优化查询条件
-- 避免范围查询,使用精确查询
SELECT * FROM account WHERE id = 10 FOR UPDATE;  -- 只锁定(5,10]
-- 而不是
SELECT * FROM account WHERE id BETWEEN 10 AND 20 FOR UPDATE;  -- 锁定(5,20]

-- 方案3:降低隔离级别
-- 如果业务允许,使用读已提交隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
-- 读已提交级别不会使用Next-Key Lock
```

**Next-Key Lock的底层实现:**

Next-Key Lock通过InnoDB的锁管理器实现,每个事务维护一个锁的哈希表:

```
锁的哈希表:
{
  trx_id=101: [
    { type='NEXT_KEY', space_id=1, page_no=100, n_bits=8, lock_mode='X' },
    { type='GAP', space_id=1, page_no=100, n_bits=8, lock_mode='X' },
    ...
  ]
}
```

#### 提问3:什么是幻读?可重复读级别下如何避免幻读?

**回答:**

幻读(Phantom Read)是指在同一事务内,前后两次执行相同的范围查询,返回的结果集不一样(记录数不一样)。因为期间有其他事务插入或删除了数据并提交。

**幻读的定义:**

幻读不是读到了错误的数据,而是读到了"本不该存在"或"本该存在但不存在的"记录。

**幻读 vs 不可重复读:**

| 问题类型 | 定义 | 触发条件 | 解决方式 |
|---------|------|---------|---------|
| 不可重复读 | 同一行数据两次读取的值不一样 | 其他事务修改并提交 | MVCC |
| 幻读 | 范围查询两次读取的记录数不一样 | 其他事务插入或删除 | Next-Key Lock |

**演示幻读:**

```sql
-- 创建测试表
CREATE TABLE account (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    balance INT
);

INSERT INTO account VALUES (1, '张三', 1000), (2, '李四', 2000), (3, '王五', 3000);

-- 读已提交级别下出现幻读
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE balance > 1500;  -- 第一次查询:返回2条记录(李四、王五)

-- 事务B插入新记录
START TRANSACTION;
INSERT INTO account VALUES (4, '赵六', 4000);
COMMIT;

-- 事务A第二次查询
SELECT * FROM account WHERE balance > 1500;  -- 第二次查询:返回3条记录(李四、王五、赵六)
-- 多出了一条记录,这就是幻读
COMMIT;
```

**可重复读级别下如何避免幻读:**

在InnoDB的可重复读隔离级别下,通过两种机制避免幻读:

**1) 快照读:使用MVCC**

```sql
-- 快照读(普通SELECT)
SELECT * FROM account WHERE balance > 1500;
-- 使用Read View,读取事务开始时的快照,不会看到其他事务新插入的数据

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE balance > 1500;  -- 第一次查询:返回2条记录

-- 事务B插入新记录
INSERT INTO account VALUES (4, '赵六', 4000);
COMMIT;

-- 事务A第二次查询
SELECT * FROM account WHERE balance > 1500;  -- 第二次查询:还是2条记录
-- 不会出现幻读
COMMIT;
```

**2) 当前读:使用Next-Key Lock**

```sql
-- 当前读(SELECT FOR UPDATE/UPDATE/DELETE)
SELECT * FROM account WHERE balance > 1500 FOR UPDATE;
-- 使用Next-Key Lock锁定整个范围,防止其他事务插入

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE balance > 1500 FOR UPDATE;  -- 锁定范围
-- Next-Key Lock锁定: (2000, 3000] (假设有非唯一索引)

-- 事务B插入新记录
START TRANSACTION;
INSERT INTO account VALUES (4, '赵六', 2500);  -- 被阻塞
-- 等待事务A释放锁
COMMIT;

-- 事务A
SELECT * FROM account WHERE balance > 1500 FOR UPDATE;  -- 第二次查询:还是2条记录
-- 不会出现幻读
COMMIT;
```

**快照读 vs 当前读:**

| 读类型 | SQL语句 | 实现方式 | 是否使用Next-Key Lock |
|--------|---------|---------|---------------------|
| 快照读 | SELECT | MVCC | 否 |
| 当前读 | SELECT FOR UPDATE | Next-Key Lock | 是 |
| 当前读 | UPDATE | Next-Key Lock | 是 |
| 当前读 | DELETE | Next-Key Lock | 是 |
| 当前读 | INSERT | 插入意向锁 | 否 |

**为什么需要两种机制?**

- 快照读:不加锁,并发性能高,通过MVCC保证一致性
- 当前读:需要读取最新数据,使用Next-Key Lock防止幻读

**演示对比:**

```sql
-- 快照读
START TRANSACTION;
SELECT * FROM account WHERE id = 1;  -- 快照读,不加锁,读取快照

-- 当前读
SELECT * FROM account WHERE id = 1 FOR UPDATE;  -- 当前读,加Next-Key Lock
UPDATE account SET balance = 2000 WHERE id = 1;  -- 当前读,加Next-Key Lock
DELETE FROM account WHERE id = 1;  -- 当前读,加Next-Key Lock
```

**可重复读级别下避免幻读的总结:**

1) **快照读通过MVCC避免幻读**:Read View保证事务看到一致的数据快照
2) **当前读通过Next-Key Lock避免幻读**:锁定整个范围,防止其他事务插入

**MySQL InnoDB的可重复读级别基本解决了幻读**,这是MySQL相比其他数据库的优势。

#### 提问4:事务的四种隔离级别各有什么优缺点?如何选择?

**回答:**

四种隔离级别各有特点,需要根据业务场景权衡一致性和性能。

**各隔离级别的详细对比:**

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 加锁 | 性能 | 适用场景 |
|---------|------|-----------|------|------|------|---------|
| 读未提交 | ✅ | ✅ | ✅ | 几乎不加 | ⭐⭐⭐⭐⭐ | 极少使用 |
| 读已提交 | ❌ | ✅ | ✅ | 读不加锁 | ⭐⭐⭐⭐ | 大多数OLTP系统 |
| 可重复读 | ❌ | ❌ | 基本解决 | 读不加锁,写加锁 | ⭐⭐⭐ | MySQL默认,大多数业务 |
| 串行化 | ❌ | ❌ | ❌ | 都加锁 | ⭐ | 极少使用 |

**各隔离级别的优缺点:**

**1) 读未提交(READ UNCOMMITTED)**

**优点:**
- 性能最好,几乎不加锁
- 可以读取最新数据

**缺点:**
- 可能读到脏数据(未提交的数据)
- 数据一致性最差
- 几乎不使用

**适用场景:**
- 极少数特殊场景,如实时统计分析
- 数据一致性要求不高的日志分析

```sql
-- 示例:实时监控,可以接受数据不准确
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
SELECT AVG(response_time) FROM logs WHERE service = 'api';
-- 即使未提交的请求也被统计
```

**2) 读已提交(READ COMMITTED)**

**优点:**
- 避免脏读,数据一致性较好
- 性能较好,读操作不加锁
- Oracle、SQL Server等数据库的默认级别

**缺点:**
- 会出现不可重复读和幻读
- 两次查询可能看到不同的数据

**适用场景:**
- 大多数OLTP(在线事务处理)系统
- 电商、金融等业务
- 需要看到最新提交的数据

```sql
-- 示例:银行转账系统
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 事务A:查询余额
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;
-- 只能看到已提交的转账记录,不会看到脏数据
```

**3) 可重复读(REPEATABLE READ)**

**优点:**
- 解决脏读和不可重复读
- InnoDB通过Next-Key Lock基本解决了幻读
- 性能较好,读操作不加锁
- MySQL默认隔离级别

**缺点:**
- 可能出现死锁(因为使用了Next-Key Lock)
- 读写并发性能略低于读已提交
- 当前读会阻塞其他事务

**适用场景:**
- 大多数业务场景
- 需要保证事务内数据一致性
- 银行转账、订单查询等

```sql
-- 示例:订单详情查询
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 事务A
START TRANSACTION;
SELECT * FROM orders WHERE order_id = 1001;
-- 即使其他事务更新了订单,这个事务内看到的订单详情保持一致
SELECT * FROM orders WHERE order_id = 1001;  -- 两次查询结果一致
COMMIT;
```

**4) 串行化(SERIALIZABLE)**

**优点:**
- 绝对的数据一致性
- 解决所有并发问题

**缺点:**
- 性能极差,吞吐量低
- 容易出现死锁
- 几乎不使用

**适用场景:**
- 极少数对一致性要求极高的场景
- 库存扣减等关键操作

```sql
-- 示例:库存扣减
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- 事务A
START TRANSACTION;
SELECT stock FROM product WHERE product_id = 100 FOR UPDATE;
-- 锁定记录,其他事务无法并发修改
UPDATE product SET stock = stock - 1 WHERE product_id = 100;
COMMIT;

-- 事务B会被阻塞,直到事务A完成
```

**如何选择隔离级别?**

**决策树:**

```
业务是否能看到脏数据?
└─ 否 → 排除读未提交
└─ 是 → 使用读未提交(极少使用)

业务是否需要看到最新提交的数据?
└─ 是 → 使用读已提交
└─ 否 → 继续评估

业务是否需要事务内数据一致?
└─ 是 → 使用可重复读(推荐)
└─ 否 → 使用读已提交

业务是否允许任何并发问题?
└─ 不允许 → 使用串行化(极少使用)
└─ 允许 → 使用可重复读
```

**实际业务场景推荐:**

**1) 普通业务系统:可重复读**

```sql
-- 推荐使用MySQL默认的REPEATABLE READ
-- 适用于大多数业务
-- 系统配置:不需要额外配置,使用默认值
```

**2) 银行金融系统:可重复读 + 业务层控制**

```sql
-- 虽然可以使用可重复读,但关键操作需要业务层额外控制
-- 使用SELECT FOR UPDATE锁定关键记录
START TRANSACTION;
SELECT balance FROM account WHERE id = 1 FOR UPDATE;
UPDATE account SET balance = balance - 100 WHERE id = 1;
COMMIT;
```

**3) 电商秒杀系统:读已提交 + 分布式锁**

```sql
-- 电商秒杀场景需要高性能,可以使用读已提交
-- 关键操作通过分布式锁控制
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 配合Redis分布式锁
SETNX lock:product:100 1
-- 获取锁后执行扣库存操作
UPDATE product SET stock = stock - 1 WHERE id = 100;
DEL lock:product:100
```

**4) 数据分析系统:读未提交**

```sql
-- 数据分析可以接受数据不准确,使用读未提交
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
SELECT COUNT(*) FROM logs WHERE level = 'ERROR';
```

**不同数据库的选择:**

```sql
-- MySQL(InnoDB):默认REPEATABLE READ,推荐使用默认值
SELECT @@transaction_isolation;  -- REPEATABLE-READ

-- Oracle:默认READ COMMITTED,推荐使用默认值
-- Oracle不支持REPEATABLE READ

-- SQL Server:默认READ COMMITTED,推荐使用默认值
-- 支持SNAPSHOT隔离级别,类似MySQL的REPEATABLE READ

-- PostgreSQL:默认READ COMMITTED
-- 可以通过配置改为SERIALIZABLE
SET default_transaction_isolation = 'serializable';
```

**总结:**

- **90%的业务场景**:使用MySQL默认的REPEATABLE READ
- **少数需要高性能的场景**:使用READ COMMITTED
- **极少数特殊场景**:使用READ UNCOMMITTED或SERIALIZABLE

**最佳实践:**

```sql
-- 1. 使用MySQL默认的REPEATABLE READ
-- 不需要额外配置,性能和一致性平衡得最好

-- 2. 关键操作使用SELECT FOR UPDATE
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
UPDATE account SET balance = balance - 100 WHERE id = 1;
COMMIT;

-- 3. 避免长事务
-- 长事务会导致Undo Log链过长,影响性能
START TRANSACTION;
-- 执行快速操作,不要长时间持有事务
COMMIT;

-- 4. 监控死锁
SHOW ENGINE INNODB STATUS;
-- 查看死锁日志,优化SQL和索引
```

## 2. MySQL默认的事务隔离级别是什么?为什么选择这个级别?

### 📌 回答重点

MySQL(InnoDB存储引擎)的默认事务隔离级别是**可重复读(REPEATABLE READ)**。

**查看当前隔离级别:**

```sql
-- 方式1:查看系统变量
SELECT @@transaction_isolation;
-- 结果: REPEATABLE-READ

-- 方式2:使用SHOW命令
SHOW VARIABLES LIKE 'transaction_isolation';
-- 结果: transaction_isolation | REPEATABLE-READ

-- 方式3:查看会话隔离级别
SELECT @@session.transaction_isolation;
```

**为什么MySQL选择可重复读作为默认级别?**

主要原因有以下几点:

1) **避免脏读和不可重复读**:可重复读级别解决了这两个最常见的数据一致性问题

2) **通过MVCC实现高性能**:快照读不需要加锁,读写不冲突,并发性能好

3) **通过Next-Key Lock基本解决幻读**:虽然理论上可重复读会出现幻读,但InnoDB通过Next-Key Lock在实际应用中基本避免了幻读

4) **平衡了一致性和性能**:相比读已提交,提供更好的一致性;相比串行化,提供更好的性能

**各数据库默认隔离级别对比:**

| 数据库 | 默认隔离级别 | 说明 |
|--------|-------------|------|
| MySQL(InnoDB) | REPEATABLE READ | 通过MVCC和Next-Key Lock基本解决幻读 |
| Oracle | READ COMMITTED | 不支持REPEATABLE READ级别 |
| SQL Server | READ COMMITTED | 支持快照隔离 |
| PostgreSQL | READ COMMITTED | 可通过配置改为SERIALIZABLE |
| DB2 | READ COMMITTED | 支持RR级别但不是默认 |

**MySQL选择RR级别的历史原因:**

早期MySQL的设计者认为,数据库事务应该保证"可重复读"的数据一致性,这是业务最基本的需求。同时,通过MVCC和Next-Key Lock技术,MySQL在保证一致性的同时也能保持较好的性能。

### 📚 扩展知识

#### 为什么不选择读已提交?

读已提交(READ COMMITTED)是Oracle、SQL Server等主流数据库的默认隔离级别,MySQL为什么不选择它?

**读已提交的问题:**

**1) 不可重复读现象**

```sql
-- 读已提交级别下
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 事务A
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 第一次读取:1000

-- 事务B更新并提交
START TRANSACTION;
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;

-- 事务A第二次查询
SELECT balance FROM account WHERE id = 1;  -- 第二次读取:2000
-- 在同一个事务内,两次查询结果不一致,可能导致业务逻辑错误
COMMIT;
```

**2) 幻读现象**

```sql
-- 读已提交级别下
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 事务A
START TRANSACTION;
SELECT * FROM orders WHERE user_id = 100;  -- 第一次查询:5条订单

-- 事务B插入新订单
START TRANSACTION;
INSERT INTO orders (user_id, amount) VALUES (100, 500);
COMMIT;

-- 事务A第二次查询
SELECT * FROM orders WHERE user_id = 100;  -- 第二次查询:6条订单
-- 多了一条订单,可能导致统计数据不准确
COMMIT;
```

**业务场景中不可重复读的危害:**

```sql
-- 场景:生成报表
START TRANSACTION;

-- 第一步:查询总订单数
SELECT COUNT(*) FROM orders;  -- 返回1000条

-- 第二步:处理其他业务逻辑...
-- 期间可能有新订单插入

-- 第三步:再次查询订单金额
SELECT SUM(amount) FROM orders;  -- 对应的订单数已经不是1000条了
-- 报表数据不一致

COMMIT;
```

**为什么MySQL坚持使用可重复读?**

1) **业务需求**:大多数业务需要事务内数据一致,不希望看到其他事务的修改

2) **技术优势**:通过MVCC,可重复读级别的快照读性能不比读已提交差

3) **减少问题**:避免不可重复读和幻读,减少业务层的复杂性

---

#### 为什么不选择串行化?

串行化(SERIALIZABLE)是最高隔离级别,完全避免所有并发问题,但MySQL不选择它作为默认级别,原因是:

**串行化的问题:**

**1) 性能极差**

```sql
-- 串行化级别下
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 1;  -- 隐式加共享锁
-- 事务A持有锁,其他读操作也会被阻塞

-- 事务B
START TRANSACTION;
SELECT * FROM account WHERE id = 1;  -- 被阻塞,等待事务A释放锁
-- 即使是读操作,也会被阻塞!
COMMIT;
```

**2) 并发度低**

串行化强制事务串行执行,即使只是简单的读查询也要加锁,导致系统并发度极低。

```sql
-- 串行化级别下,简单的读查询也会加锁
SELECT * FROM users WHERE id = 1;  -- 加共享锁
SELECT * FROM users WHERE id = 2;  -- 被阻塞
```

**3) 容易死锁**

由于所有操作都加锁,串行化级别容易出现死锁。

```sql
-- 死锁示例

-- 事务A
START TRANSACTION;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- 持有id=1的锁

-- 事务B
START TRANSACTION;
SELECT * FROM account WHERE id = 2 FOR UPDATE;
-- 持有id=2的锁

-- 事务A尝试获取id=2
SELECT * FROM account WHERE id = 2 FOR UPDATE;
-- 被阻塞,等待事务B释放id=2

-- 事务B尝试获取id=1
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- 被阻塞,等待事务A释放id=1

-- 出现死锁!
```

**性能对比测试:**

```sql
-- 测试场景:1000个并发事务,每条事务执行1次查询

-- 读已提交级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
-- 吞吐量:约8000 TPS

-- 可重复读级别
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
-- 吞吐量:约7500 TPS (略低,但差别不大)

-- 串行化级别
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;
-- 吞吐量:约1000 TPS (性能下降87.5%)
```

**结论:**

串行化虽然能完全避免所有并发问题,但性能损失太大,不适合作为默认隔离级别。只有在极少数对一致性要求极高的场景(如银行转账、库存扣减)才使用。

#### MVCC如何保证可重复读的性能?

MVCC(Multi-Version Concurrency Control,多版本并发控制)是MySQL在可重复读级别下保持高性能的关键技术。

**MVCC的核心优势:**

**1) 读写不冲突**

```sql
-- 事务A:读操作
START TRANSACTION;
SELECT * FROM account WHERE id = 1;  -- 快照读,不加锁
-- 即使事务B在修改数据,事务A也不会被阻塞

-- 事务B:写操作
START TRANSACTION;
UPDATE account SET balance = 2000 WHERE id = 1;  -- 写操作加锁
-- 只会阻塞其他写操作,不会阻塞读操作
COMMIT;

-- 事务A继续读
SELECT * FROM account WHERE id = 1;  -- 读取的是事务开始时的快照
-- 不会被事务B的修改影响
COMMIT;
```

**2) Read View的复用**

在可重复读级别下,Read View只在事务开始时生成一次,整个事务期间都使用这个Read View。

```sql
-- 可重复读级别
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

START TRANSACTION;
-- 生成Read View,包含当前活跃事务列表

SELECT * FROM account WHERE id = 1;  -- 第一次查询,使用Read View
SELECT * FROM account WHERE id = 2;  -- 第二次查询,复用Read View
SELECT * FROM account WHERE id = 3;  -- 第三次查询,复用Read View
-- 整个事务只生成一次Read View,性能好

COMMIT;
```

而在读已提交级别下,每次SELECT都要生成新的Read View。

```sql
-- 读已提交级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

START TRANSACTION;

SELECT * FROM account WHERE id = 1;  -- 第一次查询,生成Read View
SELECT * FROM account WHERE id = 2;  -- 第二次查询,生成新的Read View
SELECT * FROM account WHERE id = 3;  -- 第三次查询,再生成新的Read View
-- 每次查询都要生成Read View,性能略差

COMMIT;
```

**3) Undo Log的链式结构**

Undo Log采用链式结构,不需要拷贝整个数据页,存储开销小。

```sql
-- Undo Log链表示意
当前版本:
  id=1, balance=2000, DB_TRX_ID=102, DB_ROLL_PTR → 上一个版本
         ↓
   上一个版本:
     id=1, balance=1500, DB_TRX_ID=101, DB_ROLL_PTR → 再上一个版本
            ↓
      再上一个版本:
        id=1, balance=1000, DB_TRX_ID=100, DB_ROLL_PTR → NULL

-- 查找旧版本时,顺着DB_ROLL_PTR指针遍历Undo Log链
-- 不需要拷贝整个数据页,性能好
```

**MVCC的性能优势对比:**

```sql
-- 测试场景:1000个并发事务,每个事务读取100条数据

-- 使用锁机制(如串行化级别)
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;
-- 每次读都要加锁,阻塞其他事务
-- 吞吐量:约1000 TPS

-- 使用MVCC(可重复读级别)
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
-- 快照读不加锁,读写不冲突
-- 吞吐量:约7500 TPS (性能提升7.5倍)
```

#### Next-Key Lock如何解决幻读?

虽然MySQL通过MVCC解决了快照读的幻读问题,但当前读(SELECT FOR UPDATE/UPDATE/DELETE)仍然可能出现幻读。InnoDB通过Next-Key Lock解决了这个问题。

**Next-Key Lock = Record Lock + Gap Lock**

- **Record Lock(记录锁)**:锁定索引记录本身
- **Gap Lock(间隙锁)**:锁定索引记录之间的间隙
- **Next-Key Lock**:锁定记录和其前面的间隙,即开区间

**Next-Key Lock解决幻读的原理:**

```sql
-- 可重复读级别
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 创建测试表
CREATE TABLE account (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    balance INT
);

INSERT INTO account VALUES (10, '张三', 1000), (20, '李四', 2000), (30, '王五', 3000);

-- 事务A:当前读,锁定范围
START TRANSACTION;
SELECT * FROM account WHERE id BETWEEN 15 AND 25 FOR UPDATE;
-- Next-Key Lock锁定: (10, 20] 和 (20, 30)
-- 防止其他事务插入id在15-25之间的记录

-- 事务B:尝试插入
START TRANSACTION;
INSERT INTO account VALUES (15, '赵六', 1500);  -- 被阻塞,15在(10,20]范围内
INSERT INTO account VALUES (25, '钱七', 2500);  -- 被阻塞,25在(20,30)范围内
COMMIT;

-- 事务A:再次查询
SELECT * FROM account WHERE id BETWEEN 15 AND 25 FOR UPDATE;
-- 不会出现幻读,查询结果还是0条
COMMIT;
```

**Next-Key Lock的加锁规则:**

**1) 等值查询,记录存在**

```sql
-- id=20存在
SELECT * FROM account WHERE id = 20 FOR UPDATE;
-- 加锁: (10, 20]
-- 包含20这个记录,以及10到20之间的间隙
```

**2) 等值查询,记录不存在**

```sql
-- id=15不存在
SELECT * FROM account WHERE id = 15 FOR UPDATE;
-- 加锁: (10, 20)
-- 只锁定间隙,不包含记录(因为记录不存在)
```

**3) 范围查询**

```sql
-- 范围查询
SELECT * FROM account WHERE id >= 20 AND id <= 30 FOR UPDATE;
-- 加锁: (10, 30]
-- Next-Key Lock(20) + Next-Key Lock(30)
```

**Next-Key Lock vs Gap Lock vs Record Lock:**

| 锁类型 | 锁定范围 | 作用 | 是否阻塞插入 |
|--------|---------|------|------------|
| Record Lock | 索引记录 | 锁定单个记录 | 否 |
| Gap Lock | 索引之间的间隙 | 防止其他事务插入 | 是 |
| Next-Key Lock | 记录+前面的间隙 | 防止幻读 | 是 |

#### 不同业务场景下的隔离级别选择

虽然MySQL默认使用可重复读,但某些业务场景可能需要调整隔离级别。

**场景1:电商订单系统 → 可重复读**

```sql
-- 推荐使用默认的可重复读
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 原因:
-- 1. 订单详情查询需要事务内数据一致
-- 2. 用户希望看到稳定的订单信息
-- 3. 避免不可重复读导致的前端数据跳动

-- 示例
START TRANSACTION;
SELECT * FROM orders WHERE order_id = 1001;
-- 第一次查询:状态=待支付
-- 其他事务可能修改订单状态
SELECT * FROM orders WHERE order_id = 1001;
-- 第二次查询:状态=待支付(可重复读)
-- 前端展示的订单状态不会跳动
COMMIT;
```

**场景2:电商秒杀系统 → 读已提交**

```sql
-- 推荐使用读已提交
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 原因:
-- 1. 秒杀需要看到最新的库存数据
-- 2. 避免用户抢到重复商品
-- 3. 性能比可重复读更好

-- 配合Redis分布式锁
SETNX lock:product:100 1
-- 获取锁后执行扣库存
UPDATE product SET stock = stock - 1 WHERE id = 100 AND stock > 0;
DEL lock:product:100
```

**场景3:银行转账系统 → 可重复读 + SELECT FOR UPDATE**

```sql
-- 使用可重复读级别
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 关键操作使用SELECT FOR UPDATE
START TRANSACTION;
SELECT balance FROM account WHERE id = 1 FOR UPDATE;
-- 锁定账户,防止其他事务修改
UPDATE account SET balance = balance - 100 WHERE id = 1;
UPDATE account SET balance = balance + 100 WHERE id = 2;
COMMIT;
```

**场景4:日志分析系统 → 读未提交**

```sql
-- 可以使用读未提交
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;

-- 原因:
-- 1. 日志分析不需要绝对准确的数据
-- 2. 可以接受未提交的数据
-- 3. 性能最好

SELECT COUNT(*) FROM logs WHERE level = 'ERROR';
-- 统计错误日志,即使有些未提交的ERROR日志也被统计
```

**场景5:实时监控系统 → 读已提交**

```sql
-- 推荐使用读已提交
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 原因:
-- 1. 监控需要看到最新的数据
-- 2. 不要求事务内数据一致
-- 3. 性能好

SELECT AVG(response_time) FROM api_logs WHERE create_time > NOW() - INTERVAL 1 MINUTE;
-- 统计最近1分钟的平均响应时间
-- 需要看到最新的请求记录
```

### 💬 面试官追问

#### 提问1:MySQL默认隔离级别和Oracle有什么不同?从Oracle迁移到MySQL需要注意什么?

**回答:**

**默认隔离级别不同:**

| 数据库 | 默认隔离级别 | 可解决并发问题 |
|--------|-------------|--------------|
| MySQL(InnoDB) | REPEATABLE READ | 脏读、不可重复读、基本解决幻读 |
| Oracle | READ COMMITTED | 只解决脏读 |

**核心差异:**

**1) 不可重复读现象**

```sql
-- Oracle(读已提交)下会出现不可重复读
-- 事务A
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 第一次读取:1000

-- 事务B
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;

-- 事务A
SELECT balance FROM account WHERE id = 1;  -- 第二次读取:2000(不可重复读)
```

MySQL可重复读级别下,第二次读取还是1000。

**2) 幻读现象**

```sql
-- Oracle(读已提交)下会出现幻读
-- 事务A
START TRANSACTION;
SELECT * FROM orders WHERE user_id = 100;  -- 第一次查询:5条订单

-- 事务B
INSERT INTO orders (user_id, amount) VALUES (100, 500);
COMMIT;

-- 事务A
SELECT * FROM orders WHERE user_id = 100;  -- 第二次查询:6条订单(幻读)
```

MySQL可重复读级别下,第二次查询还是5条。

**从Oracle迁移到MySQL的注意事项:**

**1) 业务逻辑可能需要调整**

```sql
-- Oracle的业务代码
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;
-- ... 处理逻辑 ...
SELECT balance FROM account WHERE id = 1;
-- Oracle这里可能读到不同的值,业务代码依赖这个特性
COMMIT;

-- 迁移到MySQL后,需要调整业务逻辑
-- 或者降低隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

**2) 使用SELECT FOR UPDATE锁定关键数据**

```sql
-- 如果需要看到最新数据,使用SELECT FOR UPDATE
START TRANSACTION;
SELECT balance FROM account WHERE id = 1 FOR UPDATE;
-- 锁定记录,读取最新数据
-- ... 处理逻辑 ...
SELECT balance FROM account WHERE id = 1;
-- 仍然是锁定时读取的值,不会变化
COMMIT;
```

**3) 使用提示(Hint)改变隔离级别**

```sql
-- MySQL不支持Oracle的HINT语法,但可以设置会话隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 或者在连接URL中指定
-- jdbc:mysql://host:3306/db?transactionIsolation=READ_COMMITTED
```

**4) 测试并发场景**

```sql
-- 迁移前必须测试并发场景
-- 模拟多个事务并发操作,验证业务逻辑是否正确

-- 事务A
START TRANSACTION;
SELECT * FROM orders WHERE user_id = 100;
-- ... 处理 ...
COMMIT;

-- 事务B
START TRANSACTION;
INSERT INTO orders (user_id, amount) VALUES (100, 500);
COMMIT;
```

**最佳实践:**

1) **评估业务对不可重复读的容忍度**:如果业务可以接受不可重复读,可以在MySQL中使用READ COMMITTED

2) **使用可重复读 + SELECT FOR UPDATE**:关键操作使用锁定,保证一致性

3) **充分测试**:迁移前必须进行充分的并发测试

4) **渐进式迁移**:先迁移非核心业务,再逐步迁移核心业务

#### 提问2:在实际项目中,你是否遇到过因隔离级别导致的问题?如何解决的?

**回答:**

**案例1:库存超卖问题**

**问题描述:**

电商秒杀场景下,多个用户同时抢购同一商品,导致库存超卖。

```sql
-- 原始代码(可重复读级别)
START TRANSACTION;
SELECT stock FROM product WHERE id = 100;
-- 读取到的stock=10

-- 业务逻辑判断
IF stock > 0 THEN
    -- 更新库存
    UPDATE product SET stock = stock - 1 WHERE id = 100;
END IF;
COMMIT;
```

问题:多个事务同时读取stock=10,都更新为9,导致超卖。

**解决方案:使用乐观锁或SELECT FOR UPDATE**

**方案1:乐观锁(推荐)**

```sql
START TRANSACTION;
SELECT stock, version FROM product WHERE id = 100;

IF stock > 0 THEN
    -- 使用版本号控制
    UPDATE product 
    SET stock = stock - 1, version = version + 1 
    WHERE id = 100 AND version = [读取的version];
    
    -- 检查是否更新成功
    IF ROW_COUNT() = 0 THEN
        -- 版本号不匹配,说明其他事务已修改,重试或返回失败
        ROLLBACK;
        RETRY;
    END IF;
END IF;
COMMIT;
```

**方案2:SELECT FOR UPDATE**

```sql
START TRANSACTION;
SELECT stock FROM product WHERE id = 100 FOR UPDATE;
-- 锁定记录,其他事务无法并发修改

IF stock > 0 THEN
    UPDATE product SET stock = stock - 1 WHERE id = 100;
END IF;
COMMIT;
```

**方案3:UPDATE条件判断(最简单)**

```sql
START TRANSACTION;
-- 直接在UPDATE中判断库存
UPDATE product SET stock = stock - 1 
WHERE id = 100 AND stock > 0;

-- 检查是否更新成功
IF ROW_COUNT() = 0 THEN
    -- 库存不足,返回失败
    ROLLBACK;
END IF;
COMMIT;
```

**案例2:统计数据不一致**

**问题描述:**

生成报表时,第一次查询订单数量,第二次查询订单金额,但两次查询的订单集合不一致,导致统计数据错误。

```sql
-- 原始代码(读已提交级别)

START TRANSACTION;
-- 第一步:查询订单数量
SELECT COUNT(*) FROM orders WHERE create_time >= '2026-01-01';
-- 返回1000条

-- 第二步:查询订单金额(期间可能有新订单插入)
SELECT SUM(amount) FROM orders WHERE create_time >= '2026-01-01';
-- 对应的订单数已经不是1000条了

COMMIT;
```

**解决方案:使用可重复读隔离级别**

```sql
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

START TRANSACTION;
-- 生成Read View,整个事务期间使用同一快照

-- 第一步:查询订单数量
SELECT COUNT(*) FROM orders WHERE create_time >= '2026-01-01';
-- 返回1000条
```

**案例3:长事务导致性能问题**

**问题描述:**

使用可重复读级别时,长事务导致Undo Log链过长,影响查询性能。

```sql
-- 长事务示例
START TRANSACTION;
-- 生成Read View

-- 第一步:查询数据
SELECT * FROM large_table WHERE id = 1;

-- 第二步:执行其他业务逻辑,耗时5分钟
-- ...

-- 第三步:再次查询
SELECT * FROM large_table WHERE id = 1;

COMMIT;
```

问题:事务持续5分钟,期间有大量数据修改,Undo Log链很长,第二次查询需要遍历很长的Undo Log链,性能差。

**解决方案:拆分事务**

```sql
-- 拆分成多个短事务
START TRANSACTION;
SELECT * FROM large_table WHERE id = 1;
COMMIT;

-- 执行其他业务逻辑
-- ...

-- 开启新事务
START TRANSACTION;
SELECT * FROM large_table WHERE id = 1;
COMMIT;
```

**或者降低隔离级别(如果业务允许):**

```sql
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

START TRANSACTION;
SELECT * FROM large_table WHERE id = 1;

-- 第二次查询生成新的Read View,Undo Log链短
SELECT * FROM large_table WHERE id = 1;

COMMIT;
```

#### 提问3:如何判断当前系统使用的隔离级别是否合适?

**回答:**

判断隔离级别是否合适,需要从多个维度评估。

**评估指标:**

**1) 业务一致性要求**

```sql
-- 分析业务是否需要事务内数据一致
-- 如果业务代码依赖"同一事务内多次查询结果一致",则需要可重复读

-- 示例:报表生成
-- 需要可重复读
SELECT COUNT(*) FROM orders WHERE user_id = 100;
SELECT SUM(amount) FROM orders WHERE user_id = 100;
-- 两次查询应该基于同一订单集合
```

**2) 性能要求**

```sql
-- 测试不同隔离级别的性能
-- 使用性能测试工具模拟实际业务

-- 可重复读
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
-- 吞吐量:7500 TPS

-- 读已提交
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
-- 吞吐量:8000 TPS
```

**3) 死锁频率**

```sql
-- 查看死锁日志
SHOW ENGINE INNODB STATUS;

-- 查看死锁次数
SHOW STATUS LIKE 'Innodb_deadlocks';

-- 如果死锁频率高,可以考虑降低隔离级别
```

**4) 监控指标**

```sql
-- 监控事务持续时间
SELECT 
    trx_id,
    TIMESTAMPDIFF(SECOND, trx_started, NOW()) AS duration_seconds
FROM information_schema.INNODB_TRX;

-- 长事务可能导致Undo Log链过长,影响性能

-- 监控锁等待
SELECT * FROM information_schema.INNODB_LOCK_WAITS;

-- 锁等待多说明并发冲突严重
```

**评估流程:**

```
1. 分析业务需求
   └─ 是否需要事务内数据一致?
      └─ 是 → 使用可重复读
      └─ 否 → 继续评估

2. 性能测试
   └─ 可重复读性能是否满足要求?
      └─ 是 → 使用可重复读
      └─ 否 → 继续评估

3. 监控死锁和锁等待
   └─ 死锁频率是否可接受?
      └─ 是 → 使用可重复读
      └─ 否 → 考虑降低隔离级别

4. 综合评估
   └─ 如果性能满足要求,优先使用可重复读
   └─ 如果性能不足,考虑使用读已提交
```

**实际案例:**

**案例1:电商系统**

```sql
-- 需求分析:
-- 1. 订单详情查询需要一致 → 需要可重复读
-- 2. 性能要求:5000 TPS
-- 3. 测试结果:可重复读 6000 TPS

-- 结论:使用默认的可重复读
```

**案例2:秒杀系统**

```sql
-- 需求分析:
-- 1. 需要看到最新库存 → 不需要可重复读
-- 2. 性能要求:20000 TPS
-- 3. 测试结果:可重复读 8000 TPS,读已提交 12000 TPS

-- 结论:使用读已提交 + Redis分布式锁
```

**案例3:报表系统**

```sql
-- 需求分析:
-- 1. 报表数据必须一致 → 需要可重复读
-- 2. 性能要求:100 TPS
-- 3. 测试结果:可重复读 500 TPS

-- 结论:使用可重复读(性能足够)
```

## 📝 总结

**MySQL默认事务隔离级别是可重复读(REPEATABLE READ)**,选择这个级别的原因:

1) **避免脏读和不可重复读**:解决最常见的并发问题

2) **MVCC实现高性能**:快照读不加锁,读写不冲突

3) **Next-Key Lock解决幻读**:基本解决幻读问题

4) **平衡一致性和性能**:比读已提交更一致,比串行化更高效

**各隔离级别适用场景:**

- **90%场景**:使用MySQL默认的可重复读
- **少数高性能场景**:使用读已提交(如秒杀系统)
- **极少数特殊场景**:使用读未提交或串行化

**最佳实践:**

1. 使用默认的可重复读,不需要额外配置

2. 关键操作使用SELECT FOR UPDATE锁定记录

3. 避免长事务,防止Undo Log链过长

4. 定期监控死锁和锁等待,优化SQL和索引

5. 从Oracle等数据库迁移时,需要评估业务对不可重复读的容忍度

## 3. 数据库的脏读、不可重复读和幻读分别是什么?

### 📌 回答重点

脏读、不可重复读、幻读是数据库事务并发访问时可能出现的三个问题。

**1) 脏读(Dirty Read)**

一个事务读取到了另一个事务**未提交**的数据。如果那个事务回滚了,读取到的数据就是无效的。

**2) 不可重复读(Non-repeatable Read)**

在同一个事务内,两次读取同一数据得到的结果不一样。因为期间有其他事务修改了该数据并提交。

**3) 幻读(Phantom Read)**

在同一个事务内,前后两次执行相同的查询,返回的记录数不一样。因为期间有其他事务插入或删除了数据并提交。

**三者对比表格:**

| 问题类型 | 定义 | 触发条件 | 出现隔离级别 | 危害 |
|---------|------|---------|-------------|------|
| 脏读 | 读取到未提交的数据 | 读未提交隔离级别 | READ UNCOMMITTED | 数据错误,违反原子性 |
| 不可重复读 | 同一行数据两次读取值不一样 | 事务内两次读取同一数据 | READ COMMITTED | 业务逻辑错误,数据不一致 |
| 幻读 | 范围查询两次记录数不一样 | 事务内两次范围查询 | READ COMMITTED、部分REPEATABLE READ | 统计错误,数据集不一致 |

**示意图:**

```
事务A                               事务B
----------------------------------  ----------------------------------
START TRANSACTION;                  START TRANSACTION;
SELECT balance = 1000;             UPDATE balance = 2000;
(读取到事务B未提交的修改→脏读)     COMMIT;
                                    (事务B回滚,balance变成1000)
SELECT balance = 2000;             INSERT INTO account VALUES (...);
(读取的值与实际不一致)             COMMIT;

SELECT * FROM account;
(多了一条记录→幻读)
COMMIT;
```

### 📚 扩展知识

#### 脏读详解

**定义:**

脏读是指一个事务读取了另一个事务未提交的数据。如果该事务回滚,读取到的数据就是无效的。

**示例:**

```sql
-- 创建测试表
CREATE TABLE account (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    balance DECIMAL(10,2)
);

INSERT INTO account VALUES (1, '张三', 1000);

-- 事务A
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;
-- 第一次读取: balance = 1000

-- 事务B
START TRANSACTION;
UPDATE account SET balance = 2000 WHERE id = 1;
-- 此时事务B还未提交

-- 事务A再次读取(读未提交级别)
SELECT balance FROM account WHERE id = 1;
-- 第二次读取: balance = 2000 (脏读!)
-- 读取到了事务B未提交的数据

-- 事务B回滚
ROLLBACK;
-- balance恢复为1000

-- 事务A继续
-- 此时事务A认为balance是2000,但实际是1000
-- 如果基于这个错误数据执行操作,会导致严重问题
COMMIT;
```

**业务场景危害:**

**场景1:银行转账**

```sql
-- 事务A
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 读取到1000
SELECT balance FROM account WHERE id = 2;  -- 读取到脏数据2000(未提交)

-- 如果事务A基于错误的balance=2000执行转账
-- 而事务B回滚,实际balance还是1000
-- 会导致账户余额错误
```

**场景2:库存扣减**

```sql
-- 事务A
START TRANSACTION;
SELECT stock FROM product WHERE id = 1;  -- 读取到脏数据(其他事务减库存后未提交)
-- 基于错误的库存数据判断是否允许购买
-- 可能导致超卖
```

**解决方案:**

使用**读已提交**或更高级别的隔离级别。

```sql
-- 设置隔离级别为读已提交
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 事务A
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 第一次读取:1000

-- 事务B更新并提交
START TRANSACTION;
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;

-- 事务A
SELECT balance FROM account WHERE id = 1;  -- 第二次读取:2000
-- 只能读到已提交的数据,不会脏读
COMMIT;
```

#### 不可重复读详解

**定义:**

不可重复读是指在同一个事务内,两次读取同一数据得到的结果不一样。因为期间有其他事务修改了该数据并提交。

**示例:**

```sql
-- 设置隔离级别为读已提交
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 事务A
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 第一次读取:1000

-- 事务B更新并提交
START TRANSACTION;
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;

-- 事务A第二次读取
SELECT balance FROM account WHERE id = 1;  -- 第二次读取:2000 (不可重复读!)
-- 在同一个事务内,两次读取结果不一致
COMMIT;
```

**业务场景危害:**

**场景1:生成报表**

```sql
-- 事务A
START TRANSACTION;

-- 第一步:查询总余额
SELECT SUM(balance) FROM account;  -- 返回100000

-- 第二步:执行其他业务逻辑...
-- 期间有转账操作,余额变成了95000

-- 第三步:查询总笔数
SELECT COUNT(*) FROM account;  -- 订单数还是10笔
-- 但总余额已经变化,报表数据不一致

COMMIT;
```

**场景2:余额校验**

```sql
-- 事务A
START TRANSACTION;

-- 第一步:查询余额
SELECT balance FROM account WHERE id = 1;  -- 1000

-- 第二步:判断余额是否足够
IF balance >= 500 THEN
    -- 第三步:执行扣款
    -- 期间其他事务扣了100元
    UPDATE account SET balance = balance - 500 WHERE id = 1;
    -- 实际扣款前余额是900,扣款后变成400
    -- 但代码逻辑认为有1000元,导致余额不足
END IF;
COMMIT;
```

**场景3:前端数据跳动**

```sql
-- 事务A
START TRANSACTION;

-- 第一次查询:订单状态
SELECT status FROM orders WHERE order_id = 1001;
-- 返回:待支付

-- 前端展示给用户:待支付

-- 其他事务修改订单状态
UPDATE orders SET status = '已支付' WHERE order_id = 1001;

-- 第二次查询:订单状态
SELECT status FROM orders WHERE order_id = 1001;
-- 返回:已支付

-- 前端显示:已支付
-- 用户看到订单状态从"待支付"变成"已支付",用户体验差
```

**解决方案:**

使用**可重复读**隔离级别。

```sql
-- 设置隔离级别为可重复读
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 事务A
START TRANSACTION;
-- 生成Read View,整个事务期间使用同一快照

SELECT balance FROM account WHERE id = 1;  -- 第一次读取:1000

-- 事务B更新并提交
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;

-- 事务A第二次读取
SELECT balance FROM account WHERE id = 1;  -- 第二次读取:1000 (可重复读!)
-- 使用Read View,读取事务开始时的快照
-- 即使其他事务修改并提交,也看不到
COMMIT;
```

#### 幻读详解

**定义:**

幻读是指在同一个事务内,前后两次执行相同的查询,返回的记录数不一样。因为期间有其他事务插入或删除了数据并提交。

**示例:**

```sql
-- 设置隔离级别为读已提交
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 创建测试表
CREATE TABLE orders (
    id INT PRIMARY KEY,
    user_id INT,
    amount DECIMAL(10,2)
);

INSERT INTO orders VALUES (1, 100, 100), (2, 100, 200), (3, 100, 300);

-- 事务A
START TRANSACTION;
SELECT * FROM orders WHERE user_id = 100;
-- 第一次查询:返回3条记录

-- 事务B插入新订单
START TRANSACTION;
INSERT INTO orders VALUES (4, 100, 400);
COMMIT;

-- 事务A第二次查询
SELECT * FROM orders WHERE user_id = 100;
-- 第二次查询:返回4条记录 (幻读!)
-- 在同一个事务内,相同查询返回的记录数不一样
COMMIT;
```

**业务场景危害:**

**场景1:统计分析**

```sql
-- 事务A
START TRANSACTION;

-- 第一步:查询订单数量
SELECT COUNT(*) FROM orders WHERE user_id = 100;
-- 返回:10条

-- 第二步:计算订单总金额
SELECT SUM(amount) FROM orders WHERE user_id = 100;
-- 期间有新订单插入,总金额变了
-- 但订单数量还是10条,数据不一致

-- 报表显示:
-- 订单数:10条
-- 总金额:计算基于11条订单
-- 报表数据错误!
COMMIT;
```

**场景2:权限检查**

```sql
-- 事务A
START TRANSACTION;

-- 第一步:查询用户的订单
SELECT * FROM orders WHERE user_id = 100;
-- 返回:5条订单

-- 业务逻辑:允许最多5个订单
IF COUNT <= 5 THEN
    -- 第二步:执行新订单
    -- 期间其他用户插入了订单
    INSERT INTO orders (user_id, amount) VALUES (100, 500);
END IF;

-- 结果:实际有6条订单,违反了业务规则
COMMIT;
```

**场景3:数据导出**

```sql
-- 事务A
START TRANSACTION;

-- 第一步:导出前1000条订单
SELECT * FROM orders LIMIT 1000 OFFSET 0;
-- 返回:订单1001-2000

-- 第二步:导出下一批
SELECT * FROM orders LIMIT 1000 OFFSET 1000;
-- 期间有新订单插入
-- 返回:订单2001-3000,但漏掉了中间的订单

-- 导出的数据不完整!
COMMIT;
```

**解决方案:**

使用**可重复读**隔离级别,MySQL通过MVCC和Next-Key Lock解决幻读。

**快照读避免幻读:**

```sql
-- 设置隔离级别为可重复读
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 事务A
START TRANSACTION;
-- 生成Read View

SELECT * FROM orders WHERE user_id = 100;
-- 第一次查询:返回3条记录

-- 事务B插入新订单
INSERT INTO orders VALUES (4, 100, 400);
COMMIT;

-- 事务A第二次查询
SELECT * FROM orders WHERE user_id = 100;
-- 第二次查询:返回3条记录 (不会幻读!)
-- 使用Read View,看不到新插入的数据
COMMIT;
```

**Next-Key Lock避免幻读(当前读):**

```sql
-- 事务A:当前读,锁定范围
START TRANSACTION;
SELECT * FROM orders WHERE user_id = 100 FOR UPDATE;
-- Next-Key Lock锁定范围,防止其他事务插入

-- 事务B尝试插入
INSERT INTO orders (user_id, amount) VALUES (100, 400);
-- 被阻塞,等待事务A释放锁
COMMIT;

-- 事务A
SELECT * FROM orders WHERE user_id = 100;
-- 第二次查询:还是3条记录 (不会幻读!)
COMMIT;
```

#### 脏读 vs 不可重复读 vs 幻读

**核心区别:**

| 特性 | 脏读 | 不可重复读 | 幻读 |
|------|------|-----------|------|
| 定义 | 读取未提交数据 | 同一行两次读取值不一样 | 相同查询两次记录数不一样 |
| 触发原因 | 读取未提交的数据 | 其他事务修改并提交 | 其他事务插入/删除并提交 |
| 读取对象 | 单个数据 | 单个数据 | 数据集 |
| 最小隔离级别 | READ COMMITTED | REPEATABLE READ | REPEATABLE READ (需Next-Key Lock) |

**形象比喻:**

```
脏读:
你看到隔壁邻居在买东西(未结账),
你认为他买了,结果他后来没付钱又放回去了
(你看到了"虚假"的数据)

不可重复读:
你第一次看邻居家的表有3个苹果,
过一会儿再看,变成了2个苹果
(同一个对象,两次看到的不一样)

幻读:
你第一次看邻居家有3个人,
过一会儿再看,变成了4个人
(数据集的大小变了)
```

**代码示例对比:**

```sql
-- 1. 脏读示例
-- 事务A
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 1000

-- 事务B
UPDATE account SET balance = 2000 WHERE id = 1;
-- 未提交

-- 事务A
SELECT balance FROM account WHERE id = 1;  -- 2000 (脏读)
-- 读到了未提交的数据
```

```sql
-- 2. 不可重复读示例
-- 事务A
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 1000

-- 事务B
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;  -- 已提交

-- 事务A
SELECT balance FROM account WHERE id = 1;  -- 2000 (不可重复读)
-- 同一行,两次读取值不一样
```

```sql
-- 3. 幻读示例
-- 事务A
START TRANSACTION;
SELECT COUNT(*) FROM orders WHERE user_id = 100;  -- 10条

-- 事务B
INSERT INTO orders (user_id, amount) VALUES (100, 500);
COMMIT;  -- 已提交

-- 事务A
SELECT COUNT(*) FROM orders WHERE user_id = 100;  -- 11条 (幻读)
-- 相同查询,返回的记录数不一样
```

### 💬 面试官追问

#### 提问1:不可重复读和幻读有什么区别?如何区分?

**回答:**

**核心区别:**

1) **读取对象不同**
   - 不可重复读:读取的是**同一行数据**的值
   - 幻读:读取的是**数据集**的记录数

2) **触发条件不同**
   - 不可重复读:其他事务**修改**了数据并提交
   - 幻读:其他事务**插入或删除**了数据并提交

3) **SQL语句不同**
   - 不可重复读:针对单条记录的查询
     ```sql
     SELECT * FROM account WHERE id = 1;
     ```
   - 幻读:针对范围查询
     ```sql
     SELECT * FROM orders WHERE user_id = 100;
     SELECT * FROM orders WHERE create_time > '2026-01-01';
     ```

**详细对比:**

| 维度 | 不可重复读 | 幻读 |
|------|-----------|------|
| SQL类型 | 等值查询 | 范围查询 |
| 返回结果 | 值变化 | 记录数变化 |
| 事务操作 | UPDATE/DELETE | INSERT/DELETE |
| 举例 | SELECT * FROM account WHERE id = 1; | SELECT * FROM orders WHERE user_id = 100; |
| 原因 | 其他事务UPDATE | 其他事务INSERT |

**代码示例对比:**

```sql
-- 不可重复读:查询同一行,值变化
START TRANSACTION;

-- 第一次查询
SELECT balance FROM account WHERE id = 1;  -- 返回:1000

-- 其他事务修改
UPDATE account SET balance = 2000 WHERE id = 1;
COMMIT;

-- 第二次查询
SELECT balance FROM account WHERE id = 1;  -- 返回:2000 (值变化)
-- 不可重复读

COMMIT;
```

```sql
-- 幻读:相同查询,记录数变化
START TRANSACTION;

-- 第一次查询
SELECT * FROM orders WHERE user_id = 100;
-- 返回:10条记录

-- 其他事务插入
INSERT INTO orders (user_id, amount) VALUES (100, 500);
COMMIT;

-- 第二次查询
SELECT * FROM orders WHERE user_id = 100;
-- 返回:11条记录 (记录数变化)
-- 幻读

COMMIT;
```

**如何快速判断:**

记住两个关键词:

- **不可重复读** = "值"变了
- **幻读** = "数量"变了

#### 提问2:在实际项目中,你遇到过哪些并发问题?如何解决的?

**回答:**

**案例1:库存超卖问题**

**问题描述:**

电商秒杀场景下,多个用户同时抢购同一商品,导致库存超卖。

```sql
-- 原始代码(读已提交级别)
START TRANSACTION;
SELECT stock FROM product WHERE id = 100;
-- 两个事务都读取到stock=10

-- 判断库存是否充足
IF stock > 0 THEN
    UPDATE product SET stock = stock - 1 WHERE id = 100;
END IF;
COMMIT;
-- 两个事务都执行成功,库存变成8,但实际应该只卖1个
```

**原因分析:**

这是**不可重复读**问题。两个事务读取到的库存值一样,都判断可以购买,导致超卖。

**解决方案:**

**方案1:使用乐观锁(推荐)**

```sql
-- 在表中添加version字段
ALTER TABLE product ADD COLUMN version INT DEFAULT 0;

START TRANSACTION;
SELECT stock, version FROM product WHERE id = 100;
-- 读取stock=10, version=1

IF stock > 0 THEN
    -- 使用版本号控制
    UPDATE product 
    SET stock = stock - 1, version = version + 1 
    WHERE id = 100 AND version = 1;
    
    -- 检查是否更新成功
    IF ROW_COUNT() = 0 THEN
        -- 版本号不匹配,说明其他事务已修改
        -- 返回库存不足或重试
        ROLLBACK;
        RETURN '库存不足,请重试';
    END IF;
END IF;
COMMIT;
```

**方案2:使用SELECT FOR UPDATE**

```sql
-- 设置隔离级别为可重复读
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

START TRANSACTION;
SELECT stock FROM product WHERE id = 100 FOR UPDATE;
-- 锁定记录,其他事务无法并发修改

IF stock > 0 THEN
    UPDATE product SET stock = stock - 1 WHERE id = 100;
END IF;
COMMIT;
-- 第二个事务会被阻塞,直到第一个事务提交
-- 避免超卖
```

**方案3:UPDATE条件判断(最简单)**

```sql
START TRANSACTION;
-- 直接在UPDATE中判断库存
UPDATE product SET stock = stock - 1 
WHERE id = 100 AND stock > 0;

-- 检查是否更新成功
IF ROW_COUNT() = 0 THEN
    -- 库存不足,返回失败
    ROLLBACK;
    RETURN '库存不足';
END IF;
COMMIT;
```

**案例2:报表数据不一致**

**问题描述:**

生成报表时,第一次查询订单数量,第二次查询订单金额,但两次查询的订单集合不一致,导致统计数据错误。

```sql
-- 读已提交级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

START TRANSACTION;

-- 第一步:查询订单数量
SELECT COUNT(*) FROM orders WHERE create_time >= '2026-01-01';
-- 返回:1000条

-- 第二步:处理其他业务逻辑...
-- 期间有新订单插入

-- 第三步:查询订单金额
SELECT SUM(amount) FROM orders WHERE create_time >= '2026-01-01';
-- 对应的订单数已经不是1000条了

-- 报表显示:
-- 订单数:1000
-- 订单总金额:基于1100条订单计算
-- 数据不一致!
COMMIT;
```

**原因分析:**

这是**幻读**问题。事务期间有新订单插入,导致两次查询的订单集不一致。

**解决方案:**

使用**可重复读**隔离级别。

```sql
-- 设置隔离级别为可重复读
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

START TRANSACTION;
-- 生成Read View,整个事务期间使用同一快照

-- 第一步:查询订单数量
SELECT COUNT(*) FROM orders WHERE create_time >= '2026-01-01';
-- 返回:1000条

-- 第二步:处理其他业务逻辑...
-- 期间有新订单插入

-- 第三步:查询订单金额
SELECT SUM(amount) FROM orders WHERE create_time >= '2026-01-01';
-- 对应的还是那1000条订单

-- 报表数据一致!
COMMIT;
```

**案例3:转账数据错误**

**问题描述:**

银行转账场景下,读取到未提交的转账记录,导致余额计算错误。

```sql
-- 读未提交级别(很少使用)
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;

-- 事务A
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;
-- 读取到balance=1000

-- 事务B
START TRANSACTION;
UPDATE account SET balance = 2000 WHERE id = 1;
-- 未提交

-- 事务A
SELECT balance FROM account WHERE id = 1;
-- 读取到balance=2000 (脏读!)

-- 事务B回滚
ROLLBACK;
-- balance恢复为1000

-- 事务A基于错误的balance=2000执行转账
-- 导致余额错误
COMMIT;
```

**原因分析:**

这是**脏读**问题。读取到了未提交的数据,如果该事务回滚,读取到的数据就是无效的。

**解决方案:**

使用**读已提交**或更高级别的隔离级别。

```sql
-- 设置隔离级别为读已提交
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

**案例4:订单状态跳变**

**问题描述:**

前端查询订单详情,显示"待支付",刷新后显示"已支付",用户体验差。

```sql
-- 读已提交级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 事务A(对应HTTP请求1)
START TRANSACTION;
SELECT status FROM orders WHERE order_id = 1001;
-- 返回:待支付
-- 前端显示:待支付

-- 事务B(支付系统)
START TRANSACTION;
UPDATE orders SET status = '已支付' WHERE order_id = 1001;
COMMIT;

-- 事务A(对应HTTP请求2,刷新页面)
SELECT status FROM orders WHERE order_id = 1001;
-- 返回:已支付
-- 前端显示:已支付
-- 用户看到状态跳变
```

**原因分析:**

这是**不可重复读**问题。同一个用户看到订单状态不断变化,体验差。

**解决方案:**

使用**可重复读**隔离级别,确保同一个会话内数据一致。

```sql
-- 设置隔离级别为可重复读
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 事务A(对应同一个会话)
START TRANSACTION;
-- 生成Read View

SELECT status FROM orders WHERE order_id = 1001;
-- 第一次查询:待支付

-- 其他事务修改
UPDATE orders SET status = '已支付' WHERE order_id = 1001;
COMMIT;

-- 事务A(刷新页面)
SELECT status FROM orders WHERE order_id = 1001;
-- 第二次查询:待支付 (可重复读)
-- 看到的还是事务开始时的状态

COMMIT;
```

## 📝 总结

## 📝 总结

**1. MySQL中的事务隔离级别**

MySQL中有四种事务隔离级别,从低到高依次是:

1) **读未提交(READ UNCOMMITTED)**:可以读取到其他事务未提交的数据,会出现脏读、不可重复读、幻读问题。

2) **读已提交(READ COMMITTED)**:只能读取到其他事务已提交的数据,解决了脏读,但会出现不可重复读和幻读。

3) **可重复读(REPEATABLE READ)**:MySQL的默认隔离级别,保证同一事务内多次读取同一数据的结果一致,解决了脏读和不可重复读,但可能出现幻读。MySQL InnoDB通过MVCC(多版本并发控制)和Next-Key Lock解决了幻读。

4) **串行化(SERIALIZABLE)**:最高隔离级别,强制事务串行执行,解决了所有并发问题,但性能最差。

---

**2. MySQL默认的事务隔离级别**

MySQL(InnoDB存储引擎)的默认事务隔离级别是**可重复读(REPEATABLE READ)**。

**为什么选择可重复读作为默认级别?**

1) **避免脏读和不可重复读**:解决最常见的并发问题

2) **通过MVCC实现高性能**:快照读不加锁,读写不冲突

3) **通过Next-Key Lock基本解决幻读**:基本解决幻读问题

4) **平衡一致性和性能**:比读已提交更一致,比串行化更高效

**各数据库默认隔离级别对比:**

| 数据库 | 默认隔离级别 | 说明 |
|--------|-------------|------|
| MySQL(InnoDB) | REPEATABLE READ | 通过MVCC和Next-Key Lock基本解决幻读 |
| Oracle | READ COMMITTED | 不支持REPEATABLE READ级别 |
| SQL Server | READ COMMITTED | 支持快照隔离 |
| PostgreSQL | READ COMMITTED | 可通过配置改为SERIALIZABLE |

---

**3. 数据库的脏读、不可重复读和幻读**

脏读、不可重复读、幻读是数据库事务并发访问时可能出现的三个问题。

**脏读(Dirty Read)**:一个事务读取到了另一个事务未提交的数据,如果该事务回滚,读取到的数据就是无效的。

**不可重复读(Non-repeatable Read)**:在同一个事务内,两次读取同一数据得到的结果不一样,因为期间有其他事务修改了该数据并提交。

**幻读(Phantom Read)**:在同一个事务内,前后两次执行相同的查询,返回的记录数不一样,因为期间有其他事务插入或删除了数据并提交。

**三者对比:**

| 问题类型 | 定义 | 触发条件 | 最小隔离级别 | 危害 |
|---------|------|---------|-------------|------|
| 脏读 | 读取到未提交的数据 | 读未提交隔离级别 | READ COMMITTED | 数据错误,违反原子性 |
| 不可重复读 | 同一行数据两次读取值不一样 | 事务内两次读取同一数据 | REPEATABLE READ | 业务逻辑错误,数据不一致 |
| 幻读 | 范围查询两次记录数不一样 | 事务内两次范围查询 | REPEATABLE READ(需Next-Key Lock) | 统计错误,数据集不一致 |

---

**核心知识点:**

- **MVCC**:通过Read View和Undo Log实现读不阻塞写,可重复读级别下Read View只在事务开始时生成一次
- **Next-Key Lock**:Record Lock + Gap Lock,解决幻读问题
- **快照读 vs 当前读**:不加锁读 vs 加锁读最新数据
- **隔离级别选择**:90%场景使用REPEATABLE READ,少数场景使用READ COMMITTED
- **MySQL vs Oracle**:MySQL默认RR级别避免不可重复读,Oracle默认RC级别会出现不可重复读
- **从Oracle迁移**:需要评估业务对不可重复读的容忍度,可能需要调整业务逻辑或使用SELECT FOR UPDATE
- **脏读 vs 不可重复读 vs 幻读**:
  - 脏读:读取未提交的数据
  - 不可重复读:同一行数据,两次读取值不一样
  - 幻读:相同查询,两次返回的记录数不一样
- **解决并发问题**:
  - 脏读:使用READ COMMITTED及以上级别
  - 不可重复读:使用REPEATABLE READ及以上级别
  - 幻读:使用REPEATABLE READ + Next-Key Lock,或SERIALIZABLE
