## 1. MySQL的存储引擎有哪些?它们之间有什么区别?

### 📌 回答重点

MySQL存储引擎是可插拔的，不同引擎负责数据的存储和读取方式。实际工作中95%以上的场景用的都是InnoDB，面试重点把InnoDB和MyISAM的区别搞清楚就行。

**MySQL 8.4版本一共提供了10个引擎，常见的有这几个：**

1) **InnoDB**：MySQL 5.5之后的默认引擎，支持事务、行级锁、外键，MVCC也有，适合高并发的OLTP场景。数据按聚簇索引组织，主键查询贼快。

2) **MyISAM**：老版本的默认引擎，不支持事务，只有表级锁，但读性能不错。适合那种写少读多、对一致性要求不高的场景，比如早年的一些报表系统。

3) **MEMORY**：数据全放内存里，速度快但MySQL重启数据就没了。一般拿来做临时表或者会话级缓存。

4) **Archive**：专门存归档数据的，只支持INSERT和SELECT，不支持索引，但压缩率高。日志归档、历史订单这种场景用得上。

5) **NDB**：MySQL Cluster用的引擎，支持分布式和高可用，数据自动分片，适合电信级别的大规模集群。

### 📚 扩展知识

#### InnoDB和MyISAM为什么差别这么大？

这俩引擎设计目标就不一样。MyISAM是MySQL早期的引擎，那会儿互联网还没爆发，大部分场景就是读多写少的静态页面，表锁够用了。InnoDB是后来Oracle收购的，从一开始就奔着企业级事务去的，行锁、MVCC、redo log、undo log这套东西都是为了保证ACID。

**从存储结构看：**

MyISAM的索引和数据是分开存的：
- `.MYI` 文件存索引
- `.MYD` 文件存数据
- 查询时先查索引拿到数据行的物理地址，再去数据文件里捞

InnoDB用的是聚簇索引：
- 数据直接挂在B+树叶子节点上
- 主键查询一次B+树就能拿到数据，不用二次寻址

**锁机制差别更大：**

MyISAM只有表锁，一个写操作会锁住整张表，并发写性能很差。

InnoDB支持行锁，通过MVCC实现了读写不阻塞，1000个并发改同一张表的不同行，互不干扰。

---

### 🎯 存储引擎怎么选

**99%的业务场景直接用InnoDB就完事了。**

MySQL 5.5之后InnoDB就是默认引擎，官方也一直在优化它，全文索引、空间索引这些原来MyISAM有的能力，InnoDB现在也支持了。

**只有几种特殊场景才考虑其他引擎：**

1) **临时计算结果集**，比如ETL中间表，可以用MEMORY引擎加速，但要注意数据易失性。

2) **日志归档、历史流水**这种只写不改的数据，Archive引擎压缩率高，能省不少磁盘。

3) **搞MySQL Cluster分布式集群**，才会用到NDB。

**MyISAM基本可以淘汰了**，MySQL 8.0已经把系统表从MyISAM迁移到InnoDB了，官方的态度很明确。

---

### 📈 存储引擎版本演进

- **MySQL 5.5**：InnoDB成为默认引擎，之前是MyISAM。
- **MySQL 5.6**：InnoDB支持全文索引，填补了相对MyISAM的一个短板。
- **MySQL 5.7**：InnoDB支持空间索引，JSON类型也来了。
- **MySQL 8.0**：系统表全部迁移到InnoDB，MyISAM只保留用于兼容，新增了原子DDL。
- **MySQL 8.4**：版本一共提供了10个引擎，有兴趣的话可以看看官方文档。

### 💬 面试官追问

#### 提问1：InnoDB的行锁是怎么实现的?是锁住整行数据吗?

**回答**：

InnoDB的行锁其实是锁索引记录，不是锁物理行。

- 如果SQL走的是**主键索引**，就锁主键索引上的那条记录
- 如果走的是**二级索引**，会先锁二级索引记录，再锁对应的主键记录
- 如果查询**没走索引**，那就退化成表锁了，因为要扫全表，所有记录都得锁

所以建索引不光是为了查询快，也是为了锁的粒度更细。

---

#### 提问2：MyISAM既然只有表锁，为什么还说它读性能好?

**回答**：

MyISAM读性能好主要是因为它没有事务开销：
- 不用维护undo log
- 也不用做MVCC版本链查找
- 读的时候直接从B+树拿数据地址，去数据文件里顺序读就行

但这个优势是在**无并发写或者写很少**的前提下，一旦有写操作，表锁会把所有读都阻塞住，性能就崩了。

---

#### 提问3：MEMORY引擎数据放内存里，和Redis比有什么区别?

**回答**：

| 对比项 | MEMORY引擎 | Redis |
|--------|-----------|-------|
| 归属 | MySQL进程内的内存存储 | 独立的KV存储服务 |
| 访问方式 | 只能通过SQL访问 | 支持多种协议（TCP、HTTP等） |
| 数据结构 | 固定是表结构 | 支持多种数据结构（String、List、Set、Hash等） |
| 持久化 | ❌ 不支持，重启就丢 | ✅ 支持RDB、AOF持久化 |
| 跨进程 | ❌ 不能跨进程访问 | ✅ 可以跨进程访问 |
| 适用场景 | MySQL内部做临时表计算 | 分布式缓存、会话存储 |

如果只是MySQL内部做临时表计算，MEMORY引擎够用；如果是分布式缓存、会话存储这种场景，用Redis。两者定位完全不一样。

#### 提问4：生产环境怎么查看当前表用的是什么存储引擎?

**回答**：

**方法1：使用 SHOW TABLE STATUS**

```sql
SHOW TABLE STATUS LIKE 'table_name';
```

或者指定数据库：

```sql
SHOW TABLE STATUS FROM 数据库名 LIKE 'table_name';
```

结果中的 `Engine` 列就是引擎类型。

---

**方法2：使用 SHOW CREATE TABLE**

```sql
SHOW CREATE TABLE table_name;
```

在建表语句的最后可以看到 `ENGINE=InnoDB` 这样的参数。

---

**方法3：批量查询所有表的引擎**

```sql
SELECT 
    TABLE_NAME, 
    ENGINE 
FROM 
    information_schema.TABLES 
WHERE 
    TABLE_SCHEMA = '数据库名';
```

这样可以一次性查看某个数据库下所有表的存储引擎。

## 2. MySQL InnoDB引擎中的聚簇索引和非聚簇索引有什么区别?

### 📌 回答重点

在InnoDB引擎中，聚簇索引的叶子节点直接存储完整的数据行，一张表只能有一个聚簇索引，默认就是主键索引。非聚簇索引的叶子节点只存储索引列的值和主键，要拿到完整数据得先查主键再回表，一张表可以有多个非聚簇索引。

**核心区别就是叶子节点存的东西不一样：**
- **聚簇索引**：存完整数据
- **非聚簇索引**：存主键值

这导致聚簇索引查询能一步到位拿到数据，非聚簇索引可能要多一次回表操作。

---

### 📚 扩展知识

#### B+树的结构细节

InnoDB的聚簇索引和非聚簇索引底层都是B+树。

**主键索引的B+树结构：**

1) 非叶子节点存储主键和页号，用来快速定位到叶子节点

2) 叶子节点存储完整的数据行

3) 叶子节点之间有双向链表链接，方便范围查询顺序扫描

4) 叶子节点内部有页目录，记录之间是单链表链接。查找时先通过页目录二分定位，再遍历链表找到具体记录

5) B+树只能帮助快速定位到页，而不是具体记录。页大小默认16KB

---

**非聚簇索引的B+树结构：**

和主键索引的差别就在叶子节点。非聚簇索引的叶子节点只存索引列的值和主键值，不存完整的数据行。

比如一张表有 `id`、`姓名`、`性别`、`地址` 字段，给 `姓名` 建一个索引，这个索引的叶子节点长这样：

| 姓名（索引列） | 主键id |
|---------------|--------|
| 张三          | 100     |
| 李四          | 101     |

如果查询 `SELECT * FROM user WHERE 姓名 = '张三'`，先在姓名索引树里找到主键id=100，再拿id去主键索引树里查完整数据，这就叫回表。

---

#### 为什么主键要有序

页大小默认16KB，数据按主键顺序排列。如果主键无序，新插入的记录可能要塞到页中间，容量满了就会触发页分裂，性能很差。

**所以主键最好是自增的**，这样新数据永远追加到末尾，不会产生页分裂。

用UUID做主键就是个典型的反面教材，每次插入都可能触发页分裂，写入性能能差3-5倍。

**示例对比：**

```sql
-- ✅ 推荐：自增主键
CREATE TABLE user (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50)
);

-- ❌ 不推荐：UUID主键
CREATE TABLE user (
    id CHAR(36) PRIMARY KEY,  -- UUID字符串
    name VARCHAR(50)
);
```

---

#### 回表的代价

通过非聚簇索引查询时，如果需要的数据不在索引里，就得拿着主键再去聚簇索引查一遍，这个过程叫回表。

**回表的代价不小**，因为聚簇索引和非聚簇索引是两棵独立的B+树，回表意味着要额外走一遍B+树查找。

如果查询返回大量数据，每条都要回表，可能产生大量随机I/O，性能急剧下降。

**这也是为什么不推荐SELECT ***。比如通过姓名查询，只需要id和姓名，姓名索引的叶子节点已经存了这些数据，直接返回就行。但如果用 `SELECT *`，得把性别、地址这些不需要的字段也查出来，就得回表去主键索引取数据，白白浪费I/O。**

**示例对比：**

```sql
-- ✅ 推荐：只查需要的字段（可能走覆盖索引，不需要回表）
SELECT id, name FROM user WHERE name = '张三';

-- ❌ 不推荐：SELECT *（可能需要回表）
SELECT * FROM user WHERE name = '张三';
```

---

#### 和MyISAM的对比

MyISAM没有聚簇索引，它的主键索引和普通索引结构一样，叶子节点都只存数据行的物理地址。数据文件和索引文件是分开存储的，查询时先在索引里找到地址，再去数据文件里取数据。

**特性对比：**

| 特性 | InnoDB聚簇索引 | MyISAM索引 |
|------|---------------|-----------|
| 叶子节点内容 | 完整数据行 | 数据行的物理地址 |
| 数据存储 | 索引即数据，数据即索引 | 索引和数据分离 |
| 主键查询 | 一次B+树查找 | B+树查找+磁盘寻址 |
| 范围查询 | 高效，数据物理连续 | 可能产生大量随机I/O |

InnoDB选择聚簇索引的好处是：主键查询效率高，范围查询时数据物理上连续，能利用磁盘预读。坏处是二级索引查询多一次回表，插入时维护数据有序有开销。

### 💬 面试官追问

#### 提问1：如果一张表没有主键，InnoDB怎么处理聚簇索引?

**回答：**

InnoDB会找第一个非空的唯一索引作为聚簇索引。如果连唯一索引都没有，InnoDB会自动生成一个6字节的隐含列 `row_id` 作为聚簇索引。

**但这个隐藏列对业务不可见，查询时用不上，所以建表时最好显式指定主键。**

**示例：**

```sql
-- ✅ 推荐：显式指定主键
CREATE TABLE user (
    id INT PRIMARY KEY,
    name VARCHAR(50)
);

-- ⚠️ InnoDB自动处理：没有主键和唯一索引
CREATE TABLE user (
    name VARCHAR(50)  -- InnoDB会生成隐藏的row_id
);
```

---

#### 提问2：为什么InnoDB的非聚簇索引存主键而不是存数据行的物理地址?

**回答：**

因为聚簇索引的数据物理位置不是固定的。页分裂、页合并、数据更新都可能导致数据行的物理地址变化。

如果非聚簇索引存物理地址，每次数据移动都得更新所有相关的非聚簇索引，维护成本太高。

**存主键的话，主键值是稳定的，数据怎么移动都不影响二级索引。**

**示意图：**

```
数据移动时：
聚簇索引数据行位置：页A → 页B（物理地址变化）

如果非聚簇索引存物理地址：
- 需要更新所有相关二级索引 ❌ 成本高

如果非聚簇索引存主键：
- 主键值不变，不需要更新二级索引 ✅ 成本低
```

---

#### 提问3：什么时候非聚簇索引查询不需要回表?

**回答：**

当查询的列都在索引里时，不需要回表，这叫**覆盖索引**。

比如索引是 `(name, age)`，查询 `SELECT name, age FROM t WHERE name = 'xx'`，索引叶子节点存了 `name`、`age`、主键，完全够用，直接返回就行。

EXPLAIN的 `Extra` 列会显示 `Using index`，说明走了覆盖索引。

**示例：**

```sql
-- 创建复合索引
CREATE INDEX idx_name_age ON user(name, age);

-- ✅ 覆盖索引：所有列都在索引里，不需要回表
SELECT name, age FROM user WHERE name = '张三';
-- EXPLAIN Extra: Using index

-- ❌ 需要回表：address不在索引里
SELECT name, age, address FROM user WHERE name = '张三';
-- EXPLAIN Extra: 无（需要回表）
```

## 3. MySQL的索引类型有哪些?

### 📌 回答重点

MySQL索引可以从三个维度来分类：数据结构、存储方式、索引性质。面试时一般先从数据结构角度切入，重点讲B+树索引，然后再展开聊聚簇索引和非聚簇索引的区别。

---

#### 从数据结构角度看，MySQL支持这几种索引：

1) **B+树索引**：最常用的，InnoDB和MyISAM默认都用它。多层平衡树结构，叶子节点用链表串起来，既能快速定位单条记录，又能高效做范围扫描。

2) **哈希索引**：通过哈希函数直接算出数据位置，等值查询O(1)级别，但不支持范围查询和排序。Memory引擎默认用哈希索引，InnoDB有个自适应哈希索引会自动建，不用手动管。

3) **全文索引**：把文本分词后建倒排索引，类似搜索引擎的原理。适合对TEXT类型字段做关键字搜索，比如文章内容检索。

4) **空间索引**：基于R树实现，专门处理地理坐标这种多维数据，支持区域查询、距离计算。

---

#### 从InnoDB存储方式看，分两类：

1) **聚簇索引**：主键索引就是聚簇索引，叶子节点直接存完整的行数据，数据按主键顺序物理存储。一张表只能有一个聚簇索引。

2) **非聚簇索引**：也叫二级索引，叶子节点只存索引字段值和主键值。查完二级索引还得拿着主键去聚簇索引里再查一遍，这个过程叫回表。

---

#### 从索引性质看，有这几种：

1) **主键索引**：唯一且非空，每张表只能有一个。InnoDB里主键索引就是聚簇索引。

2) **唯一索引**：保证列值不重复，但允许有NULL，可以有多个NULL。

3) **普通索引**：没有唯一约束，纯粹为了加速查询。

4) **联合索引**：多列组合成一个索引，遵循最左前缀原则，列顺序很重要。

5) **全文索引**：文本搜索用。

6) **空间索引**：GIS数据用。

---

### 📚 扩展知识

#### B+树索引为什么是主力

数据库查询有两大类操作：等值查询和范围查询。哈希索引等值查询快，但范围查询搞不定。B+树两个都能扛，而且叶子节点用双向链表连起来，范围扫描的时候顺着链表走就行，不用回溯。

B+树还有个特点是树高很矮。一个3层的B+树，能存2000万左右的数据。查任何一条数据最多3次磁盘I/O，这个性能在机械硬盘时代非常重要。

**对比示意：**

| 索引类型 | 等值查询 | 范围查询 | 排序 | 适用场景 |
|---------|-----------|-----------|------|---------|
| B+树索引 | ✅ 快 | ✅ 快 | ✅ 支持 | 绝大多数场景 |
| 哈希索引 | ✅ 极快 | ❌ 不支持 | ❌ 不支持 | 等值查询多 |
| 全文索引 | ⚠️ 中等 | ❌ 不支持 | ✅ 相关性排序 | 文本搜索 |
| 空间索引 | ✅ 快 | ✅ 区域查询 | ❌ 不支持 | GIS数据 |

---

#### 聚簇索引和非聚簇索引的本质区别

聚簇索引的叶子节点存的是完整的行数据，数据和索引绑定在一起。InnoDB表的数据文件本身就是按主键组织的B+树，所以主键查询只需要一次索引查找就能拿到数据。

非聚簇索引的叶子节点只存索引列的值和主键值。比如在 `name` 列上建索引，叶子节点存的是 `(name, id)` 这样的组合。查的时候先在name索引里找到对应的id，再拿着id去聚簇索引里查完整数据，这就是回表。

**示例流程：**

```sql
-- 执行查询
SELECT * FROM user WHERE name = 'Alice';

-- 流程：
1. 在name索引树中找到name='Alice'的记录，得到id=100
2. 拿着id=100去主键索引树中查找
3. 返回完整数据行
```

如果查询只需要索引列本身的值，比如 `SELECT id FROM user WHERE name = 'Alice'`，name索引的叶子节点里已经有id了，不用回表，这就是**覆盖索引**的原理。

#### 联合索引和最左前缀原则

联合索引是把多个列按顺序组合成一个索引。比如建 `(a, b, c)` 联合索引，B+树先按a排序，a相同的再按b排序，b相同的再按c排序。

这就导致一个规则：**查询条件必须从最左边的列开始才能用上索引。**

**示例：**

假设有联合索引 `(a, b, c)`

| 查询条件 | 能用索引 | 原因 |
|---------|---------|------|
| WHERE a=1 | ✅ 能用 | 从最左边a开始 |
| WHERE a=1 AND b=2 | ✅ 能用 | a和b都匹配 |
| WHERE a=1 AND b=2 AND c=3 | ✅ 能用 | a、b、c都匹配 |
| WHERE b=2 | ❌ 用不上 | 没有a，违背最左前缀 |
| WHERE a=1 AND c=3 | ⚠️ 只能用a | a能用，c用不上 |

**建联合索引时，选择性高的列放前面**。选择性就是不重复值的比例，比如性别只有男女两种，选择性低；手机号每个都不一样，选择性高。

**示例：**

```sql
-- ✅ 推荐：高选择性列放前面
CREATE INDEX idx_phone_name ON user(phone, name);

-- ⚠️ 不推荐：低选择性列放前面
CREATE INDEX idx_gender_name ON user(gender, name);
```

---

#### 索引创建示例

```sql
-- 1. 主键索引（聚簇索引）
CREATE TABLE user (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    age INT
);

-- 2. 唯一索引
CREATE UNIQUE INDEX idx_unique_email ON user(email);

-- 3. 普通索引
CREATE INDEX idx_name ON user(name);

-- 4. 联合索引（注意列顺序）
CREATE INDEX idx_name_age ON user(name, age);

-- 5. 全文索引
CREATE FULLTEXT INDEX idx_content ON article(content);

-- 6. 空间索引
CREATE SPATIAL INDEX idx_location ON location(geo_point);

-- 7. 查看表的索引
SHOW INDEX FROM user;

-- 8. 删除索引
DROP INDEX idx_name ON user;
```

### 💬 面试官追问

#### 提问1：InnoDB里如果没有显式定义主键，聚簇索引怎么建?

**回答：**

InnoDB会找第一个非空的唯一索引当聚簇索引。如果连唯一索引都没有，InnoDB会自己生成一个6字节的隐藏列 `row_id` 作为聚簇索引的键。

**这个 `row_id` 是全局递增的，不是表级别的。**

所以建表时最好显式指定主键，否则这个隐藏主键既浪费空间，排查问题时也看不到。

**示例：**

```sql
-- ✅ 推荐：显式指定主键
CREATE TABLE user (
    id INT PRIMARY KEY,
    name VARCHAR(50)
);

-- ⚠️ InnoDB自动处理：没有主键和唯一索引
CREATE TABLE user (
    name VARCHAR(50)  -- InnoDB会生成隐藏的row_id
);
```

---

#### 提问2：哈希索引既然等值查询这么快，为什么不是默认索引?

**回答：**

三个原因。

**第一，哈希索引不支持范围查询**，`WHERE id > 100` 这种条件没法用。

**第二，不支持排序**，`ORDER BY` 用不上。

**第三，哈希冲突多的时候性能会退化**。

数据库查询大部分场景是范围查询、排序、分页，哈希索引应付不了，**B+树全能选手更适合做默认**。

**对比：**

| 特性 | 哈希索引 | B+树索引 |
|------|----------|----------|
| 等值查询 | ✅ 极快 O(1) | ✅ 快 O(log N) |
| 范围查询 | ❌ 不支持 | ✅ 支持 |
| 排序 | ❌ 不支持 | ✅ 支持 |
| 冲突处理 | ⚠️ 性能退化 | ❌ 无冲突 |
| 默认选择 | ❌ Memory引擎 | ✅ InnoDB/MyISAM |

#### 提问3：联合索引(a,b)和分别建a、b两个索引，有什么区别?

**回答：**

联合索引 `(a, b)` 是一棵B+树，先按a排序再按b排序。`WHERE a=1 AND b=2` 这种查询一次索引扫描就搞定。

分别建两个单列索引是两棵树，MySQL可能会用索引合并，但效率不如联合索引。

**联合索引还能实现覆盖索引**，比如 `SELECT b FROM t WHERE a=1`，不用回表。

**缺点是联合索引占用空间更大**，更新维护成本也高一点。

**示例对比：**

```sql
-- 方案1：联合索引（推荐）
CREATE INDEX idx_a_b ON user(a, b);
-- 优势：
-- 1. WHERE a=1 AND b=2 走一棵树，效率高
-- 2. SELECT b FROM t WHERE a=1 走覆盖索引，不回表

-- 方案2：两个单列索引（不推荐）
CREATE INDEX idx_a ON user(a);
CREATE INDEX idx_b ON user(b);
-- 劣势：
-- 1. WHERE a=1 AND b=2 可能用索引合并，效率低
-- 2. SELECT b FROM t WHERE a=1 需要回表
-- 3. 占用两棵树的存储空间
```

#### 提问4：全文索引和LIKE '%keyword%'有什么区别?

**回答：**

`LIKE 'keyword%'` 是前缀匹配，可以用B+树索引。但 `LIKE '%keyword%'` 是前后模糊匹配，没法用B+树索引，只能全表扫描。

**全文索引是分词后建倒排索引，能快速定位包含某个词的记录**。全文索引支持自然语言搜索、布尔模式搜索，还能按相关性排序。

但全文索引对中文支持不太好，需要配合ngram分词器。

**如果是正经的全文搜索需求，一般用Elasticsearch比MySQL全文索引靠谱。**

**示例对比：**

```sql
-- 1. 前缀匹配（可以用普通索引）
SELECT * FROM article WHERE title LIKE 'MySQL%';
-- 索引：idx_title (B+树)
-- EXPLAIN：type=range，能用到索引

-- 2. 前后模糊匹配（全表扫描）
SELECT * FROM article WHERE content LIKE '%优化%';
-- 无法用普通索引
-- EXPLAIN：type=ALL，全表扫描 ❌

-- 3. 全文索引（支持）
SELECT * FROM article 
WHERE MATCH(content) AGAINST('优化' IN NATURAL LANGUAGE MODE);
-- 索引：FULLTEXT idx_content
-- EXPLAIN：type=fulltext，能用到索引 ✅
```

**建议：**
- 简单前缀匹配：用普通索引 + `LIKE 'prefix%'`
- 复杂全文搜索：用Elasticsearch专业搜索引擎
- 不推荐：MySQL全文索引处理 `LIKE '%keyword%'`

## 📝 总结

| 存储引擎 | 核心特性 | 适用场景 |
|---------|---------|---------|
| InnoDB | 事务+行锁+外键+聚簇索引 | 99%的业务场景，高并发OLTP |
| MyISAM | 表锁+索引分离 | 读多写少，已基本淘汰 |
| MEMORY | 纯内存存储 | 临时表、会话缓存 |
| Archive | 高压缩+只读 | 归档数据、日志存储 |
| NDB | 分布式+高可用 | 大规模集群 |

**简单记忆：** MySQL存储引擎 = InnoDB（99%默认）+ 其他特殊场景引擎（MEMORY/Archive/NDB）
