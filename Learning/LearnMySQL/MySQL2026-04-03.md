## 1. 为什么MySQL选择使用B+树作为索引结构？

### 📌 回答重点

MySQL选B+树核心就一个原因：**磁盘I/O次数最少**。数据库的数据存在磁盘上，磁盘随机读写比内存慢10万倍，所以索引结构的设计目标就是尽量减少磁盘访问次数。

B+树能做到这一点靠三个特性：

1) **树矮。** B+树是多叉树，一个节点能存几百上千个key。3层的B+树就能存两千多万条数据，查任何一条最多3次磁盘I/O。红黑树是二叉树，存同样的数据量要二十多层，I/O次数直接爆炸。

2) **非叶子节点只存key和指针，不存数据。** 一个16KB的页能塞进更多索引项，内存里能缓存更多索引，命中率高，磁盘访问少。

3) **叶子节点用双向链表串起来。** 范围查询时，定位到起点后顺着链表往后扫就行，不用再回根节点重新查找，顺序I/O比随机I/O快太多。

### 📚 扩展知识

#### 为什么不用哈希表

哈希表等值查询O(1)，看起来比B+树的O(log n)快多了，但数据库场景下有两个致命问题：

**第一，不支持范围查询。** `WHERE id > 100 AND id < 200` 这种SQL，哈希表只能全表扫描，B+树顺着叶子链表走一遍就完事。

**第二，不支持排序。** `ORDER BY` 得把数据捞出来再排，B+树的叶子节点天然有序，扫描出来就是排好的。

**不过InnoDB也没完全放弃哈希**，它有个自适应哈希索引。如果发现某个索引值被频繁访问，会自动在内存里建一个哈希索引加速查询。这个对业务透明，不需要手动创建。

**对比表格：**

| 特性 | 哈希表 | B+树 |
|------|--------|------|
| 等值查询 | ✅ O(1) 极快 | ✅ O(log n) 快 |
| 范围查询 | ❌ 不支持 | ✅ 支持 |
| 排序 | ❌ 不支持 | ✅ 支持 |
| 适用场景 | 等值查询多 | 绝大多数场景 |

---

#### 为什么不用红黑树、AVL树

红黑树和AVL树都是二叉树，每个节点最多两个子节点。数据量一大，树的高度蹭蹭往上涨。1000万条数据，红黑树大概23层，查一条数据最坏情况要23次磁盘I/O。

B+树是多叉树，一个节点能存多少个key取决于页的大小和key的长度。InnoDB默认页大小16KB，假设主键是bigint占8字节，加上6字节的指针，一个非叶子节点能存16KB/14B≈1170个key。

**三层B+树容量计算：**
- 第一层：1个节点
- 第二层：1170个节点
- 第三层：1170×1170≈137万个叶子节点

每个叶子节点假设存16条数据，三层就能存**2000多万条数据**，查任何一条最多3次I/O。

这个差距就是为什么**内存数据库可以用红黑树，磁盘数据库必须用B+树**。

---

#### B+树和B树的区别

B树也是多叉平衡树，和B+树最大的区别是数据存储位置：

1) **B树每个节点都存完整数据，B+树只有叶子节点存数据，非叶子节点只存key和指针。** 这意味着B+树的非叶子节点更小，一页能装更多索引项，树更矮。

2) **B+树叶子节点用链表串起来，范围查询直接顺序扫描。** B树做范围查询要不断回到上层节点再往下找，I/O次数多。

3) **B树有可能在非叶子节点就找到数据直接返回，查询时间不稳定。** B+树必须查到叶子节点，时间稳定可预期。

**对比表格：**

| 特性 | B树 | B+树 |
|------|-----|------|
| 数据存储位置 | 所有节点 | 仅叶子节点 |
| 范围查询 | 多次回溯，I/O多 | 顺序扫描叶子链表，I/O少 |
| 查询稳定性 | 可能中途返回 | 必须查到叶子，稳定 |
| 适用场景 | 文件系统（如MongoDB早期版本） | 数据库（MySQL、PostgreSQL） |

### 🎯 InnoDB的B+树实现细节

InnoDB的索引分两种：**聚簇索引**和**二级索引**。

**聚簇索引**的叶子节点存的是完整的行数据，按主键排序。一张表只有一个聚簇索引，就是主键索引。如果没定义主键InnoDB会找一个唯一非空索引代替，都没有就自己生成一个6字节的rowid。

**二级索引**的叶子节点存的是索引列的值+主键值。通过二级索引查数据时，先在二级索引树上找到主键，再回聚簇索引树上查完整数据，这就是**回表**。

**聚簇索引 vs 二级索引：**

| 索引类型 | 叶子节点内容 | 查询方式 | 一张表数量 |
|---------|-------------|---------|-----------|
| 聚簇索引 | 完整数据行 | 一次B+树查找 | 只有1个 |
| 二级索引 | 索引列值+主键值 | 二次B+树查找（回表） | 可以有多个 |

**示例：**

```sql
-- 执行查询
SELECT * FROM user WHERE name = '张三';

-- 流程：
-- 1. 在name索引树中找到name='张三'的记录，得到id=100
-- 2. 拿着id=100去主键索引树中查找完整数据
-- 3. 返回完整数据行
```

### 📄 页分裂和页合并

B+树的节点就是数据页，InnoDB默认16KB。

**数据插入时**如果页满了，会触发**页分裂**：把一个页拆成两个，各存一半数据。

**删除数据时**如果页的利用率太低，会触发**页合并**：把相邻两个页合成一个。

页分裂和页合并都涉及数据搬迁和指针修改，开销不小。所以**自增主键比UUID主键好**，自增主键总是往后追加，不会触发中间页的分裂。UUID是随机的，插入位置也随机，频繁分裂影响性能。

**示例对比：**

```sql
-- ✅ 推荐：自增主键
CREATE TABLE user (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50)
);
-- 优势：新数据追加到末尾，不触发页分裂

-- ❌ 不推荐：UUID主键
CREATE TABLE user (
    id CHAR(36) PRIMARY KEY,  -- UUID字符串
    name VARCHAR(50)
);
-- 劣势：随机插入，频繁触发页分裂
```

**页分裂过程示意：**

```
原页（16KB，满了）：
┌─────────────────────────┐
│ id:1,2,3,...,1000      │
└─────────────────────────┘

插入新记录id=500后，触发页分裂：

页A（前半部分）：
┌─────────────────────────┐
│ id:1,2,3,...,500       │
└─────────────────────────┘

页B（后半部分）：
┌─────────────────────────┐
│ id:501,502,...,1001    │
└─────────────────────────┘
```

### 💬 面试官追问

#### 提问1：MySQL的B+树一个节点能存多少个key？

**回答：**

一个节点能存多少个key取决于**页的大小**和**key的长度**。

InnoDB默认页大小16KB，假设主键是bigint占8字节，加上6字节的指针：

- 每个索引项大小 = 8字节（key）+ 6字节（指针）= 14字节
- 一个页能存 = 16KB / 14B ≈ 1170个key

如果是联合索引 `(a, b, c)`，key的长度是三个字段的总和，能存的key数量会少一些。

**计算公式：**

```
索引项大小 = key长度 + 指针长度（6字节）
每页key数量 = 页大小（16KB）/ 索引项大小
```

---

#### 提问2：为什么InnoDB的B+树叶子节点是双向链表，单向链表不行吗？

**回答：**

双向链表主要是为了优化**范围查询**的性能。

如果是单向链表，从第3个记录往后扫到第100个记录，只能顺序往下走。但如果想往前扫，就得重新从根节点开始找起点。

双向链表可以：
- 向前向后灵活遍历
- 某些查询优化时可能需要向前回溯
- `ORDER BY id DESC` 降序查询时，反向遍历更高效

**示例对比：**

```sql
-- 单向链表：只能从头开始重新查找
SELECT * FROM user WHERE id < 100 AND id > 50;

-- 双向链表：可以直接在范围内双向移动
SELECT * FROM user WHERE id < 100 AND id > 50;
-- 找到id=99后，可以直接往前走到50
```

---

#### 提问3：B+树的范围查询为什么比B树快？

**回答：**

核心原因是**叶子节点的组织方式不同**。

B+树叶子节点用链表串起来，范围查询时：
1. 定位到起点（O(log n)）
2. 顺着叶子链表顺序往后扫（O(m)，m是范围大小）

B树做范围查询时：
1. 找到起点（O(log n)）
2. 每个记录都要从根节点重新往下找（O(m × log n)）

**时间复杂度对比：**

| 树类型 | 范围查询复杂度 | 说明 |
|--------|---------------|------|
| B+树 | O(log n + m) | 找起点一次，后续顺序扫描 |
| B树 | O(m × log n) | 每条记录都要重新找 |

当m很大时（比如查询1000条记录），B+树的优势非常明显。

#### 提问4：三层B+树能存多少数据，你是怎么算的？

**回答：**

假设主键是bigint占8字节，页指针6字节，一个非叶子节点能存16KB/14B≈1170个key。叶子节点存完整行，假设一行1KB，一个叶子节点存16条。

三层B+树根节点指向1170个第二层节点，每个第二层节点指向1170个叶子，总共1170×1170≈137万个叶子，乘以每个叶子16条，大概**2000多万条数据**。行数据越小能存越多。

**详细计算过程：**

```
第一层（根节点）：
- 1个节点
- 每个节点存1170个key
- 指向1170个第二层节点

第二层：
- 1170个节点
- 每个节点存1170个key
- 指向 1170×1170 ≈ 137万个第三层节点

第三层（叶子节点）：
- 约137万个叶子节点
- 每个叶子节点假设存16条数据
- 总数据量 = 137万 × 16 ≈ 2192万条
```

**影响因素：**
- 主键长度：主键越短，非叶子节点能存越多key
- 行数据大小：行越小，叶子节点能存越多数据
- 页大小：InnoDB默认16KB，可以调整（影响不大）

#### 提问5：为什么MongoDB早期用B树，后来又换成B+树了？

**回答：**

MongoDB早期定位是文档数据库，查询模式以单文档查询为主，B树非叶子节点就能存数据，命中了直接返回，看起来更快。

但后来发现范围查询和排序需求越来越多，B树这方面太吃亏。WiredTiger存储引擎引入后就换成了B+树的变种，叶子节点也串起来了，范围查询性能大幅提升。

**MongoDB存储引擎演进：**

| 版本 | 存储引擎 | 索引结构 | 原因 |
|------|---------|---------|------|
| 早期 | MMAPv1 | B树 | 单文档查询快，实现简单 |
| 3.0+ | WiredTiger | B+树变种 | 支持压缩、范围查询、事务 |

**WiredTiger的优势：**
- 文档级锁（MMAPv1是集合级锁）
- 支持压缩（节省50%+磁盘空间）
- B+树结构，范围查询更高效
- 支持事务（4.0版本开始）

#### 提问6：自适应哈希索引是怎么工作的？什么时候该关掉它？

**回答：**

InnoDB监控索引页的访问模式，如果发现某个索引值被反复查询，就在内存里给它建个哈希索引。下次查同样的值直接哈希定位，跳过B+树查找。

这个功能对高并发点查场景有帮助，但它本身也消耗内存和CPU，而且在高并发写的场景下，维护哈希索引的锁竞争反而成了瓶颈。如果监控发现AdaptiveHashIndex的锁等待很高，可以考虑关掉。

**工作原理：**

```
1. InnoDB监控索引页的访问模式
2. 检测到某些索引值被频繁访问
3. 自动在内存中为这些热点值建立哈希索引
4. 后续查询优先查哈希索引
5. 如果没命中，再走B+树
```

**查看自适应哈希索引状态：**

```sql
-- 查看自适应哈希索引使用情况
SHOW ENGINE INNODB STATUS;

-- 或者
SELECT * FROM information_schema.INNODB_METRICS
WHERE NAME LIKE 'adaptive%';
```

**关闭自适应哈希索引：**

```sql
-- 在my.cnf/my.ini中添加
[mysqld]
innodb_adaptive_hash_index=0

-- 或运行时关闭
SET GLOBAL innodb_adaptive_hash_index=0;
```

**适用场景：**
- ✅ 高并发点查（主键查询、唯一索引查询）
- ❌ 高并发写入场景（锁竞争）
- ❌ 范围查询、排序场景（用不上）

#### 提问7：为什么InnoDB要求主键尽量短？

**回答：**

两个原因。

**第一，主键越短，非叶子节点能存的key越多，树越矮，I/O次数越少。**

**第二，所有二级索引的叶子节点都存了主键值，主键越长，二级索引占的空间越大。**一个表有5个二级索引，主键从int换成36字节的UUID，索引体积能膨胀好几倍。

**示例对比：**

假设一张表有5个二级索引，每条二级索引记录都存主键值。

| 主键类型 | 主键长度 | 每条二级索引记录 | 5个索引的总开销（1000万数据） |
|---------|---------|-----------------|------------------------------|
| INT | 4字节 | 4字节 | 4B × 1000万 × 5 ≈ 190MB |
| BIGINT | 8字节 | 8字节 | 8B × 1000万 × 5 ≈ 381MB |
| CHAR(36) | 36字节 | 36字节 | 36B × 1000万 × 5 ≈ 1.7GB |

**主键选择建议：**

```sql
-- ✅ 推荐：自增INT/BIGINT
CREATE TABLE user (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50)
);

-- ⚠️ 不推荐：UUID
CREATE TABLE user (
    id CHAR(36) PRIMARY KEY,  -- 36字节
    name VARCHAR(50)
);

-- ⚠️ 不推荐：复合主键
CREATE TABLE user (
    user_id INT,
    order_id INT,
    PRIMARY KEY (user_id, order_id)  -- 8字节
);
```

**特殊情况：**
- 分库分表场景：可能需要UUID避免主键冲突
- 数据导入：可以用雪花算法生成的分布式ID
- 避免主键暴露：不能直接暴露自增ID

## 2. MySQL索引的最左前缀匹配原则是什么？

### 📌 回答重点

MySQL索引的最左前缀匹配原则是指使用联合索引时，查询条件必须从索引的最左列开始匹配。联合索引包含多个列，查询时必须先有第一个列的条件，然后是第二个列，以此类推。

这是因为联合索引在B+树中的排列方式是"从左到右"的顺序。比如联合索引 `(first_name, last_name, age)` 会先按 `first_name` 排序，`first_name` 相同再按 `last_name` 排序，`last_name` 相同再按 `age` 排序。

MySQL查找时会优先用 `first_name` 作为匹配依据，然后依次用 `last_name` 和 `age`。跳过最左侧字段，后面的列在B+树中是无序的，压根没法利用索引快速定位。

**示例说明：**

假设有联合索引 `(first_name, last_name, age)`：

| 查询条件 | 能用索引 | 原因 |
|---------|---------|------|
| WHERE first_name = '张' | ✅ 能用 | 从最左列开始匹配 |
| WHERE first_name = '张' AND last_name = '三' | ✅ 能用 | 完全匹配前两列 |
| WHERE first_name = '张' AND last_name = '三' AND age = 25 | ✅ 能用 | 完全匹配三列 |
| WHERE last_name = '三' | ❌ 用不上 | 跳过了最左列first_name |
| WHERE age = 25 | ❌ 用不上 | 跳过了前两列 |
| WHERE last_name = '三' AND age = 25 | ❌ 用不上 | 跳过了最左列 |

### 📚 扩展知识

#### 匹配与不匹配的场景

假设当前表有一个联合索引 `(a, b, c)`。

**符合最左匹配原则的查询条件：**

```sql
WHERE a = 1;
WHERE a = 1 AND b = 2;
WHERE a = 1 AND b = 2 AND c = 3;
```

**不符合最左匹配原则的查询条件：**

```sql
WHERE b = 2;
WHERE c = 3;
WHERE b = 2 AND c = 3;
```

**再看这个查询：**

```sql
WHERE a = 1 AND c = 3;
```

这个查询可以用上联合索引 `(a, b, c)`，但在MySQL 5.6之前只能利用 `a = 1` 这个条件过滤。

**MySQL 5.6版本引入了索引下推优化**，可以在引擎层查到 `a = 1` 的数据后，直接用 `c` 过滤掉不符合的数据，减少回表次数。

**索引下推示例：**

```sql
-- 假设有联合索引 (name, age)
SELECT * FROM user WHERE name = '张' AND age > 25;

-- MySQL 5.6之前：
-- 1. 在索引中找到name='张'的所有记录（可能1000条）
-- 2. 全部回表到聚簇索引，再过滤age>25

-- MySQL 5.6+（索引下推）：
-- 1. 在索引中找到name='张'的所有记录
-- 2. 在索引层直接过滤age>25（可能只剩100条）
-- 3. 只回表这100条数据
```

---

#### 范围查询的特殊规则

理解范围查询之前，得先搞清楚联合索引的排序规则。

联合索引 `(a, b, c)` 在B+树中的排序是：先按 `a` 的值排序，`a` 相同再按 `b` 排序，`b` 相同再按 `c` 排序。

**范围查询的特殊规则：**

一旦在某个列上使用了范围查询（`>`、`<`、`BETWEEN`、`LIKE 'xxx%'`），**该列后面的所有列都无法使用索引**。

**示例说明：**

假设有联合索引 `(a, b, c)`：

| 查询条件 | 能用索引 | 使用索引的列 | 原因 |
|---------|---------|-------------|------|
| WHERE a = 1 | ✅ 能用 | a | 等值查询，可以用 |
| WHERE a = 1 AND b = 2 | ✅ 能用 | a, b | 都是等值查询 |
| WHERE a = 1 AND b > 2 | ✅ 能用 | a, b | b是范围查询，但能用到 |
| WHERE a = 1 AND b > 2 AND c = 3 | ⚠️ 只能用a, b | a, b | b是范围查询，c用不上 |
| WHERE a > 1 | ✅ 能用 | a | 范围查询，可以用 |
| WHERE a > 1 AND b = 2 | ⚠️ 只能用a | a | a是范围查询，b用不上 |
| WHERE a > 1 AND b > 2 | ⚠️ 只能用a | a | a是范围查询，b用不上 |

**关键点：**

1) **等值查询 + 等值查询**：所有列都能用索引
2) **等值查询 + 范围查询**：范围查询之前的列能用，之后的用不上
3) **范围查询 + 任意条件**：后面的列都用不上

**实际案例：**

```sql
-- 假设表结构
CREATE TABLE orders (
    id INT PRIMARY KEY,
    user_id INT,
    order_date DATE,
    amount DECIMAL(10,2),
    INDEX idx_user_date_amount (user_id, order_date, amount)
);

-- 案例1：完全使用索引
SELECT * FROM orders 
WHERE user_id = 100 AND order_date = '2026-04-03' AND amount = 1000;
-- 使用索引：user_id, order_date, amount

-- 案例2：部分使用索引
SELECT * FROM orders 
WHERE user_id = 100 AND order_date > '2026-04-01';
-- 使用索引：user_id, order_date
-- amount用不上，因为是范围查询之后的列

-- 案例3：只能用第一列
SELECT * FROM orders 
WHERE user_id = 100 AND order_date > '2026-04-01' AND amount > 500;
-- 使用索引：user_id, order_date
-- amount用不上

-- 案例4：范围查询在第一列
SELECT * FROM orders 
WHERE user_id > 100 AND order_date = '2026-04-03';
-- 使用索引：只有user_id
-- order_date用不上，因为user_id是范围查询
```

**为什么范围查询后的列用不上？**

因为范围查询返回的是一个**区间**，而不是**精确值**。

```
索引 (a, b, c) 的数据结构示意：

a=1, b=1, c=1
a=1, b=1, c=2
a=1, b=1, c=3
a=1, b=2, c=1  ← b开始变化
a=1, b=2, c=2
a=2, b=1, c=1  ← a开始变化
...

查询：WHERE a = 1 AND b > 1 AND c = 3

1. 先用 a=1 定位到 a=1 的数据块 ✅
2. 用 b>1 过滤，得到 b>1 的记录 ✅
3. 但 b>1 返回的是一个区间，c在这个区间内是无序的，无法用索引定位 ❌
```

**优化建议：**

如果经常需要按 `a, b` 范围查询，再按 `c` 精确查询，可以考虑：

```sql
-- 方案1：调整索引顺序
CREATE INDEX idx_a_c_b ON table(a, c, b);
-- 这样 WHERE a = 1 AND b > 1 AND c = 3 能用到 a, c

-- 方案2：创建两个索引
CREATE INDEX idx_a_b ON table(a, b);
CREATE INDEX idx_a_c ON table(a, c);
-- MySQL会根据成本选择更优的索引
```

#### MySQL 8.0 Skip Scan 优化

MySQL 8.0.13 版本引入了 **Skip Scan Range Access Method**，在特定场景下可以绕过最左匹配原则。

**以下实验基于 MySQL 8.0.26 版本**

为了演示 Skip Scan，我们先建个表：

```sql
-- 创建测试表
CREATE TABLE orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id INT NOT NULL,
    order_date DATE NOT NULL,
    amount DECIMAL(10,2),
    INDEX idx_user_date (user_id, order_date)
);

-- 插入测试数据
INSERT INTO orders (user_id, order_date, amount)
SELECT 
    FLOOR(RAND() * 100) + 1 AS user_id,
    DATE('2026-01-01') + INTERVAL FLOOR(RAND() * 90) DAY AS order_date,
    ROUND(RAND() * 1000, 2) AS amount
FROM (
    SELECT 1 UNION SELECT 2 UNION SELECT 3 UNION SELECT 4 UNION SELECT 5
) t1,
(
    SELECT 1 UNION SELECT 2 UNION SELECT 3 UNION SELECT 4 UNION SELECT 5
) t2,
(
    SELECT 1 UNION SELECT 2 UNION SELECT 3 UNION SELECT 4 UNION SELECT 5
) t3,
(
    SELECT 1 UNION SELECT 2 UNION SELECT 3 UNION SELECT 4 UNION SELECT 5
) t4;
```

**Skip Scan 工作原理：**

传统最左匹配要求必须有 `user_id` 条件才能用索引。但 Skip Scan 可以在只有 `order_date` 条件时使用索引：

```sql
-- 传统情况：不使用索引，全表扫描
EXPLAIN SELECT * FROM orders WHERE order_date = '2026-03-15';
-- type: ALL（全表扫描）

-- Skip Scan：使用索引
EXPLAIN SELECT * FROM orders WHERE order_date = '2026-03-15';
-- type: range（使用了索引）
-- Extra: Using index for skip-scan
```

**Skip Scan 的执行过程：**

```
索引 (user_id, order_date) 数据：

user_id=1, order_date=2026-01-01
user_id=1, order_date=2026-01-02
...
user_id=1, order_date=2026-03-31
user_id=2, order_date=2026-01-01  ← user_id 变化
user_id=2, order_date=2026-01-02
...

查询：WHERE order_date = '2026-03-15'

Skip Scan 执行流程：
1. 遍历不同的 user_id 值（1, 2, 3, ...）
2. 对于每个 user_id，在索引中查找 order_date = '2026-03-15'
3. 合并所有结果
```

**Skip Scan 的适用条件：**

1) 索引必须是**联合索引**，至少2列
2) 查询条件中没有使用最左列，但使用了后面的列
3) 最左列的**基数较小**（distinct 值少）
4) 查询需要满足一定的**选择性**

**适用场景示例：**

```sql
-- ✅ 适合 Skip Scan
-- user_id 只有 100 个不同的值（基数小）
SELECT * FROM orders WHERE order_date = '2026-03-15';

-- ✅ 适合 Skip Scan
SELECT COUNT(*) FROM orders WHERE order_date BETWEEN '2026-03-01' AND '2026-03-31';

-- ❌ 不适合 Skip Scan
-- user_id 有 100 万个不同的值（基数太大，跳过成本高）
SELECT * FROM orders WHERE order_date = '2026-03-15';

-- ❌ 不适合 Skip Scan
-- 最左列基数太大，Skip Scan 不划算
SELECT * FROM big_table WHERE second_column = 'value';
```

**Skip Scan vs 全表扫描：**

| 场景 | 全表扫描 | Skip Scan |
|------|---------|----------|
| 最左列基数大（>1000） | ✅ 更快 | ❌ 跳过成本高 |
| 最左列基数小（<100） | ❌ 慢 | ✅ 更快 |
| 查询选择性好 | ⚠️ 都可以 | ✅ 更快 |
| 查询选择性差 | ✅ 全表扫描 | ❌ Skip Scan 不划算 |

**查看是否使用 Skip Scan：**

```sql
EXPLAIN SELECT * FROM orders WHERE order_date = '2026-03-15';
-- 检查 Extra 字段是否有 "Using index for skip-scan"
```

**强制/禁止 Skip Scan：**

```sql
-- 强制使用索引（可能触发 Skip Scan）
SELECT /*+ INDEX(orders idx_user_date) */ * FROM orders 
WHERE order_date = '2026-03-15';

-- 禁止使用索引（强制全表扫描）
SELECT /*+ NO_INDEX(orders idx_user_date) */ * FROM orders 
WHERE order_date = '2026-03-15';
```

**Skip Scan 的局限性：**

1) **不适用于所有场景**：最左列基数大时不如全表扫描
2) **只支持等值查询和简单范围查询**：复杂的 OR、IN 可能不触发
3) **需要索引覆盖查询列**：如果不是覆盖索引，可能回表成本高
4) **MySQL 自动决定**：不能手动指定使用 Skip Scan

**总结：**

Skip Scan 是 MySQL 8.0 的一个重要优化，可以在特定场景下绕过最左前缀原则。但它的使用是**自动的**，由优化器根据成本决定，开发者可以通过 `EXPLAIN` 查看，但不能强制启用。

### 💬 面试官追问

#### 提问1：联合索引(a,b,c)，执行WHERE b=1 AND a=2 能用上索引吗？

**回答：**

能用上。MySQL的查询优化器会自动调整条件顺序，把 `a=2` 放到前面，所以写成 `WHERE b=1 AND a=2` 和 `WHERE a=2 AND b=1` 效果一样。优化器足够聪明，不用纠结SQL里条件的书写顺序。

**示例验证：**

```sql
-- 假设有联合索引 (a, b, c)

-- 查询1：条件顺序与索引顺序一致
EXPLAIN SELECT * FROM table WHERE a = 2 AND b = 1;
-- 能用到索引

-- 查询2：条件顺序与索引顺序不一致
EXPLAIN SELECT * FROM table WHERE b = 1 AND a = 2;
-- 也能用到索引！优化器会自动调整为 a=2 AND b=1

-- 查询3：缺少最左列
EXPLAIN SELECT * FROM table WHERE b = 1 AND c = 3;
-- 用不上索引，缺少a
```

**优化器的工作原理：**

```sql
-- 优化器收到的SQL
SELECT * FROM table WHERE b = 1 AND a = 2;

-- 优化器内部处理：
-- 1. 解析WHERE条件，提取出 a=2 AND b=1
-- 2. 查看可用索引，发现有 (a, b, c)
-- 3. 判断可以用索引前缀 (a, b)
-- 4. 重写执行计划，按索引顺序匹配
-- 5. 最终效果等同于 WHERE a = 2 AND b = 1
```

**EXPLAIN 验证：**

```sql
EXPLAIN SELECT * FROM table WHERE b = 1 AND a = 2;

-- 输出示例：
-- +----+-------------+-------+-------+---------------+---------+---------+------+------+-------------+
-- | id | select_type | table | type  | possible_keys | key     | key_len | ref  | rows | Extra       |
-- +----+-------------+-------+-------+---------------+---------+---------+------+------+-------------+
-- |  1 | SIMPLE      | table | ref   | idx_a_b_c     | idx_a_b_c | 8       | const|  100  | Using where |
-- +----+-------------+-------+-------+---------------+---------+---------+------+------+-------------+
-- key 字段显示用到了索引 idx_a_b_c
```

**关键点：**

1) **优化器会自动重排序**：SQL中条件的顺序不影响索引使用
2) **必须有最左列**：缺少最左列a就一定用不上索引
3) **使用EXPLAIN验证**：不确定时用EXPLAIN查看执行计划

#### 提问2：为什么>=不会停止匹配，但>会停止匹配？

**回答：**

关键在于能不能精确定位到一条记录。

`a >= 1` 可以先精确定位到 `a = 1` 的第一条数据，这时候 `b`、`c` 在 `a = 1` 内部是有序的，可以继续匹配。

而 `a > 1` 定位到的是 `a = 2` 的第一条数据，此时已经跨过了 `a = 1` 的整个范围，不同 `a` 值之间 `b`、`c` 是无序的，所以后面的列用不上索引。

**示例说明：**

假设有联合索引 `(a, b, c)`，数据如下：

```
a=1, b=1, c=1
a=1, b=1, c=2
a=1, b=2, c=1  ← b开始变化
a=1, b=2, c=2
a=2, b=1, c=1  ← a开始变化，跨过了a=1的范围
a=2, b=1, c=2
...
```

**场景1：WHERE a >= 1 AND b = 2**

```
执行过程：
1. a >= 1 定位到 a=1 的第一条记录 ✅
2. 在 a=1 的范围内，b 是有序的（1,1,2,2,...）
3. 用 b=2 进一步过滤 ✅
4. 结果：能用上 a 和 b
```

**场景2：WHERE a > 1 AND b = 2**

```
执行过程：
1. a > 1 定位到 a=2 的第一条记录
2. 已经跨过了 a=1 的整个范围
3. 在 a=2, a=3, a=4... 之间，b 是无序的
4. 无法用 b=2 过滤 ❌
5. 结果：只能用 a
```

**对比表格：**

| 查询条件 | 定位点 | 后续列是否有序 | 能用到的列 |
|---------|--------|--------------|-----------|
| WHERE a = 1 AND b = 2 | a=1第一条 | ✅ b在a=1内有序 | a, b |
| WHERE a >= 1 AND b = 2 | a=1第一条 | ✅ b在a=1内有序 | a, b |
| WHERE a > 1 AND b = 2 | a=2第一条 | ❌ b跨多个a值无序 | a |
| WHERE a >= 1 AND b > 2 | a=1第一条 | ✅ b在a=1内有序 | a, b |
| WHERE a > 1 AND b > 2 | a=2第一条 | ❌ b跨多个a值无序 | a |

**EXPLAIN 验证：**

```sql
-- 案例1：>= 后续能用
EXPLAIN SELECT * FROM table WHERE a >= 1 AND b = 2;
-- key_len: 显示a和b的字段长度
-- 能用到两个列

-- 案例2：> 后续不能用
EXPLAIN SELECT * FROM table WHERE a > 1 AND b = 2;
-- key_len: 只显示a的字段长度
-- 只能用到a

-- 案例3：>= 后续范围查询也用不上
EXPLAIN SELECT * FROM table WHERE a >= 1 AND b > 2;
-- key_len: 显示a的字段长度（b是范围查询，但>=不影响）
-- 能用到a，b用不上（因为是范围查询）
```

**为什么会有这个差异？**

```
B+树索引结构示意：

(a, b, c) 索引：

┌─────────────────┐
│ a=1           │ ← 第一个叶子页
│   b=1         │
│     c=1       │
│     c=2       │
│   b=2         │
│     c=1       │
│     c=2       │
├─────────────────┤
│ a=2           │ ← 第二个叶子页
│   b=1         │
│     c=1       │
│     ...        │
└─────────────────┘

a >= 1:
- 定位到 a=1 的页
- 在这个页内，b 是有序的 ✅

a > 1:
- 定位到 a=2 的页
- 跨过了 a=1 的整个页
- 不同页之间，b 是无序的 ❌
```

**记忆口诀：**

- `=` 和 `>=`：能精确定位起点，后续列能用
- `>`、`<`、`BETWEEN`：跨过范围，后续列用不上
- `LIKE 'abc%'`：能精确定位，后续列能用
- `LIKE '%abc'`：没法定位，全索引扫描

#### 提问3：建联合索引时，列的顺序应该怎么考虑？

**回答：**

一般有几个原则：

1) **高频查询条件放左边**，保证常用查询能走索引

2) **区分度高的列放左边**，能更快过滤掉数据

3) **范围查询的列放最右边**，避免阻断后面列的匹配

4) **如果有排序需求，把排序字段放到索引里**，避免 file sort

**详细说明：**

**原则1：高频查询条件放左边**

```sql
-- 场景：订单表，80%的查询是按 user_id 查询
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    user_id INT,
    order_date DATE,
    amount DECIMAL(10,2),
    status VARCHAR(20)
);

-- ✅ 推荐：user_id 放前面
CREATE INDEX idx_user_date ON orders(user_id, order_date);

-- 查询1：高频查询，能用索引
SELECT * FROM orders WHERE user_id = 100;
-- 能用到索引

-- 查询2：低频查询，也能用
SELECT * FROM orders WHERE user_id = 100 AND order_date = '2026-04-03';
-- 能用到索引

-- ❌ 不推荐：order_date 放前面
CREATE INDEX idx_date_user ON orders(order_date, user_id);

-- 查询3：高频查询，用不上索引
SELECT * FROM orders WHERE user_id = 100;
-- 用不上索引，跳过了最左列 order_date
```

**原则2：区分度高的列放左边**

区分度（Selectivity）= 不重复值数量 / 总记录数

```sql
-- 假设有100万条订单数据

-- user_id: 10万个不同用户，区分度 0.1
-- status: 3种状态（待支付/已支付/已取消），区分度 0.000003
-- amount: 50万个不同金额，区分度 0.5

-- ✅ 推荐：区分度高的放左边
CREATE INDEX idx_amount_status ON orders(amount, status);

-- 查询1：先按amount过滤，能快速定位
SELECT * FROM orders WHERE amount = 100 AND status = '已支付';
-- 先按amount=100过滤，可能只剩10条，再按status过滤
-- 能用到两列

-- ❌ 不推荐：区分度低的放左边
CREATE INDEX idx_status_amount ON orders(status, amount);

-- 查询2：先按status过滤，过滤效果差
SELECT * FROM orders WHERE amount = 100 AND status = '已支付';
-- 先按status='已支付'过滤，可能还有30万条
-- 只能用到status，用不上amount
```

**查看区分度：**

```sql
-- 查看某个列的区分度
SELECT 
    COUNT(DISTINCT column_name) AS distinct_values,
    COUNT(*) AS total_rows,
    COUNT(DISTINCT column_name) / COUNT(*) AS selectivity
FROM table_name;

-- 示例：计算orders表各列的区分度
SELECT 
    'user_id' AS column_name,
    COUNT(DISTINCT user_id) AS distinct_values,
    COUNT(*) AS total_rows,
    COUNT(DISTINCT user_id) / COUNT(*) AS selectivity
FROM orders
UNION ALL
SELECT 
    'status' AS column_name,
    COUNT(DISTINCT status) AS distinct_values,
    COUNT(*) AS total_rows,
    COUNT(DISTINCT status) / COUNT(*) AS selectivity
FROM orders;
```

**原则3：范围查询的列放最右边**

```sql
-- 场景：经常按 user_id 和 order_date 范围查询

-- ✅ 推荐：范围查询放最后
CREATE INDEX idx_user_date ON orders(user_id, order_date);

-- 查询1：能用到两列
SELECT * FROM orders 
WHERE user_id = 100 
  AND order_date >= '2026-04-01' 
  AND order_date <= '2026-04-30';
-- 能用到 user_id 和 order_date

-- ❌ 不推荐：范围查询放中间
CREATE INDEX idx_date_status ON orders(user_id, order_date, status);

-- 查询2：只能用到一列
SELECT * FROM orders 
WHERE user_id = 100 
  AND order_date >= '2026-04-01' 
  AND status = '已支付';
-- 能用到 user_id 和 order_date
-- status 用不上（order_date是范围查询）

-- 更好的方案：调整顺序或建多个索引
CREATE INDEX idx_user_status ON orders(user_id, status);
CREATE INDEX idx_user_date ON orders(user_id, order_date);
-- 优化器会根据成本选择更优的索引
```

**原则4：排序字段放到索引里**

```sql
-- 场景：经常需要按某个字段排序

-- ❌ 不推荐：没有排序字段在索引里
CREATE INDEX idx_user ON orders(user_id);

-- 查询1：需要 filesort
SELECT * FROM orders 
WHERE user_id = 100 
ORDER BY order_date;
-- 能用到索引定位 user_id=100
-- 但排序需要 filesort（在内存或磁盘中排序）

-- ✅ 推荐：排序字段也在索引里
CREATE INDEX idx_user_date ON orders(user_id, order_date);

-- 查询2：不需要 filesort
SELECT * FROM orders 
WHERE user_id = 100 
ORDER BY order_date;
-- 能用到索引，数据本身就是按 user_id, order_date 排序的
-- 直接返回，不需要额外排序

-- 验证是否用了 filesort
EXPLAIN SELECT * FROM orders WHERE user_id = 100 ORDER BY order_date;
-- Extra 字段：
--   "Using filesort" ← 需要排序（不好）
--   "Using index" ← 覆盖索引，不需要排序（好）
```

**综合示例：**

```sql
-- 场景：订单表，需求如下
-- 1. 80%查询按 user_id 查询（高频）
-- 2. 60%查询需要按 status 过滤（区分度：0.000003）
-- 3. 40%查询需要按 order_date 范围查询（范围查询）
-- 4. 30%查询需要按 order_date 排序（排序需求）

-- 方案1：综合所有需求
CREATE INDEX idx_user_status_date_amount ON orders(
    user_id,        -- 高频查询，放最左
    status,         -- 虽然区分度低，但需要
    order_date      -- 范围查询，放最后
);

-- 方案2：拆分索引，更灵活
CREATE INDEX idx_user_date ON orders(user_id, order_date);
-- 适合：按用户+日期查询/排序

CREATE INDEX idx_user_status ON orders(user_id, status);
-- 适合：按用户+状态查询

CREATE INDEX idx_status_date ON orders(status, order_date);
-- 适合：按状态+日期查询（管理员报表）

-- 根据实际查询模式选择方案
```

**索引设计决策树：**

```
1. 是否有某个字段高频查询？
   └─ 是 → 放最左边
   └─ 否 → 继续

2. 是否有范围查询？
   └─ 是 → 放最后面
   └─ 否 → 继续

3. 哪个字段区分度最高？
   └─ 区分度高的放前面

4. 是否有排序需求？
   └─ 有 → 排序字段加入索引
```

**实际应用建议：**

1) **先分析查询模式**：用慢查询日志、EXPLAIN分析实际查询
2) **监控索引使用情况**：定期检查索引是否被使用
3) **避免过度索引**：每个索引占用空间，影响写入性能
4) **定期优化索引**：删除未使用的索引，合并重复索引

**查看索引使用情况：**

```sql
-- 查看表的索引
SHOW INDEX FROM orders;

-- 查看索引使用统计（需要开启）
SET GLOBAL userstat = 1;

SELECT * FROM information_schema.INNODB_SYS_INDEXES 
WHERE table_id = (
    SELECT table_id FROM information_schema.INNODB_SYS_TABLES 
    WHERE name = 'orders/数据库'
);

-- 使用 performance_schema
SELECT * FROM performance_schema.table_io_waits_summary_by_index_usage
WHERE OBJECT_SCHEMA = 'your_database'
  AND OBJECT_NAME = 'orders';
```

## 3. MySQL三层B+树能存多少数据？

### 📌 回答重点

三层B+树大约能存2000万条记录，这是面试中最常考的结论。

**具体算法如下：**

InnoDB默认页大小是16KB，非叶子节点存的是主键和指针，叶子节点存的是完整数据行。

假设主键是bigint占8字节，指针占6字节，那么一个非叶子节点能存的索引项数量是：

```
16KB ÷ (8字节 + 6字节) = 16 × 1024 ÷ 14 ≈ 1170个
```

假设每条数据记录占1KB，那么一个叶子节点能存的记录数是：

```
16KB ÷ 1KB = 16条
```

三层B+树的结构是：

```
1个根节点 → 1170个中间节点 → 1170 × 1170个叶子节点
```

**总记录数 = 1170 × 1170 × 16 ≈ 2190万条**

**示意图：**

```
第一层（根节点）：
┌─────────────────────────┐
│ 存1170个key + 1170个指针 │
└───────────┬─────────────┘
            │
            ↓
第二层（1170个中间节点）：
┌─────────────────────────┐     ┌─────────────────────────┐
│ 节点1: 存1170个key+指针  │ ... │ 节点1170: 存1170个key+指针 │
└───────────┬─────────────┘     └───────────┬─────────────┘
            │                              │
            ↓                              ↓
第三层（1170×1170≈137万个叶子节点）：
┌─────────────────┐ ... ┌─────────────────┐
│ 叶子1: 16条数据  │     │ 叶子137万: 16条数据│
└─────────────────┘     └─────────────────┘

总记录数 = 137万个叶子 × 16条/叶子 ≈ 2192万条
```

**关键参数：**

| 参数 | 值 | 说明 |
|------|-----|------|
| 页大小 | 16KB | InnoDB默认值，可配置 |
| 主键类型 | BIGINT | 8字节，影响索引项大小 |
| 指针大小 | 6字节 | InnoDB内部实现 |
| 非叶子节点key数 | 1170个 | 16KB ÷ 14B ≈ 1170 |
| 叶子节点数据量 | 16条 | 16KB ÷ 1KB = 16 |
| 总记录数 | 约2000万 | 1170 × 1170 × 16 |

**查询效率：**

- 查任意一条记录：最多3次磁盘I/O
- 第1次：读根节点
- 第2次：读中间节点
- 第3次：读叶子节点

这就是为什么B+树适合磁盘存储的核心原因：**树矮，I/O次数少**。

### 📚 扩展知识

#### 为什么B+树要设计得这么"矮胖"

磁盘I/O是数据库最大的性能瓶颈。每访问一个B+树节点，就是一次磁盘随机读，一次随机读大概要10ms。如果树高是10层，一次查询就要100ms，这谁受得了。

B+树的核心思路就是把树压扁，让每个节点尽可能多存数据，降低树的高度。

**时间对比：**

| 树高 | 磁盘I/O次数 | 假设每次10ms | 查询耗时 |
|------|------------|--------------|---------|
| 3层 | 3次 | 30ms | ✅ 快 |
| 4层 | 4次 | 40ms | ✅ 可接受 |
| 10层 | 10次 | 100ms | ❌ 太慢 |

**16KB的页大小不是随便定的**，这个值是InnoDB权衡I/O效率和内存利用率后的选择。

**所以B+树的树高一般就3-4层：**
- 3层撑2000万
- 4层能撑200亿

绝大多数互联网公司的单表数据量都在这个范围内，一次查询最多3-4次磁盘I/O就能定位到数据。

**B+树 vs 红黑树对比：**

| 树类型 | 树高（1000万数据） | I/O次数 | 查询耗时 |
|--------|-------------------|---------|---------|
| 红黑树 | 约23层 | 23次 | 230ms ❌ |
| B+树 | 3-4层 | 3-4次 | 30-40ms ✅ |

这就是为什么**内存数据库可以用红黑树，磁盘数据库必须用B+树**。

#### 页大小的影响

InnoDB默认页大小是16KB，可以通过 `innodb_page_size` 参数调整，支持4KB、8KB、16KB、32KB、64KB这几个档位。

**页越大，单个非叶子节点能存的指针越多，树的扇出越大，树高越低。**

**不同页大小的容量对比：**

假设主键是BIGINT（8字节），指针6字节：

| 页大小 | 非叶子节点key数 | 树高（1000万数据） |
|--------|----------------|-------------------|
| 4KB | 4KB ÷ 14B ≈ 291个 | 5-6层 |
| 8KB | 8KB ÷ 14B ≈ 585个 | 4-5层 |
| 16KB | 16KB ÷ 14B ≈ 1170个 | 3-4层 ✅ |
| 32KB | 32KB ÷ 14B ≈ 2340个 | 3层 |
| 64KB | 64KB ÷ 14B ≈ 4685个 | 3层 |

**但页太大也有问题：**

1) **内存利用率下降**：一个页只存了几条数据也要占64KB内存

2) **刷脏页的粒度变大，写放大更严重**：修改一行就要刷整个页到磁盘

3) **SSD的随机读性能很好，大页的优势没那么明显**：HDD时代大页优势明显，SSD时代差距缩小

**修改页大小配置：**

```sql
-- 在my.cnf/my.ini中修改
[mysqld]
innodb_page_size=32768  -- 32KB

-- 注意：修改页大小需要重新初始化数据库
-- 没法直接修改已有数据库的页大小
```

**一般保持默认16KB就行**，只有特殊场景才调整：

**场景1：大数据分析，行数据特别宽**

```sql
-- 分析型表，每行数据10KB+
CREATE TABLE analytics (
    id BIGINT,
    metric1 DECIMAL(20,4),
    metric2 DECIMAL(20,4),
    -- ... 几十个大字段
    metric100 DECIMAL(20,4)
);
-- 可以考虑调大页到32KB或64KB
```

**场景2：只读数据仓库**

```sql
-- 只读查询，不需要频繁修改
-- 大页可以减少树高，提升查询性能
-- 写放大问题不严重（因为只读）
```

**场景3：主键很短，数据量特别大**

```sql
-- 主键是INT（4字节），不是BIGINT
CREATE TABLE big_table (
    id INT PRIMARY KEY,
    -- ... 几千万条数据
);
-- 16KB页的扇出更大，树高更低
-- 保持默认即可
```

#### 不同场景下的存储量差异

上面按每行1KB算的，实际业务中行大小差异很大：

**1) 用户表、配置表这种字段少的，一行可能就几百字节**

```sql
CREATE TABLE user (
    id BIGINT PRIMARY KEY,
    username VARCHAR(50),
    email VARCHAR(100),
    status TINYINT
    -- 每行约200字节
);

-- 计算：16KB ÷ 200B = 约80条/叶子
-- 三层总容量：1170 × 1170 × 80 ≈ 1.1亿条
```

**2) 文章表、评论表这种有大TEXT字段的，一行可能好几KB**

```sql
CREATE TABLE article (
    id BIGINT PRIMARY KEY,
    title VARCHAR(200),
    content TEXT,  -- TEXT可能几KB
    author_id BIGINT
    -- 每行约5KB（TEXT字段内容）
);

-- 计算：16KB ÷ 5KB = 约3条/叶子
-- 三层总容量：1170 × 1170 × 3 ≈ 410万条
```

**3) 如果主键不是BIGINT而是UUID字符串，非叶子节点每个索引项更大**

```sql
CREATE TABLE orders (
    id CHAR(36) PRIMARY KEY,  -- UUID字符串
    user_id BIGINT,
    amount DECIMAL(10,2)
);

-- 计算：(36B + 6B) ÷ 16KB ≈ 357个key/节点
-- 三层总容量：357 × 357 × 16 ≈ 204万条
-- 树高可能变成4层
```

**对比表格：**

| 表类型 | 行大小 | 主键类型 | 叶子节点条数 | 三层总容量 |
|--------|--------|---------|-------------|-----------|
| 用户表（字段少） | 200B | BIGINT | 80条 | 约1.1亿 |
| 订单表（字段中等） | 1KB | BIGINT | 16条 | 约2200万 |
| 文章表（有大TEXT） | 5KB | BIGINT | 3条 | 约410万 |
| UUID主键表 | 1KB | CHAR(36) | 16条 | 约200万（扇出小） |

**所以那个2000万只是个估算值**，面试答出这个数就行，但要知道实际会根据数据特征波动。

**实际项目中如何评估：**

```sql
-- 1. 查看表的平均行大小
SELECT 
    AVG(ROW_LENGTH) AS avg_row_length,
    AVG(ROW_LENGTH) * COUNT(*) / 1024 / 1024 AS total_size_mb
FROM information_schema.INNODB_SYS_TABLES
WHERE NAME = 'database/table';

-- 2. 查看索引的大小
SELECT 
    INDEX_NAME,
    STAT_VALUE * @@innodb_page_size / 1024 / 1024 AS size_mb
FROM information_schema.INNODB_SYS_INDEXES
WHERE TABLE_ID = (
    SELECT TABLE_ID FROM information_schema.INNODB_SYS_TABLES
    WHERE NAME = 'database/table'
)
ORDER BY SIZE_MB DESC;

-- 3. 根据实际行大小估算容量
-- 三层容量 ≈ 1170 × 1170 × (16KB / avg_row_length)
```

#### 为什么非叶子节点不存数据

B+树和B树的核心区别就在这里。

**B树的非叶子节点也存数据，B+树只有叶子节点存数据。**

**这么设计的好处是非叶子节点能塞更多的索引项，扇出更大，树高更低。**

**B树 vs B+树对比：**

```
B树（非叶子节点存数据）：
非叶子节点：
┌─────────────────────────┐
│ key=100, data=完整行   │ ← 存了数据，占用空间
│ key=200, data=完整行   │
│ key=300, data=完整行   │
└─────────────────────────┘
问题：数据占用空间，存的key少，扇出小


B+树（非叶子节点只存key和指针）：
非叶子节点：
┌─────────────────────────┐
│ key=100, ptr=页号1001 │ ← 只存key和指针
│ key=200, ptr=页号1002 │
│ key=300, ptr=页号1003 │
│ ... (能存1000多个key)   │
└─────────────────────────┘
优势：不存数据，存的key多，扇出大
```

**假设数据占1KB，非叶子节点也存数据的话：**

| 树类型 | 非叶子节点key数 | 树高（1000万数据） |
|--------|----------------|-------------------|
| B树（存数据） | 16KB ÷ 1KB ≈ 16个 | 6-7层 |
| B+树（不存数据） | 16KB ÷ 14B ≈ 1170个 | 3-4层 |

**树高从3层变成六七层，查询效率直接减半。**

**时间对比：**

| 树类型 | 树高 | I/O次数 | 查询耗时 |
|--------|------|---------|---------|
| B树（存数据） | 6-7层 | 6-7次 | 60-70ms ❌ |
| B+树（不存数据） | 3-4层 | 3-4次 | 30-40ms ✅ |

**另外B+树的叶子节点用双向链表串起来了**，范围查询的时候顺着链表扫就行，不用回溯到上层节点，这对OLTP场景很友好。

**范围查询对比：**

```
B树范围查询：
1. 找起点：a >= 100，从根节点向下找 ✅
2. 找下一条：a = 101，需要从根节点重新找 ❌
3. 找下一条：a = 102，需要从根节点重新找 ❌
4. ... 每条都要重新找，O(m × log n)

B+树范围查询：
1. 找起点：a >= 100，从根节点向下找 ✅
2. 找下一条：a = 101，直接顺着叶子链表走 ✅
3. 找下一条：a = 102，直接顺着叶子链表走 ✅
4. ... 顺序扫描，O(log n + m)
```

**B+树的优势总结：**

1) **树更矮**：非叶子节点不存数据，扇出大，I/O少
2) **范围查询快**：叶子节点链表连接，顺序扫描
3) **查询稳定**：必须查到叶子节点，时间可预期
4) **缓存友好**：非叶子节点小，内存能缓存更多索引

**这就是为什么几乎所有磁盘数据库都用B+树的原因。**

### 💬 面试官追问

#### 提问1：如果主键用UUID字符串代替BIGINT，对存储量有什么影响？

**回答：**

影响很大。UUID是36字节，比BIGINT的8字节大了4倍多。

非叶子节点每个索引项从14字节变成42字节：
- BIGINT：8字节（主键）+ 6字节（指针）= 14字节
- UUID：36字节（主键）+ 6字节（指针）= 42字节

一个节点的扇出从1170降到390左右：
```
BIGINT：16KB ÷ 14B ≈ 1170个key
UUID：  16KB ÷ 42B ≈ 390个key
```

三层B+树只能存：
```
390 × 390 × 16 ≈ 240万条
```

比原来少了一个数量级（从2000万降到240万）。

**所以UUID做主键不光是占空间，还会让索引树变高，查询性能也受影响。**

**详细对比：**

| 主键类型 | 主键长度 | 索引项大小 | 非叶子节点key数 | 三层总容量 |
|---------|---------|-----------|----------------|-----------|
| INT | 4字节 | 10字节 | 16KB ÷ 10B ≈ 1639个 | 1639 × 1639 × 16 ≈ 4300万 |
| BIGINT | 8字节 | 14字节 | 16KB ÷ 14B ≈ 1170个 | 1170 × 1170 × 16 ≈ 2200万 |
| CHAR(36) | 36字节 | 42字节 | 16KB ÷ 42B ≈ 390个 | 390 × 390 × 16 ≈ 240万 |

**树高对比：**

假设数据量是2000万条：

| 主键类型 | 树高 | I/O次数 | 说明 |
|---------|------|---------|------|
| BIGINT | 3层 | 3次 | ✅ 正常 |
| CHAR(36) | 4-5层 | 4-5次 | ⚠️ 性能下降40-67% |

**实际测试：**

```sql
-- 创建测试表：BIGINT主键
CREATE TABLE user_bigint (
    id BIGINT PRIMARY KEY,
    username VARCHAR(50)
);
-- 插入100万条数据
-- 树高：3层，查询3次I/O

-- 创建测试表：UUID主键
CREATE TABLE user_uuid (
    id CHAR(36) PRIMARY KEY,
    username VARCHAR(50)
);
-- 插入100万条数据
-- 树高：4层，查询4次I/O
```

**UUID的其他问题：**

1) **二级索引也受影响**：每个二级索引的叶子节点都存主键值，UUID占用空间大

2) **页分裂更频繁**：UUID是随机的，插入时容易触发页分裂

3) **缓存利用率低**：一个节点存的key少，内存能缓存的索引页就少

**什么时候可以用UUID？**

```sql
-- ✅ 分库分表场景：避免主键冲突
CREATE TABLE orders (
    id CHAR(36) PRIMARY KEY,  -- UUID
    user_id BIGINT,
    amount DECIMAL(10,2)
);

-- ✅ 数据导入：批量导入时用分布式ID
-- ✅ 避免主键暴露：不能暴露自增ID

-- ❌ 单表业务：优先用自增BIGINT
CREATE TABLE user (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50)
);
```

**更好的替代方案：**

```sql
-- 方案1：雪花算法生成的分布式ID（BIGINT）
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,  -- 雪花ID
    user_id BIGINT,
    amount DECIMAL(10,2)
);
-- 优势：保持BIGINT的性能，避免主键冲突

-- 方案2：ULID（可排序的唯一标识）
-- 26字节，比UUID短，支持排序

-- 方案3：数据库自增ID + 应用层ID映射
-- 单表用自增ID，应用层维护ID映射表
```

#### 提问2：二级索引的三层B+树能存多少数据？

**回答：**

二级索引的叶子节点存的是主键值而不是完整行数据。

假设主键是BIGINT，二级索引也是BIGINT类型，叶子节点存的就是：
- 8字节的主键
- 8字节的二级索引值

一个叶子节点能存：
```
16KB ÷ (8B + 8B) = 1024条
```

三层二级索引能存：
```
1170 × 1170 × 1024 ≈ 13.5亿条
```

**所以二级索引比主键索引能支撑的数据量大得多，但查数据要回表。**

**对比表格：**

| 索引类型 | 叶子节点内容 | 每页条数 | 三层容量 | 查询特点 |
|---------|-------------|---------|---------|---------|
| 主键索引 | 完整行数据（假设1KB） | 16条 | 约2200万 | 直接返回数据 |
| 二级索引 | 主键值+索引值（16字节） | 1024条 | 约13.5亿 | 需要回表 |

**为什么二级索引能存更多？**

```
主键索引叶子节点：
┌─────────────────────────┐
│ id=1, name='张三', ...  │ ← 完整行数据，占1KB
│ id=2, name='李四', ...  │
│ ...（16条/页）          │
└─────────────────────────┘


二级索引叶子节点：
┌─────────────────────────┐
│ name='张三', id=1       │ ← 只有索引列+主键，16字节
│ name='李四', id=2       │
│ ...（1024条/页）        │ ← 条数是主键索引的64倍
└─────────────────────────┘
```

**实际案例：**

```sql
-- 订单表
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    user_id BIGINT,
    order_date DATE,
    amount DECIMAL(10,2)
);

-- 主键索引 (id)
-- 叶子节点：完整行数据（假设100字节）
-- 三层容量：1170 × 1170 × 160 ≈ 2.2亿条

-- 二级索引 (user_id)
-- 叶子节点：user_id(8B) + id(8B) = 16字节
-- 三层容量：1170 × 1170 × 1024 ≈ 13.5亿条
-- 但查询时需要回表到主键索引

-- 查询示例
SELECT * FROM orders WHERE user_id = 100;
-- 流程：
-- 1. 在user_id索引中找到id=1000, 1001, 1002...
-- 2. 拿着这些id回主键索引查完整数据
-- 3. 返回结果
```

**回表的代价：**

```sql
-- 假设通过user_id查询返回100条记录

-- 不使用二级索引：
-- 全表扫描，1次I/O（如果数据在缓存中）
-- 或多次I/O（如果需要读磁盘）

-- 使用二级索引+回表：
-- 1. 在二级索引中找：3次I/O（三层）
-- 2. 回表查每条记录：100 × 3次I/O = 300次I/O
-- 总共：303次I/O ← 可能比全表扫描还慢！

-- 使用覆盖索引：
SELECT id, user_id FROM orders WHERE user_id = 100;
-- 索引已经包含了查询的列，不需要回表
-- 总共：3次I/O ← 很快！
```

**什么时候二级索引更合适？**

```sql
-- ✅ 查询返回少量记录
SELECT * FROM orders WHERE user_id = 100;
-- 假设user_id=100只有10条记录
-- 回表10次，总共13次I/O → 可接受

-- ❌ 查询返回大量记录
SELECT * FROM orders WHERE user_id = 100;
-- 假设user_id=100有100万条记录
-- 回表100万次，总共300万次I/O → 灾难！

-- ✅ 使用覆盖索引
SELECT id, user_id FROM orders WHERE user_id = 100;
-- 不需要回表，3次I/O → 很快
```

**优化建议：**

```sql
-- 方案1：创建覆盖索引
CREATE INDEX idx_user_id_amount ON orders(user_id, amount);
-- 查询时可以避免回表

SELECT user_id, amount FROM orders WHERE user_id = 100;
-- 使用覆盖索引，不需要回表

-- 方案2：限制返回数量
SELECT * FROM orders WHERE user_id = 100 LIMIT 100;
-- 只回表100次，性能可控

-- 方案3：使用EXPLAIN分析
EXPLAIN SELECT * FROM orders WHERE user_id = 100;
-- 检查type、rows、Extra字段
-- 评估是否真的需要回表
```

**二级索引数量的影响：**

```sql
-- 假设一张表有5个二级索引，主键是BIGINT

-- 每条二级索引记录占用8字节
-- 1000万条数据，5个索引
-- 总占用：1000万 × 5 × 8B ≈ 381MB

-- 如果主键是CHAR(36)：
-- 每条二级索引记录占用36字节
-- 总占用：1000万 × 5 × 36B ≈ 1.7GB

-- 结论：主键越短，二级索引占用空间越小
```

#### 提问3：表数据量超过2000万就要分库分表吗？

**回答：**

不一定，2000万只是个参考值。

**关键看查询性能能不能满足业务需求。**

如果索引设计得好，查询都能走索引，几千万数据单表也没问题。

**真正需要分库分表的信号是：**

1) **慢查询明显增多**

```sql
-- 查看慢查询日志
SHOW VARIABLES LIKE 'slow_query%';

-- 分析慢查询
mysqldumpslow -s t /var/log/mysql/slow.log

-- 常见指标：
-- 查询耗时 > 100ms 频繁出现
-- 慢查询数量 > 1000次/小时
```

2) **单表写入QPS顶不住**

```sql
-- 查看写入QPS
SHOW GLOBAL STATUS LIKE 'Questions';
SHOW GLOBAL STATUS LIKE 'Com_insert';

-- 计算QPS
-- 单表写入QPS持续 > 5000-10000（取决于硬件）
```

3) **磁盘空间告急**

```sql
-- 查看表大小
SELECT 
    TABLE_NAME,
    ROUND(((DATA_LENGTH + INDEX_LENGTH) / 1024 / 1024), 2) AS 'Size(MB)'
FROM information_schema.TABLES
WHERE TABLE_SCHEMA = 'your_database'
ORDER BY (DATA_LENGTH + INDEX_LENGTH) DESC;

-- 单表 > 50GB 或磁盘使用率 > 80%
```

**分库分表引入的复杂度很高，能不分就不分。**

**单表性能优化：**

```sql
-- 1. 优化索引设计
-- 添加合适的索引
CREATE INDEX idx_user_date ON orders(user_id, order_date);

-- 2. 分区表（MySQL 5.7+）
CREATE TABLE orders (
    id BIGINT,
    user_id BIGINT,
    order_date DATE,
    ...
) PARTITION BY RANGE (YEAR(order_date)) (
    PARTITION p2024 VALUES LESS THAN (2025),
    PARTITION p2025 VALUES LESS THAN (2026),
    PARTITION pmax VALUES LESS THAN MAXVALUE
);
-- 按年份分区，每个分区独立管理

-- 3. 历史数据归档
-- 将冷数据迁移到归档表
CREATE TABLE orders_archive LIKE orders;
INSERT INTO orders_archive SELECT * FROM orders WHERE order_date < '2023-01-01';
DELETE FROM orders WHERE order_date < '2023-01-01';

-- 4. 数据冷热分离
-- 热数据放主表，冷数据放归档表
-- 业务层判断查询哪个表
```

**什么时候必须分库分表？**

```sql
-- 1. 单表数据量 > 1亿，且查询性能无法满足
-- 2. 单表写入QPS > 10000，且无法优化
-- 3. 磁盘空间无法扩展（单机磁盘满了）
-- 4. 需要跨地域部署（数据分片到不同地域）

-- 分库分表方案示例：
-- 按user_id取模分片：
-- user_id % 4 → 分配到4个库
-- 每个库再按user_id % 16 → 分配到16个表
-- 总共64张表

-- 路由规则：
-- 计算user_id % 4 → 确定库
-- 计算user_id % 16 → 确定表
```

**分库分表的代价：**

1) **跨库JOIN复杂**：需要在应用层实现
2) **事务处理复杂**：分布式事务，性能差
3) **数据迁移复杂**：扩容时需要重新分片
4) **运维复杂度增加**：监控、备份、恢复都更复杂

**分库分表 vs 单表优化对比：**

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 单表优化 | 简单，事务简单 | 有上限 | 数据量<1亿，性能可满足 |
| 分区表 | 对业务透明 | 单机上限 | 数据量1-10亿，可按时间分区 |
| 分库分表 | 无上限 | 复杂度高 | 数据量>10亿，性能无法满足 |

**决策树：**

```
表数据量 > 2000万？
└─ 否 → 不需要分库分表
└─ 是 → 继续评估

查询性能能否满足业务需求？
└─ 是 → 不需要分库分表
└─ 否 → 继续优化

优化索引后性能是否改善？
└─ 是 → 不需要分库分表
└─ 否 → 继续评估

使用分区表是否可行？
└─ 是 → 使用分区表
└─ 否 → 继续评估

数据量 > 1亿？
└─ 否 → 继续优化，暂不分库分表
└─ 是 → 考虑分库分表
```

**总结：**

- 2000万不是分库分表的硬性门槛
- 先优化索引和查询，再考虑分区表
- 最后才考虑分库分表
- 分库分表是最后手段，不是首选方案

#### 提问4：innodb_page_size什么时候需要调整？

**回答：**

绝大多数场景保持默认16KB就行。

需要调整的情况比较少：

**需要调整的情况：**

1) **如果行数据特别小，比如只有几十字节的监控指标表，可以考虑调成4KB或8KB，提高内存利用率**

```sql
-- 场景：监控数据表
CREATE TABLE metrics (
    timestamp BIGINT,
    metric_name VARCHAR(50),
    metric_value DOUBLE
);
-- 每行约50字节

-- 16KB页：16KB ÷ 50B ≈ 327条/页
-- 4KB页：4KB ÷ 50B ≈ 81条/页
-- 内存占用：4KB页是16KB页的1/4

-- 修改配置
[mysqld]
innodb_page_size=4096  -- 4KB
```

2) **如果行数据特别大，比如宽表场景，可以考虑调成32KB或64KB**

```sql
-- 场景：分析型宽表
CREATE TABLE analytics (
    id BIGINT,
    metric1 DECIMAL(20,4),
    metric2 DECIMAL(20,4),
    metric3 DECIMAL(20,4),
    -- ... 几十个字段
    metric100 DECIMAL(20,4),
    content TEXT
);
-- 每行约10KB

-- 16KB页：16KB ÷ 10KB ≈ 1条/页
-- 32KB页：32KB ÷ 10KB ≈ 3条/页
-- 容量提升3倍

-- 修改配置
[mysqld]
innodb_page_size=32768  -- 32KB
```

**但这个参数只能在初始化数据库时设置，一旦设定就不能改了，所以要慎重。**

**不同页大小的对比：**

| 页大小 | 适用场景 | 优点 | 缺点 |
|--------|---------|------|------|
| 4KB | 行数据<100字节 | 内存占用小 | 树高较高，I/O多 |
| 8KB | 行数据100-500字节 | 平衡 | 树高略高 |
| 16KB | 通用场景（默认） | 平衡性好 | - |
| 32KB | 行数据>1KB | 树高较低 | 内存占用大 |
| 64KB | 行数据>5KB | 树高最低 | 内存占用大，写放大 |

**修改页大小的步骤：**

```bash
# 1. 停止MySQL
systemctl stop mysql

# 2. 备份数据
mysqldump -u root -p --all-databases > backup.sql

# 3. 修改配置文件
vim /etc/my.cnf
[mysqld]
innodb_page_size=32768  # 32KB

# 4. 删除旧数据目录
rm -rf /var/lib/mysql/*

# 5. 初始化数据库
mysqld --initialize

# 6. 启动MySQL
systemctl start mysql

# 7. 恢复数据
mysql -u root -p < backup.sql
```

**注意事项：**

1) **必须在初始化时设置**：无法修改已有数据库的页大小

2) **所有库共享配置**：一个MySQL实例的所有表使用相同的页大小

3) **SSD vs HDD**：
   - HDD：大页优势明显（减少I/O次数）
   - SSD：大页优势不明显（随机读快）

4) **OLTP vs OLAP**：
   - OLTP：小页更好（内存利用率高）
   - OLAP：大页更好（减少I/O次数）

## 📝 总结
- 第1次：读根节点
- 第2次：读中间节点
- 第3次：读叶子节点

这就是为什么B+树适合磁盘存储的核心原因：**树矮，I/O次数少**。

## 📝 总结

**为什么MySQL选B+树？**

1) **树矮**：3层能存2000万条，查任何一条最多3次I/O
2) **非叶子节点不存数据**：一页能装更多索引项，缓存命中率高
3) **叶子节点链表连接**：范围查询顺序扫描，效率高

**其他索引结构的问题：**

- 哈希表：等值查询快，但不支持范围查询和排序
- 红黑树/AVL树：二叉树太深，I/O次数爆炸
- B树：范围查询要多次回溯，效率不如B+树

**简单记忆：** MySQL索引 = B+树（99%默认）+ 特殊场景补充（自适应哈希）
