# 常见HTTP状态码

## 概览

### 状态码结构（3位数）

```
状态码是3位数：
┌──┬──┬──┐
│ 1│ 2│ 3│
└──┴──┴──┘
  │  │  └── 具体含义
  │  └───── 分类
  └──────── 类别（1-5）
```

### 五大类别

| 类别 | 含义 | 常见吗 |
|------|------|--------|
| **1xx** | 信息性（服务器说"我收到了"） | 几乎不用管 |
| **2xx** | ✅ 成功！ | 最常见：200 |
| **3xx** | 🔀 重定向（你去别的地方） | 偶尔见：301、302 |
| **4xx** | ❌ 客户端错误（你发错了） | 非常常见：404、403 |
| **5xx** | ❌ 服务端错误（我的锅） | 常见：500 |

> 核心区别：4xx = 请求本身有问题；5xx = 请求没问题但服务器处理时出错了

---

## 各类详细说明

### ✅ 2xx 成功类

| 状态码 | 名称 | 场景 |
|--------|------|------|
| **200** | OK | 请求成功，返回数据（最最最常见） |
| **201** | Created | 创建成功（比如注册成功、新增记录） |
| **204** | No Content | 成功但不返回内容（删除操作常用） |

### 🔀 3xx 重定向类

| 状态码 | 名称 | 场景 |
|--------|------|------|
| **301** | Moved Permanently | 永久跳转（网站换域名了） |
| **302** | Found / Moved Temporarily | 临时跳转（登录后跳转首页） |

> 类比：301 = 搬家了新地址；302 = 出门旅游暂住酒店

### ❌ 4xx 客户端错误（你的锅）

| 状态码 | 名称 | 场景 | 记忆口诀 |
|--------|------|------|----------|
| **400** | Bad Request | 请求参数错了（前端传的数据格式不对） | 参数错 |
| **401** | Unauthorized | 没登录 / token过期 | 没登上 |
| **403** | Forbidden | 登录了但没权限 | 登错号了 |
| **404** | Not Found | 资源不存在（地址写错、数据被删） | 上错网址 |
| **405** | Method Not Allowed | 方法不对（用GET去提交数据） | 方法错了 |
| **408** | Request Timeout | 请求超时（网络慢或服务器忙，客户端等不及了） | 网卡了 |

> 口诀：400参数错、401没登录、403没权限、404找不到、405方法错、408超时
>
> 405为什么属于客户端错误？服务器代码正常运行，是你用的方式不对。
> 类比：餐厅门口写着"只接受堂食"，你非要打包 → 是你的问题不是餐厅的问题

### ❌ 5xx 服务端错误（服务器的锅）

| 状态码 | 名称 | 场景 |
|--------|------|------|
| **500** | Internal Server Error | 服务器内部出错（代码bug、数据库挂了） |
| **502** | Bad Gateway | 网关错了（Nginx和后端服务通信失败） |
| **503** | Service Unavailable | 服务器宕机/维护中/负载过高 |
| **504** | Gateway Timeout | 后端处理太慢，网关等不及了 |

---

## 易混淆的状态码对比

### 408 vs 503 vs 504（都是"超时"？）

| | **408** | **503** | **504** |
|--|---------|---------|---------|
| **谁发起的** | 你的浏览器 | 服务器自己 | 中间网关(Nginx) |
| **谁的责任** | 可能是网络差 | 服务器问题 | 后端太慢 |
| **意思** | "我不等了！" | "我不能服务" | "后端太慢我等不了了" |
| **怎么解决** | 检查网络、增加前端超时时间 | 等恢复、检查服务器状态 | 优化后端性能、增加网关超时时间 |
| **类比** | 打电话没人接挂断了 | 餐厅关门 | 快递员等不到人走了 |

> 记忆法：**我-服-网 → 408-503-504**

```
浏览器(你)  ──网络链路──→ 中间人(Nginx) ──→ 服务器(Java程序)
     ↑                       ↑                  ↑
    408                     504                503
 (你等不及了)            (中间人等不及)       (服务器挂了)
```

### 405 vs 500（都是"代码问题"？）

| | **405** | **500** |
|--|---------|---------|
| 请求方式 | GET去访问只能POST的接口 | 正常访问 |
| 服务端状态 | 代码正常，只是不让你这么用 | 代码有bug崩了 |
| 谁改 | **前端**改成POST | **后端**修bug |
| 类比 | 走了员工通道被拦住 | 餐厅厨房着火了 |

> 405 = "我会这活儿，但你找我的方式不对" → 客户端改
> 500 = "我想干活，但我自己出错了" → 服务端改

---

## 毕设中的典型场景

```
POST /api/login     → 密码正确 → 200（+token数据）
                     → 密码错误 → 401（"用户名或密码错误"）

GET  /api/user/999   → 用户存在 → 200（+用户信息）
                     → 不存在   → 404（"用户不存在"）

DELETE /api/user/1   → 有权限   → 204（无内容）
                     → 没权限   → 403（"无权操作"）

POST /api/user       → 参数缺了 → 400（"缺少必填字段"）
                     → 数据库崩 → 500（"服务器内部错误"）
```

---

> **我的总结：** HTTP状态码分五大类别（1-5）：1xx信息响应、2xx成功（200请求成功、201创建成功、204删除成功）、3xx重定向（301永久换网站、302临时换网站）、4xx客户端出错——代码没问题，请求方法有问题（400参数传错、401没登陆或登录过期、403登上了但没权限、404资源不存在、405请求方法错、408请求超时）、5xx服务器出错——请求没问题代码有问题（500内部BUG、502通信错误、503服务器卡了或在维护、504后端处理卡了）。

## 特殊情况与易错点

### 情况1：返回200但实际有问题

很多国内公司会定义统一的返回结果类 `Result`：

```java
class Result<T> {
    int code;       // 业务状态码
    String msg;     // 提示消息
    T data;         // 真正的数据
    boolean success;// 成功标志
}
```

```json
// HTTP 200 但业务失败
{
    "code": 200,
    "msg": "success",
    "data": null,
    "success": false      ← 跟200矛盾！
}
```

**原因：SpringBoot默认行为 —— 方法正常执行完没崩 → 自动给200**

```java
public Result getUser(Long id) {
    try {
        User user = userService.findById(id);
        return Result.ok(user);            // 方法正常返回 → HTTP 200 ✅
    } catch (Exception e) {
        return Result.error("查询失败");  // 方法也正常返回了 → 还是200 ❌
    }
}
```

> 类比：考试交白卷，老师说'试卷收到了(200)'，但实际上一分都没有。

### 情况2：返回502但网关本身没问题

**502大部分是后端的锅：**
| 原因 | 说明 |
|------|------|
| 后端进程崩了 | Java程序崩了 |
| 后端重启中 | 还没准备好 |
| 防火墙拦截 | 安全组把请求拦了 |

### 规范做法建议
- 真成功 → HTTP 200 + `{"success": true}`
- 用户不存在 → HTTP 404 + `{"msg": "用户不存在"}`
- 服务器出错 → HTTP 500 + `{"msg": "内部错误"}`（不要catch后返回200！）

---

> **补充理解：虽然返回结果内容错了（比如数据为空、success=false），但是返回方法本身正常执行完了（没抛异常），所以SpringBoot还是给了HTTP 200。关键区别在于：方法崩不崩决定HTTP状态码，返回什么内容决定body。**

# HTTP请求的组成（请求行、请求头、请求体）

## 概览

HTTP请求就像一封信，包含三部分：

```
┌─────────────────────────────┐
│  请求行（信封上的地址）       │  ← 去哪、干什么
├─────────────────────────────┤
│  请求头（信封上的备注信息）   │  ← 我是谁、用什么方式送
├─────────────────────────────┤
│  请求体（信里面的内容）      │  ← 实际要传送的数据
└─────────────────────────────┘

为什么需要分开：分工明确，各管各的，服务器按顺序处理。
```

## 一、请求行（Request Line）

> Line = 只有一行，解决"用什么方法去哪里"的问题

```
POST /api/login HTTP/1.1
 │      │         └─── 协议版本（基本固定是HTTP/1.1）
 │      └───────────── 请求路径（URL）
 └────────────────── 请求方法（GET/POST/PUT/DELETE/PATCH）
```

| 部分 | 含义 | 例子 |
|------|------|------|
| **请求方法** | 要做什么 | GET获取、POST提交、DELETE删除、PUT修改 |
| **请求路径** | 资源在哪里 | `/api/login`、`/user/123` |
| **协议版本** | HTTP版本 | HTTP/1.1 |

> 命名由来：Request = 请求，Line = 真的只有一行。像信封上写的那一行地址。

## 二、请求头（Headers）

> Header = 头部，在最前面先被读到。键值对形式的备注信息。

### 常见请求头

| 请求头 | 含义 | 例子 | 类比 |
|--------|------|------|------|
| **Content-Type** | 请求体的数据格式 | `application/json` | "我寄的是JSON不是Word" |
| **Authorization** | 身份验证信息（token） | `Bearer eyJhbGci...` | "这是我的身份证" |
| **Accept** | 希望返回什么格式 | `application/json` | "希望用JSON回复我" |
| **User-Agent** | 我是谁（浏览器） | `Mozilla/5.0...` | "我是Chrome" |
| **Cookie** | 浏览器存的cookie | `sessionid=abc123` | "上次给我的小票" |
| **Content-Length** | 请求体长度（字节） | `128` | "这封信128个字" |

> 命名由来：Head = 头部，正文之前先被读取。像安检——先查身份（请求头），再登机（请求体）。

### Content-Type 类型详解（毕设必用）

| Content-Type 值 | 含义 | 用在哪 |
|-----------------|------|--------|
| **application/json** | JSON格式 | 最常用！前端传对象数据 |
| **application/x-www-form-urlencoded** | 表单格式 | 传统表单提交 |
| **multipart/form-data** | 多部分格式 | **上传文件必用！** |
| **text/plain** | 纯文本 | 简单文本 |

## 三、请求体（Body）

> Body = 身体/主体，核心内容，真正的数据。

### 各方法的请求体情况

| 方法 | 有请求体？ | 说明 |
|------|-----------|------|
| **POST** | ✅ 有 | 要提交的新数据 |
| **PUT** | ✅ 有 | 要修改的完整数据 |
| **PATCH** | ✅ 有 | 要修改的部分数据 |
| **GET** | ❌ 没有 | 数据放在URL问号后面 |
| **DELETE** | ❌ 没有 | 只需要知道删谁就行 |

### 请求体的三种类型

**类型1：JSON格式（最最最常用）**
```http
POST /api/login HTTP/1.1
Content-Type: application/json
{"username": "秋霁", "password": "123456"}
```

**类型2：表单格式**
```http
Content-Type: application/x-www-form-urlencoded
username=秋霁&password=123456
```

**类型3：多部分格式（上传文件）**
```http
Content-Type: multipart/form-data; boundary=----xxx
------xxx
（文件二进制数据...）
------xxx--
```

> 命名由来：Body = 身体/干货。Head是说明，Body是真正要送的东西。

## 完整请求示例

```http
# ===== 请求行 =====
POST /api/user/register HTTP/1.1

# ===== 请求头 =====
Host: localhost:8080
Content-Type: application/json
Authorization: Bearer xxxxx.token
Accept: application/json
User-Agent: Mozilla/5.0 (Chrome)
Content-Length: 56

# ===== 请求体（空一行隔开）=====
{"name": "秋霁", "email": "qiuj@example.com", "age": 22}
```

---

> **我的总结：** HTTP请求三种结构——请求行、请求头、请求体。请求行包含请求方法（GET/POST等）、请求路径、协议版本，解决用什么方法去哪里的问题；请求头是很多键值对形式的备注信息，说明我是谁（Authorization）、输出什么格式（Content-Type）、接受什么格式（Accept）；请求体是实际要传给服务器的具体数据，POST/PUT/PATCH有请求体，GET/DELETE一般没有。三部分分工明确，服务器按顺序处理。

# GET vs POST 的区别

## 核心区别总表

| 区别 | GET | POST |
|------|-----|------|
| **用途** | 获取/查询数据 | 提交/修改数据 |
| **数据位置** | URL后面（?key=value） | 请求体Body里 |
| **安全性** | 低（数据在URL里完全可见） | 稍高（数据在Body里不可见） |
| **长度限制** | 有（约2048字符） | 几乎没有 |
| **缓存** | 会被缓存 | 不会被缓存 |
| **书签/收藏** | 可以 | 不可以 |
| **幂等性** | 幂等（查多少次结果一样） | 不幂等（提交多次产生多笔） |

## 各区别详解

### 1. 数据位置不同
```
GET：数据拼在URL后面
GET /api/search?keyword=秋霁&page=1
                  ↑ 数据在这里！明文的

POST：数据放在Body里面
POST /api/login (Body: {"username":"秋霁","password":"123456"})
                     ↑ 数据藏在这里！
```

### 2. 安全性不同
- GET的参数在地址栏、历史记录、服务器日志中**全部明文可见**
- POST的参数在Body中，地址栏和历史记录**看不到**
- 但注意：**POST不等于加密**！抓包工具还是能看到Body。真正安全要靠HTTPS。

### 3. URL是什么？
> URL = 网址 = 快递地址
```
https://www.baidu.com/s?wd=秋霁
└───┬──┘ └────┬──┘ └┬┘ └───┬──┘
  协议    域名   路径  参数(GET数据)
  
  https  baidu.com /s  ?wd=秋霁
```

### 4. "看不见"的真正含义
POST的Body"看不见"是指：
- ✅ 浏览器地址栏不显示
- ✅ 历史记录不留Body内容
- ❌ 但你自己写代码当然能看到
- ⚠️ 技术人员用F12开发者工具/抓包工具还是能看到（除非用HTTPS）

### 5. HTTPS下的情况
```
HTTPS只加密"传输过程"，不加密"起点和终点"

GET + HTTPS：路上安全了，但地址栏/历史/日志还是明文的
POST + HTTPS：路上安全 + 地址栏/历史都不留痕迹

所以：密码必须用 POST + HTTPS
```

## 面试常见追问

**Q: GET能用来提交数据吗？**
A: 技术上可以（把数据拼在URL后），但不应该——不安全、有长度限制、会被缓存。

**Q: POST绝对安全吗？**
A: 不是！POST只是不在URL显示而已。抓包工具照样能看到Body。**真正安全要靠HTTPS。**

**Q: 为什么登录用POST而不是GET？**
A: 密码不能出现在URL中（会留在历史记录/日志），所以必须用POST。
