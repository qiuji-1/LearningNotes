# 一、TCP/IP 四层模型

## 1.1 为什么要分层？

用**餐厅类比**理解：

- 食客（应用层）→ 点菜 → 服务员（传输层）→ 厨房（网络层）→ 实际运输（链路层）
- **不分层的后果**：所有事情混在一起，出问题不知道哪里出错
- 分层的好处：每层各司其职，改哪层不影响其他层

## 1.2 四层模型总览

| 层次 | 名称       | 类比          | 核心职责                                  |
| :--: | :--------- | :------------ | :---------------------------------------- |
|  4  | 应用层     | 食客          | 决定"传什么"（HTTP/FTP/DNS等）            |
|  3  | 传输层     | 服务员        | 加上端口号，送到目标程序（TCP/UDP）       |
|  2  | 网络层     | 导航          | 加上IP + 规划路线，找到目标主机（IP协议） |
|  1  | 网络接口层 | 司机/交通工具 | 加上MAC + 转物理信号，实际传输            |

## 1.3 数据封装过程

发送方从应用层→网络接口层层层打包（加头部），接收方反向逐层解包。
类比：寄快递 —— 信件→信封(端口)→地址标签(IP)→交给快递公司(MAC+运输)

## 1.4 IP地址 vs MAC地址

|        | IP 地址              | MAC 地址             |
| :----- | :------------------- | :------------------- |
| 性质   | 逻辑地址（软件分配） | 物理地址（硬件出厂） |
| 类比   | 收货地址（可变）     | 身份证号（唯一固定） |
| 会变吗 | ✅ 换环境就变        | ❌ 全球唯一不变      |
| 所在层 | 网络层               | 网络接口层           |

**配合工作**：IP 负责"宏观导航"，MAC 负责"微观交付"。MAC 每经过一跳都会变化；IP 从头到尾不变。

## 1.5 OSI 七层 vs TCP/IP 四层对比

| TCP/IP 四层 | 对应 OSI 七层            |
| :---------- | :----------------------- |
| 应用层      | 应用层 + 表示层 + 会话层 |
| 传输层      | 传输层                   |
| 网络层      | 网络层                   |
| 网络接口层  | 数据链路层 + 物理层      |

> OSI 是理论理想模型，TCP/IP 是实际使用的模型

---

> 💬 **我的总结（TCP/IP四层模型）**：
>
> **原始版**：应用层（传数据的内容）、传输层（传数据的工具）、网络层（目的地IP地址）、网络接口层（目的地MAC地址）
>
> **修正版**：
>
> - 应用层：决定**传什么内容**
> - 传输层：加上**端口号**，保证送到**目标程序**
> - 网络层：加上**IP地址 + 规划路线**，找到**目标主机**
> - 网络接口层：加上**MAC地址 + 转成物理信号**，在**链路上实际传输**

# 二、Cookie vs Session vs Token

## 2.1 为什么需要它们？

**HTTP 是无状态的** —— 服务器不认识你，每次请求都当作第一次见面。

用**餐厅类比**：

- ❌ 没有"身份记录"：每次来服务员都当你是新客人
- ✅ 有了记录：会员卡(Cookie) / 店内登记(Session) / 带签名通行证(Token)

## 2.2 三者核心对比

|              | **Cookie**       | **Session**        | **Token (JWT)**    |
| :----------- | :--------------------- | :----------------------- | :----------------------- |
| 类比         | 会员卡（信息写在卡上） | 店内登记本（信息存店里） | 带签名的通行证           |
| 数据存储位置 | 客户端（浏览器）       | 服务端（内存/数据库）    | 客户端（localStorage）   |
| 安全性       | 较低（明文可伪造）     | 较高（数据不在用户手里） | 高（签名防篡改）         |
| 跨域支持     | 需要配置               | 天然支持                 | 天然支持                 |
| 服务器压力   | 小                     | 大（每个用户存一份）     | 小（无状态不存数据）     |
| 适用场景     | 存非敏感偏好           | 传统Web应用              | 前后端分离/移动端/微服务 |

## 2.3 三者工作流程

### Cookie 方式

```
客户端 ──登录──► 服务端生成Cookie ──返回Cookie──► 浏览器自动保存
客户端 ──(每次请求自动带Cookie)──► 服务端验证 ──► 返回数据
```

### Session 方式

```
客户端 ──登录──► 服务端创建Session ──返回SessionID──► 客户端存Cookie中
客户端 ──(带SessionID)──► 服务端查Session表 ──► 返回数据
```

### Token (JWT) 方式

```
客户端 ──登录──► 服务端签名生成Token ──返回Token──► 客户端存localStorage
客户端 ──(手动带Token)──► 服务端验签名 ──► 返回数据（无需查库！）
```

## 2.4 JWT 结构 = Header.Payload.Signature

```
┌──────────────────────────────────────────────────┐
│  eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9           │ ← Header（算法类型）
│  .eyJ1c2VyTmFtZSI6IuWwjueCouIsInJvbGUiOiJhZG0i │ ← Payload（用户数据）
│  .SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c  │ ← Signature（防伪签名）
└──────────────────────────────────────────────────┘
```

⚠️ **重要误区**：Payload 是 Base64 编码不是加密！人人可以解码看到内容！

- Token 的安全性来自"**签名防篡改**"，而不是"内容加密"
- 所以千万别把密码等敏感信息放进 Token 里！

## 2.5 怎么选？

| 场景              | 推荐方案         |
| :---------------- | :--------------- |
| 传统单体 Web 应用 | Session + Cookie |
| 前后端分离项目    | Token (JWT)      |
| 移动端 App        | Token (JWT)      |
| 微服务架构        | Token (JWT)      |
| 只需记住简单偏好  | 纯 Cookie        |

---

> 💬 **我的总结（Cookie/Session/Token）**：
>
> **原始版**：三者都是让服务器记住用户的方式。
>
> - **Cookie**：最简单，空间小，无隐私性
> - **Session**：给服务器数据起个名字放客户端，隐私性好，但多服务器不好存
> - **Token**：通过密钥签名生成，现代常用，安全很高
>
> ⚠️ **修正点**：Token 的 Payload **不是加密的**！是 Base64 编码，可以被解码读取。
> Token 的安全性来自"签名防篡改"，而非"内容保密"。所以别把敏感信息放进去！

# 三、从输入网址到网页显示 —— 完整流程

> 📌 **知识点导航**：本节串联了之前学过的多个知识点，点击可跳转复习：
>
> - [HTTP 状态码](./HTTP2026-04-18.md)
> - [HTTP 和 TCP 基础概念](./HTTP2026-04-19.md)
> - [TCP 与 UDP 对比](./HTTP2026-04-20.md)
> - [TCP 四次挥手详解](./HTTP2026-04-21.md)

## 3.1 全貌总览

用**去餐厅吃饭**类比整个过程：

```
1. 查地址   → "这家店在哪？"（DNS解析）
2. 打车过去 → "怎么走？"（TCP三次握手）
3. 点菜     → "我要吃什么"（发送HTTP请求）
4. 厨房做菜 → 后厨处理（服务器处理 + 返回响应）
5. 吃饭     → 在桌上享用（浏览器渲染页面）
6. 结账走人 → 离开（TCP四次挥手）
```

## 3.2 六步详细拆解

### 第①步：DNS 解析（域名 → IP 地址）

**问题**：输入的是 `www.baidu.com`（域名），但网络通信需要 **IP 地址**

就像你知道餐厅叫"海底捞"，但得先查到它的具体地址才能出发

```
浏览器缓存 → 操作系统缓存 → 本地hosts文件 → 本地DNS服务器 → 根域 → 顶级域 → 权威域
```

### 第②步：TCP 三次握手 🤝

> 🔗 详细原理见 [TCP 与 UDP 对比](./HTTP2026-04-20.md)

```
📱 客户端                          🖥️ 服务器
   │ ──── SYN(我想建立连接) ────►    │ 第一次
   │ ◄─── SYN+ACK(我也同意) ──      │ 第二次
   │ ──── ACK(开始吧) ──────────►    │ 第三次
   │ ✅ 连接建立！                    │
```

类比打电话："喂？能听到吗？"→ "能听到,你能听到吗？"→ "能听到！" → 开始聊天

### 第③步：发送 HTTP 请求

> 🔗 HTTP 请求结构见 [HTTP 和 TCP 基础概念](./HTTP2026-04-19.md)

```http
GET /index.html HTTP/1.1
Host: www.example.com
Cookie: sessionid=abc123
```

### 第④步：服务器处理 & 返回响应

> 🔗 状态码含义见 [常见 HTTP 状态码](./HTTP2026-04-18.md)

```http
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8

<!DOCTYPE html>
<html><body>你好！</body></html>
```

| 状态码  | 含义          |
| :------ | :------------ |
| 200     | 成功 ✅       |
| 301/302 | 重定向        |
| 404     | 找不到资源 ❌ |
| 500     | 服务器错误 💥 |

### 第⑤步：浏览器渲染页面 ⚠️ 注意：是客户端渲染，不是服务器！

```
HTML → DOM树
CSS  → CSSOM树
       ↘ 合并
     渲染树 → 布局(Layout) → 绘制(Paint) → 显示在屏幕 ✅
```

> 类比：服务器=后厨做菜（生成HTML/CSS等数据），浏览器=你在餐桌上享用（渲染展示）

### 第⑥步：TCP 四次挥手 👋

> 🔗 详细原理见 [四次挥手详解](./HTTP2026-04-21.md)

```
📱 客户端                          🖥️ 服务器
   │ ──── FIN(我说完了) ────────►    │ 第一次
   │ ◄─── ACK(知道了) ─────────      │ 第二次
   │ ◄─── FIN(我也说完了) ─────      │ 第三次
   │ ──── ACK(拜拜) ───────────►     │ 第四次
   │ 🔌 连接断开                      │
```

为什么是4次？因为 TCP 是全双工的（双向通道），两边各自关闭自己的方向。

## 3.3 面试补充点

**HTTPS 会多一步 TLS 握手**（加密协商），在三次握手之后、HTTP 请求之前

**Keep-Alive（持久连接）**：

- ❌ HTTP/1.0：每个请求新建连接（慢）
- ✅ HTTP/1.1 默认 Keep-Alive：一次连接发多个请求（快）

**渲染优化建议**：

- CSS 放 `<head>` 里（先加载样式）
- JS 放 `</body>` 前（避免阻塞首屏）

---

> 💬 **我的总结（输入网址到显示）**：
>
> 从输入网址到网页显示共六步：
>
> 1. **DNS 解析**：查域名对应的 IP 地址（查地址）
> 2. **TCP 三次握手**：建立可靠连接（打电话确认对方在线）
> 3. **发送 HTTP 请求**：告诉服务器你要什么资源（点菜）
> 4. **服务器处理 & 响应**：服务器处理后返回数据（厨房做菜+上菜）
> 5. **浏览器渲染页面**：**客户端**解析 HTML/CSS/JS 绘制到屏幕（你在餐桌上吃饭）
> 6. **TCP 四次挥手**：断开连接（挂电话）
