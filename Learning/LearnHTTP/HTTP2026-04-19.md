# 一、HTTP 和 TCP 的概念

## 1. HTTP 是什么？

**H**yper**T**ext **T**ransfer **P**rotocol（超文本传输协议）

| 单词               | 含义      | 通俗理解       |
| ------------------ | --------- | -------------- |
| **Hyper**    | 超级      | 不只是普通文字 |
| **Text**     | 文本      | 文字内容       |
| **Transfer** | 传输      | 从A搬到B       |
| **Protocol** | 协议/规矩 | 双方约定的规则 |

**"超文本"** = 普通文字 + 链接 + 图片 + 视频（比纯文字更"超"级）

**作用**：浏览器和服务器之间通讯的语言/规则。

```
┌──────────┐         ┌──────────┐
│  浏览器   │  ←→→   │   服务器   │
│ (你的电脑) │  HTTP  │  (网站)    │
└──────────┘         └──────────┘
```

## 2. TCP 是什么？

**T**ransmission **C**ontrol **P**rotocol（传输控制协议）

**作用**：保证数据可靠地从一端传到另一端（不丢、不乱、不重复）。

> TCP 是传数据的工具，保证数据一定送到。

## 3. HTTP 和 TCP 的关系（上下层）

```
┌─────────────────────────────────┐
│          应用层                  │
│     HTTP / HTTPS / FTP ...      │  ← "说什么话"
├─────────────────────────────────┤
│        传输层                   │
│       TCP / UDP                │  ← "怎么把话传过去"
├─────────────────────────────────┤
│        网络层                   │
│           IP                    │  ← "去哪里"
├─────────────────────────────────┤
│      数据链路层 / 物理层         │
│     WiFi / 光纤 / 网线          │  ← "走哪条路"
└─────────────────────────────────┘
```

**类比**：

- HTTP = 信件的**写法格式**（双方能读懂）
- TCP = **快递公司的配送服务**（保证送到）

---

> **我的总结：** HTTP是浏览器跟网站之间通讯的规则，而TCP是保证它们通讯的数据的安全的（修正：TCP是传数据的工具，保证数据可靠送达，但TCP不负责安全，安全是HTTPS/TLS做的事）。

# 二、HTTP 版本演进

## 1. HTTP 1.0

### 核心特点

#### (1) 短连接

每次请求都要重新建立连接（TCP握手），用完就断开。

```
第1次：建立连接 → 请求 → 响应 → 断开
第2次：建立连接 → 请求 → 响应 → 断开
第3次：建立连接 → 请求 → 响应 → 断开
```

**类比**：每次去银行都要重新取号排队。

**问题**：一个网页有20个资源，就要建立连接20次、断开20次 → 慢！

#### (2) 无状态

服务器不记得你之前做过什么，每次请求都当"新人"处理。

```
你：登录，我是秋霁 → ✅ 成功
你：看我的个人主页 → 服务器：你是谁？？？
```

**类比**：一个失忆的服务员，每次对话都当第一次见面。

> 后来通过 Cookie / Session / Token 解决了这个问题。

#### (3) 串行请求

请求只能一个一个来，不能同时发多个。

```
请求图片1 → 等 → 响应 → 请求图片2 → 等 → 响应 → ...
```

#### (4) 没有Host头

一个IP地址只能运行一个网站。要搞3个网站需要3台服务器、3个IP。

---

## 2. HTTP 1.1

在 1.0 基础上的优化：

#### (1) 长连接（Keep-Alive）

连接建立后可以多次使用，不用反复握手。

```
建立连接 → 请求1 → 响应1 → 请求2 → 响应2 → 请求3 → 响应3 → 断开
                ↑ 不用重新建立连接了！
```

**类比**：取一次号，可以办好几件事。

#### (2) 管线化（Pipelining）

不用等上一个响应回来就能发下一个请求。

```
没有管线化：请求1 → 等响应1 → 请求2 → 等响应2 → ...
有管线化：  请求1 → 请求2 → 请求3（连续发！）→ 响应1 → 响应2 → 响应3
```

**类比**：一口气发三条微信，不用一条一条等对方回。

**缺陷**：响应必须按顺序回来（队头阻塞），请求2慢了，请求3再快也得等。

#### (3) Host头

一个IP可以托管多个网站。

```
没有Host头：192.168.1.100 → 只能放一个网站
有Host头：  192.168.1.100 → 可以放 taobao.com / baidu.com / jd.com
```

请求时会带上 `Host: www.taobao.com`，服务器据此分发到对应网站。

**类比**：一栋楼开了多家店，Host头就是"我要去几零几室"。

---

> **我的总结：** HTTP1.0就是一次连接只能让服务器干一件事情，服务器干就忘记，而且它一个IP只能跑一个网站，而1.1就是优化了它的使用，一次连接可以干好几件事，一个IP可以有好几个网站。（补充：1.1还是串行，只是不用反复建立连接了，响应还是要按顺序回来）

---

## 3. HTTP 2.0

质的飞跃！核心思想：**多路复用**

```
┌─────────────────────────────────────┐
│           HTTP/2.0 连接              │
│  ┌─────────┐ ┌─────────┐ ┌──────┐ │
│  │ 流1     │ │ 流2     │ │ 流3  │ │
│  │ 图片1    │ │ 图片2    │ │ HTML │ │
│  └─────────┘ └─────────┘ └──────┘ │
│         同一个 TCP 连接内并行        │
└─────────────────────────────────────┘
```

#### (1) 多路复用（Multiplexing）

一个TCP连接上，多个请求**并行**传输，互不阻塞。

```
HTTP/1.1：请求1 → 响应1 → 请求2 → 响应2 → 请求3 → 响应3（串行排队）
HTTP/2.0：请求1、2、3 同时发出去，谁先好谁先回来！（并行）
```

**类比**：有多个窗口同时给你办事。

#### (2) 二进制分帧

把数据拆成小帧（Frame），用二进制格式传输。

```
HTTP 1.x（文本格式）：GET /index.html HTTP/1.1 （人能看懂，机器处理慢）
HTTP 2.0（二进制格式）：00010001 01001010 11010011 （人看不懂，机器处理快）
```

**类比**：拆沙发成零件装箱运输，到了目的地再拼回来。

| 对比     | 文本格式（1.x）      | 二进制分帧（2.0）      |
| -------- | -------------------- | ---------------------- |
| 解析速度 | 慢（要逐字判断边界） | 快（固定格式，直接读） |
| 体积     | 大（文本占空间多）   | 小（二进制更紧凑）     |
| 出错率   | 容易出错             | 不容易出错             |

> 注意：计算机的母语就是二进制！不是"不能解析二进制"，是"人类看不懂"。

#### (3) 头部压缩（HPACK）

三招压缩请求头：

| 招数                 | 做法                             | 类比                       |
| -------------------- | -------------------------------- | -------------------------- |
| **静态表**     | 常用头部预设好，只发索引号       | 用序号代替菜名点菜         |
| **动态表**     | 第一次发过的记住，之后只发索引号 | 自我介绍一次后不用再说名字 |
| **哈夫曼编码** | 重复字符用更短的代码代替         | "中国"简写成"CN"           |

**效果**：头部体积减少 70%+！

#### (4) 服务端推送（Server Push）

服务器主动把客户端"肯定需要"的资源提前发过去。

```
没有推送：请求HTML → 解析 → 发现需要CSS → 请求CSS → 发现需要JS → 请求JS（一步步等）
有推送：  请求HTML → 服务器顺便把CSS和JS一起推过来！
```

**类比**：聪明服务员知道套餐一般要配饮料和甜点，不用你开口就一起上了。

#### (5) 优先级控制

重要资源先传（比如CSS比图片优先），优化页面渲染顺序。

#### (6) TCP 层队头阻塞（HTTP 2.0 的缺陷）

HTTP 2.0 解决了 HTTP 层的阻塞，但底层 TCP 有一个铁规矩：**丢了一个包，后面的全部等着**。

```
HTTP 2.0 把多个流的数据混在一条TCP连接里传：
  流1 ✅  流2 ❌丢了  流3 ✅  流4 ✅  流5 ✅
  
  TCP：流2没到，流3、4、5虽然到了也不许往上交！
  → HTTP层想处理流3、4、5？对不起，TCP层卡住了
```

> 类比：出菜通道必须按顺序出菜，菜3没做好，菜4、5做好了也出不来。

---

> **我的总结：** HTTP2.0主要解决信息传递速度、数据大小和内容重复的问题的。（对应关系：速度→多路复用+服务端推送，大小→二进制分帧+HPACK，重复→HPACK）

## 4. HTTP 3.0

### 为什么现在 HTTP 1.1/2.0 还是最多？

| 原因                 | 说明                                                                   |
| -------------------- | ---------------------------------------------------------------------- |
| **兼容性**     | 升级到 HTTP 3.0 需要整条链路（浏览器→运营商→CDN→服务器）都支持 QUIC |
| **老系统**     | 很多公司的老代码、老 Nginx 版本不支持 QUIC，"能用就不改"               |
| **UDP 被拦截** | 有些防火墙直接拦截 UDP，有些运营商给 UDP 限速                          |
| **升级成本**   | 改代码、改配置、测试、上线，代价大风险高                               |

### 核心改变：抛弃 TCP，用 QUIC！

```
HTTP 2.0：HTTP/2 → TCP → IP（TCP 的队头阻塞解决不了，内核写死的）
HTTP 3.0：HTTP/3 → QUIC → UDP → IP（在 UDP 上自己实现一切）
```

> TCP 是成品汽车（发动机焊死改不了），UDP 是汽车底盘，QUIC 自己造了一辆更好的车。

### UDP 的问题

| UDP 的问题             | 说明                                         |
| ---------------------- | -------------------------------------------- |
| **不可靠**       | 发了不管，丢了不重传，到了顺序可能乱         |
| **没有流量控制** | 接收方处理不过来？不管！继续发               |
| **没有拥塞控制** | 网络堵了？不管！继续高速冲，挤占其他应用带宽 |

### QUIC 如何解决 UDP 的问题

| UDP 的问题 | QUIC 的解决方案 | 具体做法 |
|-----------|----------------|---------|
| **不可靠（丢了不管）** | 自己实现确认和重传 | 每个包都有编号，接收方确认收到，没确认的自动重传 |
| **不管顺序** | 自己实现按序交付 | 每个流独立排序，该流丢包只影响该流，不影响其他流 |
| **没有流量控制** | 自己实现流量控制 | 根据接收方的处理能力动态调整发送速度 |
| **没有拥塞控制** | 自己实现拥塞控制 | 监测网络拥堵情况，拥堵时主动降速，比 TCP 更智能 |

> 核心思路：UDP 提供轻量级的传输通道，QUIC 在应用层（而非内核层）自己补上所有可靠性机制。放在应用层的好处是可以灵活定制，不像 TCP 那样改不了。

### QPACK（头部压缩升级版）

QPACK 是 HPACK 的升级版，解决了 HPACK 动态表丢包同步问题：

1. 允许客户端和服务器的动态表**暂时不同步**
2. 用"引用"的方式（类似指针），即使表不完全一致也能安全解析
3. 如果遇到真的解析不了的情况，会**缓冲等待**，而不是像 HPACK 那样直接报错

> HPACK = 双方必须背一样的通讯录，不同步就打错人 💥
> QPACK = 允许通讯录暂时不一样，能容错，打错了也不崩溃 ✅

### HTTP 3.0 的其他优势

- **连接建立快**：TCP 需要 3 次握手 + TLS 握手（多次往返），QUIC 1 次搞定
- **网络切换不断开**：QUIC 用 Connection ID 标识连接，不依赖 IP+端口，WiFi 切 4G 无缝衔接

---

> **我的总结：** 现在用HTTP1.1和HTTP2.0的比较多，因为兼容性好；HTTP3.0把传输层从TCP改成了UDP，但UDP只管速度不管质量（不可靠、不管顺序、不管拥堵），所以HTTP3.0设计了QUIC，在UDP的基础上自己实现了可靠性机制，让各个流可以独立传输互不影响；对于头部压缩，HTTP2.0的HPACK在动态表丢包时无法同步导致解析失败，HTTP3.0的QPACK允许动态表不同步，通过容错机制解决了丢包问题（允许暂时不同步、用引用方式安全解析、缓冲等待而非直接报错）。

## 5. 三个版本完整对比

| 对比项               | HTTP 1.1              | HTTP 2.0                        | HTTP 3.0                         |
| -------------------- | --------------------- | ------------------------------- | -------------------------------- |
| **传输层**     | TCP                   | TCP                             | **QUIC（基于UDP）**        |
| **并行**       | 串行                  | 多路复用                        | 多路复用（**无队头阻塞**） |
| **数据格式**   | 文本                  | 二进制帧                        | 二进制帧                         |
| **头部压缩**   | 无                    | HPACK                           | **QPACK**                  |
| **连接建立**   | 慢（TCP+TLS多次握手） | 慢（同1.1）                     | **快（1次搞定）**          |
| **队头阻塞**   | HTTP层有              | HTTP层解决，**TCP层还有** | **彻底解决**               |
| **网络切换**   | 断开重连              | 断开重连                        | **无缝切换**               |
| **服务端推送** | 无                    | 支持                            | 支持                             |
| **无状态**     | ✅ 无状态             | ✅ 无状态                       | ✅ 无状态                       |
| **Host头**     | ✅ 有（一IP多网站）   | ✅ 有                           | ✅ 有                           |

### 直观对比

```
HTTP/1.1：请求1 → 响应1 → 请求2 → 响应2 → 请求3 → 响应3 ...（长连接，但仍排队）
HTTP/2.0：═══════════════════════════════════════════（一个连接，全部并行）
           请求1 请求2 请求3 请求4 请求5（同时发出！）
           ↓     ↓     ↓     ↓     ↓
           响应3  响应1  响应5  响应2  响应4（谁快谁先回！）
HTTP/3.0：═══════════════════════════════════════════（QUIC，各流完全独立）
           每个流有自己的通道，丢包互不影响！
```

### 一句话总结

> **HTTP 1.0** = 单车道土路，一辆车过了才能过下一辆
> **HTTP 1.1** = 单车道柏油路，可以排队连续走但还是堵
> **HTTP 2.0** = 多车道高速，所有车同时跑，谁快谁先到
> **HTTP 3.0** = 高架桥，每条路独立通行，一个出事不影响其他

# 三、补充概念

## 1. 可靠性 vs 安全性（重要区分！）

|                | TCP（可靠性）                | HTTPS/TLS（安全性）        |
| -------------- | ---------------------------- | -------------------------- |
| **意思** | 数据一定送到，不会丢、不会乱 | 数据不会被偷看、不会被篡改 |
| **类比** | 快递保证包裹送达 ✓          | 快递给包裹上了锁 🔒        |

完整关系：

```
HTTP（格式规范）+ TLS（加密保护）= HTTPS（安全通讯）
TCP（可靠传输）在底层运送它们
```

## 2. TCP vs UDP

| | TCP（靠谱老二） | UDP（随性老大） |
|--|----------------|----------------|
| **性格** | 认真负责、事无巨细 | 随心所欲、爱咋咋地 |
| **送包裹** | 送到为止，丢了重发，按序到达 | 扔出去就不管了 |
| **确认机制** | 每次都要对方说"收到了" | 发完就走，不问结果 |
| **流量控制** | 有（根据对方能力调速） | 无 |
| **拥塞控制** | 有（网络堵了就减速） | 无 |
| **类比** | 顺丰快递（签收确认） | 扔飞镖（扔了就算） |

# 四. HTTP vs HTTPS

## 核心区别：安全性

```
HTTP：
  你 → [我要登录，用户名秋霁，密码123456] → 服务器
  明文传输！路上任何人都能看到
  类比：写在明信片上寄出去

HTTPS = HTTP + TLS（加密层）
  你 → [#$%^&*!@#$(加密后的乱码)$%^&*] → 服务器
  只有你和服务器能看懂
  类比：装进密封信封里寄出去
```

| 对比项 | HTTP | HTTPS |
|--------|------|-------|
| 全称 | HyperText Transfer Protocol | HyperText Transfer Protocol **Secure** |
| 数据传输 | **明文**（谁都能看） | **加密**（只有双方能看） |
| 默认端口 | 80 | 443 |
| 需要证书 | ❌ 不需要 | ✅ 需要（CA证书） |

> S = Secure（安全的）

## HTTPS 的加密方式（两种配合使用）

### (1) 非对称加密 —— 安全交换密钥

```
一对钥匙：
  公钥（锁）→ 谁都能拿到，用来锁东西
  私钥（钥匙）→ 只有服务器有，用来开东西
  
  公钥锁上的东西，只有私钥能打开！

类比：
  服务器给每人发一把锁（公钥），自己留着钥匙（私钥）
  你把密码用锁锁好寄过去 → 路人打不开 → 服务器用钥匙打开 ✓
```

### (2) 对称加密 —— 快速传输数据

```
特点：双方用同一把钥匙
  优点：速度快（比非对称快几百倍）
  缺点：这把钥匙怎么安全地给到对方？
  → 用非对称加密来交换密钥！
```

### 配合流程

```
第1步（非对称加密，慢但安全）：
  用公钥加密对称密钥，安全传给对方

第2步（对称加密，快）：
  之后所有数据都用这个对称密钥加密/解密

总结：
  非对称 = 安全地交换密钥（只做一次）
  对称   = 快速地传输数据（后续全用它）
```

## HTTPS 完整握手流程

```
客户端                          服务器
  │                                │
  │── 1. 我要建立HTTPS连接 ────────→│
  │                                │
  │← 2. 这是我的CA证书（含公钥）───│
  │                                │
  │   3. 验证证书（可信？正确？过期？）│
  │                                │
  │── 4. 用公钥加密对称密钥发给你 ──→│
  │                                │
  │←── 5. 加密通道建立成功 ────────│
  │                                │
  │═══ 6. 之后全部用对称密钥加密传输 ═══│
```

## CA 证书是什么？

CA = Certificate Authority（证书颁发机构），就是"第三方担保人"。

```
没有CA：黑客可以冒充"我是淘宝"
有CA：浏览器验证证书是否是可信CA颁发的 → 假的会被识别出来 ❌

类比：
  CA证书 = 身份证
  银行信任公安局发的身份证 → 所以信任你
  黑客没有身份证 → 银行不给他办卡
```

## 为什么地址栏看不到 `https://`？

现代浏览器（Chrome/Edge/Firefox/Safari）默认隐藏协议前缀：

```
你看到的：    baidu.com
实际访问的：  https://www.baidu.com
```

**加密的标志**：地址栏左边的 🔒 锁头图标

| 图标 | 含义 |

## 同一网络的人能看到什么？

```
能看到的：
  ✓ 你在访问 baidu.com（域名在TLS握手时暴露）
  ✓ 大概的数据量
  ✓ 什么时候访问的

不能看到的：
  ❌ 搜索的内容 / 密码 / 页面具体内容

类比：路人知道你去银行了，但不知道你存了多少钱。
```

---

> **我的总结：** 现代都是用的HTTPS来加密运输，加密的内容只有我和服务器能看到，跟我连同一个网络的人看不到。（补充：同一网络的人能看到你在跟谁通信（域名），但看不到通信的具体内容。）
