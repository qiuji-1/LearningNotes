# 一、TCP 与 UDP 深入理解

## 1. 基本概念回顾

| | TCP | UDP |
|--|-----|-----|
| 全称 | Transmission Control Protocol | User Datagram Protocol |
| 性格 | 靠谱老二 | 随性老大 |
| 可靠性 | ✅ 保证送达 | ❌ 不管送达 |
| 连接 | 面向连接（要先握手） | 无连接（直接发） |
| 类比 | 打电话 | 寄明信片 |

> **两者都是传输层用来传输数据的工具。**

---

## 2. TCP 保证可靠的 4 大机制

### (1) 连接管理：三次握手（建立）+ 四次挥手（断开）

**三次握手（建连接）**

```
客户端                          服务器
  │                                │
  │── SYN（我想跟你建立连接）─────→│    第1次：你好，在吗？
  │                                │
  │← SYN+ACK（在的，我也准备好了）──│    第2次：在的，你准备好了吗？
  │                                │
  │── ACK（好的，开始吧！）────────→│    第3次：好了，开始聊！
  │                                │
  │═══ 连接建立成功，可以传输数据 ═══│
```

> 为什么3次？1次对方不知道你来了；2次可能因延迟导致重复连接；3次双方都确认了状态。
> 类比：微信视频通话 —— 你问能视频吗？朋友说可以问你好了没？你说好了拨通了。

**四次挥手（断连接）**

```
客户端（主动断开）              服务器（被动断开）
  │                              │
  │── FIN（我说完了，不发了）───→│    第1次：我不说了
  │                              │
  │← ACK（知道了）──────────────│    第2次：好的，我知道了
  │                              │     （服务器可能还有剩余数据处理）
  │                              │
  │← FIN（我也说完了）──────────│    第3次：我也说完了
  │                              │
  │── ACK（好的，再见！）───────→│    第4次：好的，拜拜！
  │                              │
  │═══ 连接完全关闭 ═════════════│
```

> 为什么4次？因为 TCP 是双向通道，每一边要单独关闭自己的方向。第2步和第3步之间服务器可能还要发数据，不能合并。

### (2) 可靠传输：序列号 + 确认应答 + 超时重传

```
每个数据包都有一个编号（序列号 Sequence Number）：

发送方                    接收方
  │                          │
  │── 包1（seq=1000）──────→│
  │← ACK=2001（包1收到了）──│   ← ACK = 下一个期望收到的序号
  │                          │
  │── 包2（seq=2001）──────→│
  │← ACK=3001（包2收到了）──│
  │                          │
  │── 包3（seq=3001）──────→│
  │   （这个包丢了❌）         │
  │                          │
  │← 超时了没收到ACK ────────│
  │                          │
  │── 包3 重传（seq=3001）──→│
  │← ACK=4001（包3收到了）──│
```

> 关键点：每个包都有编号 → 接收方知道顺序；收到必须回复 ACK → 发送方知道到了；超时没 ACK → 自动重传。
> 类比：给信纸标页码，朋友收到后说"下一页请给我第X页"，丢了就重寄。

### (3) 流量控制：滑动窗口（对方处理不过来？减速）

```
接收方告诉发送方："我的窗口大小是 X"
窗口大 = 能接收更多数据（处理得快）
窗口小 = 只能接收少量数据（处理得慢）
窗口0  = 别发了！我先消化一下

发送方                      接收方
  │                            │
  │── 我的窗口大小是 5个包 ──→│
  │── 包1~5 ──────────────────→│  （发了5个）
  │                            │
  │←── 窗口缩小到2个包 ──────│  （缓冲区快满了）
  │                            │
  │── 包6~7 ──────────────────→│  （只能发2个了）
  │  （等等，窗口满了）          │
  │←── 窗口恢复到5个包 ──────│  （消化完了）
  │                            │
  │── 继续发...                │
```

> 关注的是：**对方能不能接收**
> 类比：吃饭时服务员上菜的速度 —— 你吃得快就多上，吃得慢就少上，说"停"就停。

### (4) 拥塞控制：网络堵了？（整体减速）

```
流量控制 vs 拥塞控制：
  流量控制 = 对方电脑慢 → 少发点给他（关注接收方）
  拥塞控制 = 整条高速公路堵了 → 大家都减速（关注网络整体）

TCP 的拥塞策略：
  开始：慢启动，指数增长试探（1→2→4→8...）
  丢包了：立刻减半！（10→5→...然后慢慢增长）
  再丢包：再减半...

类比：高速公路上的车流 —— 出事故了前面的车减速，后面的看到刹车灯也减速，全线减速避免更堵。
```

---

> **我的总结：** TCP和UDP都是传输层用来传输数据的工具，TCP通过4种方法来保证连接的可靠，分别是：连接管理（三次握手建连接、四次挥手断连接）、可靠传输（序列号+确认应答+超时重传）、流量控制（滑动窗口，对方电脑处理不过来就减速）、拥塞控制（网络堵了就整体减速）；而UDP是纯粹的传输数据，只注重效率，延迟低开销小，但是数据不会给你排好序甚至可能还会丢失。

## 3. UDP 详细介绍

### 为什么需要 UDP？

TCP 这么靠谱，为什么不都用 TCP？

```
TCP 的代价：
  - 三次握手 = 建立连接要时间
  - 确认应答 = 每个包都要等回复
  - 重传机制 = 丢包了要重新发
  - 各种控制 = 额外的计算开销

有些场景不需要可靠，但必须要快！→ 用 UDP
```

### UDP 的核心特点

| 特点 | 说明 | 影响 |
|------|------|------|
| **无连接** | 不用握手，直接发 | 极低延迟 |
| **无确认** | 发完就走 | 快 |
| **无重传** | 丢了不管 | 实时性好 |
| **头部只有8字节** | 极简格式 | 开销极小 |

### TCP 头部 vs UDP 头部

```
TCP 头部（20-60 字节）：复杂！
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│源端口│目标端口│序列号 │确认号 │标志位 │窗口大小│校验和 │紧急指针│选项  │填充  │
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘

UDP 头部（8 字节）：简单！
┌──────┬──────┬──────┬──────┐
│源端口│目标端口│长度  │校验和 │
└──────┴──────┴──────┴──────┘

UDP 头部只有 TCP 的 1/3 到 1/7 大小！
```

### UDP 适用场景

| 场景 | 为什么用 UDP |
|------|-------------|
| **实时音视频/直播** | 偶尔丢帧没关系，重传反而卡顿 |
| **网络游戏** | 子弹位置晚到就没意义了 |
| **DNS 查询** | 就一个请求一个响应，不值得三次握手 |
| **语音通话** | 实时性最重要 |
| **HTTP 3.0 (QUIC)** | 在 UDP 之上自己实现可靠性 |

### 选择指南

```
选 TCP：数据不能错、不能丢、不能乱
  ✅ 文件下载（少一字节文件坏了）
  ✅ 网页浏览（HTML/JS/CSS 必须完整）
  ✅ 邮件 / 文件传输(FTP) / 远程登录(SSH)

选 UDP：速度第一，偶尔丢包无所谓
  ✅ 直播 / 视频通话 / 游戏 / DNS / HTTP 3.0
```

## 4. TCP vs UDP 终极对比

| 对比项 | TCP | UDP |
|--------|-----|-----|
| **连接** | 面向连接（需握手） | 无连接（直接发） |
| **可靠性** | ✅ 可靠（确认+重传） | ❌ 不可靠（尽力而为） |
| **有序性** | ✅ 保证顺序 | ❌ 不保证 |
| **速度** | 较慢（有各种机制开销） | **快**（几乎没有开销） |
| **资源消耗** | 高 | 低 |
| **头部大小** | 20-60 字节 | **8 字节** |
| **流量控制** | ✅ 有（滑动窗口） | ❌ 无 |
| **拥塞控制** | ✅ 有 | ❌ 无 |
| **模式** | 一对一（点对点） | 支持一对一、一对多、多对多 |
| **适用场景** | 文件、网页、邮件 | 视频、游戏、DNS |
| **类比** | 打电话 | 广播/喊话 |

> **一句话总结：TCP 是认真负责的老大哥，虽然慢但绝不出错；UDP 是自由奔放的年轻人，虽然粗心但速度快效率高。不是谁更好，而是不同场景选不同的工具。**

# 二、三次握手详解

## 1. 标志位（Flag）是什么？

SYN / ACK / FIN 都是 TCP 头部里的标志位，每个占 1 个 bit：

| 标志 | 全称 | 含义 | 一句话 |
|------|------|------|--------|
| **SYN** | **S**ynchronize | 同步 | "我想跟你建立连接" |
| **ACK** | **Acknowledge** | 确认 | "我收到了 / 我知道了" |
| **FIN** | **F**inish | 结束 | "我说完了，要断开" |

> 用打电话类比：SYN = 拨号"喂在吗"；ACK = 回应"在！"；FIN = 说"挂了吧"

## 2. 三次握手完整过程

```
客户端                          服务器
  │                                │
  │  状态：CLOSED                  │  状态：LISTEN（等待来电）
  │                                │
  │══ 第1次握手 ═══                │
  │── SYN=1, seq=x ─────────────→│    你拨电话："喂？在吗？"
  │  状态：SYN_SENT               │
  │   （已拨出，等接听）            │
  │                                │
  │══ 第2次握手 ═══                │
  │← SYN=1, ACK=1, seq=y ────────│    对方接了："在的！你好了吗？"
  │   ack=x+1                     │     （对方也确认想和你通话）
  │                                │
  │══ 第3次握手 ═══                │
  │── ACK=1, seq=x+1 ───────────→│    你说："好了！开始吧！"
  │  ack=y+1                      │
  │                                │
  │  双方状态：ESTABLISHED        │  双方状态：ESTABLISHED
  │  （✅ 电话接通）                │  （✅ 电话接通）
  │                                │
  │═══ 可以开始传输数据啦！🎉 ═════│
```

### 为什么每一步这样设计？

| 步骤 | 为什么只有 SYN 没有 ACK？ | 类比 |
|------|--------------------------|------|
| 第1次只有SYN | 第一次对话，还没东西可确认 | 你第一次给陌生人发微信，只能说"你好"，不能说"收到" |
| 第2次有SYN+ACK | 既要回复你(ACK)，也要发起自己的连接(SYN) | 朋友回："在的！我们视频吧？"——双向确认 |
| 第3次只要ACK | 双方都SYN过了，只需最后确认 | 你只需说"好，开始"——不用再说"我想视频"了 |

> 三次握手全是"确认关系"阶段，全部确认完之后才开始传数据。第3次只是最后确认一下，真正传数据是之后的事。

## 3. 状态变化（面试常考）

```
客户端：CLOSED → SYN_SENT → ESTABLISHED
         （关着）  （已拨号）  （接通了）

服务器：LISTEN → SYN_RCVD → ESTABLISHED
        （等来电）（已接听）（接通了）
```

## 4. 为什么不能简化为2次？

**问题：旧的连接请求因为网络延迟迟到了**

```
时间线（用打电话类比）：
  t=0:   你打了一个电话（请求A）→ 信号不好，卡住了...
  t=5:   你等不及了，重新打了（请求B）→ 接通了 ✅
  t=7:   那个旧电话（请求A）信号终于到了！

  如果是2次握手：
  朋友接到旧电话："喂？" → 以为你要重新聊
  但你根本没有这个打算！→ 白白浪费时间 💀

  有3次握手就不会这样：
  朋友接到旧电话后说："你确定吗？请确认"
  你一看："我都忘了这事了" → 不理它 → 问题解决 ✅

生活类比：
  你给女朋友发了一条去年的微信："今晚一起吃饭吗？"
  （消息在网络里迷路了一年才送到 😂）
  
  女朋友收到："好啊！" → 一个人去餐厅等你
  结果你没去 → 白等了
  
  有3次握手的话：
  女朋友收到："好啊！你确定吗？请确认一下"
  你："？？？我都忘了这回事" → 取消 → 不白跑 ✅
```

# 三、三次握手的经典问题与异常处理

## 问题总览

| 问题 | 说明 | 处理方式 |
|------|------|---------|
| **SYN 洪泛攻击** | 黑客用假IP大量发起连接请求 | SYN Cookie、限速 |
| **握手丢包** | SYN/SYN+ACK/ACK 丢了 | 自动重传（指数退避） |
| **历史重复SYN** | 旧的请求延迟到达 | 3次握手能识别并丢弃 |
| **半打开连接** | 建立连接后一方突然失联 | 心跳/保活机制检测 |

## 1. SYN 洪泛攻击（最重要）

### 攻击原理（用打电话类比）

```
正常情况：
  你打电话给朋友 → 朋友接听 → 聊天 ✅
  朋友为你分配了一个通话线路

黑客攻击：
  黑客用10000个假号码给朋友打电话：
  假号码1 打进来 → 朋友接起电话等对方说话
  假号码2 打进来 → 朋友又接起一个等对方说话
  假号码3 打进来 → 朋友再接起一个...
  ... 10000个假电话
  
  因为号码都是假的 → 没有人会回应
  朋友的电话线路全被占满了（都在等回应）
  → 真正的你打不进来了 😱
```

### 解决方案（继续用电话类比）

| 方案 | 原理 | 电话类比 |
|------|------|---------|
| **SYN Cookie** | 不立即分配资源，先验证对方身份 | 不直接"接通"，而是先问"请报上暗号" |
| **限制速率** | 同一IP每秒只接受N个SYN | 同一个人每分钟只能打一次 |
| **增加队列容量** | 扩大半连接池 | 多开几条备用线路 |

## 2. 握手过程丢包怎么办？

TCP 有自动重传机制，超时没收到回复就重发（1s → 2s → 4s → 8s... 指数增长）

| 丢哪一包 | 发生了什么 | 影响 |
|---------|-----------|------|
| **第1次 SYN 丢** | 客户端重传 SYN | 最慢，要等超时 |
| **第2次 SYN+ACK 丢** | 客户端重传 SYN | 较慢 |
| **第3次 ACK 丢** | 几乎无影响 | 客户端认为已连上，开始传数据；服务器收到数据就知道没问题了 |

## 3. 历史 SYN 迟到

前面已讲过（见"为什么不能简化为2次"），3次握手能识别过期的旧请求并丢弃。

## 4. 半打开连接（Half-Open Connection）

### 什么是半打开连接？（用打电话类比）

```
正常情况：你和朋友正在通电话（双向都能听到）

半打开：
  你说的话 → 朋友能听到 ✅
  朋友说的话 → 你听不到 ❌

原因可能是：
  - 朋友那边手机故障了
  - 信号单向中断
  - 朋友挂断了但你不知道
```

### 如何检测？（继续用电话类比）

```
TCP 的心跳机制 = 时不时问一句"你还在吗？"

  你和 friend 通话中...
  一段时间没听到对方说话了
  → 你主动问一句："喂？还在吗？"
  → 有回应 → 继续保持通话 ✅
  → 没回应 → 认为对方掉线了 → 挂断电话 ❌
```

---

> **我的总结：** 一般情况下握手丢包了会自己自动重新传输；被SYN欺骗会有验证机制（SYN Cookie等）；迟到的问题3次握手可以解决；半打开连接可以通过心跳/保活机制检查对方是否存活。
