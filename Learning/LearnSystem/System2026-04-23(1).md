## 一、进程 (Process)

### 为什么叫"进程"？
英文 **Process** = "进行中的过程"。一个程序从开始执行到结束的整个过程。

### 生活类比（工厂）
| 概念 | 类比 |
|------|------|
| 程序 (Program) | 一份**设计图纸**（静态的，放在柜子里不动） |
| 进程 (Process) | 图纸被拿出来**真正开工生产**的一条**生产线**（动态的） |

- 电脑上存着微信安装包 → 这是**程序**
- 双击打开微信 → 这时系统创建了一个**进程**

### 关键特点
```
进程 = 程序 + 运行时的资源（内存、文件句柄、CPU时间片...）
```
- **进程是资源分配的基本单位** —— 操作系统给每个进程分配独立的内存空间
- 进程之间**相互隔离**

## 二、线程 (Thread)

### 为什么叫"线程"？
英文 **Thread** 原意是"线、线索、脉络"。进程中一条独立的执行流/执行线索。

### 工厂类比续
| 概念 | 类比 |
|------|------|
| 进程 | 一条完整的**生产线**（有厂房、设备、原材料） |
| 线程 | 生产线上的**工人**（真正干活的人） |

### 关键特点
```
线程是 CPU 调度的基本单位
```

## 三、核心区别对比表

| 对比维度 | 进程 Process | 线程 Thread |
|----------|-------------|-------------|
| 本质 | 资源分配的单位 | CPU调度的单位 |
| 内存空间 | ✅ 各自独立 | ❌ 共享所属进程的内存 |
| 通信方式 | 复杂（IPC：管道、消息队列等） | 简单（直接读/写共享变量） |
| 创建开销 | 大（要分配内存、拷贝资源） | 小（共享进程资源即可） |
| 切换开销 | 大（要保存/恢复整个上下文） | 小（只需保存少量寄存器） |
| 独立性 | 崩溃不影响其他进程 | 一个线程崩溃可能拖垮整个进程 |
| 能否单独存在 | ✅ 可以 | ❌ 必须依附于进程 |

### 关系图
```
┌─────────────────────────────────────┐
│            进程（工厂）              │
│  ┌──────────┐  ┌──────────┐       │
│  │  线程1   │  │  线程2   │       │
│  │  (工人A) │  │  (工人B) │       │
│  └──────────┘  └──────────┘       │
│                                     │
│  共享：内存空间、文件、代码段...      │
└─────────────────────────────────────┘
```

## 四、深入理解两个关键术语

### 4.1 为什么叫「分配资源」，不叫「分配内存」？

因为**资源 ≫ 内存**！操作系统给进程分配的不只是内存：

| 资源类型 | 举例 |
|----------|------|
| 内存空间 | 堆、栈、代码段 |
| 文件描述符/句柄 | 打开的文件、网络socket |
| I/O 设备权限 | 键盘、打印机、显卡 |
| PID（进程标识符） | 独立的身份ID |
| 安全凭证 | 用户权限、访问控制列表 |
| 信号处理方式 | SIGKILL、SIGINT等 |
| 环境变量 | PATH、HOME等 |

如果只说"分配内存"，就**以偏概全**了。OS实际上是在说："这个进程，我给你一套完整的生存包。"

### 4.2 为什么叫「CPU调度」(Scheduling)，不叫「CPU控制」？

| 词 | 含义 | 合适吗？ |
|----|------|---------|
| 控制 (Control) | 操纵、支配、直接操控行为 | ❌ CPU自己执行指令，OS不能"控制"每条指令怎么跑 |
| 调度 (Schedule) | 安排顺序、分配时间片、决定谁先谁后 | ✅ OS做的就是：选线程→给它时间片→到时间换下一个 |

**调度员类比** 🚌：
- 有10辆公交车（= CPU核心）
- 有100个线路请求（= 就绪队列里的线程）
- 调度员决定：下一趟让哪辆车跑哪条线

OS不亲自执行指令，它只做**决策和安排**——这就是"调度"的本质。

## 六、线程底层长什么样

### 6.1 TCB (Thread Control Block) — 线程控制块

每个线程都有一个TCB，是线程在操作系统的"档案袋"：

```
┌─────────────────────────────────────┐
│           TCB 线程控制块             │
├─────────────────────────────────────┤
│  Thread ID     (线程唯一标识)        │  ← 相当于工号
│  Program Counter(程序计数器/PC)       │  ← 下一条要执行的指令地址
│  寄存器状态     (通用寄存器值)        │  ← 干到哪了、中间结果是什么
│  栈指针         (Stack Pointer)      │  ← 自己的栈在哪
│  线程状态       (运行/阻塞/就绪...)   │  ← 当前什么情况
│  优先级                            │  ← 谁先谁后
│  所属进程指针                       │  ← 归哪个工厂管
│  调度信息                          │  ← 上次什么时候跑的、跑了多久
│  信号掩码                          │  ← 哪些信号我要忽略
└─────────────────────────────────────┘
```

### 6.2 栈 (Stack) 是什么？

**栈就是内存！** 只是用法特殊的一块内存区域——专门用来管理**函数调用**。

#### 内存布局
```
┌─────────────────────────────────────── 高地址
│           命令行参数 / 环境变量          │
├───────────────────────────────────────
│               栈 (Stack)  ▼ 向下增长     │  ← 每个线程各有一份（私有）！
│            ↓ 空闲空间                    │
│               堆 (Heap)   ▲ 向上增长    │  ← 所有线程共享
├───────────────────────────────────────
│         BSS段 / 数据段 / 代码段         │  ← 共享
└─────────────────────────────────────── 低地址
```

#### 栈的作用：记录每次函数调用

每次调用函数 = 创建一个 **栈帧 (Stack Frame)**，存：局部变量、参数、返回地址。

> **类比** 📋：栈就像一摞盘子——调用函数放一个盘子（压栈），函数结束拿走一个（出栈）。每个线程有自己的一摞盘子，互不干扰。

### 6.3 每个线程私有 vs 共享

| 区域 | 私有还是共享 | 说明 |
|------|------------|------|
| 栈 (Stack) | 🔒 私有 | 每个线程有自己的调用栈 |
| 寄存器 | 🔒 私有 | 切换时保存/恢复 |
| PC计数器 | 🔒 私有 | 每个线程执行到不同位置 |
| 堆 (Heap) | 🌐 共享 | new出来的对象大家都能访问 |
| 代码段 | 🌐 共享 | 同一个程序的代码 |
| 全局/静态变量 | 🌐 共享 | ⚠️ 线程安全的重灾区 |

## 七、线程切换的完整流程（A → B）

### 触发原因
- 时间片用完（定时器中断）最常见
- 线程主动让出（Thread.yield()）
- 等待I/O阻塞
- 更高优先级线程抢占

### 切换步骤

```
① 时钟中断发生，CPU暂停当前指令

② 保存线程A现场 → 写入A的TCB：
   - PC值（执行到了哪一行）
   - 寄存器值（中间计算结果）
   - 栈指针（栈顶位置）
   - 状态标记 → 改为"就绪"或"阻塞"

③ OS调度器从就绪队列中选出下一个线程B

④ 从B的TCB恢复B的现场：
   - 把B的PC写入CPU的程序计数器
   - 把B的寄存器值写回CPU各寄存器
   - 把B的栈指针写回CPU

⑤ 中断返回，CPU开始执行线程B的代码
```

### 模式转换
```
用户态(A在跑) → 内核态(OS介入做切换) → 用户态(B在跑)
整个过程涉及一次「陷入内核」和一次「返回用户态」
```

### ⭐ 关键理解：各自存各自取，不存在继承！

```
A的TCB              B的TCB
┌──────────┐        ┌──────────┐
│PC = 行5  │        │PC = 行12 │
│R1 = 49   │   ≠   │R1 = 2    │
│R2 = 1225 │       │R2="aaa"  │
└──────────┘        └──────────┘
  ↑ 存A的              ↑ 取B的
  各自独立！

CPU寄存器就像一张办公桌：
- A用的时候铺的是A的文件
- 到时间了收进A柜子(TCB)
- B来的时候从B柜子取出B的文件铺上
- A的原数据完好保存在A的柜子里
```

> 就像游戏存档：每个人有独立的存档，读档时从自己的进度继续，不会混在一起。

## 八、用户线程 vs 内核线程

### 三种映射模型

| 模型 | 关系 | 特点 | 代表 |
|------|------|------|------|
| N:1 多对一 | N个用户线程 → 1个内核线程 | 切换快；但一个阻塞全阻塞 | 已淘汰 |
| **1:1 一对一** | 1个用户线程 ↔ 1个内核线程 | 并行强；创建开销大 | **Java用的这个** |
| M:N 多对多 | M个用户线程 → N个内核线程 | 结合两者优点 | Go goroutine |

### Java中 1:1 的协作流程

```java
你写: new Thread(() -> { doSomething(); }).start();
           │
           ▼
    JVM线程库(用户态)
    ├── 创建Thread对象
    ├── 分配Java栈
    └── 调用 pthread_create() ──▶ 进入内核态
                                      │
                               操作系统内核
                              ├── 创建TCB
                              ├── 分配内核栈
                              └── 加入调度队列 → 等待上CPU
```

**一句话总结**：用户线程是你编程时的抽象，内核线程是OS真正调度的实体，两者通过系统调用桥接。
