# 九、Java 线程的实现方式

## 9.1 三种核心实现方式

| 方式                    | 怎么写                            | 底层本质         | 适用场景            |
| ----------------------- | --------------------------------- | ---------------- | ------------------- |
| 继承 Thread             | `class MyThread extends Thread` | 1:1 映射内核线程 | 简单场景            |
| **实现 Runnable** | `new Thread(new MyRunnable())`  | 同上             | **最常用** ✅ |
| 实现 Callable           | `FutureTask + Callable`         | 同上             | 需要返回值时用      |

**注意：JDK 1.2 之后，Java线程全部采用 1:1 模型一一映射到内核线程。**

## 9.2 方式1：继承 Thread

```java
public class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("我是线程: " + getName());
    }
}
// 使用
MyThread t = new MyThread();
t.start();   // ← 注意是 start() 不是 run()
```

- **缺点**：Java单继承，继承了Thread就不能继承别的类了。

## 9.3 方式2：实现 Runnable ← 最推荐！

```java
public class MyRunnable implements Runnable {
    @Override
    public void run() {
        System.out.println("Runnable任务在运行");
    }
}
// 使用
Thread t = new Thread(new MyRunnable());
t.start();
```

**优点**：不影响继承其他类 ✅；任务和线程对象分离更灵活 ✅

```java
// Lambda 写法（最简洁）
new Thread(() -> {
    System.out.println("Lambda线程");
}).start();
```

## 9.4 方式3：Callable — 有返回值的线程

```java
import java.util.concurrent.*;

public class MyCallable implements Callable<String> {
    @Override
    public String call() throws Exception {
        return "计算结果: 42";
    }
}

// 使用（需要FutureTask包装）
FutureTask<String> task = new FutureTask<>(new MyCallable());
new Thread(task).start();
String result = task.get();  // 获取结果（会阻塞直到跑完）
```

**对比**：

|          | Runnable      | Callable        |
| -------- | ------------- | --------------- |
| 返回值   | ❌ void       | ✅ 泛型返回值V  |
| 抛异常   | ❌ 不能throws | ✅ 可以throws   |
| 获取结果 | ❌ 没有       | ✅ Future.get() |

## 9.5 方式4：线程池 ← 生产环境必用！

```java
ExecutorService pool = Executors.newFixedThreadPool(3);
pool.execute(() -> System.out.println("任务1"));
pool.execute(() -> System.out.println("任务2"));
pool.shutdown();  // 关闭线程池
```

| 不用线程池           | 用线程池           |
| -------------------- | ------------------ |
| 每次new Thread开销大 | 线程复用，省开销   |
| 无限创建可能OOM      | 控制最大线程数     |
| 手动管理麻烦         | 自动调度/复用/回收 |

# 十、Runnable 和 Callable 的来源与区别

## 10.1 都是 JDK 自带的

```java
// java.lang.Runnable — JDK 1.0 就有了（@FunctionalInterface）
public interface Runnable {
    public abstract void run();   // 返回void，无参数无返回值
}

// java.util.concurrent.Callable — JDK 1.5 加入（@FunctionalInterface）
public interface Callable<V> {
    V call() throws Exception;     // 有返回值V，还能抛异常
}
```

## 10.2 为什么 Runnable 没返回值而 Callable 有？

**时间线原因：**

```
1996年 JDK 1.0 → 设计Runnable时觉得线程就是"跑个任务"，不需要返回值
2004年 JDK 1.5 → 后来发现很多场景需要拿结果 → 新增Callable
```

**为什么不直接改 Runnable？**

- 给 run() 加上返回值 → 所有已实现 Runnable 的代码全部报错 💥
- 所以保留 Runnable 不动，新增 Callable 作为补充

## 10.3 Callable\<V\> 的 V 是什么？

**V 是泛型类型占位符，想返回什么类型就写什么！**

```java
// V = Integer
Callable<Integer> task1 = () -> 42;

// V = String
Callable<String> task2 = () -> "Hello";

// V = 自定义对象
Callable<User> task3 = () -> new User("张三", 20);
```

**泛型命名惯例：**

| 字母  | 含义                 | 常见位置                        |
| ----- | -------------------- | ------------------------------- |
| T     | Type（类型）         | `class Box<T>`                |
| E     | Element（元素）      | `List<E>`                     |
| K / V | Key / Value（键/值） | `Map<K, V>` / `Callable<V>` |

所以 `Callable<V>` 的 V 表示 **Value（返回值的类型）**。

## 10.4 无论哪种写法底层殊途同归

```
new Thread(...) / new Thread(Runnable) / pool.execute(task)
        │
        └────── 全部 ──────▶ pthread_create() ─▶ 创建内核线程
```

# 十一、「组合」vs「继承」

## 11.1 基本概念

|        | 继承 (extends)           | 组合                      |
| ------ | ------------------------ | ------------------------- |
| 关系   | **is-a**（是什么） | **has-a**（有什么） |
| 写法   | `extends`              | 成员变量持有对象          |
| 耦合度 | 高                       | 低                        |

```java
// 继承：Dog 是一种 Animal
class Dog extends Animal { ... }

// 组合：Computer 里面有一个 CPU
class Computer {
    CPU cpu;       // Computer has-a CPU
    Screen screen; // Computer has-a Screen
}
```

## 11.2 为什么 Runnable 用组合绕过了单继承？

```java
// ❌ 继承方式：你的类变成了 Thread 的一种
class MyTask extends Thread { ... }   // 占用了唯一继承位

// ✅ 组合方式：你的类还是你自己，被 Thread 持有
class MyTask implements Runnable {     // 不占用继承位！
    @Override
    public void run() { /* 干活 */ }
}

// 使用：
MyTask task = new MyTask();
Thread t = new Thread(task);   // Thread 里面持有了 task → 组合！
t.start();
```

**关系图：**

```
┌─────────────┐      持有       ┌──────────┐
│   Thread     │  ──────────▶  │  MyTask  │
│  (工人)      │                │ (任务单)  │
│             │                 │          │
│ 负责：启动、调度、管理线程    │ 负责：具体要干什么活           │
└─────────────┘                 └──────────┘
  不是父子关系！是"工人拿着任务单干活"
```

> 类比 🏭：继承=你自己变成工人；组合=你写任务单交给工人去执行。

# 十二、default 方法能解决单继承问题吗？

**不能！这是两个不同的问题。**

## 12.1 default 解决的是什么？

default 解决的是 **接口演进问题** —— 让接口可以新增方法而不破坏已有实现：

```java
// JDK 8 之前 List 只有 add/get/remove
// JDK 8 想加 forEach 方法：

public interface List<E> {
    boolean add(E e);         // 老方法
    E get(int index);         // 老方法
  
    default void forEach(Consumer<? super E> action) {  // 新方法用default
        for (E e : this) { action.accept(e); }          // 已有实现不用改！✅
    }
}
```

## 12.2 为什么不能用来解决类的单继承？

**原因1：Thread 是 class 不是 interface**

- Thread 包含大量字段(tid, threadStatus)、native方法(start())等复杂逻辑
- 不可能塞进一个 interface 的 default 里

**原因2：default 不能让 class 多继承**

- default 解决的是"接口多实现"，不是"类的多继承"

**结论：绕过单继承的正确方式是用 Runnable（组合），而不是去改造 Thread。**

# 十三、start() vs run() — 很多人踩过的坑！

## 结论：用 start()！千万别直接调 run()！

| | `start()` | `run()` |
|--|-----------|---------|
| 做了什么 | 启动新线程，然后在新线程中调用 run() | 普通方法调用 |
| 会创建新线程吗？ | ✅ 会（OS创建内核线程） | ❌ 不会，在当前线程执行 |
| 谁在执行？ | 新创建的线程 | 调用它的那个线程 |

## 代码对比

```java
Thread t = new Thread(() -> {
    System.out.println("我是谁？" + Thread.currentThread().getName());
});

t.start();   // 输出: 我是谁？Thread-0    ← 新线程在跑
t.run();     // 输出: 我是谁？main        ← main自己跑了
```

## start() 底层流程

```
你调用 t.start()
    │
    ▼
Thread.start() (Java层)
    ├── 检查线程状态
    └── 调用 native start0()
            │ (JNI 进入C/C++代码)
            ▼
JVM/操作系统层
    ├── 调用 pthread_create()
    ├── 创建TCB + 内核线程
    └── 加入OS就绪队列
            │
            ▼
OS调度器选中 → 新线程运行 → 自动回调 run()
```

> **类比 🏭：start() = "招人干"（新线程来干活）；run() = "自己干"（还是你在干活）**

# 十四、Native 方法 — Java 和 C/C++ 之间的桥梁

## 14.1 一句话定义

**native 方法 = 用C/C++实现的方法，Java只负责声明，不写具体代码。**

```java
// 普通方法：Java写的，有方法体
public String getName() {
    return this.name;
}

// native方法：只有声明，没有方法体（实现在C/C++）
public native void start0();   // ← 没有{}！只有分号
```

## 14.2 为什么需要 native？

Java 自己干不了（或干不好）的事情：

| 能力 | Java能做吗？ | 谁来做？ |
|------|-------------|---------|
| 业务逻辑、数据处理 | ✅ | Java |
| 和操作系统打交道（创建线程、文件系统） | ❌ | C/C++ |
| 底层硬件操作（内存管理、网络协议） | ❌ | C/C++ |
| 高性能计算（图像处理、加密算法） | 🐢 能但慢 | C/C++ 快 |

> 类比 🏭：Java = 业务部门（接单谈客户）；Native(C/C++) = 工厂车间（真正生产、操作机器）

## 14.3 JNI — Java和C之间的桥梁

**JNI = Java Native Interface（Java本地接口）**，让Java代码可以调用C/C++：

```
┌─────────────┐       JNI       ┌──────────────────┐
│   Java代码    │  ◀─────────▶  │   C/C++ 代码      │
│              │   (翻译/桥接)   │                  │
│ start0()     │               │ pthread_create()  │
│ System.gc()  │               │ GC_impl()         │
└─────────────┘                 └──────────────────┘
```

## 14.4 常见的 native 方法（你天天在用的）

```java
// java.lang.Object
public class Object {
    public native int hashCode();
    // wait()/notify()/notifyAll() 也是 native
}

// java.lang.Thread
public class Thread {
    private native void start0();           // 创建线程
    public static native void yield();      // 让出CPU
    public static native void sleep(long);  // 睡眠
}

// java.lang.System
public class System {
    public static native long currentTimeMillis();  // 系统时间
    public static native void arraycopy(...);        // 数组拷贝（比for循环快）
}
```

## 14.5 完整流程图（以Thread.start()为例）

```
new Thread(...).start()
       │
       ▼ Thread.start()          ← Java层
       │   调用 start0()
       ▼
       Thread.start0()          ← native声明，无方法体
       │
       │ ★ JNI ★
       ▼
       JVM内部 C代码             ← 真正实现在这里
       │
       │ pthread_create()       ← Linux系统调用
       ▼
       操作系统内核              ← 完成线程创建
```

## 14.6 设计哲学

```
95%的事情Java自己做（跨平台+安全+开发效率高）
5%底层的事交给native（通过JNI找C/C++帮忙）
```

| 角色 | 类比 | 职责 |
|------|------|------|
| Java | 公司前台 | 接待客户、处理常规业务 |
| Native(C/C++) | 后勤/技术部 | 维修设备、操作机器 |
| JNI | 对讲机 | 前台和技术部之间的通信工具 |
