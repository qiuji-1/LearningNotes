# 单例模式

## 概览

常见的单例模式实现方式主要有五种：

**1. [饿汉式](#什么是饿汉式它为什么叫这个名字)**

- 类加载时就创建实例
- 优点：实现简单，线程安全（JVM类加载机制保证）
- 缺点：类加载时就初始化，可能造成资源浪费

**2. [懒汉式](#什么是懒汉式它为什么叫这个名字)**

- 首次使用时才创建实例
- 优点：延迟加载，节省资源
- 缺点：线程不安全（需要加同步锁）

**3. [双重检查锁](#什么是双重检查锁为什么叫这个名字)**

- 懒汉式的改进版，使用双重检查 + synchronized
- 优点：既实现了延迟加载，又保证了线程安全，性能较好
- 缺点：实现稍复杂，需要volatile关键字防止指令重排序

**4. [静态内部类](#什么是静态内部类为什么叫这个名字)**

- 利用类加载机制，通过内部类持有实例
- 优点：延迟加载 + 线程安全 + 实现简单
- 缺点：无法传参初始化

**5. [枚举](#什么是枚举单例为什么叫这个名字)**

- 利用Java枚举特性实现单例
- 优点：实现最简单，天然防止反射和反序列化破坏单例
- 缺点：不支持延迟加载

## 为什么需要单例模式？

主要是为了节省资源，制造一个全局都能访问的入口，避免多个实例之间的管理冲突。

## 第一个问题：private static Counter instance = new Counter(); 这句话里，第一个Counter和末尾的new Counter()有什么区别？

第一个Counter是类型名，说明instance这个变量是什么类型；末尾的new Counter()是创建一个新的Counter对象，调用Counter类的构造器。

- 第一个Counter：只是说明要放一个钱包
- 末尾的new Counter()：才是真的放一个钱包上去

## 第三个问题：为什么只是加了一个静态的访问入口，对象就唯一，而不是它通过入口进去还是各自拿各自的对象呢？

因为静态变量只保存一个对象引用，而且构造器是私有的。

- 静态变量：整个类只有一个副本
- 私有构造器：阻止外部创建新对象
- getInstance()：不new对象，直接返回已有的instance

所以永远是同一个对象。

**用教室例子来理解：**

- **教室（instance）** - 唯一
- **黑板（count）** - 唯一
- **学生（c1, c2, c3）** - 通过入口进入教室

学生都进入同一个教室，都在同一个黑板上写字：

```java
// 学生1写字
c1.write();  // 黑板上有1个字

// 学生2写字
c2.write();  // 黑板上有2个字

// 学生3写字
c3.write();  // 黑板上有3个字
```

特点就是：教室唯一、黑板唯一，但内容在变。类比过来就是仓库（instance）唯一、对象唯一、状态可以修改。

## 第四个问题：instance相当于一个仓库，构造器将对象赋值给这个仓库，后面想要修改，只能从入口getInstance()里进去，修改仓库而不能直接修改对象。这样理解对吗？

部分正确。

**正确部分：**

- instance相当于仓库
- 构造器将对象赋值给仓库

**错误部分：**

- 不是修改仓库，而是修改仓库里的对象状态
- 从入口拿到对象，然后修改对象的状态
- 仓库本身不用改

访问流程是这样的：

```
类加载时：
  new Counter() → 调用构造器 → 创建对象 → 赋值给instance

后续访问时：
  getInstance() → 返回instance → 不调用构造器
```

---

## 第五个问题：修改的是仓库里的对象 但是 你不是说对象是唯一的吗 还是说唯一的是仓库？

**都唯一！**

- 仓库（instance）：唯一
- 仓库里的对象：唯一

整个程序只有一个instance，instance指向的对象也只有一个。修改的是对象的状态，不是创建新对象，所以还是唯一的。

完整代码长这样：

```java
public class Counter {
    // 1. 静态变量：保存唯一实例
    private static Counter instance = new Counter();
  
    // 2. 私有构造器：阻止外部创建新对象
    private Counter() {}
  
    // 3. 公共入口：提供访问方法
    public static Counter getInstance() {
        return instance;
    }
  
    // 对象属性
    private int count = 0;
  
    public void add() {
        count++;
    }
  
    public int getCount() {
        return count;
    }
}
```

使用的时候是这样的：

```java
// 获取对象（永远得到同一个）
Counter c1 = Counter.getInstance();
Counter c2 = Counter.getInstance();
Counter c3 = Counter.getInstance();

// 验证是否是同一个对象
System.out.println(c1 == c2);  // 输出：true

// 修改对象状态
c1.add();  // count从0变成1
c2.add();  // count从1变成2
c3.add();  // count从2变成3

System.out.println(c1.getCount());  // 输出：3
System.out.println(c2.getCount());  // 输出：3
System.out.println(c3.getCount());  // 输出：3
```

## 什么是饿汉式？它为什么叫这个名字？

就像一个"饿汉"一样，迫不及待地想吃饭。类一加载，就立即创建实例，不管你用不用，先创建出来再说。

### 实现原理：

- 静态变量直接初始化：在类加载时就创建实例
- 私有构造器：防止外部new
- 公共静态方法：提供访问入口

代码其实上面已经写过了，核心就一行：

```java
private static Singleton instance = new Singleton();
```

这行代码在类加载时就执行，JVM保证类加载过程线程安全，所以天然线程安全。

### 我的总结：

饿汉式就是实例在类被加载时就创建，实现很简单，核心就一行代码，JVM会保证类在加载中的线程安全，所以饿汉式创建出来的实例天然就很安全。

### 那所谓的"浪费"是什么意思？

饿汉式创建时机不是你调用getInstance()时创建，而是类被加载时就创建了。类加载的触发条件有很多，比如调用getInstance()会触发，或者别的代码引用了Singleton类名也会触发。所以可能出现的情况是——类被加载了，实例创建了，但你从没真正使用这个实例。

不过话说回来，如果用不上这个单例类，我根本就不会写它对吧。既然写了就说明打算用，那提前创建也没什么大不了的。饿汉式的"缺点"其实是理论上的缺点，实际开发中大部分单例都是核心功能，肯定要用，创建开销也不大。

### 拓展：JVM相关疑问

**问题：为什么JVM可以保证类加载时的线程安全呢？**

核心原因：**JVM在类加载时，会自动加一把隐式的锁**

```
类加载的内部流程（JVM自动完成）：

1. JVM检测到某个类需要加载
2. JVM内部加锁（类似synchronized）
3. 执行类的初始化（包括静态变量赋值、静态代码块）
4. 释放锁

如果两个线程同时触发同一个类的加载：
  线程A：拿到锁 → 初始化 → 释放锁
  线程B：等锁 → 等到A完成后 → 发现已经初始化好了 → 直接用
```

对比一下就清楚了：

|        | 类加载过程                      | 懒汉式的getInstance()            |
| ------ | ------------------------------- | -------------------------------- |
| 谁控制 | **JVM内部控制**，自动加锁 | **你自己写的代码**，没加锁 |
| 所以   | 天然安全                        | 不安全                           |

简单说就是JVM帮你在类加载的时候偷偷加了同步机制，你不用操心。而懒汉式那个 `if (instance == null)` 是你自己写的逻辑，JVM不会帮你加锁。

**问题：JVM到底是什么？它怎么这么厉害？**

JVM就是**Java虚拟机（Java Virtual Machine）**，也就是Java程序运行的地方。

```
你的电脑硬件
    └── 操作系统（Windows/Mac/Linux）
            └── JVM（Java虚拟机）
                    └── 你的Java程序
```

你写的 `.java` 代码 → 编译成 `.class` 文件（字节码） → **JVM来执行**

它之所以"厉害"，其实就是帮你干了很多脏活累活：

| 你不用操心的事 | JVM帮你做的                    |
| -------------- | ------------------------------ |
| 内存分配和回收 | 自动垃圾回收（GC）             |
| 类什么时候加载 | 类加载机制（包括那个自动加锁） |
| 不同系统的差异 | 屏蔽操作系统差异               |

**我的总结：**

JVM就是Java虚拟机，也就是java运行程序的地方。简单来说就是你先在Windows系统上写代码，然后这个代码会被编译成.class文件（也就是字节码），这个文件可以在任何装了JVM的系统上跑，不管是Windows还是Mac还是Linux。它可以帮我完成一些幕后工作，比如内存分配、垃圾回收、类怎么加载、不同系统环境之间的差异这些事。JVM就像个管家，你只管写逻辑，剩下的它帮你搞定。

## 什么是懒汉式？它为什么叫这个名字？

跟饿汉式反过来。就像一个"懒汉"，不到万不得已不动手。你不去找我（调用getInstance()），我就不创建实例，能拖就拖。

### 实现原理：

- 静态变量先不初始化：声明为null
- 首次调用时才创建：在getInstance()里判断，如果为null就new一个
- 私有构造器：防止外部new

代码长这样：

```java
public class Singleton {
    // 先不创建，等着
    private static Singleton instance = null;
  
    // 私有构造器
    private Singleton() {}
  
    // 调用时才判断要不要创建
    public static Singleton getInstance() {
        if (instance == null) {
            instance = new Singleton();  // 第一次才创建
        }
        return instance;
    }
}
```

核心就一句话：`if (instance == null)` 这个判断——第一次进来是空的，创建；之后每次进来都不是空了，直接返回已有的。

和饿汉式的对比：

|              | 饿汉式   | 懒汉式                  |
| ------------ | -------- | ----------------------- |
| 创建时机     | 类加载时 | 首次调用getInstance()时 |
| 是否延迟加载 | 否       | 是                      |
| 线程安全     | 天然安全 | **不安全**        |

### 我的总结：

懒汉式一开始不会初始化静态变量，会先声明成null，然后在要调用的getInstance()时先进行一个判断，如果变量是空，就将它实例化再返回它。好处是如果不用的话就一直不会被创建，可以节省一点空间。但是坏处是线程不会那么安全，容易出现同时创建多个实例的问题。

---

### 拓展：线程安全问题

**问题：两个线程同时调用getInstance()真的会出问题吗？这种情况多见吗？**

其实这个情况在Web开发中很常见。比如两个请求同时进来：

```
时间线：

T1时刻：请求A 调用 getInstance() → 进入if判断 → instance是null → 准备new
T1时刻：请求B 调用 getInstance() → 进入if判断 → instance是null → 准备new
                                                              ↑ 两个线程同时到这里

T2时刻：请求A new了一个实例（地址0x001）
T2时刻：请求B new了另一个实例（地址0x002）

结果：A拿到的是0x001，B拿到的是0x002，不是同一个对象了
```

更生活化的例子就是抢厕所：厕所门上没锁，小明和小红同时看了一眼都没人，结果两个人都进去了，一个厕所里两个人，尴尬了。

单线程环境下确实不会出这个问题，但现在基本都是Web应用、服务器程序，Tomcat每个请求就是一个线程，多线程是常态，所以这个风险是真实的。

## 什么是双重检查锁？为什么叫这个名字？

**"双重" = 两个if判断 + "检查" + "锁"**

就是检查了两次，而且加了把锁。

### 解决了什么问题？

懒汉式线程不安全 → 直接加synchronized又太慢（每次调用都要排队）→ **折中方案：只在第一次创建时加锁**

代码长这样：

```java
public class Singleton {
    // volatile关键字很重要！后面说为什么
    private static volatile Singleton instance = null;
  
    private Singleton() {}
  
    public static Singleton getInstance() {
        // 第一次检查：如果已经有实例了，直接返回（大部分情况走这里）
        if (instance == null) {          // ← 检查1
            // 加锁：保证只有一个线程能进去创建实例
            synchronized (Singleton.class) {
                // 第二次检查：防止两个线程同时通过第一次检查
                if (instance == null) {  // ← 检查2
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

流程是这样的：

```
getInstance()被调用
       ↓
   第一次if检查 ──── 不为null ──→ 直接返回（快！不用加锁）
       ↓ 为null
     加synchronized锁
       ↓
   第二次if检查 ──── 不为null ──→ 返回（别的线程已经创建了）
       ↓ 为null
      new实例
       ↓
     释放锁 → 返回
```

### 那个volatile是什么？

`private static volatile Singleton instance = null;`

volatile的作用是**禁止指令重排序**。因为 `instance = new Singleton()` 这一行其实分了3步：

```java
// 1. 分配内存空间
// 2. 调用构造器，初始化对象
// 3. 把内存地址赋值给instance
```

没有volatile的话，JVM可能优化成 1 → 3 → 2（指令重排序）。问题是执行完第3步后，instance已经不是null了，但对象还没初始化完！此时别的线程拿到的就是个半成品对象。有volatile就强制按顺序执行，不会乱。

### 我的总结：

双重检查锁就是在第一次判断的地方加一个锁和二次判断来防止多线程导致的安全问题，缺点就是实现有点麻烦（还多了一个volatile关键字）。

---

### 拓展：为什么必须要有第二次检查？

**问题：A创建完成后instance不是已经不为null了吗？B为什么还会再创建一个新的？**

关键就在于两个线程都通过了第一次检查之后、拿到锁之前这段空隙。

```
初始状态：instance = null

T1时刻：
  线程A：第一次 if (instance == null) → true → 准备抢锁
  线程B：第一次 if (instance == null) → true → 准备抢锁
                    ↑ 两个线程都通过了第一次检查！

T2时刻：
  线程A：抢到锁了
  线程B：没抢到，在门口等着

T3时刻：
  线程A：进入锁内，new了一个实例，instance = 0x001
  线程B：还在等着...

T4时刻：
  线程A：释放锁，走了
  线程B：终于抢到锁了！
```

线程B进入锁的那一刻，它只知道一件事——自己通过了第一次if检查，它不知道线程A在里面干了什么！

- **没有第二次检查**：B直接 `new Singleton()` → 又创建了一个实例 → 单例被破坏
- **有第二次检查**：B进来先看一眼 `instance == null?` → 发现不是null了 → 跳过，不new了

简单说就是：**第一次检查是"大概看看有没有"，第二次检查是"进屋之后再确认一遍"**。因为从大概看到真正进屋之间，别人可能已经把事办完了。

---

**问题：为什么双重检查锁一定要volatile关键字？前两种就不需要？还有顺序的问题，如果不调用构造器初始化对象，instance的值怎么会被赋呢？**

### 为什么前两种不需要volatile？

- **饿汉式不需要**：类加载时JVM已经帮你搞定一切（自动加锁 + 顺序保证），你写 `new Singleton()` 在静态变量里，JVM保证它完整初始化后才对外可用
- **懒汉式不需要**：懒汉式本身就线程不安全，加volatile也没用——问题是多线程重复创建，不是指令重排序。就像房子门都坏了（线程不安全），你修窗户（加volatile）有什么用？
- **双重检查锁需要volatile**：因为它**几乎安全了**（有锁），唯一剩下的漏洞就是指令重排序。所以volatile是补上最后一块拼图

```
饿汉式   → JVM全包了，不用管
懒汉式   → 本身就烂透了，修一个点没用
双重检查 → 就差这一个点，必须补上
```

### 对象都不存在怎么赋值？

你的直觉完全对——逻辑上肯定是先创建对象再赋值。但问题是实际执行顺序可能被JVM打乱：

```java
instance = new Singleton();

理想顺序：
  1. 分配一块内存空间（比如地址0x100）
  2. 在这块内存上调用构造器，初始化属性（把对象"装修"好")
  3. 把0x100赋给instance

JVM优化后的顺序（指令重排序）：
  1. 分配一块内存空间（0x100）
  3. 把0x100赋给instance              ← instance指向0x100了！
  2. 在这块内存上调用构造器           ← 还没装修...
```

问题就出在这：由于内存提前分配了，变量里面会提前有值，那if判断的时候就会拿到内存地址而不是初始化后的属性。

```
T1时刻：线程A执行到 new Singleton()
         分配内存0x100 → 赋给instance(0x100) → 构造器还没执行完...

T1时刻（同时）：线程B调用 getInstance()
         第一次if (instance == null) → false！（因为instance已经是0x100了）
         直接返回instance → 拿到了一个 **没初始化完的对象**

结果：B拿到的是个"毛坯房"，里面的属性都是默认值，还没装修呢
```

用生活例子理解：

```
正常流程：
  1. 盖房子（分配内存）
  2. 装修（构造器初始化）
  3. 发钥匙（赋值给instance）

JVM重排后：
  1. 盖房子
  3. 先发钥匙 ← 问题在这！
  2. 装修

别人拿到钥匙进去了，发现墙没刷、地板没铺...
```

所以volatile的作用就是告诉JVM：别乱改顺序，老老实实按123来。

### 我的总结：

对于volatile，饿汉式不需要是因为有JVM兜底，懒汉式是要了也没用，只有双重检查锁刚好需要。因为JVM认为的顺序是先分配空间、再把空间赋值给变量、最后才初始化属性，但是这样有一个问题——由于内存被提前分配了，导致变量里面会提前有值，if判断的时候就会拿到内存地址而不是初始化的属性。

## 什么是静态内部类？为什么叫这个名字？

因为它是用一个静态的内部类来持有单例实例。

### 核心思路：

利用JVM类加载机制——外部类加载时不会触发内部类加载，只有用到内部类时才加载。

代码长这样：

```java
public class Singleton {
    
    private Singleton() {}
    
    // 静态内部类，专门用来装实例
    private static class Holder {
        // 内部类被加载时才创建实例
        private static final Singleton INSTANCE = new Singleton();
    }
    
    public static Singleton getInstance() {
        return Holder.INSTANCE;  // 第一次调用时才会触发Holder类加载
    }
}
```

### 为什么能同时做到延迟加载 + 线程安全？

```
调用 Singleton.getInstance() 之前：
  外部类Singleton被加载了 → 但Holder类没被加载 → 实例没创建 ✓ 延迟加载

第一次调用 getInstance() 时：
  触发Holder类加载 → JVM自动加锁初始化 → 创建实例 → 返回 ✓ 线程安全

之后再次调用 getInstance() 时：
  Holder已经加载过了 → 直接返回已有的INSTANCE ✓ 高性能
```

简单说就是把"饿汉式的线程安全"和"懒汉式的延迟加载"结合在了一起。

和前几种方式的对比：

| | 饿汉式 | 懒汉式 | 双重检查锁 | **静态内部类** |
|---|---|---|---|---|
| 延迟加载 | 否 | 是 | 是 | **是** |
| 线程安全 | 是 | 否 | 是 | **是** |
| 性能（无锁） | 是 | 是 | 否 | **是** |
| 写法简单 | 是 | 是 | 否 | **是** |

### 流程图解：

```
程序启动，什么都没调用：

  ┌─────────────┐
  │ Singleton类  │ ← 被引用了（比如import），JVM加载它
  │  (外部类)    │   但此时不加载内部类！
  └──────┬──────┘
         │
         ↓ Holder没被用到，不会被加载
    ┌─────────┐
    │  Holder  │ ← 没人用它，不加载，INSTANCE不存在
    │ (内部类) │
    └─────────┘

         状态：还没创建任何实例 ✓ 延迟加载


第一次调用 getInstance() 时：

  ┌─────────────┐
  │ getInstance()│
  └──────┬──────┘
         ↓ 访问 Holder.INSTANCE
    ┌────┴──────┐
    │  触发Holder  │ ← JVM检测到Holder需要被使用
    │  类加载！   │
    └────┬──────┘
         ↓
  ┌────────────────────────────┐
  │     JVM自动加锁 + 初始化      │
  │                             │
  │   1. 分配内存给INSTANCE      │
  │   2. new Singleton()        │
  │   3. 赋值给INSTANCE          │
  │                             │
  └────────────┬───────────────┘
               ↓
         返回 INSTANCE ✓ 线程安全（JVM保证）


第二次（及以后）调用 getInstance() 时：

  ┌─────────────┐
  │ getInstance()│
  └──────┬──────┘
         ↓ 访问 Holder.INSTANCE
    ┌────┴──────┐
    │  Holder已加载 │ ← 不再加载了！直接用
    │  直接返回    │
    └────┬──────┘
         ↓
       返回同一个 INSTANCE ✓ 高性能（无锁、无判断）
```

一句话总结：第一次用才加载（延迟），加载时JVM自动加锁（线程安全），之后直接返回（高性能）。

缺点：无法传参初始化。

### 我的总结：

静态内部类就是在一开始的类里面嵌套一个静态类，然后把实例化的代码放到静态类里面。这样只要里面的类不运行，变量就不会实例化，资源也就不会浪费。相当于给饿汉式的核心代码又套了个静态类。但是这样变量也就只能在里面这个静态类里改，值基本就是写死了，写的是什么就是什么，外部没法改了。

### 拓展：为什么无法传参？

**问题：实例化的代码跟前面三种不是一样的吗？都是 `private static xxx instance = new xxx();` 为什么这个就不能传参？**

区别不在于那行代码本身，而在于它在哪里被执行、什么时候被执行。

```java
// 懒汉式：你在自己的方法里控制创建过程
public static Singleton getInstance(String name) {
    if (instance == null) {
        instance = new Singleton(name);  // ← 你可以传参数！
    }
    return instance;
}

// 静态内部类：创建过程被"锁死"在类加载里了
private static class Holder {
    private static final Singleton INSTANCE = new Singleton();  // ← 你碰不到这行！
}
```

懒汉式的new是在你的方法里写的，你想怎么写都行；静态内部类的new是写在静态变量声明里的，JVM类加载时自动跑，你没法干预。

举个传参的例子：

```java
// 假设这是一个数据库连接单例，需要传配置文件路径
public class DatabaseManager {
    private static DatabaseManager instance = null;
    private String configPath;
    
    // 私有构造器，接收参数
    private DatabaseManager(String path) {
        this.configPath = path;
    }
    
    // 传参创建实例
    public static DatabaseManager getInstance(String path) {
        if (instance == null) {
            instance = new DatabaseManager(path);  // ← 参数传进来了
        }
        return instance;
    }
}

// 使用时：
DatabaseManager db = DatabaseManager.getInstance("db-config.properties");
```

如果换成静态内部类：

```java
private static class Holder {
    private static final DatabaseManager INSTANCE = new DatabaseManager(???)  // 写不了
}
```

那行代码写完就定死了，运行时没法改。而且就算你在静态类里面改，改的还是写死的值——path本身也是个静态变量，只能写死。更麻烦的是，你想要从外面传参进去还会遇到顺序问题：你要先有参数才能创建实例，但实例创建时机你控制不了（Holder什么时候加载你说了不算）。所以结论就是：不是不能改那行代码，而是改了也没用，没地方让你从外部动态地把参数送进去。

## 什么是枚举单例？为什么叫这个名字？

因为它是用Java的**enum关键字**来实现单例的。

### 代码：

```java
public enum Singleton {
    
    INSTANCE;  // 就这一行，完事了
    
    // 你可以加方法
    public void doSomething() {
        System.out.println("我是单例");
    }
}
```

**就这？对，就这。**

使用方式：

```java
// 获取实例
Singleton s1 = Singleton.INSTANCE;
Singleton s2 = Singleton.INSTANCE;

System.out.println(s1 == s2);  // true，永远是同一个

s1.doSomething();
```

### 为什么它天然就是单例？

enum在JVM层面的本质是这样的：

```java
public final class Singleton extends Enum<Singleton> {
    public static final Singleton INSTANCE = new Singleton();  // JVM保证唯一
    
    private Singleton() {}  // 构造器是私有的！
}
```

Java规定：
- enum的构造器**自动私有**，你写不了public
- enum的每个值都是 `public static final` 的——全局唯一、不可变
- JVM保证enum值只会被创建一次

### 为什么说它能防反射和反序列化？

这是其他四种都做不到的，只有枚举可以。

**反射攻击（其他方式的漏洞）：**

```java
// 正常单例可以被反射破坏
Constructor<Singleton> constructor = Singleton.class.getDeclaredConstructor();
constructor.setAccessible(true);
Singleton reflectionInstance = constructor.newInstance();  // 哦豁，又创建了一个！
```

**枚举不怕：**

```java
// 构造器的newInstance()方法里有一行代码：
if (clazz.isEnum()) {
    throw new IllegalArgumentException("Cannot reflectively create enum objects");
}
// 直接抛异常！Java规范层面封死了
```

**反序列化攻击（其他方式的漏洞）：**

```java
// 把单例序列化成文件，再读回来 → 可能得到不同的对象
// 因为反序列化会调用构造器重新创建对象
```

**枚举不怕：**

```java
// Java规范规定：enum反序列化时直接返回已有的INSTANCE
// 不会重新创建对象
```

### 和饿汉式的对比：

| | 饿汉式 | 枚举 |
|---|---|---|
| 创建时机 | 类加载时 | 类加载时（一样） |
| 延迟加载 | 否 | 否（一样） |
| 线程安全 | 是 | 是（一样） |
| 防反射 | 不能 | **能** |
| 防反序列化 | 不能 | **能** |

### 缺点：
- 不支持延迟加载
- 不能继承其他类（已经继承了Enum类）

### 我的总结：

枚举类就是强化班的饿汉式单例模式。它属于Java独有的关键字，可以通过Java语言规范来进行防反射和防反序列化的工作。本质上就是饿汉式 + 安全保护罩。

### 拓展

**问题：enum这个关键字是JVM自带的吗？反射跟序列化是什么意思？它是不是跟饿汉式一样运行就实例化？**

### enum是Java语言自带的，不是JVM专属的

enum是 Java 5（2004年）新增的一个关键字，属于Java语言本身的一部分。就像class、int、if这些关键字一样，写在代码里的。

JVM不认识"enum"这个词，编译器会把它翻译成JVM能看懂的普通类：

```
你写的：public enum Singleton { INSTANCE; }
编译器翻译成：public final class Singleton extends Enum<Singleton>
              { public static final Singleton INSTANCE = new Singleton(); }
```

所以enum是**语法糖**——写起来简单好看，底层还是类。

### 什么是反射和序列化？

**反射（Reflection）：**

正常方式是你通过"门"进房间——`Singleton.getInstance()` 走正规入口。

反射就是你把墙拆了直接进来：

```java
Constructor<?> c = Singleton.class.getDeclaredConstructor();
c.setAccessible(true);  // 强制打开私有构造器！
Object s2 = c.newInstance();  // 绕过getInstance()，直接new了一个
```

简单说就是**反射能让你访问本来不该访问的东西**——私有构造器、私有变量都能被强行打开。

**序列化和反序列化：**

序列化就是把对象转成字节流，存到文件或网络传输，就像把一个人拍照存成照片。反序列化就是从字节流恢复成对象，就像从照片里把人变回来（但可能变出另一个人）。问题是反序列化可能创建了新对象，导致原来的单例被破坏。

### 跟饿汉式一样吗？

对，一样的。都是程序启动时就创建好了，不支持延迟加载。区别只是枚举更安全（防反射、防反序列化），其他都差不多。所以枚举本质上就是饿汉式的加强版——同样提前创建，但多了层保护罩。

## 如何防止反序列化破坏单例？

### 问题：如果单例类实现了Serializable接口，怎么防止反序列化破坏单例？

```java
public class Singleton implements Serializable {  // 实现了Serializable
    private static final Singleton INSTANCE = new Singleton();
    // ...
}

// 序列化 → 写到文件
Singleton s1 = Singleton.INSTANCE;           // 原来的实例
Singleton s2 = (Singleton) readFromFile();     // 从文件读出来的

s1 == s2  // false！！反序列化创建了新对象！
```

### 解决方法：添加 `readResolve()` 方法

```java
public class Singleton implements Serializable {
    
    private static final Singleton INSTANCE = new Singleton();
    
    private Singleton() {}
    
    public static Singleton getInstance() {
        return INSTANCE;
    }
    
    // 关键！这个方法会在反序列化时自动被调用
    private Object readResolve() {
        return getInstance();  // 返回已有的单例实例，而不是新创建的
    }
}
```

原理是这样的：

```
反序列化的正常流程：
  ObjectInputStream读取数据 → 调用构造器创建新对象 → 返回新对象 ❌

加了readResolve后的流程：
  ObjectInputStream读取数据 → 创建了一个临时对象（还没返回）
  → 发现类里有readResolve()方法 → 调用它
  → readResolve()返回了INSTANCE → 丢弃临时对象，用返回的替代 ✓
```

简单说就是：反序列化想偷偷建个新房？readResolve告诉你——别建了，直接住老房吧。

五种单例模式的防反序列化能力：

| | 防反序列化 |
|---|---|
| 饿汉式 | 需要手动加 `readResolve()` |
| 懒汉式 | 需要手动加 `readResolve()` |
| 双重检查锁 | 需要手动加 `readResolve()` |
| 静态内部类 | 需要手动加 `readResolve()` |
| **枚举** | **天然防，不用加** |

这也是推荐用枚举的原因之一——少操心。

### 拓展

**问题：Serializable这个接口名字是什么意思？是自带的还是自己起的？**

**Serializable 是 Java 自带的接口。**

名字的意思：Serial（序列） + ize（使...化） + able（能够...的）= "能够被序列化的"。就是告诉JVM这个类的对象可以被转成字节流。

它长这样：

```java
// Java自带的接口，在 java.io 包里
public interface Serializable {
    // 里面什么方法都没有！空的！它只是一个"标记"
}
```

所以 **Serializable 只是一个标记接口（Marker Interface）**——你只需要说"我能序列化"，不需要实现任何东西：

```java
// 打个标记就行，表示允许被序列化
public class Person implements Serializable {
    private String name;
}

// 没加标记的类不能序列化，强行序列化会报错 NotSerializableException
```

就像在快递盒子上贴个"可邮寄"的标签一样，贴了标签快递公司知道可以运输，没贴标签拒绝收件。但贴不贴标签不影响盒子本身的内容，它就是个标记。

### 我的总结：

想让另外4种方式应对反序列化就需要添加一个 `readResolve()` 方法，它会强制让系统不调用构造器去创建新对象，而是直接返回老对象。而 **Serializable 是 Java 自带的一个类似于标记的接口，里面什么都没有，只是告诉 JVM 这个类里的对象可以被转换成字节流（也就是可以序列化）**。

## 线程安全的两种保护方式

### 问题：关于保护线程安全，有哪些方式？

**方式一：靠JVM（你不用操心）**

通过Java的JVM来保护——运行类时顺便创建实例，利用JVM类加载机制自动加锁：

- **饿汉式**：类加载时创建
- **静态内部类**：内部类加载时创建
- **枚举**：类加载时创建

共同点：实例在类加载时创建 → JVM的类加载机制自带同步。你就写一行 `new xxx()`，剩下的加锁、保证顺序、只初始化一次全是JVM帮你干的。

**方式二：靠自己写（手动控制）**

通过限制锁 + volatile关键字的方式来保护：

- **双重检查锁**：synchronized + volatile + 双重if

因为创建时机是你控制的（在方法里），不是类加载时，所以JVM不管，需要自己手动加锁、手动防重排序、手动做双重检查。

### 两种方式对比：

| | 靠JVM | 靠自己 |
|---|---|---|
| 代表 | 饿汉式、静态内部类、枚举 | 双重检查锁 |
| 原理 | 类加载时JVM自动加锁 | 手动synchronized + volatile |
| 写法 | 简单 | 复杂 |
| 创建时机 | 类加载时 | 方法调用时（可延迟加载） |

简单说就是：要么让JVM帮你管（类加载机制），要么自己动手管（锁 + 关键字）。

# 策略模式

## 概览

策略模式是一种**行为型设计模式**，核心思想：**把一组算法封装成独立的类，让它们可以互相替换**。

## 三个角色

| 角色 | 是什么 | 作用 |
|------|--------|------|
| **策略接口** | 一个接口定义 | 规定统一的方法签名（比如`travel()`） |
| **具体策略** | 多个实现类 | 每个类实现接口的方法，提供不同算法 |
| **上下文** | 一个普通类 | 持有接口引用，负责调用策略 |

## 完整流程

### 1. 定义策略接口
```java
interface TravelStrategy {
    void travel();  // 就这一个方法签名
}
```

### 2. 编写具体策略类（实现接口）
```java
class CarStrategy implements TravelStrategy {
    public void travel() { System.out.println("开车去"); }
}

class BikeStrategy implements TravelStrategy {
    public void travel() { System.out.println("骑车去"); }
}
```

### 3. 编写上下文类（持有接口引用）
```java
class Navigator {
    private TravelStrategy strategy;
    
    public void setStrategy(TravelStrategy strategy) {
        this.strategy = strategy;
    }
    
    public void go() {
        strategy.travel();
    }
}
```

### 4. 客户端调用
```java
Navigator nav = new Navigator();

// 客户端决定用哪个策略
nav.setStrategy(new CarStrategy());
nav.go();  // 输出：开车去

// 换策略
nav.setStrategy(new BikeStrategy());
nav.go();  // 输出：骑车去
```

## 好处

- **开闭原则**：新增策略不改原有代码
- **避免if-else地狱**：不用写一堆`if (type == "car") else if ...`
- **策略可复用**：同一个策略类可以在多个地方使用

## 坏处

- 客户端需要知道所有策略，自己选哪个
- 策略类增多时，系统复杂度上升

## 我的总结

策略模式就是先定义一个大接口，在用好几个实现类来把这个大接口实现，然后通过一个上下文类听客户端的指令——客户端选择哪个策略，上下文就听他的来执行。核心就是"调用者只管调，执行者只管做"。

---

## 拓展

### 代码逐词解析

**问题：具体策略类和上下文类的代码每个单词是什么意思？**

### 具体策略类

```java
class ConcreteStrategyA implements Strategy {
    void doSomething() { System.out.println("策略A的算法"); }
}
```

| 单词 | 含义 |
|------|------|
| **class** | 定义一个新类 |
| **ConcreteStrategyA** | 类名：Concrete=具体的，Strategy=策略，A=第一个 |
| **implements** | 实现某个接口 |
| **Strategy** | 要实现的接口名 |
| **void** | 方法不返回任何值 |
| **doSomething** | 方法名：做某事 |
| **()** | 不接收参数 |
| **System.out.println(...)** | 在屏幕上打印一行文字 |

整行意思：创建一个叫ConcreteStrategyA的类，它实现了Strategy接口，里面有个doSomething方法执行时打印文字。

---

### 上下文类的方法

```java
// 设置策略
public void setStrategy(Strategy strategy) {
    this.strategy = strategy;
}

// 执行策略
public void execute() {
    strategy.doSomething();
}
```

| 单词 | 含义 |
|------|------|
| **public** | 公开的，谁都能调用 |
| **setStrategy** | 方法名：设置策略 |
| **(Strategy strategy)** | 参数：接收一个Strategy类型的对象，变量名叫strategy |
| **this** | 当前对象自己（Navigator） |
| **this.strategy** | Navigator自己的成员变量 |
| **=** | 赋值，右边给左边 |
| **strategy**（等号右边） | 外面传进来的参数 |
| **execute** | 方法名：执行 |
| **strategy.doSomething()** | 让保存的策略对象去调用它的方法 |

**setStrategy的意思**：外面传进来一个策略对象，存到我的成员变量里。
**execute的意思**：让我存的那个策略对象去干活。

用老板员工打比方：上下文就是老板，`setStrategy`是你给老板派个员工（记下来），`execute`是老板让那个员工去干活。

---

### 为什么需要策略模式？用在什么场景？

**核心原因：避免代码里写一堆 if-else / switch-case**

没有策略模式时：
```java
public void pay(String type) {
    if (type.equals("alipay")) {
        System.out.println("支付宝支付");
    } else if (type.equals("wechat")) {
        System.out.println("微信支付");
    } else if (type.equals("card")) {
        System.out.println("银行卡支付");
    }
    // 每加一种支付方式，就要加一个else if...
}
```

问题：每新增一种方式就要改原代码、if-else越写越长、各种逻辑耦合在一起。

用了策略模式，新增支付方式只需要新建一个类，原来的代码不用动。

**常见场景：**

| 场景 | 例子 |
|------|------|
| 支付方式 | 支付宝、微信、银行卡、现金 |
| 出行方式 | 开车、骑车、走路、坐地铁 |
| 排序算法 | 冒泡排序、快速排序、归并排序 |
| 折扣促销 | 打8折、满减、买一送一、无折扣 |

**判断标准：同一个事情有多种做法可以互相替换 → 就可以用策略模式。只要在写if-else判断"用哪种方式做同一件事"的时候，就该考虑策略模式了。**

### 重要理解

不是一个类里写好几个pay()方法，而是**好几个类，每个类里有一个pay()方法**：

```java
// 一个接口，定义一个pay()方法
interface PayStrategy { void pay(); }

// 好几个类，每个类的pay()内容不同
class AlipayStrategy implements PayStrategy {
    public void pay() { System.out.println("支付宝支付"); }
}

class WechatStrategy implements PayStrategy {
    public void pay() { System.out.println("微信支付"); }
}
```

总结：一个接口里面有一个pay()方法，然后弄好几个类，每个类实现pay()的内容不一样。

### 策略模式的缺点

**问题：策略模式有什么缺点？感觉是比较麻烦？**

主要缺点有两个：

**1. 客户端得自己选策略（选择逻辑外移）：**

```java
// 你得知道有哪些策略，还得自己判断用哪个
if (type.equals("alipay")) {
    cashier.setStrategy(new AlipayStrategy());
} else if (type.equals("wechat")) {
    cashier.setStrategy(new WechatStrategy());
}
```

说好的避免if-else，结果if-else从上下文里转移到了客户端里。只是换了个地方写而已。

**2. 类越来越多（类爆炸）：**

每种策略就是一个类：

```
AlipayStrategy.java      ← 1个文件
WechatStrategy.java      ← 1个文件
CardStrategy.java        ← 1个文件
CashStrategy.java        ← 1个文件
ApplePayStrategy.java    → 又加一种支付？新建一个类
...
```

策略一多，项目里全是小类文件。

**和模板方法模式缺点的对比：**

| | 策略模式缺点 | 模板方法模式缺点 |
|---|---|---|
| 核心问题 | **类爆炸** + **选择逻辑外移** | **继承耦合严重** |
| 换来的好处 | 灵活、可运行时切换 | 结构清晰、步骤固定 |

所以感觉麻烦是对的——**策略模式把复杂度分散到了多个类里，每个类简单了，但整体类数量多了。**这也是为什么实际开发中常和工厂模式搭配使用，让工厂帮你选策略，客户端就不用写那一堆if-else了。

# 模板方法模式

## 概览

模板方法模式是一种**行为型设计模式**，核心思想：**在父类里定义一个算法的骨架（模板），把一些步骤延迟到子类去实现**。

## 为什么叫"模板"？

就像**PPT模板**——格式和结构都定好了，你只需要填内容就行：

```
PPT模板：
  [标题] ← 你填
  [正文] ← 你填
  [结尾] ← 你填
  格式、字体、布局 → 已经定死不动

代码里的模板：
  步骤1：做某事 ← 子类决定怎么做
  步骤2：做某事 ← 子类决定怎么做
  步骤3：做某事 ← 子类决定怎么做
  执行顺序 → 父类已经定死不动
```

父类的方法就是**模板**，子类往里面**填内容**，所以叫"模板方法"。

## 角色（就两个）

| 角色 | 是什么 | 作用 |
|------|--------|------|
| **抽象类（AbstractClass）** | 定义了"模板"的类 | 定义模板方法（算法骨架），声明抽象步骤 |
| **具体子类（ConcreteClass）** | 继承抽象类的类 | 实现抽象步骤的具体逻辑 |

比策略模式少一个角色——因为**没有"上下文"这个中间人**，子类直接干活。

## 和策略模式的区别

| | 策略模式 | 模板方法模式 |
|---|---|---|
| 核心 | **组合**（用接口） | **继承**（用抽象类） |
| 关系 | 上下文"有"一个策略 | 子类"是"一种模板 |
| 变化的东西 | 整个算法替换 | 算法步骤不变，每步的实现不同 |
| 比喻 | 换不同的工人干同一份活 | 同一份流程，不同的人做不同的环节 |

## 具体代码

以"做菜流程"为例：

```java
// 1. 抽象类：定义模板（做菜流程）
abstract class CookRecipe {
    
    // 模板方法：定死执行顺序，final不让子类改
    public final void cook() {
        prepareFood();   // 步骤1：备菜
        addSeasoning();  // 步骤2：加调料（子类决定）
        cookDish();      // 步骤3：烹饪（子类决定）
        serve();         // 步骤4：上菜
    }
    
    // 固定的步骤，子类不用改
    void prepareFood() { System.out.println("洗菜、切菜"); }
    void serve() { System.out.println("端上桌"); }
    
    // 抽象步骤：子类必须自己实现
    abstract void addSeasoning();
    abstract void cookDish();
}

// 2. 具体子类：填内容
class SweetSourPork extends CookRecipe {
    void addSeasoning() { System.out.println("加糖、加醋"); }     // 糖醋排骨的做法
    void cookDish() { System.out.println("红烧收汁"); }
}

class MapoTofu extends CookRecipe {
    void addSeasoning() { System.out.println("加豆瓣酱、花椒"); }  // 麻婆豆腐的做法
    void cookDish() { System.out.println("翻炒勾芡"); }
}
```

## 调用

```java
CookRecipe dish1 = new SweetSourPork();   // 父类类型引用子类对象
dish1.cook();  // 洗菜切菜 → 加糖醋 → 红烧收汁 → 端上桌

CookRecipe dish2 = new MapoTofu();       // 父类类型引用子类对象
dish2.cook();  // 洗菜切菜 → 加豆瓣酱花椒 → 翻炒勾芡 → 端上桌
```

## 流程图示

```
调用 cook()
    ↓
prepareFood() ← 固定的，大家都一样
    ↓
addSeasoning() ← 子类自己实现（糖醋？麻辣？）
    ↓
cookDish() ← 子类自己实现（红烧？翻炒？）
    ↓
serve() ← 固定的，大家都一样
```

**父类管"先做什么后做什么"，子类管"每一步具体怎么做"。**

## 拓展

### 为什么模板方法要用final修饰？

**问题：模板方法为什么要用final修饰？不用final会怎么样？**

final的作用是**防止子类重写模板方法，把执行顺序搞乱**。

不用final的话，子类可以这样干：

```java
class BadCook extends CookRecipe {
    // 子类重写了模板方法！顺序全变了！
    @Override
    public void cook() {
        serve();         // 先上菜？？？
        addSeasoning();  // 再加调料？？？
        prepareFood();   // 最后才备菜？？？？
        cookDish();      // 这都什么鬼
    }
}
```

用了`final`，子类想改顺序直接报错，编译都过不了。

**一句话总结：final就是给模板方法上锁——骨架（顺序）归父类管，子类只能填内容，不能动结构。就像考试答题卡，格式和题号是定死的（final），你只能在括号里填ABCD。**

### 模板方法和步骤方法的关系

cook()里面的那些方法算是子方法吗？cook算它们的父方法？

更准确的说法是**调用关系**：

```
cook()（模板方法/主方法）
  ├── 调用 → prepareFood()
  ├── 调用 → addSeasoning()
  ├── 调用 → cookDish()
  └── 调用 → serve()
```

Java里的标准叫法：

| 叫法 | 指的是 | 例子 |
|------|--------|------|
| **模板方法** | 那个定死顺序的方法 | `cook()` |
| **步骤方法** | 被模板方法调用的那些步骤 | `prepareFood()`、`addSeasoning()`... |

**模板方法管顺序，步骤方法管内容。**

### 钩子方法

之前说的步骤方法分两种（具体方法、抽象方法），其实还有第三种——**钩子方法**：

| 类型 | 关键字 | 子类要不要实现 | 例子 |
|------|--------|-------------|------|
| **具体方法** | 普通 | 不用，直接继承 | `prepareFood()` |
| **抽象方法** | abstract | **必须**实现 | `addSeasoning()` |
| **钩子方法** | 普通 + 默认值 | **可选**，想用就重写 | `wantSpice()` |

**什么是钩子方法？**

钩子方法就是"可选的步骤"——子类可以选择实现，也可以选择不实现。一般默认返回false（不执行），子类重写后返回true就会触发对应逻辑。

```java
abstract class CookRecipe {
    
    public final void cook() {
        prepareFood();
        addSeasoning();
        if (wantSpice()) {     // ← 钩子方法！决定要不要加辣
            addSpice();        // 加辣椒
        }
        cookDish();
        serve();
    }
    
    // 钩子方法：默认返回false（不加辣）
    protected boolean wantSpice() {
        return false;
    }
    
    void addSpice() { System.out.println("加辣椒"); }
}

// 子类1：不加辣（不重写钩子，用默认的false）
class SweetSourPork extends CookRecipe {
    // 没有重写wantSpice() → 默认false → 不加辣
}

// 子类2：加辣（重写钩子）
class MapoTofu extends CookRecipe {
    @Override
    protected boolean wantSpice() {
        return true;   // 我要加辣！
    }
}
```

调用效果：

```java
new SweetSourPork().cook();  // 洗菜→调料→[跳过加辣]→烹饪→上桌
new MapoTofu().cook();      // 洗菜→调料→[加辣椒]→烹饪→上桌
```

**为什么叫"钩子"？**

就像墙上挂了一个挂钩：你可以往上面挂东西（重写它）触发额外逻辑，也可以不挂（不重写）跳过。它是给子类一个控制权的小开关，让模板方法更灵活。

**一句话总结：钩子方法就是子类的一个特殊选项开关，默认返回false（不执行），子类可以重写成true来触发父类模板方法里的某一步骤。**

### @Override 注解

**问题：@Override 是什么？第一次见。**

`@Override` 是 Java 的一个**注解（Annotation）**，意思是"我要重写父类的方法"。

```java
class MapoTofu extends CookRecipe {
    @Override                          // ← 告诉编译器：我要重写 wantSpice()
    protected boolean wantSpice() {
        return true;
    }
}
```

**为什么要加？**

| | 有 `@Override` | 没有 |
|---|---|---|
| 写错方法名 | **编译报错**，立刻发现 | 不报错，静默失败（方法没生效你都不知道） |
| 可读性 | 一眼看出是重写的方法 | 得自己猜 |

```java// 加了 @Override，写错直接报错@Overrideprotected boolean wantSpcie() {   // 编译器：父类里没有这个方法！报错！
    return true;}// 不加 @Override，写错了也不报错（静默失败）protected boolean wantSpcie() {   // 拼写错误！少了个i，但编译通过，方法没生效
    return true;}
```

**就是给方法贴个标签："我是重写的，不是新建的"。**

### 应用场景与缺点

**判断标准：流程顺序固定，但每步的实现不同。只要遇到这种情况就可以考虑用模板方法模式。**

常见场景：

| 场景 | 例子 |
|------|------|
| 做菜/做饭 | 备菜→加调料→烹饪→上菜，步骤一样做法不同 |
| 数据库连接 | 加载驱动→建立连接→执行SQL→关闭连接 |
| Http请求处理 | 解析请求→验证权限→处理业务→返回响应 |
| 游戏角色行为 | 初始化→攻击→移动→死亡（不同角色实现不同） |
| Servlet生命周期 | init() → service() → destroy（经典应用） |
| 框架初始化 | Spring、MyBatis等框架的启动流程 |

**缺点：父类和子类绑定严重，这是最大的问题。**

| 缺点 | 说明 |
|------|------|
| 继承耦合 | 子类依赖父类，父类改了子类可能全受影响 |
| 增加系统复杂度 | 类数量增多，抽象类+多个子类 |
| 不够灵活 | 父类定了骨架就不好改，想调整顺序得动父类本身 |

和策略模式的对比：

| | 模板方法模式 | 策略模式 |
|---|---|---|
| 耦合方式 | **继承耦合**（强绑定） | **组合耦合**（松绑定） |
| 改算法 | 得改父类或新增子类 | 新建一个策略类就行 |
| 运行时切换 | 不行，编译时就定死了 | 可以，随时换策略对象 |

所以：如果步骤固定用模板方法，如果需要灵活切换用策略模式。一句话总结——**模板方法用继承换来了结构清晰，但代价是父子绑定紧密。**

### 面试题：已继承其他类还想用模板方法怎么办？

Java是**单继承**的，一个类只能有一个父类。如果已经继承了别的类（比如 `extends HttpServlet`），就不能再继承模板方法的抽象类了。

#### 解法一：改用策略模式（首选推荐）

**最推荐的解法。** 把"变化的部分"抽成策略接口，用组合代替继承。

```java
// 策略接口：把变化的步骤抽象出来
interface CookStrategy {
    void addSeasonings();
    void cook();
}

// 模板类（不再是抽象类了，就是普通类）
class RecipeTemplate {
    private CookStrategy strategy;

    public RecipeTemplate(CookStrategy strategy) {
        this.strategy = strategy;
    }

    // 模板方法
    public final void cook() {
        prepareIngredients();
        strategy.addSeasonings();  // 委托给策略
        strategy.cook();          // 委托给策略
        serve();
    }

    private void prepareIngredients() { System.out.println("洗菜切菜"); }
    private void serve() { System.out.println("上菜！"); }
}

// 具体策略
class KungPaoStrategy implements CookStrategy {
    public void addSeasonings() { System.out.println("加花椒、辣椒"); }
    public void cook() { System.out.println("大火爆炒"); }
}

// 使用：随便继承什么类都行
public class MyServlet extends HttpServlet {  // 已经占用了继承
    public void doGet() {
        RecipeTemplate template = new RecipeTemplate(new KungPaoStrategy());
        template.cook();
    }
}
```

**核心思路：继承不够，组合来凑。**

#### 解法二：父类去继承抽象类

让**已有的父类**去继承模板抽象类，子类自然就有了。

```java
abstract class AbstractRecipe {
    public final void cook() { /* 模板方法 */ }
}

// 已有的父类继承抽象类
public class BaseService extends AbstractRecipe {
    // 公共业务逻辑...
}

// 子类正常继承
public class UserService extends BaseService {
    @Override
    protected void cookStep() { /* 自己的实现 */ }
}
```

**局限：必须能控制父类才行。** 比如 `HttpServlet` 是 JDK 写的类，你碰不到它的源码，没法给它加 `extends AbstractRecipe`。只有你自己写的父类才能这么改。

#### 解法三：接口 + 默认方法（Java 8+）

用接口的 `default` 方法模拟模板方法。

```java
interface Recipe {
    default void cook() {
        prepareIngredients();     // 调用自己的default方法
        addSeasonings();         // 调用子类实现
        doCook();                // 调用子类实现
        serve();
    }

    default void prepareIngredients() { System.out.println("备菜"); }
    default void serve() { System.out.println("上菜"); }

    // 子类必须实现的步骤（抽象方法）
    void addSeasonings();
    void doCook();
}

// 实现接口就行，不影响继承
public class MyServlet extends HttpServlet implements Recipe {
    @Override
    public void addSeasonings() { System.out.println("加调料"); }
    @Override
    public void doCook() { System.out.println("烹饪"); }
}
```

**优点：接口可以多实现，完全不影响单继承限制。**
**缺点：接口不能有实例字段（状态），只能有方法。**

#### 三种解法对比

| 解法 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **策略模式** | 最灵活，运行时可切换 | 类稍微多一点 | **首选推荐** |
| **父类继承** | 子类不用改 | 必须能控制父类（JDK/框架的类不行） | 自己写的父类 |
| **接口+默认方法** | 不影响继承 | 不能有状态字段 | Java 8+环境 |

**面试回答要点：** "Java 单继承导致无法同时继承模板抽象类。首选策略模式用组合解耦；如果父类可控可以让父类继承；Java 8 还可以用接口 default 方法绕过。"

### 组合 + 回调（函数式接口 + Lambda 替代继承）

#### 为什么需要这种方式？

传统模板方法每写一个实现都要新建一个子类：

```java
class KungPaoChicken extends Recipe { ... }   // 一个文件
class MapoTofu extends Recipe { ... }          // 又一个文件
```

**就几行代码也要一个新类，太啰嗦了。** 能不能不写子类，直接把"那几行代码"传给模板？

#### 核心思路：把抽象方法变成"回调参数"

以前：变化的部分是 **抽象方法** → 子类去重写
现在：变化的部分是 **回调函数（Lambda）** → 构造时传进来

```java
import java.util.function.Consumer;

// 模板类——不需要子类了！
class CookTemplate {

    // Consumer 是 Java 自带的函数式接口，含义是"接收一个参数、处理一下、不返回值"
    private final Consumer<CookContext> addSeasonings;
    private final Consumer<CookContext> doCook;

    public CookTemplate(Consumer<CookContext> addSeasonings,
                        Consumer<CookContext> doCook) {
        this.addSeasonings = addSeasonings;   // 把传入的 Lambda 存到字段里
        this.doCook = doCook;
    }

    // 模板方法——流程还是固定的
    public final void cook() {
        CookContext ctx = new CookContext();
        System.out.println("1. 备菜");
        System.out.println("2. → 调用回调：加调料");
        addSeasonings.accept(ctx);            // ← 执行你传入的 Lambda
        System.out.println("4. → 调用回调：烹饪");
        doCook.accept(ctx);                   // ← 执行你传入的 Lambda
        System.out.println("6. 上菜！");
    }
}

// 上下文对象：在回调和模板之间传递数据
class CookContext {
    private String dishName;
    private int spicyLevel;
    // getter/setter...
}
```

#### 关键概念解释

**什么是回调（Callback）？**

> 回调 = 你把一段代码交给我，我在合适的时机帮你执行它。

就像你去餐厅：
1. 你告诉服务员："好了叫我" → 给了通知方式（**注册回调**）
2. 餐厅做好饭后按你的方式通知你（**执行回调**）

代码中也是一样：

```java
// Lambda 这行不会立刻执行，而是被"保存"起来
new CookTemplate(
    ctx -> System.out.println("加调料"),  // 注册回调：把代码交给模板
    ctx -> System.out.println("大火炒")
);
// 等调用 cook() 时，模板才会在第2步和第4步执行这两段代码（触发回调）
```

#### 使用方式——Lambda 一行搞定

```java
public class Main {
    public static void main(String[] args) {
        // 宫保鸡丁 —— 不需要新建子类！Lambda 直接传入
        CookTemplate kungPao = new CookTemplate(
            ctx -> System.out.println("   加花椒、干辣椒、花生米"),
            ctx -> System.out.println("   大火爆炒3分钟")
        );
        kungPao.cook();

        System.out.println("---");

        // 麻婆豆腐 —— 同样不需要新类
        CookTemplate mapoTofu = new CookTemplate(
            ctx -> System.out.println("   加豆瓣酱、花椒粉"),
            ctx -> System.out.println("   小火慢炖5分钟")
        );
        mapoTofu.cook();

        System.out.println("---");

        // 甚至可以做一次性的菜，连变量名都不用起
        new CookTemplate(
            ctx -> System.out.println("   加盐"),
            ctx -> System.out.println("   白煮")
        ).cook();   // 直接调用，用完就扔
    }
}
```

输出：
```
1. 备菜
```

**参数是怎么传进去的？**

```java
CookTemplate kungPao = new CookTemplate(
    ctx -> System.out.println("加花椒..."),     // 参数1 → addSeasonings 字段
    ctx -> System.out.println("大火爆炒3分钟")    // 参数2 → doCook 字段
);
// Lambda 本身就是一个"值"，类型是 Consumer<CookContext>
// 和 String name = "张三" 一样，只不过存的是一段代码而不是文字
```

#### 四个核心概念

| 概念 | 解释 | 类比 |
|------|------|------|
| **函数式接口** | 只有一个抽象方法的接口，可以用 Lambda 表示 | `Consumer<T>` = 接收T，处理，不返回 |
| **回调 Callback** | 把代码作为参数传入，在合适时机执行 | 餐厅做好了叫你 |
| **组合 Has-a** | 模板类"持有"回调函数，而不是"是"某个子类 | 有一个 vs 是一个 |
| **Lambda** | `(参数) -> {代码}` 匿名表示一段逻辑 | 代替写一整个类 |

#### 与传统写法对比

| | 继承式模板方法 | 组合+回调式 |
|---|---|---|
| 写法 | 每个子类一个新类 | Lambda 传入即可 |
| 类数量 | 多（每个实现都要新类） | 少（只要一个模板类） |
| 运行时切换 | 不能 | 随时创建新的模板实例 |

#### 原理总结

> 利用回调函数方法，把原本需要抽象重写的子类改成一行 Lambda 传回给模板类。main 方法中只需要用一开始的模板类就可以了。

## 我的总结

模板方法模式也是一种行为型设计模式，它的核心思想是创建好一个父类模板，然后里面的方法顺序是固定的，内容可以我们自己来写。例如上面的例子，菜谱就是抽象父类，cook()方法就是做菜的模板，里面有固定的步骤和抽象的步骤，固定的步骤是写在父类里的，调用的时候会自动实现；然后写两个子类对应两个菜谱里的调料和烹饪过程；最后调用的时候只需要注意用父类类型引用正确的子类对象就可以了。
