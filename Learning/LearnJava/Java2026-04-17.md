# 工厂模式

## 概览

### 它解决什么问题？

不用工厂模式的代码：

```java
public class Main {
    public static void main(String[] args) {
        String type = "宫保鸡丁";
        Dish dish;

        // 客户端要自己决定怎么创建对象——这就是问题所在
        if (type.equals("宫保鸡丁")) {
            dish = new KungPaoChicken();     // 客户端要知道具体的类名
        } else if (type.equals("麻婆豆腐")) {
            dish = new MapoTofu();           // 又一个具体类名
        } else if (type.equals("番茄炒蛋")) {
            dish = new TomatoEggs();         // 再一个...
        }

        dish.cook();
    }
}
```

**问题：**

1. **客户端暴露了细节** —— `new KungPaoChicken()` 这种具体类名到处都是
2. **加新产品要改客户端** —— 每加一道菜就要加一个 `else if`
3. **创建逻辑散落各处** —— 如果多处都要创建菜，这段 if-else 要复制粘贴好几遍

### 工厂模式的核心思想

> **把"创建对象"这件事，交给一个专门的类（工厂）来管，客户端只管"要"，不关心"怎么造"。**

```java
// 对比：用了工厂之后
public class Main {
    public static void main(String[] args) {
        Dish dish = DishFactory.create("宫保鸡丁");  // 只传名字就行！
        dish.cook();
        // 不需要知道是 new KungPaoChicken() 还是别的
    }
}
```

### 已学设计模式对比

| 模式               | 核心思想                         | 一句话                     |
| ------------------ | -------------------------------- | -------------------------- |
| 单例模式           | 整个程序只有一个实例             | 全局只有一个               |
| 策略模式           | 用组合代替继承，算法可切换       | 算法封装好随时换           |
| 模板方法模式       | 父类定流程骨架，子类填细节       | 流程固定，步骤不同         |
| **工厂模式** | **把创建对象的活交给工厂** | **你只管要，我来造** |

> **注意："工厂模式"不是一个模式，而是一族模式。**

| 名称               | 复杂度 | 说明                      |
| ------------------ | ------ | ------------------------- |
| **简单工厂** | ⭐     | 一个工厂类 + if-else 创建 |
| **工厂方法** | ⭐⭐   | 每个产品一个工厂          |
| **抽象工厂** | ⭐⭐⭐ | 创建一组相关的产品        |

## 一、简单工厂模式（Simple Factory）

### 角色和结构

```
客户端(Main)  ──调用──→  工厂类(DishFactory)  ──创建──→  产品(Dish接口)
                        (负责造东西)                  ↑
                                                    |
                                              具体产品(各个菜)
```

三个角色：

| 角色                      | 职责                       | 类比                         |
| ------------------------- | -------------------------- | ---------------------------- |
| **产品接口/抽象类** | 定义产品的规范             | "菜"的统一标准（能做、能吃） |
| **具体产品**        | 实现具体功能               | 宫保鸡丁、麻婆豆腐...        |
| **工厂类**          | 根据传入参数创建对应的产品 | 厨房，你点啥我做啥           |

### 代码

```java
// ① 产品接口
interface Dish {
    void cook();
}

// ② 具体产品
class KungPaoChicken implements Dish {
    public void cook() { System.out.println("做宫保鸡丁：加花椒、大火炒"); }
}
class MapoTofu implements Dish {
    public void cook() { System.out.println("做麻婆豆腐：加豆瓣酱、小火炖"); }
}
class TomatoEggs implements Dish {
    public void cook() { System.out.println("做番茄炒蛋：先炒蛋再放番茄"); }
}

// ③ 工厂类 —— 核心！
class DishFactory {

    // 静态方法，根据名字创建对应的对象
    public static Dish create(String type) {
        switch (type) {
            case "宫保鸡丁":
                return new KungPaoChicken();
            case "麻婆豆腐":
                return new MapoTofu();
            case "番茄炒蛋":
                return new TomatoEggs();
            default:
                throw new IllegalArgumentException("没有这道菜：" + type);
        }
    }
}
```

### 使用

```java
public class Main {
    public static void main(String[] args) {
        // 客户端只传名字，完全不碰具体的 new xxx()
        Dish dish1 = DishFactory.create("宫保鸡丁");
        dish1.cook();

        Dish dish2 = DishFactory.create("麻婆豆腐");
        dish2.cook();
    }
}
```

### 不用工厂 vs 用了工厂

```java
// ❌ 不用工厂：创建逻辑写在客户端
Dish dish;
if (type.equals("宫保鸡丁")) {
    dish = new KungPaoChicken();      // 客户端要知道具体类名
} else if (type.equals("麻婆豆腐")) {
    dish = new MapoTofu();            // 又是具体类名
}
dish.cook();


// ✅ 用了工厂：创建逻辑交给工厂
Dish dish = DishFactory.create("宫保鸡丁");   // 只传字符串，不知道内部怎么创建的
dish.cook();
```

### 优点和缺点

**优点：**

| 优点               | 说明                                |
| ------------------ | ----------------------------------- |
| **解耦**     | 客户端不依赖具体类名，只知道字符串  |
| **集中管理** | 创建代码全在工厂里，改一处就行      |
| **使用简单** | 一行 `Factory.create("xxx")` 搞定 |

**缺点：**

| 缺点                     | 说明                                |
| ------------------------ | ----------------------------------- |
| **违反开闭原则**   | 加新产品要改工厂类的 switch/if-else |
| **工厂类职责太重** | 所有产品的创建都堆在一个类里        |
| **不易扩展**       | 产品一多，switch 会变得很长         |

> **开闭原则 = 对扩展开放，对修改关闭。** 简单工厂加新菜时得改工厂源码，所以违反了这个原则。

---

### 与策略模式的对比

简单工厂和策略模式结构很像，但目的不同：

|                    | **简单工厂**           | **策略模式**           |
| ------------------ | ---------------------------- | ---------------------------- |
| **核心目的** | **造东西**（创建对象） | **换行为**（使用对象） |
| 关注点             | `new` 的过程               | 怎么用这个对象               |
| 创建完后呢？       | 交给你了，工厂没用了         | 对象还在容器里，随时能换     |
| 能运行时切换吗？   | ❌ 不能（一次性的）          | ✅ 能                        |

> **如果只是要选一个实现来用，策略模式比简单工厂更好。**
>
> 打个比方：**简单工厂 = 工厂车间**（下订单造出来给你）；**策略模式 = 手机壳**（套上不喜欢了随时换）。

## 二、工厂方法模式（Factory Method）

### 为什么需要它？—— 名字怎么来的？

回顾简单工厂的问题：

```java
// 简单工厂：所有产品都在一个类里造
class DishFactory {
    public static Dish create(String type) {
        switch (type) {          // ← 问题就在这！
            case "宫保鸡丁": return new KungPaoChicken();
            case "麻婆豆腐": return new MapoTofu();
            // 加新菜？改这里！违反开闭原则
        }
    }
}
```

简单工厂是一个**大杂烩工厂**，什么都在里面造。加新产品就得**修改原来的代码**。

解决思路：**每个产品配一个专属工厂！**

```
简单工厂（一个大工厂干所有事）：
┌──────────────────────┐
│   DishFactory         │  ← 一个工厂类
│   - create("宫保鸡丁") │
│   - create("麻婆豆腐") │
│   - create("番茄炒蛋") │
└──────────────────────┘

工厂方法模式（每种产品一个专门工厂）：
┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
│ KungPaoFactory   │  │ MapoTofuFactory  │  │ TomatoFactory   │
│ 创建宫保鸡丁      │  │ 创建麻婆豆腐      │  │ 创建番茄炒蛋     │
└─────────────────┘  └─────────────────┘  └─────────────────┘
```

所以叫 **"工厂方法"** = **把工厂的创建逻辑变成一个方法**，让子工厂各自实现。

### 结构 —— 四个角色

```
                    ┌──────────────┐
                    │ Dish 接口     │  ← 产品接口
                    └──────┬───────┘
                           │ 实现
              ┌────────────┼────────────┐
              ▼            ▼            ▼
       ┌──────────┐ ┌──────────┐ ┌──────────┐
       │ 宫保鸡丁   │ │ 麻婆豆腐   │ │ 番茄炒蛋   │  ← 具体产品
       └──────────┘ └──────────┘ └──────────┘

                    ┌──────────────┐
                    │ DishFactory   │  ← 抽象工厂接口（定义了一个方法）
                    │ +create()     │
                    └──────┬───────┘
                           │ 实现
              ┌────────────┼────────────┐
              ▼            ▼            ▼
       ┌──────────┐ ┌──────────┐ ┌──────────┐
       │ 宫保工厂   │ │ 麻婆工厂   │ │ 番茄工厂   │  ← 具体工厂（每个造一种产品）
       └──────────┘ └──────────┘ └──────────┘
```

### 代码

```java
// ① 产品接口（和之前一样）
interface Dish {
    void cook();
}

// ② 具体产品（和之前一样）
class KungPaoChicken implements Dish {
    public void cook() { System.out.println("做宫保鸡丁"); }
}
class MapoTofu implements Dish {
    public void cook() { System.out.println("做麻婆豆腐"); }
}

// ③ 抽象工厂接口 —— 核心新增！定义了"创建方法"
interface DishFactory {
    Dish create();   // 这就是"工厂方法"，每个具体工厂都要实现它
}

// ④ 具体工厂 —— 每个工厂只负责一种产品
class KungPaoFactory implements DishFactory {
    public Dish create() { return new KungPaoChicken(); }  // 只造宫保鸡丁
}

class MapoTofuFactory implements DishFactory {
    public Dish create() { return new MapoTofu(); }        // 只造麻婆豆腐
}
```

### 使用

```java
public class Main {
    public static void main(String[] args) {
        // 想要宫保鸡丁？先拿宫保工厂
        DishFactory factory = new KungPaoFactory();
        Dish dish = factory.create();   // 工厂方法！
        dish.cook();

        // 想要麻婆豆腐？换个工厂就行
        DishFactory factory2 = new MapoTofuFactory();
        Dish dish2 = factory2.create();
        dish2.cook();
    }
}
```

### 为什么厉害？—— 解决开闭原则问题！

|          | 简单工厂                             | 工厂方法模式                                              |
| -------- | ------------------------------------ | --------------------------------------------------------- |
| 加新产品 | **改原代码**（加 switch 分支） | **新建两个类**（产品+工厂），**不改原有代码** |
| 开闭原则 | ❌ 违反                              | ✅ 遵守                                                   |

```java
// 简单工厂加新菜：必须打开 DishFactory 改代码 ❌
class DishFactory {
    public static Dish create(String type) {
        switch (type) {
            case "番茄炒蛋": ...
            case "回锅肉": return new HuiRouRou();   // ← 改了原有代码！
        }
    }
}

// 工厂方法模式加新菜：新建文件就行，不动原有代码 ✅
class HuiRouRou implements Dish { ... }           // 新建产品
class HuiRouRouFactory implements DishFactory {    // 新建工厂
    public Dish create() { return new HuiRouRou(); }
}
```

> **本质是用"类的数量"换"代码的可扩展性"。**

### 缺点

| 缺点                 | 说明                                        |
| -------------------- | ------------------------------------------- |
| **类爆炸**     | 加一个产品要加**两个类**（产品+工厂） |
| **复杂度上升** | 从一个工厂变成一堆工厂                      |
| **使用稍麻烦** | 要先 `new` 工厂，再 `factory.create()`  |

---

> **我的总结：** 工厂方法模式跟策略模式比较像，前两部分跟简单工厂一样（产品接口+具体产品），只是把工厂类分成了抽象父工厂接口和具体子工厂类。也是一个接口一个方法，然后子类去重写它的方法，重写的方法直接返回具体产品里的值。一个子工厂对应一个具体产品，最后客户端只需要通过工厂接口来选择工厂方法就可以。但是这样开发后期会发现类超级多，使用很麻烦，复杂度也很高。

## 三、抽象工厂模式（Abstract Factory）

### 它解决什么新问题？

前面两个工厂都是造**一种东西**。但现实中经常遇到：**一次要造一组相关的东西（产品族）**。

举个例子——开一家餐厅：

```
中式餐厅 = 中式菜品 + 中式餐具 + 中式装修风格
西式餐厅 = 西式菜品 + 西式餐具 + 西式装修风格
日式餐厅 = 日式菜品 + 日式餐具 + 日式装修风格
```

这些是**一整套**，不能混搭：

- 你不会在中式餐厅用法式刀叉吃寿司 ❌
- 每个餐厅提供的是 **一个"产品族"**

> **抽象工厂的核心：不是造一个产品，而是造一整套相关的产品（产品族）。**

### 结构图

```
              产品族A(中)          产品族B(西)
           ┌──────────┐       ┌──────────┐
产品1(菜): │ 中式炒菜   │       │ 西式牛排   │
           └─────┬────┘       └─────┬────┘
                │ 同一工厂           │ 同一工厂
                ↓                    ↓
           ┌──────────┐       ┌──────────┐
产品2(餐具):│ 筷子      │       │ 刀叉       │
           └──────────┘       └──────────┘
                ↑                    ↑
                └────────┬───────────┘
                         ↓
                  ┌──────────────────┐
                  │ 抽象工厂接口       │
                  │ - createDish()    │
                  │ - createTableware()│
                  └────────┬─────────┘
                           │ 实现
              ┌────────────┼────────────┐
              ▼            ▼            ▼
       ┌──────────┐ ┌──────────┐ ┌──────────┐
       │ 中式工厂   │ │ 西式工厂   │ │ 日式工厂   │
       └──────────┘ └──────────┘ └──────────┘
```

### 四个角色

| 角色         | 数量                     | 职责                   |
| ------------ | ------------------------ | ---------------------- |
| 抽象产品接口 | 多个（每种产品类型一个） | 定义各产品的规范       |
| 具体产品     | 多个                     | 实现具体功能           |
| 抽象工厂接口 | 1个                      | 定义创建所有产品的方法 |
| 具体工厂     | 多个（每个产品族一个）   | 创建一整套配套产品     |

### 代码

```java
// ========== 产品接口（多种）==========

// 产品1：菜品
interface Dish {
    void cook();
}

// 产品2：餐具
interface Tableware {
    void use();
}


// ========== 中式产品族 ==========
class ChineseDish implements Dish {
    public void cook() { System.out.println("做中式炒菜"); }
}
class Chopsticks implements Tableware {     // 筷子
    public void use() { System.out.println("使用筷子"); }
}

// ========== 西式产品族 ==========
class WesternDish implements Dish {
    public void cook() { System.out.println("煎西式牛排"); }
}
class KnifeFork implements Tableware {      // 刀叉
    public void use() { System.out.println("使用刀叉"); }
}


// ========== 抽象工厂接口（核心！）==========
interface RestaurantFactory {
    Dish createDish();              // 创建菜品
    Tableware createTableware();    // 创建餐具
}


// ========== 具体工厂：每个工厂创建一个完整的产品族 ==========

class ChineseRestaurantFactory implements RestaurantFactory {
    public Dish createDish() {
        return new ChineseDish();       // 造中式菜
    }
    public Tableware createTableware() {
        return new Chopsticks();        // 造筷子 → 配套！
    }
}

class WesternRestaurantFactory implements RestaurantFactory {
    public Dish createDish() {
        return new WesternDish();       // 造西餐
    }
    public Tableware createTableware() {
        return new KnifeFork();         // 造刀叉 → 配套！
    }
}
```

### 使用

```java
public class Main {
    public static void main(String[] args) {

        // 客户端选择一家餐厅（= 选择了一个产品族）
        RestaurantFactory factory = new ChineseRestaurantFactory();

        // 工厂会自动给你配套的产品——不用担心混搭问题
        Dish dish = factory.createDish();
        Tableware tool = factory.createTableware();

        dish.cook();       // 做中式炒菜
        tool.use();        // 使用筷子   ← 自动配套！


        // 换一家餐厅 = 换整个产品族
        RestaurantFactory factory2 = new WesternRestaurantFactory();

        Dish dish2 = factory2.createDish();
        Tableware tool2 = factory2.createTableware();

        dish2.cook();      // 煎西式牛排
        tool2.use();       // 使用刀叉   ← 自动配套！
    }
}
```

输出：

```
做中式炒菜
使用筷子
煎西式牛排
使用刀叉
```

> **客户端只选了一次工厂，拿到的所有东西都是天然配套的。这就是抽象工厂的威力。**

### 三种工厂对比总表

|                                            | 简单工厂 | 工厂方法 | 抽象工厂                          |
| ------------------------------------------ | -------- | -------- | --------------------------------- |
| **造什么**                           | 一种产品 | 一种产品 | **一族相关产品**            |
| **几个工厂**                         | 1个      | N个      | N个                               |
| **新增产品组**                       | 改原代码 | 新增工厂 | ✅ 只需新增一套                   |
| **新增产品种类（如加一种产品类型）** | 加方法   | 改接口   | **❌ 很麻烦，要改所有工厂** |
| **复杂度**                           | ⭐       | ⭐⭐     | ⭐⭐⭐                            |

> **抽象工厂最大的缺点：如果要在已有的产品族里加一种新产品（比如再加"装修风格"），所有工厂都要改。**

---

> **我的总结：** 抽象工厂感觉就像是工厂方法的升级版，它可以有好几个接口，每个接口对应一个产品。产品族按类分好，重写对应的接口里的方法就可以。在抽象工厂接口里放好所有产品对应的方法，最后具体工厂类按照产品族分类来重新写抽象接口里的方法。客户端只需要选好子工厂类就可以，产品工厂会一套输出出来。
>
> 但是有一个缺陷：虽然新增产品组很方便（只需新增类），但是一旦想要改产品（新增产品类型），那就得从最初的产品接口一下子改到结尾的所有子工厂类，非常麻烦。后期的类也很多，很复杂。

---

## 我的总结

三种工厂模式的关系：

| 模式               | 核心特点                   | 优点           | 缺点               |
| ------------------ | -------------------------- | -------------- | ------------------ |
| **简单工厂** | 一个工厂 + switch 造所有   | 简单、集中管理 | 违反开闭原则       |
| **工厂方法** | 每种产品一个专属工厂       | 遵循开闭原则   | 类变多             |
| **抽象工厂** | 每个工厂造一整套（产品族） | 保证产品配套   | 新增产品类型很麻烦 |

一句话总结：**简单工厂最简单但不够灵活；工厂方法用类灵活性；抽象工厂解决"成套产品"的创建问题。选哪种看场景——产品少用简单工厂，需要扩展用工厂方法，有产品族用抽象工厂。**

# 常见模式的应用场景

## 各模式的典型应用场景

| 模式               | 典型场景               | 例子                                               |
| ------------------ | ---------------------- | -------------------------------------------------- |
| **单例模式** | 全局只需要一个实例     | 数据库连接池、配置管理器、日志器                   |
| **策略模式** | 算法/行为需要切换      | 支付方式（微信/支付宝）、排序算法、优惠折扣        |
| **模板方法** | 流程固定、步骤不同     | Servlet的doGet/doPost、JUnit的测试流程、做菜的步骤 |
| **简单工厂** | 产品少、创建逻辑简单   | 不同类型的解析器（JSON/XML）                       |
| **工厂方法** | 需要扩展、遵循开闭原则 | 框架中的各种Factory                                |
| **抽象工厂** | 成套产品、不能混搭     | UI主题切换（Windows风/Mac风）、数据库驱动          |

## 单独用 vs 混着用？

**单独用的情况：** 大部分时候一个模式就够解决一个问题。

**实际开发中经常混着用！**

### 1. 工厂 + 单例（最常见）

```java
// 工厂本身是单例 —— 全局只有一个工厂
class DishFactory {
    private static final DishFactory INSTANCE = new DishFactory();  // 单例
    private DishFactory() {}
  
    public static DishFactory getInstance() { return INSTANCE; }
    public Dish create(String type) { ... }                          // 工厂方法
}
```

### 2. 工厂 + 策略

```java
// 工厂创建策略对象，然后客户端用策略来切换行为
Dish dish = DishFactory.create("宫保鸡丁");   // 工厂：造对象
dish.cook();                                    // 对象本身就是策略的实现
```

### 3. 模板方法 + 工厂方法

```java
// 模板方法的某个步骤，用工厂方法来创建具体对象
abstract class CookingTemplate {
    final void cook() {
        Dish dish = createDish();      // ← 这是工厂方法（子类实现）
        dish.prepare();
        dish.cooking();
        dish.serve();
    }
    protected abstract Dish createDish();  // 子类决定造什么菜
}
```

### 4. Spring框架里的经典组合

| Spring组件             | 用了什么模式                                                |
| ---------------------- | ----------------------------------------------------------- |
| `ApplicationContext` | **单例** + **工厂**（Bean容器既是单例又是工厂） |
| `JdbcTemplate`       | **模板方法**（固定流程，你只写SQL）                   |
| `HandlerAdapter`     | **策略模式**（不同Controller类型用不同适配策略）      |
| 各种 `BeanFactory`   | **工厂方法 / 抽象工厂**                               |
| `ProxyFactoryBean`   | **工厂 + 代理**                                       |

> **一句话：学的时候分开学，用的时候组合用。** 就像学汉字和组词语——先认单个字，写作文时才组合起来。

## 场景相关名词解释

- **数据库连接池**：提前建好一批数据库连接放"池子"里复用，不用每次都新建销毁
- **配置管理器**：统一读取配置文件的类，全局只需要一个
- **Servlet的doGet/doPost**：Java Web开发中处理请求的方法，流程固定（接收→处理→响应），中间的doGet/doPost由你实现 → 模板方法
- **JUnit测试流程**：`@Before`(准备) → `@Test`(测试) → `@After`(清理)，固定流程你填内容 → 模板方法
- **JSON/XML解析器**：根据传入类型返回对应的数据格式解析器 → 简单工厂
- **数据库驱动**：不同数据库有配套的一套驱动（Connection、Statement等） → 抽象工厂（产品族）
- **Spring ApplicationContext**：管理所有Bean的容器，默认单例 + 工厂功能
- **Spring JdbcTemplate**：帮你操作数据库，固定流程只写SQL → 模板方法

## 模式组合使用的真实场景

> 核心规律：**简单工具用单一模式，成体系的系统需要多个模式配合。**

### 场景1：电商支付系统（工厂 + 策略 + 单例）

```
用户下单 → 选择支付方式 → 完成支付
```

| 步骤           | 用了什么模式       | 为什么                             |
| -------------- | ------------------ | ---------------------------------- |
| 获取支付处理器 | **工厂方法** | 根据类型创建：微信/支付宝/银行卡   |
| 切换支付逻辑   | **策略模式** | 每种支付的扣款、回调逻辑不同       |
| 支付配置读取   | **单例**     | 配置（密钥、接口地址）全局只有一份 |

```java
PayStrategy payment = PayFactory.create("支付宝");  // 工厂造对象
payment.pay(100);                                     // 策略切换行为
Config config = Config.getInstance();                  // 单例拿配置
```

### 场景2：日志系统（单例 + 策略 + 模板方法）

```
程序运行时记录日志 → 可以输出到不同地方
```

| 步骤       | 用了什么模式       | 为什么                                 |
| ---------- | ------------------ | -------------------------------------- |
| 日志器本身 | **单例**     | 全程只需要一个日志器                   |
| 输出到哪   | **策略**     | 控制台？文件？数据库？随时切           |
| 记录流程   | **模板方法** | 固定流程：时间+级别+内容→格式化→输出 |

```java
Logger logger = Logger.getInstance();           // 单例
logger.setTarget(new FileLogTarget());          // 策略：输出到文件
logger.info("用户登录了");                       // 内部走模板方法流程
```

### 场景3：数据库操作（Spring全家桶，毕设相关！）

| 层             | 用了什么模式              | 说明                          |
| -------------- | ------------------------- | ----------------------------- |
| DataSource     | **单例**            | 数据源（连接池）全局一个      |
| JdbcTemplate   | **模板方法**        | 固定JDBC流程，你只写SQL       |
| Mapper/DAO     | **工厂**            | MyBatis帮你创建Mapper代理对象 |
| 不同数据库切换 | **抽象工厂 + 策略** | 换数据库 = 换一整套驱动和方言 |

### 场景4：数据导出功能（简单工厂 + 模板方法 + 策略）

```
把数据导出成不同格式：Excel / PDF / Word
```

| 步骤         | 用了什么模式       | 为什么                               |
| ------------ | ------------------ | ------------------------------------ |
| 选择导出格式 | **简单工厂** | 传"excel"/"pdf"，返回对应的导出器    |
| 导出流程固定 | **模板方法** | 准备数据→渲染格式→写入文件         |
| 渲染引擎     | **策略**     | Excel渲染器和PDF渲染器的实现完全不同 |

### 总结对比

| 规模                    | 模式使用  | 例子                         |
| ----------------------- | --------- | ---------------------------- |
| **小工具/单功能** | 1个模式   | 工具类、解析器、简单的管理器 |
| **中型应用**      | 2-3个组合 | 支付模块、日志系统、导出功能 |
| **大型框架**      | 全家桶    | Spring用了几乎所有设计模式   |

> 就像搭积木：搭一个小房子几块积木就行，搭一个城堡需要各种形状的积木组合起来。

---

> **我的总结：** 模式单独用的情况一般都是当成设计一个简单的工具，而一个成体系的系统往往是要用好几个模式来进行设计。
补充一个判断标准：问自己：这个东西以后会变吗？
├── 不会怎么变 → 单一模式就够了
└── 会经常变/扩展 → 多个模式配合，提前留好"可变的口子"

# 好代码的标准（SOLID原则）

## SOLID五原则

| 原则 | 全称 | 核心含义 | 一句话 |
|------|------|----------|--------|
| **S** | Single Responsibility | 单一职责原则 | 一个类只做一件事 |
| **O** | Open/Closed | 开闭原则 | 对扩展开放，对修改关闭 |
| **L** | Liskov Substitution | 里氏替换原则 | 子类能完美替代父类，行为不变 |
| **I** | Interface Segregation | 接口隔离原则 | 接口小而精，不要大而全 |
| **D** | Dependency Inversion | 依赖倒置原则 | 依赖抽象，不依赖具体 |

### 各原则详解

**S - 单一职责**：一个类只做一件事，别让厨师收银扫地一起干

**O - 开闭原则**：加新功能加新代码，不改旧代码（工厂方法模式的思路）

**L - 里氏替换**：父类能用的地方子类一定能用，行为不能变（企鹅不该继承会飞的鸟）

**I - 接口隔离**：接口要小，能单点就别强制套餐（机器人不需要吃饭睡觉的方法）

**D - 依赖倒置**：依赖接口不依赖具体类（Spring的@Autowired就是这个思想）

## 其他重要标准

| 标准 | 含义 |
|------|------|
| **DRY** | Don't Repeat Yourself，别重复代码 |
| **KISS** | Keep It Simple, Stupid，保持简单，别过度设计 |
| **高内聚低耦合** | 内部紧密关联，外部依赖少，结构清晰 |
| **命名规范** | 见名知意，`getUserById` 比 `get` 好 |
| **适当注释** | 解释"为什么"，复杂逻辑才加注释 |

## 设计模式与原则的关系

> **设计模式的本质就是帮你实现这些原则的工具。**

---

> **我的总结：** 好的代码要符合SOLID原则，在这个基础上保证代码尽量简单（KISS）、不重复（DRY）、结构清晰不混乱（高内聚低耦合）、命名见名知意、复杂逻辑加通俗易懂的注释。另外还要注意依赖抽象不依赖具体（依赖倒置），这是Spring最核心的思想。
