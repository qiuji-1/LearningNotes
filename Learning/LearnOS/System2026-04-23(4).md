## 一、什么是调度算法

**"调度"源于交通运输领域** —— 火车站调度员决定"谁先进站、谁等等"。

**为什么需要？**
- 进程多（100个），CPU 少（4核8线程）
- 要决定：谁先用 CPU？用多久？什么时候换人？

**调度目标：**
| 目标 | 含义 |
|------|------|
| 吞吐量 | 单位时间完成更多进程 |
| 周转时间 | 进程从来到走要等多久 |
| 响应时间 | 你点一下，程序多久有反应 |
| 公平性 | 大家都有机会用 CPU |

## 二、七种调度算法

### 1. FCFS（先来先服务）

**核心思想：谁先来谁先用，不插队**

**例子：**
| 进程 | 到达时间 | 需要 CPU 时间 |
|------|----------|---------------|
| P1 | 0 | 10 |
| P2 | 1 | 5 |
| P3 | 2 | 3 |

执行顺序：P1(0-10) → P2(10-15) → P3(15-18)

**优点：** 简单，容易实现
**缺点：** 护航效应（convoy effect）—— 大任务堵前面，小任务全得等

---

### 2. SJF（短作业优先）

**核心思想：谁用时间短谁先来**

**例子：** 同样的数据
执行顺序：P1(0-2) → P3(2-5, 3秒) → P2(5-10, 5秒) → P1(10-20, 10秒)

**优点：** 平均等待时间最短
**缺点：** 长任务可能"饿死"—— 一直被插队

---

### 3. SRTF（最短剩余时间优先）

**核心思想：SJF 升级版 —— 可以"抢"CPU**

- SJF：等一个任务跑完再选下一个
- SRTF：随时盯着，有更短的就抢过来

**例子：**
| 进程 | 到达时间 | 需要 CPU 时间 |
|------|----------|---------------|
| P1 | 0 | 10 |
| P2 | 1 | 5 |
| P3 | 2 | 3 |

执行过程：P1(0-1) → P2(1-6) → P3(6-9) → P1(9-20)

**优点：** 短任务响应更快，能及时响应新来的紧急任务
**缺点：** 切换频繁开销大，长任务可能饿死

---

### 4. RR（时间片轮转）

**核心思想：大家轮流来，每人用一小段时间**

**关键概念：时间片（Time Quantum）**

**例子：** 时间片 = 4
| 进程 | 到达时间 | 需要 CPU 时间 |
|------|----------|---------------|
| P1 | 0 | 10 |
| P2 | 0 | 5 |
| P3 | 0 | 3 |

执行顺序：P1(0-4) → P2(4-8) → P3(8-12) → P1(12-15) → P2(15-16) → P3(16-20) → P1(20-22)

**优点：** 响应时间快、公平、不会饿死
**缺点：** 时间片小 → 切换频繁 → 开销大；时间片大 → 响应慢

**时间片怎么选？** 经验公式：时间片 ≈ 80% 的进程 CPU 突发时间

---

### 5. 优先级调度

**核心思想：谁重要谁先用**

**两种模式：**
| 模式 | 数字含义 | 例子 |
|------|----------|------|
| 低数字 = 低优先级 | 1最低，5最高 | Linux（nice值） |
| 低数字 = 高优先级 | 0最高，5最低 | Windows |

**解决饿死：aging（老化）技术**
- 等待越久，优先级慢慢提升
- 排得够久总能轮到

---

### 6. MLFQ（多级反馈队列）

**核心思想：自动"观察"进程行为，动态调整优先级**

**3 个队列示例：**
| 队列 | 调度方式 | 时间片 |
|------|----------|--------|
| Q1（高优先级） | RR | 8ms |
| Q2（中优先级） | RR | 16ms |
| Q3（低优先级） | FCFS | ∞ |

**规则：**
1. 新进程全部进入 Q1
2. 用完时间片还没完成 → 降级到下一队列
3. 没做完主动放弃 CPU（如等待 I/O）→ 保留当前优先级
4. 最高队列为空才执行下一队列

**为什么 I/O 密集型要保留优先级？**
- 它用 CPU 时间很短，大部分时间在等
- 如果每次等完都降级，永远升不上去
- 但它是交互式任务（鼠标、键盘），需要响应快

**优点：** 自适应、无需预设优先级、防止长任务饿死
**缺点：** 复杂度高、参数难调

**实际应用：** Linux 的 CFS（完全公平调度器）

---

### 7. HRRN（高响应比优先）

**核心思想：兼顾等待时间 + 执行时间，不饿死**

**公式：**
```
响应比 = (等待时间 + 执行时间) / 执行时间
       = 等待时间/执行时间 + 1
```

**例子：**
| 进程 | 执行时间 | 到达时刻 |
|------|----------|----------|
| P1 | 10 | 0 |
| P2 | 5 | 1 |
| P3 | 3 | 2 |

P1 先到先跑完（0-10），10时刻计算响应比：
- P2：响应比 = 9/5 + 1 = 2.8
- P3：响应比 = 8/3 + 1 ≈ 3.67

→ P3 先执行

**优点：** 不会饿死、兼顾长短任务
**缺点：** 计算响应比开销大

## 三、算法对比总结

| 算法 | 核心思想 | 会不会饿死 | 适用场景 |
|------|----------|-----------|----------|
| FCFS | 先来先到 | 会 | 批处理、任务长度相近 |
| SJF | 短的最先 | 会（长任务） | 批处理、可预估任务长度 |
| SRTF | 可抢的 SJF | 会（长任务） | 实时系统 |
| RR | 轮流排队 | 不会 | 分时系统、交互式 |
| 优先级 | 重要的先来 | 会（低优先级） | 实时系统、服务器 |
| MLFQ | 分层 + 自动调整 | 不会 | 通用操作系统 |
| HRRN | 响应比决定 | 不会 | 批处理 |

---

## 四、重要概念

### 饿死（Starvation）
- 长任务或低优先级任务长期得不到调度
- 解决方案：aging（老化）技术

### I/O 密集型 vs CPU 密集型

| 类型 | CPU 在干啥 | 调度策略 |
|------|-----------|----------|
| CPU 密集型 | 一直算（解压视频、跑游戏） | 用完时间片就降级 |
| I/O 密集型 | 大部分时间在等（等输入、等网络） | 保留优先级，响应快 |

## 五、我的总结

> 进程的调度算法主要是帮助CPU来判断哪个进程先跑，要跑多久。主要有7类：
> 1、FCFS：先来先服务，无论大小长短重要与否全都排队，先来的先运行
> 2、SJF：短作业优先，只要你任务时间短，你就先来，这样平均等待时间最短，但是容易饿死长任务
> 3、SRTF：抢占版SJF，时间短的直接抢时间长的位置，问题跟SJF一样
> 4、RR：时间片轮转，每个进程跑固定的时间，时间一到就切换下一个，保证每个程序都能响应
> 5、优先级调度：给每个进程打分，分高的先来，低分任务容易饿死
> 6、MLFQ：多级反馈队列，把上面几种算法融合到一起，新来的先放高优先级队列，用完时间片降级，I/O密集型保留优先级
> 7、HRRN：最高响应比优先，通过计算响应比来决定下一个被调度的进程是谁，根据等待时间和执行时间来计算响应比，兼顾短任务和长任务

## 六、思考题答案

1. **SJF 中长任务会怎样？** 如果不断有新短任务进来，长任务永远等不到（饿死）

2. **时间片 1ms vs 100ms 的区别？** 1ms 响应快但切换频繁开销大；100ms 响应慢但开销小

3. **MLFQ 中长任务最终在哪？** 降到 Q3（最低级队列），用 FCFS 跑完
